# ClickHouse 物化视图实战

> 📚 本模块：[README](../README.md) · [原理](原理.md) · **实战（本文）**

## 一、完整预聚合链路搭建

### 1.1 场景描述

构建一个实时 PV/UV 预聚合系统：
- 基表接收用户行为日志（每秒数万条）
- MV 自动按分钟粒度预聚合 PV、UV、平均停留时长
- 查询端直接读取预聚合结果，毫秒级响应

### 1.2 Step 1: 创建明细基表

```sql
CREATE TABLE user_behavior_log ON CLUSTER '{cluster}'
(
    event_time    DateTime64(3, 'Asia/Shanghai'),
    user_id       String,
    page_id       UInt32,
    action_type   LowCardinality(String),  -- 'view','click','scroll'
    duration_ms   UInt32,
    device_type   LowCardinality(String),
    region        LowCardinality(String)
)
ENGINE = MergeTree()
PARTITION BY toYYYYMMDD(event_time)
ORDER BY (page_id, event_time)
TTL event_time + INTERVAL 90 DAY
SETTINGS index_granularity = 8192;
```

**设计要点**：
- `event_time` 用 DateTime64(3) 保留毫秒精度
- `action_type` / `device_type` 用 LowCardinality 减少存储和加速过滤
- ORDER BY 把 page_id 放前面，因为大部分查询按页面维度分析
- TTL 设置 90 天自动清理

### 1.3 Step 2: 创建 AggregatingMergeTree 目标表

```sql
CREATE TABLE page_metrics_minute_agg ON CLUSTER '{cluster}'
(
    minute_ts     DateTime,          -- 截断到分钟的桶时间
    page_id       UInt32,
    action_type   LowCardinality(String),
    pv            AggregateFunction(sum, UInt64),
    uv            AggregateFunction(uniq, String),
    avg_duration  AggregateFunction(avg, Float64),
    click_count   AggregateFunction(sumIf, UInt8, UInt8)  -- 条件聚合示例
)
ENGINE = AggregatingMergeTree()
PARTITION BY toYYYYMM(minute_ts)
ORDER BY (minute_ts, page_id, action_type);
```

**关键设计决策**：
- `minute_ts` 是 GROUP BY 的时间维度，必须包含在 ORDER BY 中
- `action_type` 也作为维度加入，支持按动作类型拆分指标
- `uv` 使用 uniq 而非 uniqExact，牺牲极小精度换取大幅性能提升
- ORDER BY 的列顺序决定了索引效率，高频过滤列放前面

### 1.4 Step 3: 创建物化视图绑定

```sql
CREATE MATERIALIZED VIEW mv_page_metrics_minute
ON CLUSTER '{cluster}'
TO page_metrics_minute_agg
AS
SELECT
    toStartOfMinute(event_time) AS minute_ts,
    page_id,
    action_type,
    sumState(toUInt64(1)) AS pv,              -- 每行计数 = PV
    uniqState(user_id) AS uv,                  -- 用户去重
    avgState(toFloat64(duration_ms)) AS avg_duration,
    sumIfState(toUInt8(1), action_type = 'click') AS click_count
FROM user_behavior_log
GROUP BY minute_ts, page_id, action_type;
```

**注意细节**：
- `sumState(toUInt64(1))` 等价于 countState()，但更显式
- `toStartOfMinute` 将时间截断到分钟级别作为聚合桶
- SELECT 中的非聚合列必须出现在 GROUP BY 中
- GROUP BY 的列必须覆盖 target 表 ORDER BY 的所有列

## 二、查询验证

### 2.1 正确查询方式（使用 -Merge 函数）

```sql
-- 查询某页面最近 1 小时的 PV、UV、平均停留时长
SELECT
    minute_ts,
    page_id,
    sumMerge(pv) AS total_pv,
    uniqMerge(uv) AS approx_uv,
    avgMerge(avg_duration) AS avg_dur_ms,
    sumMerge(click_count) AS total_clicks
FROM page_metrics_minute_agg
WHERE minute_ts >= now() - INTERVAL 1 HOUR
  AND page_id = 10086
GROUP BY minute_ts, page_id
ORDER BY minute_ts;
```

### 2.2 错误查询方式（常见坑）

```sql
-- ❌ 错误：直接 SELECT 聚合列，得到的是二进制序列化状态
SELECT pv, uv FROM page_metrics_minute_agg WHERE page_id = 10086;
-- 输出类似: \x00\x01\x00... （无法阅读的二进制）

-- ❌ 错误：不带 GROUP BY 直接 sumMerge
SELECT sumMerge(pv) FROM page_metrics_minute_agg;
-- 这会把所有不同维度的状态合并在一起，结果无意义

-- ✅ 正确：必须在正确的维度上 GROUP BY 后再 Merge
SELECT page_id, sumMerge(pv) FROM page_metrics_minute_agg
GROUP BY page_id;
```

### 2.3 跨时间窗口聚合

由于 AggregatingMergeTree 的状态是可合并的，可以在更粗粒度上再次聚合：

```sql
-- 从分钟级预聚合数据计算小时级指标
SELECT
    toStartOfHour(minute_ts) AS hour_ts,
    page_id,
    sumMerge(pv) AS hourly_pv,
    uniqMerge(uv) AS hourly_uv,     -- uniq 状态可以跨时间段合并！
    avgMerge(avg_duration) AS hourly_avg_dur
FROM page_metrics_minute_agg
WHERE minute_ts >= today() - INTERVAL 7 DAY
GROUP BY hour_ts, page_id
ORDER BY hour_ts DESC, page_id;
```

⚠️ **注意**：avgMerge 跨时间段合并的结果是加权平均（按样本数加权），不是简单平均。这在大多数业务场景中是正确的语义。

## 三、验证 INSERT 触发行为与非回溯性

### 3.1 验证 MV 只在 INSERT 时触发

```sql
-- 1. 确认目标表为空
SELECT count() FROM page_metrics_minute_agg;
-- 返回: 0

-- 2. 向基表插入历史时间的数据
INSERT INTO user_behavior_log VALUES
    ('2025-01-01 10:00:00', 'user_001', 10086, 'view', 3500, 'mobile', '华东');

-- 3. 检查目标表 —— 应该有数据了
SELECT minute_ts, page_id, sumMerge(pv) as pv, uniqMerge(uv) as uv
FROM page_metrics_minute_agg
GROUP BY minute_ts, page_id;
-- 返回: 2025-01-01 10:00:00 | 10086 | 1 | 1

-- 4. 再插入一批相同维度但不同用户的数据
INSERT INTO user_behavior_log VALUES
    ('2025-01-01 10:00:00', 'user_002', 10086, 'view', 2000, 'desktop', '华北'),
    ('2025-01-01 10:00:00', 'user_003', 10086, 'click', 100, 'mobile', '华南');

-- 5. 等待后台 Merge（或手动触发）
OPTIMIZE TABLE page_metrics_minute_agg FINAL;

-- 6. 查看合并后的结果
SELECT minute_ts, page_id, sumMerge(pv) as pv, uniqMerge(uv) as uv
FROM page_metrics_minute_agg
GROUP BY minute_ts, page_id;
-- 返回: 2025-01-01 10:00:00 | 10086 | 3 | 3
-- PV=3（三次访问），UV=3（三个不同用户）
```

### 3.2 验证非回溯性

```sql
-- 1. 先插入一些数据到基表
INSERT INTO user_behavior_log VALUES
    ('2025-06-01 09:00:00', 'user_A', 20001, 'view', 5000, 'mobile', '华东'),
    ('2025-06-01 09:00:00', 'user_B', 20001, 'view', 3000, 'desktop', '华北');

-- 2. 此时创建 MV（不带 POPULATE）
CREATE MATERIALIZED VIEW mv_test_nonretro TO test_target AS
SELECT toStartOfMinute(event_time) AS minute_ts, page_id,
       sumState(toUInt64(1)) AS pv
FROM user_behavior_log
GROUP BY minute_ts, page_id;

-- 3. 查目标表 —— 应该为空！之前写入的 user_A/user_B 不会被处理
SELECT count() FROM test_target;
-- 返回: 0 ← 证明了非回溯性

-- 4. 新插入一条数据
INSERT INTO user_behavior_log VALUES
    ('2025-06-01 09:00:00', 'user_C', 20001, 'view', 1000, 'tablet', '华南');

-- 5. 查目标表 —— 只有 user_C 的数据
SELECT minute_ts, page_id, sumMerge(pv) as pv FROM test_target
GROUP BY minute_ts, page_id;
-- 返回: 2025-06-01 09:00:00 | 20001 | 1 ← 只有新写入的那条
```

## 四、实时 UV/PV 预聚合完整方案

### 4.1 多级粒度预聚合架构

```
                    ┌─────────────────┐
                    │  user_behavior   │ ← Flink/Kafka 实时写入
                    │  _log (明细表)    │
                    └────────┬────────┘
                             │ INSERT 触发
              ┌──────────────┼──────────────┐
              ▼              ▼              ▼
     ┌────────────┐  ┌────────────┐  ┌────────────┐
     │ MV → 分钟级 │  │ MV → 小时级 │  │ MV → 天级   │
     │ agg_minute │  │ agg_hour   │  │ agg_day    │
     └────────────┘  └────────────┘  └────────────┘
              │              │              │
         近实时查询      趋势分析        长期报表
        (延迟~1min)   (延迟~1hour)   (延迟~1day)
```

### 4.2 小时级和天级 MV 定义

```sql
-- 小时级目标表
CREATE TABLE page_metrics_hour_agg ON CLUSTER '{cluster}'
(
    hour_ts       DateTime,
    page_id       UInt32,
    pv            AggregateFunction(sum, UInt64),
    uv            AggregateFunction(uniq, String)
)
ENGINE = AggregatingMergeTree()
PARTITION BY toYYYYMM(hour_ts)
ORDER BY (hour_ts, page_id);

-- 小时级 MV（注意：挂在同一个基表上）
CREATE MATERIALIZED VIEW mv_page_metrics_hour
ON CLUSTER '{cluster}'
TO page_metrics_hour_agg
AS
SELECT
    toStartOfHour(event_time) AS hour_ts,
    page_id,
    sumState(toUInt64(1)) AS pv,
    uniqState(user_id) AS uv
FROM user_behavior_log
GROUP BY hour_ts, page_id;

-- 天级同理，toStartOfDay
```

### 4.3 多级粒度的写放大评估

```
假设基表每秒写入 10,000 条记录：
- 分钟级 MV: 每秒产出 ~167 条聚合记录（10000/60）
- 小时级 MV: 每秒产出 ~3 条聚合记录（10000/3600）
- 天级 MV:   每秒产出 ~0.1 条聚合记录

总写放大 ≈ 1 + 0.0167 + 0.0003 + 0.00001 ≈ 1.017x
→ 聚合粒度越粗，写放大越小，多级预聚合的成本可控
```

## 五、MV vs 查询时聚合的选择指南

### 5.1 决策矩阵

| 判断维度 | 选 MV 预聚合 | 选查询时聚合 |
|---------|-------------|-------------|
| 查询频率 | 高频（QPS > 10） | 低频 / Ad-hoc |
| 数据量 | 大（亿级以上） | 中小（千万级以下） |
| 延迟要求 | 严格（< 100ms） | 宽松（秒级可接受） |
| 维度组合 | 固定几种 | 多变、探索式 |
| 写入吞吐 | 高（需要分摊成本） | 低 |
| 运维复杂度 | 能接受额外对象管理 | 希望尽量简单 |
| 指标变更频率 | 稳定 | 频繁新增指标 |

### 5.2 混合策略

实际项目中通常两者结合：

```
固定报表 / Dashboard  → MV 预聚合（保证 SLA）
Ad-hoc 分析 / 排查问题 → 直接查明细表（灵活）
新增实验指标          → 先查询时聚合验证价值，确认有价值后加 MV
```

### 5.3 何时不应该用 MV

1. **维度组合爆炸**：如果 GROUP BY 有 10+ 个高基数列，聚合后数据量可能接近甚至超过明细
2. **指标频繁变更**：每次改指标都要 DROP + CREATE MV，且历史数据需要补算
3. **写入瓶颈已经是问题**：MV 会进一步增加写入压力
4. **需要 JOIN**：ClickHouse MV 不支持 JOIN，多表关联预聚合需要用 StarRocks 异步 MV

## 七、面试话术

### Q: 你们的实时指标是怎么做的？

> 我们用 ClickHouse 物化视图做实时预聚合。基表接收 Flink 写入的用户行为明细，然后通过 MV 自动按分钟粒度预聚合 PV、UV 等指标到 AggregatingMergeTree 表中。查询时用 sumMerge/uniqMerge 直接读预聚合结果，P99 延迟在 50ms 以内。相比直接查明细表再做 GROUP BY，查询速度提升了两个数量级。

### Q: 为什么不用 POPULATE？

> POPULATE 只在创建 MV 的那一刻对现有数据做一次全量扫描，期间如果有新数据写入可能被遗漏，存在数据不一致风险。我们的做法是先创建空的 MV，然后用离线任务把历史数据重新 INSERT 一份到基表来触发 MV 填充，这样可以精确控制数据范围和一致性。

### Q: 预聚合的 UV 准确吗？

> 我们用的是 uniqState 而非 uniqExactState。uniq 基于 HyperLogLog 算法，在百万级 UV 下误差约 1-2%，但内存占用和计算速度比精确去重好一个数量级。对于业务 Dashboard 来说这个精度完全够用。如果需要精确 UV，可以改用 uniqExactState，但要注意内存开销。

### Q: 多个 MV 会不会影响写入性能？

> 会的，这就是写放大问题。我们有 3 个 MV（分钟/小时/天级），实测写入吞吐下降约 15%。缓解措施包括：使用异步 MV 降低写入延迟、合理设计聚合粒度减少 MV 输出量、监控 Part 数量防止 Merge 积压。如果写放大成为瓶颈，可以考虑将部分预聚合移到 Flink 侧完成。
