# ClickHouse 物化视图原理

> 📚 本模块：[README](../README.md) · **原理（本文）** · [实战](实战.md)

## 一、核心认知：物化视图 ≠ 传统数据库的物化视图

### 1.1 最关键的本质定义

ClickHouse 的 Materialized View（以下简称 MV）本质上是一个 **INSERT 触发器**，而不是一个"缓存查询结果的自动刷新视图"。

```
┌─────────────────────────────────────────────────────┐
│           ClickHouse MV 的本质模型                    │
│                                                     │
│   INSERT INTO base_table                            │
│         │                                           │
│         ▼                                           │
│   ┌──────────────┐                                  │
│   │ 写入 base     │                                  │
│   │ table 数据块   │                                  │
│   └──────┬───────┘                                  │
│          │  触发（同步/异步）                           │
│          ▼                                          │
│   ┌──────────────┐    ┌──────────────┐              │
│   │  MV_1 SELECT  │    │  MV_2 SELECT  │             │
│   │  (只处理新块)   │    │  (只处理新块)   │            │
│   └──────┬───────┘    └──────┬───────┘             │
│          ▼                   ▼                      │
│   ┌──────────────┐    ┌──────────────┐              │
│   │ target_table_1│    │ target_table_2│             │
│   └──────────────┘    └──────────────┘              │
└─────────────────────────────────────────────────────┘
```

**三个"不是"**：
- ❌ **不是**对历史数据回溯计算（除非使用 POPULATE）
- ❌ **不是**在查询时动态计算
- ❌ **不是**定期全量刷新的快照

**一个是**：
- ✅ **是**在每次 INSERT 写入 base table 时，自动用新写入的数据块执行 MV 定义的 SELECT，并将结果写入目标表

### 1.2 与传统数据库对比

| 特性 | PostgreSQL MV | Oracle MV | ClickHouse MV |
|------|--------------|-----------|---------------|
| 刷新方式 | REFRESH 手动/定时 | FAST/COMPLETE/FORCE | INSERT 时自动触发 |
| 增量能力 | 有限 | 支持 PCT 等 | 天然增量（逐块触发） |
| 历史数据 | 创建时填充 | 创建时填充 | 默认不填充（需 POPULATE） |
| 查询透明改写 | 支持 | 支持 | 不支持（需手动查目标表） |
| 一致性保证 | 事务级 | 可配置 | 最终一致（异步模式） |
| 性能开销 | 刷新时集中开销 | 刷新时集中开销 | 写入时分散摊销 |

## 二、MV 内部工作机制详解

### 2.1 INSERT 时的触发流程

当向 base table 写入一个数据块（Data Part）时：

```
Step 1: 客户端发送 INSERT 请求
        ↓
Step 2: ClickHouse 将数据写入 base table 的新 Data Part
        ↓
Step 3: 检查 base table 上是否有绑定的 MV
        ↓
Step 4: 对每个 MV，用刚写入的数据块作为输入执行 MV 的 SELECT 语句
        ↓
Step 5: 将 SELECT 的结果写入 MV 对应的 target table
        ↓
Step 6: 如果所有 MV 都写入成功，INSERT 返回成功
        （异步模式下 Step 4-5 放入后台队列，INSERT 立即返回）
```

**关键点**：MV 的 SELECT 语句只能看到**本次 INSERT 写入的那批数据**，看不到 base table 中已有的其他数据。这就是为什么 MV 不会回溯处理历史数据。

### 2.2 同步 vs 异步执行

```sql
-- MV 在每次向基表写入 Data Part 时触发（与 INSERT 同一链路）
CREATE MATERIALIZED VIEW mv_sync TO target_table AS
SELECT ... FROM base_table;
```

**同步语义**：默认情况下，MV 处理与本次 INSERT 写入同一 Part 绑定；若 MV 写入失败，可能导致整次 INSERT 失败（取决于配置与版本）。

**与 `async_insert` 的区别**（勿混淆）：
- `async_insert` 是**客户端攒批**设置，先把多条 INSERT 缓冲再落盘，落盘后才触发 MV
- 它不是"MV 异步执行开关"，而是延迟基表 Part 产生时机

```sql
-- 客户端异步攒批（影响何时触发 MV，不是 MV 专用参数）
SET async_insert = 1;
```

### 2.3 POPULATE 关键字的行为

```sql
-- 不带 POPULATE：只对创建 MV 之后写入的数据生效
CREATE MATERIALIZED VIEW mv_new TO target AS
SELECT event_date, count() as cnt FROM base GROUP BY event_date;

-- 带 POPULATE：创建时把 base table 现有数据也跑一遍
CREATE MATERIALIZED VIEW mv_all TO target AS
POPULATE
SELECT event_date, count() as cnt FROM base GROUP BY event_date;
```

⚠️ **POPULATE 的注意事项**：
1. POPULATE 只在 MV **创建那一刻**执行一次
2. 如果在 POPULATE 执行期间有新数据写入 base table，这些数据**可能被遗漏**（取决于执行时序）
3. POPULATE 会对 base table 做一次全表扫描，大表场景耗时很长
4. 生产环境建议：**不用 POPULATE**，先创建空 MV，再手动补历史数据

```
时间线风险示意：

T0: CREATE MV ... POPULATE 开始执行
T1: 开始扫描 base table 已有数据 [data_A, data_B]
T2: 新 INSERT 写入 data_C ← 可能被 POPULATE 遗漏！
T3: POPULATE 完成，MV 开始正常监听后续 INSERT
T4: data_D 写入 → MV 正常捕获

结果：target 包含 data_A + data_B + data_D，缺少 data_C
```

## 三、AggregatingMergeTree + State/Merge 函数体系

### 3.1 为什么需要 AggregatingMergeTree

普通 MergeTree 做预聚合的问题：相同 key 的多条记录只是物理存储在一起，不会自动合并。查询时必须再 GROUP BY，失去了预聚合的意义。

AggregatingMergeTree 的核心能力：**在后台 Merge 过程中，自动按照 ORDER BY key 对聚合状态进行合并**。

```
┌────────────────────────────────────────────────────┐
│          AggregatingMergeTree Merge 过程            │
│                                                    │
│  Part A:                    Part B:                │
│  ┌──────────────────┐      ┌──────────────────┐   │
│  │ user_id=1        │      │ user_id=1        │   │
│  │ pv_state=5       │      │ pv_state=3       │   │
│  │ uv_state={a,b,c} │      │ uv_state={c,d}   │   │
│  └──────────────────┘      └──────────────────┘   │
│              │                      │              │
│              └──────────┬───────────┘              │
│                         ▼                          │
│              Background Merge                      │
│              sumMerge(5,3)=8                       │
│              uniqMerge({a,b,c},{c,d})={a,b,c,d}   │
│                         ▼                          │
│  Merged Part:                                      │
│  ┌──────────────────┐                              │
│  │ user_id=1        │                              │
│  │ pv_state=8       │  ← 自动合并！                 │
│  │ uv_state={a,b,c,d}│                             │
│  └──────────────────┘                              │
└────────────────────────────────────────────────────┘
```

### 3.2 -State / -Merge 函数配对机制

这套函数是 MV 预聚合体系的灵魂：

| 写入阶段（MV 中） | 存储类型 | 读取阶段（查询时） | 说明 |
|-------------------|---------|-------------------|------|
| `sumState(x)` | `AggregateFunction(sum, Int64)` | `sumMerge(col)` | 求和预聚合 |
| `countState()` | `AggregateFunction(count)` | `countMerge(col)` | 计数预聚合 |
| `uniqState(x)` | `AggregateFunction(uniq, String)` | `uniqMerge(col)` | 去重预聚合 |
| `minState(x)` | `AggregateFunction(min, T)` | `minMerge(col)` | 最小值预聚合 |
| `maxState(x)` | `AggregateFunction(max, T)` | `maxMerge(col)` | 最大值预聚合 |
| `avgState(x)` | `AggregateFunction(avg, Float64)` | `avgMerge(col)` | 平均值预聚合 |
| `quantilesState(0.5,0.99)(x)` | `AggregateFunction(...)` | `quantilesMerge(...)(col)` | 分位数预聚合 |
| `groupArrayState(x)` | `AggregateFunction(groupArray, T)` | `groupArrayMerge(col)` | 数组收集预聚合 |

**工作原理**：
- `-State` 函数不计算最终结果，而是生成一个**序列化的中间聚合状态**
- 这个中间状态可以被后续的 Merge 操作合并
- `-Merge` 函数读取时，将多个中间状态合并并输出最终结果

```sql
-- 示例：sumState 存储的不是数字 8，而是一个可合并的状态对象
-- 两个 part 各自有 sumState=5 和 sumState=3
-- Merge 时自动变成 sumState=8
-- 查询时 sumMerge 取出 8
```

### 3.3 AggregatingMergeTree 建表语法

```sql
CREATE TABLE agg_target ON CLUSTER cluster_name
(
    event_date  Date,
    page_id     UInt32,
    pv          AggregateFunction(sum, UInt64),
    uv          AggregateFunction(uniq, String),
    avg_duration AggregateFunction(avg, Float64)
)
ENGINE = AggregatingMergeTree()
PARTITION BY toYYYYMM(event_date)
ORDER BY (event_date, page_id);
```

⚠️ **关键注意点**：
- `ORDER BY` 定义了聚合的维度（GROUP BY 的列必须包含 ORDER BY 的所有列）
- 非 ORDER BY 的非聚合列在 Merge 时取值不确定
- 查询时**必须**使用 `-Merge` 函数，直接 SELECT 聚合列会得到二进制乱码

## 四、多 MV 写放大问题

### 4.1 写放大模型

```
base table 收到 1 次 INSERT（1MB 数据）
  ├── MV_1: SELECT ... GROUP BY date       → 写入 target_1（10KB）
  ├── MV_2: SELECT ... GROUP BY date, page → 写入 target_2（100KB）
  ├── MV_3: SELECT ... GROUP BY date, user → 写入 target_3（500KB）
  └── MV_4: SELECT * (明细复制)            → 写入 target_4（1MB）

总写入量 = 1MB + 10KB + 100KB + 500KB + 1MB ≈ 2.6MB
写放大系数 = 2.6x
```

### 4.2 写放大的影响

1. **写入延迟增加**：同步模式下，INSERT 延迟 = base 写入 + 所有 MV 写入之和
2. **磁盘 IO 增加**：每个 MV 产生独立的 Data Part，加速小文件积累
3. **Merge 压力增大**：更多 Part 意味着更频繁的后台 Merge
4. **资源竞争**：MV 写入与正常查询共享 CPU/IO 资源

### 4.3 缓解策略

| 策略 | 做法 | 效果 |
|------|------|------|
| 减少 MV 数量 | 合并维度相近的 MV | 降低写放大系数 |
| 客户端攒批写入 | `async_insert=1` 降低 Part 产生频率 | 降低 Part 爆炸风险，但 MV 触发有延迟 |
| 合理设计粒度 | 避免过细粒度的 MV | 减少 MV 输出数据量 |
| 监控 Part 数量 | `system.parts` 表 | 及时发现 Merge 积压 |
| 分区对齐 | MV 与 base 使用相同分区键 | 减少跨分区 Merge |

## 五、幂等性与 ReplacingMergeTree

### 5.1 MV 的重试问题

当 Flink 等上游系统发生故障恢复时，可能会重新发送已经写入过的数据。MV 作为触发器会再次处理这些重复数据，导致预聚合结果偏大。

```
第一次写入: batch_id=100, records=[r1,r2,r3] → MV 累加 pv+3
故障恢复后重试: batch_id=100, records=[r1,r2,r3] → MV 又累加 pv+3
结果: pv 多了 3，数据不正确！
```

### 5.2 ReplacingMergeTree 实现幂等

思路：给每条预聚合结果加上唯一标识（如 batch_id + 维度组合），利用 ReplacingMergeTree 的去重能力。

```sql
CREATE TABLE agg_idempotent
(
    batch_id    String,        -- 用于去重的版本标识
    event_date  Date,
    page_id     UInt32,
    pv          UInt64,
    updated_at  DateTime       -- ReplacingMergeTree 的版本列
)
ENGINE = ReplacingMergeTree(updated_at)
PARTITION BY toYYYYMM(event_date)
ORDER BY (batch_id, event_date, page_id);
```

**局限性**：
- ReplacingMergeTree 只在后台 Merge 时去重，查询前可能有重复
- 需要配合 `FINAL` 或 `argMax` 使用
- 不适合高频更新的场景

### 5.3 更好的幂等方案

对于严格的 exactly-once 需求，推荐：

1. **Flink 侧保证**：使用 Flink Checkpoint + 两阶段提交，确保不重复写入
2. **StarRocks Primary Key Model**：原生支持 upsert，天然幂等
3. **业务层去重**：在 MV 之前加一层去重逻辑（如 Redis 记录已处理 batch_id）

## 六、面试话术要点

### Q: ClickHouse 物化视图是怎么工作的？

> ClickHouse 的物化视图本质上是一个 INSERT 触发器。当数据写入基表时，ClickHouse 会自动用新写入的数据块执行 MV 定义的 SELECT 语句，并将结果写入目标表。它不会回溯处理历史数据，也不是查询时才计算。如果要处理历史数据，需要使用 POPULATE 关键字，但 POPULATE 只在创建时执行一次，且存在并发写入时数据遗漏的风险。

### Q: AggregatingMergeTree 和普通 MergeTree 有什么区别？

> AggregatingMergeTree 在后台 Merge 数据块时，会自动按照 ORDER BY 定义的维度，对标记了 AggregateFunction 类型的列执行合并操作。比如两个 Part 中同一个维度的 sumState 分别是 5 和 3，Merge 后自动变成 8。普通 MergeTree 只是简单地将 Part 拼接排序，不做任何聚合运算。

### Q: MV 有什么缺点或注意事项？

> 主要有四点：第一，写放大，多个 MV 会增加写入延迟和磁盘 IO；第二，不回溯，创建前的历史数据不会自动处理；第三，不支持查询透明改写，应用层需要知道查目标表而非基表；第四，异步模式下存在数据丢失风险，同步模式下 MV 写入失败会导致整个 INSERT 回滚。

### Q: 如何保证 MV 预聚合的准确性？

> 需要关注三个方面：一是幂等性，上游重试会导致重复累加，可以用 ReplacingMergeTree 或在 Flink 侧保证 exactly-once；二是 POPULATE 的数据遗漏风险，生产环境建议先建空 MV 再手动补数据；三是 AggregatingMergeTree 的查询必须使用 -Merge 函数，否则读到的是序列化的中间状态而非最终结果。
