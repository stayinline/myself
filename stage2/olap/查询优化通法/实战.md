## 一、慢查询分析完整流程

### 1.1 问题场景

运营反馈：品类销售报表查询很慢，经常超时。

```sql
-- 原始慢查询
SELECT
    o.order_date,
    p.category_name,
    p.brand,
    COUNT(DISTINCT o.user_id) AS buyer_cnt,
    COUNT(*) AS order_cnt,
    SUM(o.amount) AS total_amount,
    AVG(o.amount) AS avg_amount
FROM fact_orders o
JOIN dim_product p ON o.product_id = p.product_id
WHERE o.order_date >= '2025-01-01'
  AND o.order_date < '2025-07-01'
  AND p.category_name IN ('电子产品', '服装', '食品')
GROUP BY o.order_date, p.category_name, p.brand
ORDER BY o.order_date DESC, total_amount DESC
LIMIT 100;

-- 执行时间: 45秒 ❌ (目标 < 1秒)
```

## 二、逐步优化过程

### 2.1 优化 Round 1: 确认分区裁剪

```sql
-- 检查分区情况
SHOW PARTITIONS FROM fact_orders;
-- 发现: 按月分区, 2025-01 ~ 2025-06 共 6 个分区, 每个分区约 5300万行

-- 当前 WHERE 已覆盖分区列, 裁剪生效但仍扫 6 个分区
-- → 数据量本身大, 需排序键 / MV 进一步优化

-- 验证: 缩小到一个月
SELECT ... WHERE o.order_date >= '2025-06-01' AND o.order_date < '2025-07-01' ...
-- 执行时间: 8秒 (从45秒降到8秒, 但仍然慢)
```

**优化效果**: 45s → 8s（缩小范围后），说明分区裁剪在工作，但单月数据量仍然很大。

### 2.2 优化 Round 2: 排序键 / 前缀索引优化

```sql
-- 检查当前排序键
SHOW CREATE TABLE fact_orders;
-- ORDER BY (order_id)  ← 问题! order_id 不在 WHERE 中

-- 创建新表, 优化排序键
CREATE TABLE fact_orders_optimized
(
    order_date   DATE NOT NULL,
    product_id   BIGINT NOT NULL,
    user_id      BIGINT NOT NULL,
    amount       DECIMAL(18,2),
    quantity     INT,
    order_id     BIGINT NOT NULL
)
DUPLICATE KEY(order_date, product_id, user_id)  -- 按查询模式重排
PARTITION BY RANGE(order_date) (...)
DISTRIBUTED BY HASH(order_id) BUCKETS 16;

-- 迁移数据后重新测试
-- 执行时间: 3秒 (从8秒降到3秒)
```

**优化效果**: 8s → 3s。排序键从 `order_id` 改为 `(order_date, product_id, user_id)` 后，前缀索引+ZoneMap 大幅减少了扫描行数。

### 2.3 优化 Round 3: 添加 Skip Index / BloomFilter

```sql
-- ClickHouse: 添加 Skip Index
ALTER TABLE fact_orders_optimized
ADD INDEX idx_user_id user_id TYPE bloom_filter(0.01) GRANULARITY 4;

-- StarRocks: 添加 BloomFilter
ALTER TABLE fact_orders_optimized
SET ("bloom_filter_columns" = "user_id");

-- 对于需要 COUNT DISTINCT user_id 的查询
-- BloomFilter 可以帮助跳过不包含目标用户的 Block
-- 但在本查询中是全局 COUNT DISTINCT, BloomFilter 帮助有限

-- 更有效的做法: 对 category_name 加索引 (如果它是非排序键过滤列)
-- CH:
ALTER TABLE fact_orders_optimized
ADD INDEX idx_category category_name TYPE set(64) GRANULARITY 4;

-- SR: Bitmap Index
CREATE INDEX idx_category ON fact_orders_optimized (category_name) USING BITMAP;

-- 执行时间: 2.5秒 (小幅改善)
```

**优化效果**: 3s → 2.5s。Skip Index 对当前查询的帮助有限，因为主要瓶颈是数据量大而非过滤效率低。

### 2.4 优化 Round 4: 预聚合 MV ⭐ (收益最大)

```sql
-- 分析查询模式: 固定 GROUP BY (order_date, category_name, brand)
-- 非常适合预聚合!

-- === StarRocks Async MV ===
CREATE MATERIALIZED VIEW mv_daily_category_brand_sales
PARTITION BY (order_date)
DISTRIBUTED BY HASH(order_date) BUCKETS 8
REFRESH ASYNC EVERY (INTERVAL 1 HOUR)
AS
SELECT
    o.order_date,
    p.category_name,
    p.brand,
    bitmap_union(to_bitmap(o.user_id)) AS buyer_bitmap,
    COUNT(*) AS order_cnt,
    SUM(o.amount) AS total_amount
FROM fact_orders_optimized o
JOIN dim_product p ON o.product_id = p.product_id
GROUP BY o.order_date, p.category_name, p.brand;

-- 等待 MV 刷新完成后, 原始查询自动被改写
-- 执行时间: 150ms 🎉 (从2.5秒降到150ms!)

-- === ClickHouse MV ===
-- 需要提前打宽表或在 Flink 侧完成 JOIN
-- 然后创建 AggregatingMergeTree + MV
```

**优化效果**: 2.5s → 150ms。**预聚合是收益最大的优化手段**，将运行时 JOIN + GROUP BY 变为直接读取预计算结果。

## 三、高基数列作为主键的问题

### 3.1 问题描述

```sql
-- 错误示范: 用 UUID 做排序键/主键
CREATE TABLE events_bad
(
    event_id String,       -- UUID v4, 完全随机
    event_time DateTime,
    user_id UInt64,
    data String
)
ENGINE = MergeTree()
ORDER BY (event_id);  -- ❌ UUID 作为排序键

-- 问题1: UUID 随机性导致写入时数据分散
--   每次 INSERT 的数据可能落在不同的 Granule
--   → 大量随机 IO, 写入性能下降 5-10x

-- 问题2: 稀疏索引区分度虽高但无法利用局部性
--   相邻索引条目对应的数据在物理上不连续
--   → 范围扫描退化为随机读

-- 问题3: 字符串排序键占用大量索引空间
--   UUID 36 bytes × (rows/granularity) = 巨大的 primary.idx
```

### 3.2 解决方案

```sql
-- ✅ 方案1: 用时间 + 业务ID 替代纯随机键
ORDER BY (event_time, event_id)
-- 时间保证写入有序, event_id 只用于二级定位

-- ✅ 方案2: 用数字型 ID 替代字符串 UUID
-- 如果必须用 UUID, 转为 UInt128 或用 UUIDStringToNum()
ORDER BY (toUInt128(event_id))

-- ✅ 方案3: 分离存储键和业务键
-- 存储用自增 ID 或时间戳做排序键
-- 业务查询通过 Skip Index (BloomFilter) 加速 UUID 查找
ORDER BY (event_time)
INDEX idx_event_id event_id TYPE bloom_filter(0.001) GRANULARITY 4

-- ✅ StarRocks PK Model: 主键可以不是排序键
PRIMARY KEY(event_id)          -- 用于 Upsert 去重
ORDER BY(event_time, user_id)  -- 用于查询加速
-- PK 和 Sort Key 解耦!
```

### 3.3 量化对比

```
数据集: 1亿行, UUID 主键 vs 时间排序键

UUID 作为 ORDER BY:
  写入吞吐: ~5万 rows/s (大量随机IO)
  索引大小: 3.2 GB
  按 UUID 点查: ~5ms (二分查找快)
  按时间范围扫描: ~30s (数据分散, 大量随机读)

时间作为 ORDER BY + UUID BloomFilter:
  写入吞吐: ~50万 rows/s (顺序写入) ✅ 10x
  索引大小: 800 MB ✅ 4x smaller
  按 UUID 点查: ~15ms (BloomFilter + 扫描) 略慢但可接受
  按时间范围扫描: ~500ms ✅ 60x faster
```

## 四、面试话术："一条聚合查询很慢，你的优化路径"

### 4.1 结构化回答

> **面试官**: 如果一条聚合查询很慢，你会怎么优化？

> **回答**: 我会按照"先看计划、再做裁剪、然后索引、最后预聚合"的四步法来优化。
>
> **第一步，看执行计划**。用 EXPLAIN（StarRocks）或 EXPLAIN PIPELINE（ClickHouse）查看执行计划，重点关注三个指标：扫描了多少分区和行、有没有走索引、JOIN 策略是什么。这一步确定瓶颈在哪里。
>
> **第二步，检查分区裁剪和排序键**。确认 WHERE 条件是否触发了分区裁剪，如果没有就调整 SQL 写法或者修改分区策略。然后检查排序键是否与查询的过滤列匹配，如果不匹配考虑重建表或用 Projection。这两步通常能把扫描量减少一到两个数量级。
>
> **第三步，添加辅助索引**。对非排序键的高频过滤列加 Skip Index（ClickHouse）或 BloomFilter/Bitmap Index（StarRocks）。这一步是锦上添花，通常再减少 2-5 倍扫描。
>
> **第四步，预聚合**。如果前三步优化后仍然不够快，且查询模式比较固定，就建物化视图做预聚合。这是终极武器，可以把秒级查询降到毫秒级。StarRocks 的异步 MV 还支持多表 JOIN 预聚合和透明改写，对应用层零侵入。
>
> 在实际项目中我用这套方法把一个 45 秒的品类销售报表优化到了 150 毫秒，其中预聚合贡献了最大的收益。

### 4.2 追问准备

**Q: 如果查询模式不固定怎么办？**
> 如果 Ad-hoc 查询为主，预聚合就不太适用了。这时候重点做好前三步：分区裁剪、排序键设计、辅助索引。同时可以利用 StarRocks 的 Query Cache 缓存热点查询结果。另外考虑是否需要引入 Elasticsearch 等专门的搜索引擎来处理灵活的过滤需求。

**Q: EXPLAIN 看到全表扫描一定是坏事吗？**
> 不一定。如果表本身很小（百万级以下），全表扫描可能比走索引还快，因为避免了索引查找的开销。另外如果是 COUNT(*) 不带 WHERE，引擎可能有元数据级别的优化，不需要真正扫描数据。所以要看绝对扫描量和耗时，而不是单纯看是否全表扫描。

**Q: 排序键改了之后存量数据怎么办？**
> ClickHouse 需要重建表（INSERT INTO new_table SELECT * FROM old_table），大表迁移成本高，可以用 OPTIMIZE TABLE 逐步重组但非常慢。StarRocks 支持 Light Schema Change 但不能改排序键，也需要重建。所以排序键一定要在建表时就设计好，后期修改代价很大。我们的做法是建表前分析 Top 20 查询的 WHERE 条件，确保排序键覆盖主要查询模式。

**Q: Runtime Filter 什么时候生效？**
> Runtime Filter 在 Hash Join 时自动生成。Build 侧（通常是较小的维表）构建完 Hash Table 后，提取 Join Key 生成 BloomFilter，下推到 Probe 侧（事实表）的 Scan 算子。Scan 时用这个 Filter 提前排除不匹配的行，减少进入 JOIN 的数据量。它在事实表远大于维表、且 Join Key 选择性高的时候效果最好。StarRocks CBO 会自动决定是否启用以及使用哪种 Filter（Bloom/In/MinMax）。

## 六、常用诊断 SQL 汇总

```sql
-- === ClickHouse ===

-- 查看表的 Part 信息和索引大小
SELECT table, partition, count() AS parts,
       formatReadableSize(sum(bytes)) AS size,
       formatReadableSize(sum(primary_key_bytes_in_memory)) AS pk_mem
FROM system.parts WHERE database = 'db' AND active
GROUP BY table, partition ORDER BY size DESC;

-- 查看最近执行的慢查询
SELECT query, query_duration_ms, read_rows, read_bytes, memory_usage
FROM system.query_log
WHERE type = 'QueryFinish' AND query_duration_ms > 1000
ORDER BY event_time DESC LIMIT 20;

-- 查看索引使用情况
EXPLAIN indexes = 1 SELECT ... ;

-- === StarRocks ===

-- 查看查询 Profile
SET enable_profile = true;
SELECT ...;
-- FE Web UI → Queries → Profile

-- 查看 Tablet 大小分布（检测数据倾斜）
SELECT BackendId, count(*) AS tablet_count,
       sum(DataSize) / 1024/1024 AS total_mb
FROM information_schema.be_tablets
WHERE TableName = 'fact_orders'
GROUP BY BackendId;

-- 查看物化视图状态
SHOW MATERIALIZED VIEWS\G

-- 查看 Compaction 分数（过高说明 Merge 积压）
SHOW PROC '/compactions';
```
