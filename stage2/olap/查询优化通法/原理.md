# OLAP 查询优化通法原理

## 一、OLAP 查询优化方法论总览

### 1.1 优化层次模型

```
┌─────────────────────────────────────────────────────────┐
│              OLAP 查询优化金字塔                           │
│                                                         │
│                    ┌─────────┐                           │
│                    │ SQL改写  │ ← 最上层，应用层可做       │
│                  ┌─┴─────────┴─┐                        │
│                  │  索引/Skip   │ ← 存储层优化            │
│                ┌─┴─────────────┴─┐                      │
│                │  预聚合/MV      │ ← 数据模型层           │
│              ┌─┴─────────────────┴─┐                    │
│              │  分区/排序键设计      │ ← 建表时决定         │
│            ┌─┴─────────────────────┴─┐                  │
│            │  列存+向量化+稀疏索引     │ ← 引擎基础能力     │
│            └─────────────────────────┘                   │
│                                                         │
│  原则: 越底层的优化收益越大，但变更成本越高                   │
│        优先做底层优化，再用上层手段微调                       │
└─────────────────────────────────────────────────────────┘
```

| 查询缓存 | ✅ Query Cache | ✅ Query Cache | ∞(命中时) | 配置 |

## 二、分区裁剪（Partition Pruning）

### 2.1 原理

分区裁剪是在查询规划阶段就排除不相关分区，避免扫描无关数据。这是**收益最大、成本最低**的优化手段。

```
无分区裁剪:
  Scan → Part_202501 + Part_202502 + ... + Part_202506 → 全量扫描
  扫描量: 100GB

有分区裁剪 (WHERE date >= '2025-06-01'):
  Scan → Part_202506 → 只扫一个分区
  扫描量: 17GB (节省 83%)
```

### 2.2 有效 vs 无效的分区过滤

```sql
-- ✅ 有效：直接对分区列使用比较运算符
WHERE event_date = '2025-06-01'
WHERE event_date >= '2025-06-01' AND event_date < '2025-06-08'
WHERE toYYYYMM(event_date) = 202506    -- CH 支持函数推导

-- ❌ 无效：分区列被包裹在无法推导的函数中
WHERE toUnixTimestamp(event_date) > 1717200000
WHERE event_date IN (SELECT max_date FROM config_table)  -- 子查询
WHERE date_diff('day', event_date, now()) < 7           -- CH不支持推导

-- ✅ StarRocks 的动态分区裁剪
WHERE event_date IN (SELECT date FROM hot_dates)
-- CBO 可以将子查询结果转为 IN list，仍然触发分区裁剪
```

### 2.3 分区设计最佳实践

- 分区粒度匹配查询模式：大部分查询查 1-7 天 → 按天分区
- 避免分区过多：单表分区数 < 1000
- 使用动态分区自动管理生命周期
- 分区列选择高选择性且查询必带的字段（通常是时间）

## 三、排序键与前缀索引

### 3.1 ClickHouse 稀疏索引

```
┌──────────────────────────────────────────────────────────┐
│           ClickHouse 稀疏索引原理                          │
│                                                          │
│  ORDER BY (page_id, event_time)                          │
│                                                          │
│  Data Block (granule):                                   │
│  ┌──────────────────────────────────────────┐            │
│  │ Granule 0: page_id=1, time=10:00~10:05  │ ← index    │
│  │ Granule 1: page_id=1, time=10:05~10:10  │   entry    │
│  │ Granule 2: page_id=2, time=10:00~10:03  │            │
│  │ Granule 3: page_id=2, time=10:03~10:08  │            │
│  │ ...                                      │            │
│  └──────────────────────────────────────────┘            │
│                                                          │
│  Primary.idx (稀疏索引文件):                               │
│  每个 Granule 记录一行索引条目                              │
│  索引大小 = 总行数 / index_granularity × 索引列宽度          │
│                                                          │
│  查询 WHERE page_id = 2:                                  │
│  1. 二分查找 primary.idx → 定位 Granule 2,3               │
│  2. 只读取这两个 Granule 的数据                             │
│  3. 跳过 Granule 0,1 → 减少 50% IO                       │
└──────────────────────────────────────────────────────────┘
```

**关键参数**：`index_granularity`（默认 8192），表示每个索引条目覆盖的行数。越小索引越精确但占用越大。

### 3.2 StarRocks 前缀索引 + ZoneMap

```
┌──────────────────────────────────────────────────────────┐
│           StarRocks 前缀索引原理                           │
│                                                          │
│  SORT KEY (page_id, event_time, user_id)                 │
│                                                          │
│  Segment File:                                           │
│  ┌──────────────────────────────────────────┐            │
│  │ Prefix Index (每 1024 行一个条目):         │            │
│  │  Row 0:    page_id=1, time=10:00         │            │
│  │  Row 1024: page_id=1, time=10:05         │            │
│  │  Row 2048: page_id=2, time=10:00         │            │
│  │  Row 3072: page_id=2, time=10:03         │            │
│  └──────────────────────────────────────────┘            │
│                                                          │
│  ZoneMap Index (每个 Column 每 Block):                    │
│  ┌──────────────────────────────────────────┐            │
│  │ page_id Block 0: min=1, max=1            │            │
│  │ page_id Block 1: min=1, max=2            │            │
│  │ amount  Block 0: min=10.5, max=999.0     │            │
│  │ amount  Block 1: min=5.0, max=500.0      │            │
│  └──────────────────────────────────────────┘            │
│                                                          │
│  查询 WHERE page_id = 2 AND amount > 800:                 │
│  1. Prefix Index → 定位 page_id=2 的行范围                │
│  2. ZoneMap on amount → 排除 Block 1 (max=500 < 800)    │
│  3. 只读取 Block 0 中 page_id=2 的行                     │
└──────────────────────────────────────────────────────────┘
```

### 3.3 ORDER BY / SORT KEY 设计原则

| 原则 | 说明 | 示例 |
|------|------|------|
| 高频过滤列放前面 | 前缀索引只对前缀有效 | `(tenant_id, date)` 优于 `(date, tenant_id)` 如果 tenant 过滤更频繁 |
| 等值查询列在前，范围查询列在后 | 范围条件后的列无法利用前缀索引 | `(status, date)` 优于 `(date, status)` |
| 基数从高到低 | 高基数列区分度大，缩小范围更有效 | `(user_id, date)` 优于 `(date, user_id)` |
| 与分区键互补 | 分区已处理的维度不必再放最前 | 已按天分区，ORDER BY 可以 `(category, user_id)` |
| 避免过大列 | 字符串列作为排序键会增加索引大小 | 用 ID 代替名称 |

## 四、Skip Index / 二级索引

### 4.1 ClickHouse Skip Index

```sql
-- 创建 Skip Index
ALTER TABLE events ADD INDEX idx_action_type action_type TYPE set(0) GRANULARITY 4;
ALTER TABLE events ADD INDEX idx_duration duration TYPE minmax GRANULARITY 4;
ALTER TABLE events ADD INDEX idx_url_bf url TYPE bloom_filter(0.01) GRANULARITY 4;

-- Skip Index 类型
-- minmax: 记录区间内的最大最小值，适合范围过滤
-- set: 记录区间内的去重值集合（最多 N 个），适合低基数等值过滤
-- bloom_filter: 布隆过滤器，适合高基数等值过滤
-- tokenbf_v1: Token 级布隆过滤器，适合文本搜索
-- ngrambf_v1: N-gram 布隆过滤器，适合模糊搜索
```

### 4.2 StarRocks 索引体系

| 索引类型 | 自动创建 | 适用场景 | 创建方式 |
|---------|---------|---------|---------|
| ZoneMap | ✅ 每列自动 | 范围过滤、等值过滤 | 无需手动 |
| BloomFilter | ❌ | 高基数等值过滤 | `PROPERTIES ("bloom_filter_columns" = "col1,col2")` |
| Bitmap Index | ❌ | 低基数等值/IN 过滤 | `CREATE INDEX ... USING BITMAP` |
| Prefix Index | ✅ 排序键自动 | 排序键前缀过滤 | 由 SORT KEY 决定 |
| Inverted Index | ❌ | 全文检索、数组包含 | `CREATE INDEX ... USING INVERTED` |

## 五、预聚合与采样

### 5.1 预聚合决策框架

```
是否需要预聚合？
  │
  ├── 查询是否固定模式（GROUP BY 维度稳定）？
  │   ├── 否 → 不做预聚合，靠引擎原生性能
  │   └── 是 ↓
  │
  ├── 原始数据量是否导致查询超时？
  │   ├── 否 → 暂不预聚合，监控趋势
  │   └── 是 ↓
  │
  ├── 是否多表 JOIN？
  │   ├── 是 → StarRocks Async MV
  │   └── 否 ↓
  │
  └── 单表 → CH MV + AggregatingMergeTree
              或 SR Aggregate Model
```

### 5.2 Sampling 适用场景

```sql
-- ClickHouse Sampling
SELECT count() * 100 AS estimated_total  -- 乘以采样倍率还原
FROM events
SAMPLE 1/100;   -- 只扫描 1% 的数据

-- StarRocks Sampling
SELECT count(*) * 10 AS estimated_total
FROM events
TABLESAMPLE SYSTEM (10 PERCENT);

-- 适用场景:
-- • Ad-hoc 探索性分析（不需要精确结果）
-- • 超大数据集的近似统计
-- • A/B 测试的快速验证
-- • 数据质量检查

-- 不适用:
-- • 精确报表
-- • 小数据集（采样误差大）
-- • 需要精确 COUNT DISTINCT
```

## 六、列存 + 向量化 + 稀疏索引 "三件套"

### 6.1 为什么这三者缺一不可

```
┌─────────────────────────────────────────────────────────┐
│          OLAP 高性能查询三件套                             │
│                                                         │
│  ① 列式存储                                              │
│  • 只读取查询涉及的列（而非整行）                           │
│  • 同类型数据连续存储 → 高压缩比（5x~20x）                 │
│  • SIMD 友好的内存布局                                    │
│                                                         │
│  ② 向量化执行                                            │
│  • 一次处理一批数据（4096 rows/batch）而非逐行              │
│  • 利用 CPU SIMD 指令并行计算                              │
│  • 减少函数调用开销和分支预测失败                            │
│  • 充分利用 CPU L1/L2 Cache                               │
│                                                         │
│  ③ 稀疏索引                                              │
│  • 不是每行都有索引条目（否则索引太大）                      │
│  • 每 N 行一个索引点，通过二分查找定位数据块                  │
│  • 配合分区裁剪，将扫描范围从 TB 级缩小到 GB 级              │
│                                                         │
│  三者协同效果:                                             │
│  分区裁剪: 100TB → 1TB                                   │
│  稀疏索引: 1TB → 10GB                                    │
│  列式存储: 10GB → 500MB (只读需要的列)                     │
│  向量化: 500MB 数据在毫秒内完成计算                         │
│  总计: 从 TB 级数据中毫秒级返回结果                         │
└─────────────────────────────────────────────────────────┘
```

### 6.2 与行存的本质区别

```
行存 (MySQL/PostgreSQL OLTP):
┌────────────────────────────────────┐
│ Row1: id|name|age|city|amount|...  │
│ Row2: id|name|age|city|amount|...  │
│ Row3: id|name|age|city|amount|...  │
└────────────────────────────────────┘
→ SELECT SUM(amount) 需要读取所有行的所有列
→ 大量无用 IO

列存 (ClickHouse/StarRocks):
┌────────┬────────┬─────┬──────┬────────┐
│ id col │name col│age  │city  │amount  │
│ [1,2,3]│[A,B,C]│[25, │[BJ, │[100,   │
│        │        │ 30, │ SH, │ 200,   │
│        │        │ 28] │ GZ] │ 150]   │
└────────┴────────┴─────┴──────┴────────┘
→ SELECT SUM(amount) 只读 amount 列
→ IO 减少 80%+ (假设 5 列)
```

## 七、ORDER BY 设计常见陷阱

### 7.1 陷阱列表

| 陷阱 | 问题 | 正确做法 |
|------|------|---------|
| 把低基数字段放最后 | `ORDER BY (high_card, low_card)` → low_card 无法利用索引 | 低基数等值字段放前面 |
| 排序键与查询过滤不匹配 | 建表时没考虑主要查询模式 | 分析 Top 10 查询的 WHERE 条件后设计 |
| 排序键太多列 | 索引膨胀，写入变慢 | 控制在 3-5 列以内 |
| 用大字符串做排序键 | 索引文件巨大 | 用字典编码的 ID 替代 |
| 忽略分区键的影响 | 分区已过滤的列重复放在 ORDER BY 最前 | 分区键可以不放在 ORDER BY 首位 |
| 改了查询模式没改排序键 | 新查询走不到索引 | 定期 Review 查询模式与排序键匹配度 |
| CH 中 ORDER BY ≠ GROUP BY (MV) | AggregatingMergeTree 合并语义错误 | 确保 GROUP BY 包含全部 ORDER BY 列 |

### 7.2 排序键优化案例

```sql
-- ❌ 糟糕的设计
CREATE TABLE orders (
    order_id UInt64,
    create_date Date,
    user_id UInt64,
    status String,
    amount Float64
)
ENGINE = MergeTree()
PARTITION BY create_date
ORDER BY (order_id);  -- order_id 几乎不会用于 WHERE 过滤！

-- 典型查询: WHERE create_date = ? AND status = ? AND user_id = ?
-- ORDER BY (order_id) 完全无法加速这些查询

-- ✅ 优化后的设计
CREATE TABLE orders (
    order_id UInt64,
    create_date Date,
    user_id UInt64,
    status LowCardinality(String),
    amount Float64
)
ENGINE = MergeTree()
PARTITION BY create_date           -- 时间分区裁剪
ORDER BY (status, user_id);        -- 状态等值 + 用户ID范围

-- 查询 WHERE status = 'paid' AND user_id = 12345
-- → 前缀索引精确定位 status='paid' 段
-- → 在该段内二分查找 user_id=12345
-- → 只扫描极少量数据
```

## 八、面试话术要点

### Q: OLAP 查询优化的通用方法有哪些？

> 我总结为"五板斧"。第一是分区裁剪，通过合理的分区设计让查询只扫描相关分区，这通常能减少 90% 以上的扫描量。第二是排序键设计，把高频过滤列放在 ORDER BY 前面，利用稀疏索引或前缀索引快速定位数据。第三是 Skip Index，对非排序键的过滤列加 BloomFilter 或 MinMax 索引进一步缩小扫描范围。第四是预聚合，通过物化视图提前计算好常用指标，将运行时聚合变为直接读取。第五是采样，对探索性分析使用 SAMPLE 以精度换速度。实际优化中我会先用 EXPLAIN 看执行计划，找到瓶颈再针对性地选择优化手段。

### Q: 列存+向量化+稀疏索引为什么被称为三件套？

> 因为三者缺一不可，共同构成了 OLAP 高性能的基础。列存解决了"读多少数据"的问题，只读需要的列，IO 减少一个数量级。稀疏索引解决了"从哪里开始读"的问题，不用全表扫描就能定位目标数据块。向量化解决了"读到的数据怎么算得快"的问题，利用 SIMD 批量处理，CPU 利用率最大化。三者叠加的效果是：从 TB 级数据中毫秒级返回聚合结果。如果缺少任何一个，比如只有列存没有索引，还是要扫全列；只有索引没有向量化，计算还是慢。
