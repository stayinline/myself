# StarRocks 实操原理

## 一、异步物化视图（Async Materialized View）

### 1.1 核心概念：与 ClickHouse MV 的本质区别

StarRocks 的异步物化视图是**真正支持多表 JOIN 预聚合**的机制，这与 ClickHouse 的 INSERT 触发器式 MV 有本质不同。

```
┌─────────────────────────────────────────────────────────────┐
│         ClickHouse MV vs StarRocks Async MV                  │
│                                                             │
│  ClickHouse MV:                                             │
│  ┌──────────┐   INSERT触发   ┌────────────┐                 │
│  │ Base Table│ ──────────→   │ Target Table│                │
│  └──────────┘  (仅处理新块)   └────────────┘                │
│  ❌ 不支持 JOIN                                              │
│  ❌ 不支持透明改写                                            │
│  ✅ 写入时同步/异步触发                                       │
│                                                             │
│  StarRocks Async MV:                                        │
│  ┌──────────┐   ┌──────────┐                                │
│  │ Table A  │   │ Table B  │                                 │
│  └────┬─────┘   └────┬─────┘                                │
│       │              │                                      │
│       └──────┬───────┘                                      │
│              ▼                                              │
│       ┌─────────────┐   定时/手动刷新   ┌────────────┐      │
│       │ MV Definition│ ──────────────→  │ MV Storage │      │
│       │ (A JOIN B)   │  (增量/全量)     │            │      │
│       └─────────────┘                   └────────────┘      │
│  ✅ 支持多表 JOIN                                            │
│  ✅ CBO 自动透明改写                                          │
│  ✅ 分区级增量刷新                                            │
│  ⚠️ 非实时（有刷新间隔延迟）                                   │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 详细对比表

| 维度 | ClickHouse MV | StarRocks Async MV |
|------|--------------|-------------------|
| 触发方式 | INSERT 时自动触发 | 定时调度 / 手动 REFRESH |
| 数据源 | 单表 | 多表 JOIN |
| 增量刷新 | 天然增量（逐块） | 分区级增量（基于分区变更检测） |
| 查询透明改写 | ❌ 不支持 | ✅ CBO 自动识别并改写 |
| 聚合函数 | -State/-Merge 体系 | 标准 SQL 聚合函数 |
| 刷新一致性 | 与 INSERT 同步或异步 | 最终一致（刷新窗口内） |
| 嵌套 MV | 支持（MV on MV） | 支持（MV on MV） |
| 适用场景 | 单表实时预聚合 | 多表 JOIN 加速、ETL 简化 |

### 1.3 异步 MV 内部工作原理

```
创建异步 MV 后的系统行为：

1. FE Job Scheduler 按设定的 refresh_interval 触发刷新任务
                    ↓
2. 检查基表分区是否有变更（通过 partition version 比较）
                    ↓
3. 只刷新变更过的分区对应的数据（增量刷新）
   如果无法增量则退化为全量刷新
                    ↓
4. 执行 MV 定义的 SELECT 语句（可能包含 JOIN、GROUP BY）
                    ↓
5. 将结果写入 MV 的内部存储（本质上是一个内部表）
                    ↓
6. 更新 MV 的元数据（最新版本、刷新状态等）
                    ↓
7. 后续查询时 CBO 判断是否可以命中此 MV
   → 如果可以，自动改写查询计划使用 MV 数据
   → 如果不能，回退到原始查询
```

### 1.4 分区级增量刷新机制

```sql
-- 假设 MV 定义如下
CREATE MATERIALIZED VIEW mv_order_daily AS
SELECT
    o.order_date,
    p.category,
    COUNT(*) AS order_cnt,
    SUM(o.amount) AS total_amount
FROM fact_order o
JOIN dim_product p ON o.product_id = p.product_id
GROUP BY o.order_date, p.category;

-- 增量刷新逻辑：
-- 1. 检测 fact_order 哪些分区有新数据（如 p20250601, p20250602）
-- 2. 检测 dim_product 是否有变更（维表通常较小，可能全量参与）
-- 3. 只重新计算受影响分区的聚合结果
-- 4. 替换 MV 中对应分区的数据
```

⚠️ **增量刷新的前提条件**：
- MV 的 PARTITION BY 必须能映射到基表的 PARTITION BY
- JOIN 的维表不能有分区级别的变更追踪（通常全量参与）
- 某些复杂的 SQL 模式（如窗口函数）可能不支持增量刷新

## 二、异步 MV 创建与管理

### 2.1 基本语法

```sql
-- 创建异步 MV（定时刷新）
CREATE MATERIALIZED VIEW mv_user_page_stats
DISTRIBUTED BY HASH(stat_date) BUCKETS 8
REFRESH ASYNC EVERY (INTERVAL 5 MINUTE)
AS
SELECT
    to_date(u.login_time) AS stat_date,
    u.city,
    COUNT(DISTINCT u.user_id) AS uv,
    COUNT(*) AS pv,
    SUM(p.price) AS gmv
FROM user_behavior u
JOIN product_info p ON u.product_id = p.product_id
GROUP BY to_date(u.login_time), u.city;

-- 创建异步 MV（手动刷新）
CREATE MATERIALIZED VIEW mv_manual_refresh
REFRESH MANUAL
AS
SELECT ... ;

-- 创建带分区的异步 MV（推荐，支持增量刷新）
CREATE MATERIALIZED VIEW mv_partitioned
PARTITION BY (stat_date)
DISTRIBUTED BY HASH(stat_date) BUCKETS 8
REFRESH ASYNC EVERY (INTERVAL 1 HOUR)
AS
SELECT
    date_trunc('day', event_time) AS stat_date,
    category,
    COUNT(*) AS cnt,
    SUM(amount) AS total
FROM orders
GROUP BY date_trunc('day', event_time), category;
```

### 2.2 管理命令

```sql
-- 手动触发刷新
REFRESH MATERIALIZED VIEW mv_user_page_stats;

-- 查看 MV 状态
SHOW MATERIALIZED VIEWS WHERE NAME = 'mv_user_page_stats';
-- 关注字段: task_id, last_refresh_state, last_refresh_finished_time, rows

-- 查看 MV 刷新历史
SELECT * FROM information_schema.task_runs
WHERE task_name LIKE '%mv_user_page_stats%'
ORDER BY create_time DESC LIMIT 10;

-- 暂停/恢复自动刷新
ALTER MATERIALIZED VIEW mv_user_page_stats PAUSE;
ALTER MATERIALIZED VIEW mv_user_page_stats RESUME;

-- 修改刷新间隔
ALTER MATERIALIZED VIEW mv_user_page_stats
REFRESH ASYNC EVERY (INTERVAL 10 MINUTE);

-- 删除 MV
DROP MATERIALIZED VIEW mv_user_page_stats;
```

### 2.3 透明改写验证

```sql
-- 开启透明改写（默认开启）
SET enable_materialized_view_rewrite = true;

-- 执行原始查询
EXPLAIN SELECT
    to_date(login_time) AS stat_date,
    city,
    COUNT(DISTINCT user_id) AS uv
FROM user_behavior
JOIN product_info ON user_behavior.product_id = product_info.product_id
GROUP BY to_date(login_time), city;

-- 如果 EXPLAIN 输出中出现 "OlapScanNode: mv_user_page_stats"
-- 说明 CBO 成功将查询改写为读取 MV，而非原始表
-- 这就是透明改写的威力：应用层无需修改 SQL
```

## 三、数据导入方法详解

### 3.1 四种主要导入方式

```
┌──────────────────────────────────────────────────────────┐
│              StarRocks 数据导入方式全景                      │
│                                                          │
│  ┌─────────────────┐                                     │
│  │  Stream Load     │ ← HTTP PUT，小批量/测试/脚本         │
│  │  (同步, <10GB)   │                                     │
│  └─────────────────┘                                     │
│                                                          │
│  ┌─────────────────┐                                     │
│  │  Broker Load     │ ← HDFS/S3/OSS，大批量离线导入        │
│  │  (异步, TB级)     │                                     │
│  └─────────────────┘                                     │
│                                                          │
│  ┌─────────────────┐                                     │
│  │  Routine Load    │ ← Kafka 持续消费，实时流式导入         │
│  │  (持续运行)       │                                     │
│  └─────────────────┘                                     │
│                                                          │
│  ┌─────────────────┐                                     │
│  │  Flink Connector │ ← Flink 作业写入，精确一次语义        │
│  │  (Checkpoint)    │                                     │
│  └─────────────────┘                                     │
└──────────────────────────────────────────────────────────┘
```

### 3.2 Stream Load

**定位**：轻量级同步导入，适合小批量数据和调试。

```bash
# 基本用法
curl --location-trusted \
  -u root: \
  -H "label:stream_load_$(date +%s)" \
  -H "column_separator:," \
  -H "columns:event_time,user_id,page_url,action_type,duration_ms" \
  -T data.csv \
  http://fe_host:8030/api/demo_db/events_duplicate/_stream_load

# JSON 格式导入
curl --location-trusted \
  -u root: \
  -H "label:stream_load_json_$(date +%s)" \
  -H "format:json" \
  -H "strip_outer_array:true" \
  -T data.json \
  http://fe_host:8030/api/demo_db/events_duplicate/_stream_load
```

**关键参数**：
| 参数 | 说明 | 建议值 |
|------|------|--------|
| label | 导入标签，唯一标识一次导入 | 业务含义+时间戳 |
| max_batch_size | 最大批次大小 | 默认 10GB |
| timeout | 超时时间(秒) | 根据数据量调整 |
| merge_condition | Upsert 时的合并条件 | DELETE ON condition |

### 3.3 Broker Load

**定位**：大批量从分布式文件系统导入，适合离线 ETL。

```sql
LOAD LABEL demo_db.load_orders_20250601
(
    DATA INFILE("hdfs://namenode:8020/data/orders/2025-06-01/*.parquet")
    INTO TABLE fact_order
    FORMAT AS "parquet"
    (order_date, order_id, user_id, amount, status)
)
WITH BROKER "hdfs_broker"
PROPERTIES (
    "timeout" = "3600"
);

-- 查看导入进度
SHOW LOAD WHERE LABEL = "load_orders_20250601";
```

### 3.4 Routine Load（Kafka 消费）

**定位**：持续消费 Kafka Topic，实现近实时数据入仓。

```sql
CREATE ROUTINE LOAD rl_order_events ON fact_order
COLUMNS(order_date, order_id, user_id, amount, status),
COLUMNS TERMINATED BY ","
PROPERTIES (
    "desired_concurrent_number" = "3",          -- 并发消费线程数
    "max_batch_interval" = "20",                -- 最大攒批时间(秒)
    "max_batch_rows" = "300000",                -- 最大攒批行数
    "max_batch_size" = "209715200",             -- 最大攒批大小(200MB)
    "strict_mode" = "false"                     -- 宽松模式，脏数据不中断
)
FROM KAFKA (
    "kafka_broker_list" = "kafka1:9092,kafka2:9092,kafka3:9092",
    "kafka_topic" = "order_events",
    "property.group.id" = "sr_order_consumer",
    "property.auto.offset.reset" = "latest"
);

-- 查看消费状态
SHOW ROUTINE LOAD FOR rl_order_events\G
-- 关注: State, Progress, Lag, ErrorLogUrls

-- 暂停/恢复
PAUSE ROUTINE LOAD FOR rl_order_events;
RESUME ROUTINE LOAD FOR rl_order_events;
```

### 3.5 Flink Connector

**定位**：Flink 作业写入 StarRocks，支持 Checkpoint + Exactly-Once。

```java
// Flink Sink 配置示例
StarRocksSinkConfig config = StarRocksSinkConfig.builder()
    .withProperty("jdbc-url", "jdbc:mysql://fe_host:9030/demo_db")
    .withProperty("load-url", "fe_host:8030")
    .withProperty("username", "root")
    .withProperty("password", "")
    .withProperty("table-name", "fact_order")
    .withProperty("database-name", "demo_db")
    .withProperty("sink.buffer-flush.max-rows", "50000")
    .withProperty("sink.buffer-flush.interval", "10")      // 秒
    .withProperty("sink.properties.format", "json")
    .withProperty("sink.properties.strip_outer_array", "true")
    .build();

// 绑定到 DataStream
dataStream.sinkTo(StarRocksSink.sink(config));
```

## 四、导入方式选择决策矩阵

| 维度 | Stream Load | Broker Load | Routine Load | Flink Connector |
|------|------------|-------------|-------------|----------------|
| 数据来源 | 本地文件/API | HDFS/S3/OSS | Kafka | Flink DataStream |
| 数据量 | MB ~ 几GB | GB ~ TB | 持续流 | 持续流 |
| 时效性 | 手动触发 | 手动/调度触发 | 秒~分钟级 | 秒~分钟级 |
| 一致性 | At-Least-Once | At-Least-Once | At-Least-Once | Exactly-Once |
| 复杂度 | 低 | 中 | 中 | 高 |
| 适用阶段 | 开发/测试/小批量 | 离线批量导入 | 简单 Kafka 消费 | 复杂 ETL + 精确一次 |
| 错误处理 | 返回错误信息 | SHOW LOAD 查看 | Error Log URL | Flink Checkpoint |

### 选择决策树

```
需要导入数据到 StarRocks
  │
  ├── 一次性/测试/小批量？ → Stream Load
  │
  ├── 从 HDFS/S3 批量导入？ → Broker Load
  │
  ├── 从 Kafka 持续消费？
  │   ├── 不需要 Exactly-Once？ → Routine Load（更简单）
  │   └── 需要 Exactly-Once？ → Flink Connector
  │
  └── 已有 Flink ETL 作业？ → Flink Connector
```

## 五、面试话术要点

### Q: StarRocks 的异步物化视图和 ClickHouse 的物化视图有什么区别？

> 最大的区别有三点。第一，StarRocks 异步 MV 支持多表 JOIN，ClickHouse MV 只能基于单表。第二，StarRocks 支持查询透明改写，CBO 会自动识别并将查询路由到 MV，应用层不需要改 SQL；ClickHouse 必须手动查目标表。第三，StarRocks 用定时调度刷新，ClickHouse 是 INSERT 触发。代价是 StarRocks 的 MV 有刷新延迟（分钟级），不像 ClickHouse 那样近乎实时。在我们的架构中，单表实时预聚合仍然用 ClickHouse MV，多表 JOIN 加速用 StarRocks 异步 MV，各取所长。

### Q: 你们的数据是怎么导入 StarRocks 的？

> 我们有三种导入链路。离线 T+1 的大批量数据通过 Broker Load 从 HDFS 导入。实时 CDC 数据先写入 Kafka，然后通过 Flink Connector 写入 StarRocks Primary Key Model，利用 Flink Checkpoint 保证 Exactly-Once。少量运维数据和手工补数用 Stream Load。Routine Load 在早期用过，后来为了统一 ETL 框架和保证精确一次语义，全部迁移到了 Flink Connector。

### Q: Routine Load 和 Flink Connector 怎么选？

> 如果只是简单地把 Kafka 消息原样写入 StarRocks，不需要复杂转换，Routine Load 更简单，不需要额外维护 Flink 集群。但如果需要在写入前做 ETL 转换、多流 JOIN、或者要求 Exactly-Once 语义，就必须用 Flink Connector。我们的选择是 Flink Connector，因为我们的实时链路本身就有 Flink 做清洗和关联，直接在 Flink 里 Sink 到 StarRocks 是最自然的架构。
