## 一、Flink 窗口聚合结果批量写入 ClickHouse

### 1.1 完整 Flink Job 示例

```java
/**
 * Flink 窗口聚合 → ClickHouse 批量写入
 * 场景: 每 5 分钟统计各页面的 PV/UV，写入 CH 预聚合表
 */
public class WindowAggToClickHouse {

    public static void main(String[] args) throws Exception {
        StreamExecutionEnvironment env = StreamExecutionEnvironment.getExecutionEnvironment();

        // Checkpoint 配置
        env.enableCheckpointing(30_000);  // 30秒一次 Checkpoint
        env.getCheckpointConfig().setMinPauseBetweenCheckpoints(10_000);
        env.getCheckpointConfig().setCheckpointTimeout(60_000);

        // Kafka Source
        KafkaSource<String> source = KafkaSource.<String>builder()
            .setBootstrapServers("kafka1:9092,kafka2:9092")
            .setTopics("user_behavior")
            .setGroupId("flink-ch-sink-group")
            .setStartingOffsets(OffsetsInitializer.latest())
            .build();

        DataStream<PageMetric> metrics = env.fromSource(source, WatermarkStrategy.noWatermarks(), "Kafka Source")
            .map(new BehaviorParser())           // JSON → PageMetric POJO
            .assignTimestampsAndWatermarks(
                WatermarkStrategy.<PageMetric>forBoundedOutOfOrderness(Duration.ofSeconds(10))
                    .withTimestampAssigner((m, ts) -> m.getEventTime())
            )
            .keyBy(PageMetric::getPageId)
            .window(TumblingEventTimeWindows.of(Time.minutes(5)))
            .aggregate(new PageMetricAggregator()); // 增量聚合

        // ClickHouse Sink (攒批写入)
        ClickHouseSinkConfig sinkConfig = ClickHouseSinkConfig.builder()
            .withUrl("http://ch-node1:8123")
            .withDatabase("analytics")
            .withTable("page_metrics_5min")
            .withUsername("writer")
            .withPassword("xxx")
            .withBatchSize(50_000)          // 5万行攒批
            .withFlushInterval(10_000)      // 10秒超时flush
            .withMaxRetries(3)
            .build();

        metrics.sinkTo(new ClickHouseBatchSink<>(sinkConfig));

        env.execute("Window Agg to ClickHouse");
    }
}
```

### 1.2 ClickHouse 目标表设计

```sql
-- 接收 Flink 窗口聚合结果的表（Flink 已做聚合，用 ReplacingMergeTree 幂等去重）
CREATE TABLE page_metrics_5min ON CLUSTER '{cluster}'
(
    window_start  DateTime NOT NULL,
    window_end    DateTime NOT NULL,
    page_id       UInt32 NOT NULL,
    pv            UInt64,
    uv            UInt64,           -- Flink 窗口内已去重计数
    avg_duration  Float64,
    checkpoint_id UInt64,           -- 用于幂等去重
    write_time    DateTime DEFAULT now()
)
ENGINE = ReplacingMergeTree(checkpoint_id)
PARTITION BY toYYYYMMDD(window_start)
ORDER BY (window_start, page_id);
```

### 1.3 自定义攒批 Sink 核心逻辑

```java
/**
 * 异步攒批 Sink 核心实现（简化版）
 */
public class ClickHouseBatchSink<T> implements SinkFunction<T>, CheckpointedFunction {

    private transient List<T> buffer;
    private transient ScheduledExecutorService scheduler;
    private transient Object lock = new Object();

    private final int batchSize;
    private final long flushIntervalMs;
    private long lastFlushTime;

    @Override
    public void invoke(T value, Context context) throws Exception {
        synchronized (lock) {
            buffer.add(value);
            if (buffer.size() >= batchSize) {
                doFlush();
            }
        }
    }

    @Override
    public void open(Configuration parameters) throws Exception {
        buffer = new ArrayList<>(batchSize);
        lastFlushTime = System.currentTimeMillis();

        // 定时 flush，防止低流量时数据滞留
        scheduler = Executors.newSingleThreadScheduledExecutor();
        scheduler.scheduleAtFixedRate(() -> {
            synchronized (lock) {
                if (!buffer.isEmpty() &&
                    System.currentTimeMillis() - lastFlushTime >= flushIntervalMs) {
                    try { doFlush(); } catch (Exception e) { LOG.error("Flush failed", e); }
                }
            }
        }, flushIntervalMs, flushIntervalMs, TimeUnit.MILLISECONDS);
    }

    private void doFlush() throws Exception {
        if (buffer.isEmpty()) return;

        List<T> batch = new ArrayList<>(buffer);
        buffer.clear();
        lastFlushTime = System.currentTimeMillis();

        // 序列化为 CSV/JSON
        String body = serialize(batch);

        // HTTP Stream Load 写入 ClickHouse
        HttpPut put = new HttpPut(url + "/?query=" + URLEncoder.encode(insertSQL));
        put.setEntity(new StringEntity(body));
        put.setHeader("Content-Type", "text/plain");

        HttpResponse resp = httpClient.execute(put);
        int statusCode = resp.getStatusLine().getStatusCode();
        if (statusCode != 200) {
            throw new RuntimeException("CH write failed: " + statusCode);
        }

        // Metrics
        metricGroup.counter("rows_written").inc(batch.size());
        metricGroup.gauge("batch_size", () -> batch.size());
    }

    @Override
    public void snapshotState(FunctionSnapshotContext ctx) throws Exception {
        // Checkpoint 时强制 flush，保证数据不丢
        synchronized (lock) { doFlush(); }
    }

    @Override
    public void initializeState(FunctionInitializationContext ctx) throws Exception {
        // 状态初始化（如有需要）
    }
}
```

## 三、自研异步攒批 Sink 面试话术

### 3.1 项目背景阐述

> **面试官**: 你提到自研了异步攒批 Sink，能详细讲讲吗？

> **回答**: 好的。我们早期的实时链路是 Flink → ClickHouse，用的开源 Connector。随着业务增长遇到了几个痛点：一是开源 Connector 的攒批策略是固定的，无法根据下游负载动态调整；二是错误处理粗糙，网络抖动时整批丢弃；三是缺少细粒度监控，出了问题很难定位。所以我们决定自研一个通用的异步攒批 Sink 框架，同时支持 ClickHouse 和 StarRocks。

### 3.2 核心技术点

> **面试官**: 自研 Sink 的核心设计是什么？

> **回答**: 有三个核心设计点。
>
> **第一，自适应攒批**。传统做法是固定 batch size 和 flush interval，但实际流量波动很大。我们实现了一个基于反馈的自适应算法：监控最近 N 次 flush 的平均 RT，如果 RT 上升说明下游压力大，自动增大 batch size 减少请求频率；如果 RT 下降且 Buffer 经常满，减小 batch size 降低延迟。这样在高峰期保吞吐，低谷期保时效。
>
> **第二，分级重试与容错**。把异常分为三类：可重试（网络超时、连接断开）、需降级（限流、Part 过多）、不可恢复（Schema 不匹配、认证失败）。可重试走指数退避重试；需降级时自动增大 batch size 并降速；不可恢复直接告警并路由到死信队列。这比开源方案的一刀切重试精细很多。
>
> **第三，完善的可观测性**。暴露了十几个 Metrics：写入速率、Buffer 水位、Flush 耗时分布、Batch 大小分布、错误分类计数、重试次数等。对接 Prometheus + Grafana，做到写入链路的白盒化监控。

### 3.3 效果量化

> **面试官**: 上线后效果怎么样？

> **回答**: 三个核心指标都有显著提升。写入成功率从 99.5% 提升到 99.99%，主要得益于分级重试避免了不必要的批量丢弃。P99 写入延迟从 25 秒降到 12 秒，自适应攒批在低谷期能自动缩小批次。运维效率提升最大，以前写入问题排查要半天，现在通过 Grafana Dashboard 5 分钟就能定位原因。整个 Sink 框架后来还被其他团队复用，接入了 Doris 和 Elasticsearch 的写入。

### 3.4 防御性问题准备

**Q: 为什么不直接用官方 Flink Connector？**
> 官方 Connector 解决了基本功能，但在生产级场景中不够用。比如它不支持自适应攒批、错误处理粒度粗、Metrics 不够细。我们的场景要求 99.99% 的写入成功率和秒级的故障定位能力，这些都需要深度定制。当然如果团队规模小、写入量不大，官方 Connector 完全够用，没必要重复造轮子。

**Q: 自适应攒批会不会引入额外复杂度导致 bug？**
> 确实有这个风险。我们的应对方式是：首先设好上下界，batch size 只在 [min, max] 范围内调整，不会无限变化；其次有兜底机制，如果自适应算法异常会回退到固定参数；最后充分测试，用混沌工程注入各种异常验证稳定性。上线半年没有因为自适应逻辑出过故障。

**Q: Checkpoint 时怎么处理 Buffer 中的数据？**
> Checkpoint 触发时在 snapshotState 方法中强制 flush Buffer，确保所有已接收的数据都写入目标存储。这是保证 Exactly-Once 的关键：Checkpoint 之前的数据一定已经持久化，恢复后从 Checkpoint 位点重新消费不会产生丢失。代价是 Checkpoint 耗时会增加一个 flush 的时间，我们通过控制 Buffer 上限来限制这个影响。

## 四、常见问题排查手册

### 4.1 ClickHouse 写入问题

| 症状 | 可能原因 | 排查命令 | 解决方案 |
|------|---------|---------|---------|
| Too many parts | 写入太频繁 | `SELECT count() FROM system.parts WHERE table='xxx' AND active` | 增大 batch size |
| Memory limit exceeded | 单次 INSERT 太大 | 查看 CH server log | 减小 batch size 或增大内存限制 |
| Timeout | 网络/CH 过载 | curl 测试连通性 | 检查网络和 CH 负载 |
| 数据重复 | Flink 重放 | 检查 checkpoint 配置 | 加幂等键 + ReplacingMT |
| 数据丢失 | Sink 未 flush | 检查 Checkpoint 是否成功 | 确认 snapshotState 中有 flush |

### 4.2 StarRocks 写入问题

| 症状 | 可能原因 | 排查命令 | 解决方案 |
|------|---------|---------|---------|
| Transaction timeout | Checkpoint 间隔太长 | SHOW TRANSACTION | 缩短 Checkpoint 间隔或增大事务超时 |
| Tablet writer write failed | BE 磁盘满/Compaction 积压 | SHOW BACKENDS | 清理磁盘/等待 Compaction |
| Label already exists | 重复提交 | SHOW LOAD | 正常现象(幂等)，检查是否误报 |
| Stream load error | 数据格式错误 | 查看 Error Log URL | 修正数据格式或增加容错 |
| PK Model 写入慢 | 主键过大/索引未持久化 | SHOW TABLET | 开启 persistent_index |

### 4.3 通用排查流程

```
写入异常排查 SOP:

Step 1: 检查 Flink Job 状态
  → Web UI 看是否有 Restart / Backpressure
  → Checkpoint 是否成功

Step 2: 检查 Sink Metrics
  → rows_written 是否稳定
  → error_count 是否突增
  → flush_latency 是否飙升

Step 3: 检查目标引擎
  → CH: system.parts / system.metrics
  → SR: SHOW BACKENDS / SHOW PROC '/compactions'

Step 4: 检查中间件
  → Kafka Lag
  → 网络连通性
  → DNS 解析

Step 5: 查看日志
  → Flink TaskManager log
  → CH/SR server log
  → 错误信息关键字搜索
```
