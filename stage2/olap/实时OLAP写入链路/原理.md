## 一、Flink → ClickHouse 写入链路

### 1.2 ClickHouse Sink 的攒批机制

ClickHouse 不适合高频小批量写入，必须攒批：

```
❌ 错误做法: 每条记录一次 INSERT
   → 每次 INSERT 产生一个 Data Part
   → 每秒 1000 次 INSERT = 每秒 1000 个 Part
   → Part 过多 → Merge 跟不上 → Too many parts 报错

✅ 正确做法: 攒批后批量写入
   → Buffer 积累到 N 行或 T 秒后 flush
   → 每次 flush 产生一个 Part
   → 每秒 1-5 次 flush = 可控的 Part 增长速率
```

**关键参数**：

| 参数 | 说明 | 推荐值 | 影响 |
|------|------|--------|------|
| batch_size / max_rows | 攒批行数上限 | 5万~50万 | 越大吞吐越高，延迟越大 |
| flush_interval | 攒批时间上限(秒) | 5~30s | 保证低流量时也能及时 flush |
| max_retries | 失败重试次数 | 3~5 | 网络抖动容错 |
| retry_backoff | 重试退避(ms) | 1000~5000 | 避免雪崩 |

### 1.3 幂等性保障

由于 Flink Checkpoint 恢复后可能重复发送数据，ClickHouse 侧需要幂等处理：

```sql
-- 方案1: ReplacingMergeTree + version
CREATE TABLE agg_metrics_idempotent
(
    window_start  DateTime,
    metric_key    String,
    metric_value  Float64,
    version       UInt64        -- Flink 的 checkpoint_id 或 event_time
)
ENGINE = ReplacingMergeTree(version)
ORDER BY (window_start, metric_key);

-- 查询时必须用 argMax 或 FINAL
SELECT window_start, metric_key,
       argMax(metric_value, version) AS latest_value
FROM agg_metrics_idempotent
GROUP BY window_start, metric_key;

-- 方案2: CollapsingMergeTree（适合增量更新）
CREATE TABLE agg_metrics_collapsing
(
    window_start  DateTime,
    metric_key    String,
    metric_value  Float64,
    sign          Int8          -- +1 表示新增，-1 表示撤销
)
ENGINE = CollapsingMergeTree(sign)
ORDER BY (window_start, metric_key);

-- Flink 写入时：旧值发 sign=-1，新值发 sign=+1
-- Merge 后自动抵消
```

⚠️ **两种方案的局限**：
- ReplacingMergeTree: 查询必须加 FINAL 或用 argMax，性能开销大
- CollapsingMergeTree: 要求严格按顺序写入，乱序会导致数据错误
- 两者都依赖后台 Merge，存在"中间态不一致"窗口

### 1.4 频繁 Merge 问题

```
高频小批量写入的典型症状：

Time 0:00  Part_1 (100 rows)
Time 0:01  Part_2 (100 rows)
Time 0:02  Part_3 (100 rows)
...
Time 0:59  Part_60 (100 rows)

→ 1 分钟内产生 60 个小 Part
→ 后台 Merge 线程开始合并
→ 如果写入速度 > Merge 速度 → Part 持续累积
→ system.parts 中 active parts > 300 → 告警
→ active parts > 1000 → "Too many parts" 拒绝写入

解决方案层级：
L1: 增大 batch_size，减少 Part 产生频率
L2: 调整 merge_tree 参数（parts_to_throw_insert 等）
L3: 使用 Buffer 引擎做前置缓冲
L4: 改用 StarRocks（Compaction 机制更优）
```

```
│  ✅ 无额外依赖                                          │
└────────────────────────────────────────────────────────┘
```

### 2.2 Flink Connector 的 Exactly-Once 实现

```
Checkpoint 与 Stream Load 的两阶段提交：

Phase 1 (Pre-Commit):
  1. Flink 触发 Checkpoint
  2. Sink 将 Buffer 中的数据通过 Stream Load 发送到 BE
  3. Stream Load 使用 label = "txn_{checkpoint_id}" 标识
  4. BE 接收数据但不 commit（数据可见但未确认）
  5. Sink 返回 pre-commit 成功

Phase 2 (Commit):
  6. 所有算子 Checkpoint 完成
  7. JobManager 通知 Sink commit
  8. Sink 调用 FE 的 commit transaction API
  9. FE 标记事务为 committed，数据正式可见

故障恢复:
  - Checkpoint 未完成就崩溃 → 回滚未 commit 的事务
  - 恢复后从上一个 Checkpoint 重新消费
  - 重复的数据因为相同 label 被去重 → Exactly-Once
```

## 三、Batch Size vs Latency vs Write Amplification 权衡

### 3.1 三角权衡模型

```
              Batch Size ↑
                 /    \
                /      \
    Throughput ↑        ↓ Latency
               \        /
                \      /
         Write Amplification ↑

• Batch Size ↑ → Throughput ↑（单位时间内更多有效数据）
• Batch Size ↑ → Latency ↑（数据在 Buffer 中等待更久）
• Batch Size ↓ → Write Amplification ↑（更多小 Part/Rowset）
• Batch Size ↑ → Write Amplification ↓（更少更大的 Part/Rowset）
```

### 3.2 量化关系参考

```
测试环境: 单节点, 16C 64GB NVMe SSD
数据: 每行 ~200 bytes JSON

Batch Size    Flush Interval   写入吞吐(rows/s)   P99延迟    Part/hour
─────────────────────────────────────────────────────────────────────
1,000         1s               50,000             ~1s        3,600 ⚠️
10,000        5s               200,000            ~5s        720
50,000        10s              500,000            ~10s       360 ✅
100,000       15s              800,000            ~15s       240 ✅
500,000       30s              1,200,000          ~30s       120 ✅✅
1,000,000     60s              1,500,000          ~60s       60 ✅✅✅

观察:
• 吞吐随 Batch Size 增长但边际递减（>50万后收益变小）
• Part 数量与 Batch Size 近似反比
• P99 延迟 ≈ Flush Interval（最坏情况就是等到超时才 flush）
• 最佳平衡点通常在 5万~50万行 / 10~15秒
```

### 3.3 不同场景的参数建议

| 场景 | Batch Rows | Flush Interval | 优先级 |
|------|-----------|----------------|--------|
| 实时 Dashboard | 10,000 | 5s | 延迟优先 |
| 实时数仓入仓 | 50,000 | 10s | 均衡 |
| 离线补数/回放 | 500,000 | 30s | 吞吐优先 |
| CDC 同步 | 20,000 | 5s | 延迟+幂等 |

## 四、Exactly-Once 语义实现对比

### 4.1 三种实现路径

```
┌─────────────────────────────────────────────────────────┐
│           Exactly-Once 实现方式对比                        │
│                                                         │
│  方式1: 幂等键 + ReplacingMergeTree (CH)                  │
│  ┌────────────────────────────────────────────┐         │
│  │ Flink At-Least-Once 写入                    │         │
│  │  ↓                                          │         │
│  │ CH ReplacingMergeTree(batch_id as version)   │         │
│  │  ↓ Merge                                    │         │
│  │ 相同 batch_id 只保留最新版本                   │         │
│  └────────────────────────────────────────────┘         │
│  优点: 实现简单                                          │
│  缺点: 查询需 FINAL/argMax; Merge 前数据不一致             │
│                                                         │
│  方式2: 两阶段提交 (StarRocks Flink Connector)            │
│  ┌────────────────────────────────────────────┐         │
│  │ Flink Checkpoint 触发 Pre-Commit            │         │
│  │  ↓                                          │         │
│  │ Stream Load with Label (事务性写入)           │         │
│  │  ↓                                          │         │
│  │ Checkpoint 完成后 Commit                     │         │
│  └────────────────────────────────────────────┘         │
│  优点: 真正的 Exactly-Once; 查询强一致                    │
│  缺点: 事务开销; Checkpoint 间隔影响延迟                   │
│                                                         │
│  方式3: 自研异步攒批 Sink + 幂等键                         │
│  ┌────────────────────────────────────────────┐         │
│  │ 自研 Sink 内部攒批                            │         │
│  │  ↓                                          │         │
│  │ 按 window_end + key 生成幂等 ID               │         │
│  │  ↓                                          │         │
│  │ 批量 Upsert (PK Model / ReplacingMT)         │         │
│  └────────────────────────────────────────────┘         │
│  优点: 完全可控; 可优化到极致                              │
│  缺点: 开发维护成本高                                     │
└─────────────────────────────────────────────────────────┘
```

### 4.2 选择建议

| 需求 | 推荐方案 | 理由 |
|------|---------|------|
| 快速上线，容忍最终一致 | CH + ReplacingMergeTree | 最简单 |
| 严格 Exactly-Once | SR + Flink Connector | 开箱即用 |
| 极致性能 + 定制逻辑 | 自研 Sink | 完全掌控 |
| 多目标写入 | Flink Side Output + 各自 Connector | 解耦 |

## 五、面试话术要点

### Q: Flink 写 ClickHouse 有什么注意事项？

> 最重要的三点。第一，必须攒批，不能逐条写入，否则 Part 爆炸导致 "Too many parts" 报错。我们通常设 batch_size=10万行、flush_interval=10秒。第二，要做幂等处理，Flink 故障恢复会重放数据，我们用 ReplacingMergeTree + checkpoint_id 作为 version 列来去重。第三，注意 Merge 压力，高频写入时要监控 system.parts 表的 active part 数量，超过 300 就要考虑增大 batch size 或者降频。

### Q: Flink 写 StarRocks 怎么保证 Exactly-Once？

> 用官方 Flink Connector，它基于两阶段提交实现。Checkpoint 触发时先 pre-commit（Stream Load 带 label），Checkpoint 成功后再 commit。如果中途失败，未 commit 的事务会被回滚，恢复后用相同 label 重新写入会自动去重。这样保证了端到端的 Exactly-Once。代价是 Checkpoint 间隔会影响数据可见延迟，我们设的 30 秒 Checkpoint，加上攒批时间，端到端延迟约 40 秒。

### Q: 你们有没有自研过 Sink？为什么？

> 有。我们在早期用过开源的 ClickHouse Flink Connector，但遇到了几个问题：一是攒批策略不够灵活，无法根据下游负载动态调整；二是错误处理太粗糙，部分失败会丢整批数据；三是缺少写入指标暴露，不好做监控告警。所以我们自研了异步攒批 Sink，核心改进包括：自适应攒批（根据 RT 动态调整 batch size）、分级重试（区分可重试和不可重试异常）、完善的 Metrics 暴露（写入速率/延迟/错误率/Part 大小）。自研 Sink 上线后写入稳定性提升了 80%，P99 延迟降低了 40%。

### Q: 高频小批量写入 ClickHouse 会有什么问题？

> 核心问题是 Part 爆炸。ClickHouse 每次 INSERT 都会产生一个新的 Data Part，后台 Merge 线程负责将小 Part 合并成大 Part。如果写入频率太高（比如每秒几十次小 INSERT），Part 产生速度超过 Merge 速度，就会不断累积。当 active part 超过阈值（默认 300）时会报警，超过 1000 时直接拒绝写入。解决方法首先是增大攒批粒度，其次是调整 MergeTree 参数放宽限制，最后如果业务确实需要高频小批量写入，应该考虑换 StarRocks，它的 Compaction 机制对这种场景更友好。
