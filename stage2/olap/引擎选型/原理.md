# OLAP 引擎选型原理

## 一、三大引擎架构总览

### 1.1 ClickHouse：单表极致性能派

```
┌─────────────────────────────────────────────────┐
│              ClickHouse 架构                      │
│                                                  │
│   Client                                         │
│     │                                            │
│     ▼                                            │
│   ┌──────────┐    ┌──────────┐    ┌──────────┐  │
│   │ Shard 1  │    │ Shard 2  │    │ Shard 3  │  │
│   │ Replica A│    │ Replica A│    │ Replica A│  │
│   │ Replica B│    │ Replica B│    │ Replica B│  │
│   └──────────┘    └──────────┘    └──────────┘  │
│                                                  │
│   特点：                                          │
│   • 每个 Shard 独立执行，结果在客户端/Coordinator合并 │
│   • 无全局优化器，JOIN 靠手动优化                    │
│   • 向量化执行 + 稀疏索引 + 列存储 = 单表扫描极快      │
│   • MergeTree 家族提供丰富存储引擎                   │
└─────────────────────────────────────────────────┘
```

**核心优势**：单表宽表查询性能天花板极高，单机数十亿行秒级扫描。
**核心短板**：多表 JOIN 能力弱，缺乏 CBO，分布式 JOIN 需要大量手工调优。

### 1.2 Apache Doris：统一分析平台

```
┌─────────────────────────────────────────────────┐
│              Doris 架构                           │
│                                                  │
│   Client (MySQL Protocol)                        │
│     │                                            │
│     ▼                                            │
│   ┌──────┐                                       │
│   │ FE   │ ← Master/Follower/Observer           │
│   │ CBO  │    (元数据 + 查询规划 + MPP 调度)        │
│   └──┬───┘                                       │
│      │                                           │
│   ┌──▼──┐  ┌─────┐  ┌─────┐                     │
│   │BE 1 │  │BE 2 │  │BE 3 │ ← 存算一体           │
│   │Storage│ │Storage│ │Storage│                  │
│   │+Exec │  │+Exec │  │+Exec │                   │
│   └─────┘  └─────┘  └─────┘                     │
│                                                  │
│   特点：                                          │
│   • MySQL 协议兼容，生态友好                         │
│   • CBO + MPP，多表 JOIN 能力强                    │
│   • Unique Key Model 支持实时 Upsert               │
│   • 运维简单，FE 自带高可用                          │
└─────────────────────────────────────────────────┘
```

### 1.3 StarRocks：极速全场景 MPP

```
┌─────────────────────────────────────────────────┐
│              StarRocks 架构                       │
│                                                  │
│   Client (MySQL Protocol)                        │
│     │                                            │
│     ▼                                            │
│   ┌──────┐                                       │
│   │ FE   │ ← Leader / Follower / Observer         │
│   │ CBO  │    (全面CBO + Pipeline执行引擎)          │
│   └──┬───┘                                       │
│      │                                           │
│   ┌──▼──┐  ┌─────┐  ┌─────┐                     │
│   │BE 1 │  │BE 2 │  │BE 3 │ ← 存算一体           │
│   │Vector│  │Vector│  │Vector│ ← 全面向量化        │
│   │Exec  │  │Exec  │  │Exec  │                   │
│   └─────┘  └─────┘  └─────┘                     │
│                                                  │
│   特点：                                          │
│   • Pipeline 执行模型，充分利用多核                   │
│   • 全面 CBO（含 Runtime Filter、Colocate Join）    │
│   • Primary Key Model 支持实时 Upsert/Delete       │
│   • 异步物化视图支持多表 JOIN 预聚合                  │
│   • 向量化引擎覆盖全部算子                           │
└─────────────────────────────────────────────────┘
```

## 二、核心维度深度对比

### 2.1 JOIN 能力对比

| 能力 | ClickHouse | Doris | StarRocks |
|------|-----------|-------|-----------|
| JOIN 算法 | Hash Join（基础） | Shuffle/Broadcast/Colocate | Shuffle/Broadcast/Colocate/Bucket Shuffle |
| CBO | ❌ 无（仅 RBO） | ✅ 有（Nereids） | ✅ 有（Cascades 框架） |
| Runtime Filter | ❌ 无 | ✅ Bloom Filter | ✅ Bloom/In/MinMax |
| Colocate Join | ❌ 不支持 | ✅ 支持 | ✅ 支持 |
| 大表 JOIN 大表 | 极慢，需手动优化 | 良好 | 优秀 |
| 多表 JOIN（5+表） | 基本不可用 | 可用 | 高效 |
| JOIN 与聚合混合 | 差 | 好 | 优秀 |

**ClickHouse JOIN 痛点详解**：

```sql
-- ClickHouse: 默认分布式 JOIN 只在各 Shard 本地匹配
-- 若 sharding_key 不一致，会丢匹配；大表需 GLOBAL JOIN（广播右表）或字典/宽表SELECT a.*, b.name
FROM huge_table_a AS a
GLOBAL JOIN huge_table_b AS b ON a.id = b.id;  -- 右表广播，右表过大易 OOM
-- 更稳妥：小维表用 dictGet，或 ETL 打宽表

-- StarRocks/Doris: CBO 自动选择最优 JOIN 策略
-- 大表 JOIN 大表 → Shuffle Join
-- 大表 JOIN 小表 → Broadcast Join
-- 同分布键表 → Colocate Join（零 Shuffle）
```

### 2.2 实时更新能力对比

| 能力 | ClickHouse | Doris | StarRocks |
|------|-----------|-------|-----------|
| UPDATE/DELETE | CollapsingMergeTree / ReplacingMergeTree | Unique Key Model | Primary Key Model |
| 实现方式 | 标记删除 + 后台 Merge | Merge-on-Read/Write | Delete Vector + Merge-on-Write |
| 更新延迟 | 分钟级（等 Merge） | 秒级 | 毫秒~秒级 |
| 部分列更新 | ❌ 不支持 | ✅ 支持 | ✅ 支持 |
| DELETE 语句 | 有限（ALTER DELETE，重量级） | ✅ 支持 | ✅ 支持 |
| 高频小批量写入 | Part 过多，Merge 压力大 | 较好 | 较好（Compaction 优化） |
| CDC 对接复杂度 | 高（需处理 sign/version） | 中 | 低（原生 Upsert） |

**ClickHouse 更新的本质问题**：

```
ClickHouse 的 "更新" 其实是 "追加新版本 + 后台去重"

ReplacingMergeTree:
  INSERT (id=1, value='A', version=1)
  INSERT (id=1, value='B', version=2)
  → 两个版本共存直到后台 Merge
  → 查询时必须加 FINAL 或 argMax 才能读到最新值
  → FINAL 性能开销巨大

StarRocks Primary Key Model:
  UPSERT (id=1, value='A')
  UPSERT (id=1, value='B')
  → 立即覆盖，下次查询一定读到 'B'
  → 无需 FINAL，无额外查询开销
```

### 2.3 并发能力对比

| 指标 | ClickHouse | Doris | StarRocks |
|------|-----------|-------|-----------|
| 推荐并发 QPS | 10-50 | 100-500 | 200-1000+ |
| 连接管理 | 粗糙（无连接池内置） | MySQL 协议 + 连接池 | MySQL 协议 + 连接池 |
| 资源隔离 | 弱（Quota 粗粒度） | 中（Resource Group） | 强（Resource Group + Workload Group） |
| 短查询优化 | 弱 | 中 | 强（Short Query Optimization） |
| 缓存机制 | 文件系统缓存 | Page Cache + Query Cache | Page Cache + Query Cache + Data Cache |

**为什么 ClickHouse 不适合高并发**：
1. 每个查询占用大量 CPU/内存，不做精细的资源管控
2. 没有 Short Query Path，简单查询也要走完整执行链路
3. 连接管理原始，通常需要外部 Proxy（如 CHProxy）
4. 设计哲学是"吞吞吐量大"而非"服务请求数多"

### 2.4 运维复杂度对比

| 维度 | ClickHouse | Doris | StarRocks |
|------|-----------|-------|-----------|
| 部署复杂度 | 高（ZK + 分片 + 副本手动配） | 中（FE/BE 两类角色） | 中（FE/BE 两类角色） |
| 扩缩容 | 痛苦（需手动 rebalance） | 较易（自动均衡） | 较易（自动均衡） |
| Schema 变更 | 重（Mutation，异步且耗资源） | 轻（Light Schema Change） | 轻（Light Schema Change） |
| 监控告警 | 需自建（system tables） | 内置 Manager | 内置 Manager + Prometheus |
| SQL 兼容性 | ClickHouse SQL（非标） | MySQL 兼容 | MySQL 兼容 |
| 社区/文档 | 活跃但分散 | 中文社区强 | 中英文都活跃 |
| 升级风险 | 较高（版本间行为变化大） | 较低 | 较低 |

## 三、明细 vs 聚合场景适配

### 3.1 明细查询场景

需求特征：按条件过滤查看原始记录、日志检索、Trace 分析

| 引擎 | 适用度 | 说明 |
|------|--------|------|
| ClickHouse | ⭐⭐⭐⭐⭐ | 单表扫描极快，ORDER BY + Skip Index 加速过滤 |
| Doris | ⭐⭐⭐⭐ | 良好，但不如 CH 极致 |
| StarRocks | ⭐⭐⭐⭐ | 良好，Pipeline 执行在高并发下优势明显 |

### 3.2 聚合分析场景

需求特征：GROUP BY + 多维分析、Dashboard 报表

| 引擎 | 适用度 | 说明 |
|------|--------|------|
| ClickHouse | ⭐⭐⭐⭐⭐ | 单表预聚合（MV）极致性能 |
| Doris | ⭐⭐⭐⭐ | MPP 并行聚合 + Rollup |
| StarRocks | ⭐⭐⭐⭐⭐ | MPP + 异步 MV + Pipeline |

### 3.3 多表关联分析场景

需求特征：星型/雪花模型、维表关联、跨域分析

| 引擎 | 适用度 | 说明 |
|------|--------|------|
| ClickHouse | ⭐⭐ | JOIN 是最大短板 |
| Doris | ⭐⭐⭐⭐ | CBO + Colocate Join |
| StarRocks | ⭐⭐⭐⭐⭐ | 最强 JOIN 能力 + 异步 MV |

## 四、面试话术要点

### Q: ClickHouse、Doris、StarRocks 怎么选？

> 选型要看具体场景。如果是固定维度的宽表分析、日志扫描、写入吞吐极高的场景，ClickHouse 单表性能无可匹敌。如果需要多表 JOIN、高并发点查、实时 Upsert，StarRocks 和 Doris 更合适，它们有完整的 CBO 和 MPP 能力。StarRocks 在多表 JOIN 性能和异步物化视图方面略胜一筹，Doris 在中文社区生态和运维工具链上更成熟。实际项目中我们采用的是"CH 做宽表明细层 + StarRocks 做多表分析和实时应用层"的分层架构。

### Q: 为什么不能只用一个引擎？

> 因为没有一个引擎在所有维度都是最优的。ClickHouse 的单表扫描和预聚合性能是最强的，但 JOIN 和高并发是硬伤。StarRocks 全能但在极端单表扫描场景下不如 CH。我们的做法是让每个引擎做自己最擅长的事：CH 负责海量明细数据的存储和预聚合，StarRocks 负责面向应用的交互式分析和实时看板。通过 ETL 链路把两者串起来，既保证了性能又保证了灵活性。

### Q: Doris 和 StarRocks 有什么区别？

> StarRocks 是从 Doris fork 出来的，底层架构相似但发展路线不同。StarRocks 在 CBO 优化器（Cascades 框架）、Pipeline 执行引擎、Primary Key Model 的实时更新性能、以及异步物化视图等方面投入更多，整体查询性能尤其是多表 JOIN 场景更强。Doris 则在 Unique Key Model、Multi-Catalog 数据湖集成、以及中文社区运营方面更有优势。如果团队对 JOIN 性能和实时更新要求高，倾向 StarRocks；如果重视社区支持和数据湖生态，倾向 Doris。
