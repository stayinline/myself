## 二、典型场景分析与选型决策

### 2.1 场景 A：宽表固定报表 → ClickHouse

**场景特征**：
- 数据已提前 ETL 打成大宽表（50+ 列）
- 查询模式固定，主要是 GROUP BY + 过滤
- 日增数据量千万级，总量百亿级
- 不需要 JOIN，不需要 UPDATE
- Dashboard QPS < 20

**为什么选 ClickHouse**：
```
✅ 宽表扫描是 CH 的绝对强项
✅ MV 预聚合可以将查询从秒级降到毫秒级
✅ 写入吞吐极高，轻松应对千万级/天
✅ 压缩比优秀（LZ4/ZSTD），存储成本低
❌ 但如果未来需要 JOIN 或高并发，会陷入困境
```

**架构示意**：
```
Kafka → Flink ETL → ClickHouse 宽表 → MV 预聚合 → Grafana/Superset
```

### 2.2 场景 B：多表 JOIN + 高并发 + 实时更新 → StarRocks/Doris

**场景特征**：
- 星型模型：事实表 + 多个维表
- 需要实时 JOIN 查询
- 面向 C 端/B 端用户，QPS > 100
- 部分维度需要实时更新（如用户信息变更）
- 需要 DELETE 能力（GDPR/数据修正）

**为什么选 StarRocks**：
```
✅ CBO 自动选择最优 JOIN 策略，无需手动优化
✅ Colocate Join 避免 Shuffle，JOIN 性能极好
✅ Primary Key Model 支持毫秒级 Upsert/Delete
✅ Pipeline 执行引擎在高并发下表现优异
✅ MySQL 协议兼容，BI 工具无缝对接
✅ 异步物化视图可以加速高频 JOIN 查询
```

**架构示意**：
```
CDC ──────────→ StarRocks (Primary Key Model)
                ↕
Kafka → Flink → StarRocks (Duplicate Model 事实表)
                ↓
           Async MV (JOIN 预聚合)
                ↓
         API Server / BI Tool
```

### 2.3 场景 C：日志分析 → ClickHouse

**场景特征**：
- 海量日志（TB/天），只追加不更新
- 查询以时间范围过滤 + 关键词搜索为主
- 保留周期短（7-30 天）
- 偶尔做 Ad-hoc 分析

**为什么选 ClickHouse**：
```
✅ 写入吞吐无敌，单机可达数百 MB/s
✅ TTL 自动过期，无需手动清理
✅ 稀疏索引 + Skip Index 加速时间范围查询
✅ 压缩比极高，日志存储成本最低
❌ 如果需要全文检索，考虑 Elasticsearch 补充
```

### 2.4 场景 D：湖仓一体 → Doris/StarRocks

**场景特征**：
- 已有 Iceberg/Hudi/Paimon 数据湖
- 需要在 OLAP 引擎中直接查询湖表
- 冷热分层，热数据在 OLAP，冷数据在湖中

**推荐**：Doris（Multi-Catalog 成熟度高）或 StarRocks（External Catalog 性能好）

## 三、ClickHouse 分布式 JOIN 痛点详解

### 3.1 问题复现

```sql
-- 两张分布式表 JOIN
SELECT a.order_id, b.user_name
FROM orders_dist AS a
JOIN users_dist AS b ON a.user_id = b.user_id
WHERE a.create_date = '2025-06-01';

-- 实际执行过程：
-- 1. 每个 Shard 本地 JOIN（只匹配本 Shard 的数据）
-- 2. 结果汇总到 Coordinator
-- ⚠️ 如果数据没有按 user_id 分片，大量匹配丢失！
```

### 3.2 三种"解决"方案及其代价

| 方案 | 做法 | 代价 |
|------|------|------|
| GLOBAL JOIN | 右表广播到所有节点 | 右表大时 OOM / 网络爆 |
| 手动 Co-location | 两表用相同 sharding_key | 写入时需保证分布一致 |
| 提前打宽表 | ETL 阶段 JOIN 好写入单表 | 丧失灵活性，ETL 复杂度高 |
| Dictionary | 小维表用 dictGet | 仅适合小表，内存常驻 |

### 3.3 与 StarRocks Colocate Join 对比

```sql
-- StarRocks: 建表时指定相同的 Distribution Key
CREATE TABLE orders (...) DISTRIBUTED BY HASH(user_id) BUCKETS 32;
CREATE TABLE users (...) DISTRIBUTED BY HASH(user_id) BUCKETS 32;

-- 创建 Colocation Group
ALTER TABLE orders SET ("colocate_with" = "user_group");
ALTER TABLE users SET ("colocate_with" = "user_group");

-- JOIN 时自动走 Colocate Join，零 Shuffle
SELECT a.order_id, b.user_name
FROM orders a JOIN users b ON a.user_id = b.user_id;
-- EXPLAIN 显示: COLOCATE JOIN ← 最优执行计划
```

**核心差异**：StarRocks 在引擎层面原生支持数据对齐和自动路由，ClickHouse 需要应用层手动保证。

## 四、StarRocks Primary Key Model 资源开销分析

### 4.1 PK Model 的额外开销

Primary Key Model 为了实现实时 Upsert/Delete，引入了以下额外开销：

| 开销项 | 说明 | 影响 |
|--------|------|------|
| Delete Vector | 标记被更新的旧版本行 | 额外存储空间 |
| Compaction | 后台合并新旧版本 | CPU + IO 消耗 |
| 主键索引 | 维护主键到行的映射 | 内存占用（约 主键大小 × 行数） |
| Write-Ahead Log | 保证写入持久性 | 写入延迟增加 |
| Merge-on-Write | 写入时立即处理去重 | 写入吞吐低于 Duplicate Model |

### 4.2 资源开销量化参考

```
测试环境: 3 BE, 16C 64GB each, NVMe SSD
数据集: 1亿行, 主键 BIGINT, 20个列

Duplicate Model:
  写入吞吐: ~50万 rows/s
  内存占用: ~20GB
  查询 P99: 50ms

Primary Key Model:
  写入吞吐: ~15万 rows/s  ← 约为 Duplicate 的 30%
  内存占用: ~35GB          ← 多出主键索引 + Delete Vector
  查询 P99: 60ms           ← 略慢但差距不大
  Compaction CPU: ~20%     ← 持续后台消耗
```

### 4.3 何时不该用 PK Model

- 纯追加不更新的日志数据 → 用 Duplicate Model
- 只需要最终一致的去重 → 用 Aggregate Model
- 主键过大（如 UUID + 多个字段组合）→ 内存开销不可接受
- 写入吞吐要求极高且可容忍分钟级可见 → 用 Duplicate + 异步去重

## 五、面试话术："为什么有 ClickHouse 了还要上 StarRocks"

### 5.1 结构化回答框架

> **背景铺垫**：我们最早用 ClickHouse 做数据分析，它在宽表扫描和日志分析上表现非常出色。但随着业务发展，遇到了几个 ClickHouse 难以解决的问题。
>
> **痛点一：多表 JOIN**。业务分析越来越多地涉及事实表关联维表，ClickHouse 没有 CBO，大表 JOIN 经常 OOM 或者超时。我们不得不把数据提前打成宽表，ETL 链路越来越复杂，新需求响应周期从小时级变成了天级。
>
> **痛点二：实时更新**。运营需要修正历史数据、用户画像需要实时更新，ClickHouse 的 ReplacingMergeTree 只能做到最终一致，查询时必须加 FINAL 导致性能下降 10 倍。而 StarRocks 的 Primary Key Model 支持毫秒级 Upsert，查询无需任何额外处理。
>
> **痛点三：高并发服务**。我们的产品需要向终端用户提供交互式分析功能，QPS 要求 200+。ClickHouse 的设计哲学是大吞吐低并发，超过 50 QPS 就频繁出现查询排队和资源争抢。StarRocks 的 Pipeline 执行引擎和 Resource Group 天然适合高并发场景。
>
> **落地效果**：引入 StarRocks 后，多表 JOIN 查询 P99 从 30s 降到 200ms，实时数据更新可见延迟从分钟级降到秒级，支撑了 300+ QPS 的在线分析服务。ClickHouse 继续承担日志分析和离线宽表预聚合的职责，两个引擎各司其职。

### 5.2 一句话总结版

> ClickHouse 是"单表极致"，StarRocks 是"全场景均衡"。当业务从简单的宽表分析演进到多表关联、实时更新、高并发服务时，需要一个 JOIN 能力强、支持 Upsert、并发友好的引擎，这就是引入 StarRocks 的原因。

### 5.3 防御性问题准备

**Q: 为什么不用 Doris？**
> Doris 也是很好的选择，我们在选型时也做了对比测试。在我们的场景中，StarRocks 在多表 JOIN 的 P99 延迟上比 Doris 低 30% 左右，而且异步物化视图对多表 JOIN 预聚合的支持更完善。不过 Doris 的中文社区和文档确实更好，如果团队规模较小、更看重运维便利性，Doris 是完全合理的选择。

**Q: 两个引擎运维成本不是更高了吗？**
> 是的，多引擎确实增加了运维复杂度。我们的缓解措施是：统一监控告警平台、标准化的 ETL 调度、以及明确划分两个引擎的职责边界。CH 只做宽表和日志，SR 只做应用层分析，避免同一个需求在两个引擎上都实现。长远来看，如果 StarRocks 的日志分析能力追上来，可以考虑收敛到一个引擎。

**Q: 有没有考虑过 ClickHouse 新版本改进的 JOIN？**
> 关注了，ClickHouse 24.x 加入了 experimental analyzer 和改进的 JOIN 算法。但在我们的测试中，跟 StarRocks 的 CBO + Colocate Join 相比仍有明显差距。而且 CH 的 JOIN 改进还在实验阶段，生产环境不敢用。等成熟了可以再评估。

## 六、选型决策树

```
开始选型
  │
  ├── 是否需要多表 JOIN？
  │   ├── 否 → 是否只需追加写入？
  │   │       ├── 是 → ClickHouse ✅
  │   │       └── 否 → 是否需要高并发(>100 QPS)？
  │   │               ├── 是 → StarRocks/Doris ✅
  │   │               └── 否 → ClickHouse + ReplacingMergeTree ✅
  │   └── 是 → 是否需要实时更新(Upsert/Delete)？
  │           ├── 是 → StarRocks Primary Key ✅
  │           └── 否 → 是否需要数据湖联邦查询？
  │                   ├── 是 → Doris/StarRocks ✅
  │                   └── 否 → StarRocks (JOIN性能优先) ✅
  │
  └── 特殊场景
      ├── 纯日志/时序 → ClickHouse ✅
      ├── 点查/KV查询 → StarRocks Primary Key / HBase
      └── 湖仓一体 → Doris Multi-Catalog / StarRocks External Catalog
```
