## 一、FE/BE 架构详解

### 1.1 整体架构

```
┌─────────────────────────────────────────────────────────┐
│                    StarRocks Cluster                     │
│                                                         │
│   ┌──────────────────────────────────────────────┐      │
│   │              FE (Frontend)                    │      │
│   │                                              │      │
│   │  ┌──────────┐ ┌──────────┐ ┌──────────────┐ │      │
│   │  │Metadata  │ │Query     │ │Transaction   │ │      │
│   │  │Manager   │ │Planner   │ │Manager       │ │      │
│   │  │(BDB JE)  │ │(CBO/Cascades)│ │(2PC)       │ │      │
│   │  └──────────┘ └──────────┘ └──────────────┘ │      │
│   │                                              │      │
│   │  Leader ←→ Follower ←→ Observer              │      │
│   │  (读写)    (读写,HA)    (只读,扩展)            │      │
│   └──────────────────────┬───────────────────────┘      │
│                          │ MySQL Protocol                │
│                          │ RPC (Thrift/gRPC)             │
│   ┌──────────────────────▼───────────────────────┐      │
│   │              BE (Backend)                     │      │
│   │                                              │      │
│   │  ┌──────────┐ ┌──────────┐ ┌──────────────┐ │      │
│   │  │Storage   │ │Vectorized│ │Compaction    │ │      │
│   │  │Engine    │ │Execution │ │Engine        │ │      │
│   │  │(Segment) │ │Engine    │ │              │ │      │
│   │  └──────────┘ └──────────┘ └──────────────┘ │      │
│   │                                              │      │
│   │  BE-1 ←→ BE-2 ←→ BE-3  (数据副本+并行执行)    │      │
│   └──────────────────────────────────────────────┘      │
└─────────────────────────────────────────────────────────┘
```

### 1.2 FE 核心职责

| 组件 | 功能 | 技术实现 |
|------|------|---------|
| Metadata Manager | 表结构、分区、用户权限、集群拓扑 | BDB JE（嵌入式 KV 存储，Raft 共识） |
| Query Planner | SQL 解析 → 逻辑计划 → 物理计划 → 分布式调度 | Cascades 框架 CBO |
| Transaction Manager | 导入事务的 2PC 协调 | 两阶段提交协议 |
| Job Manager | 异步物化视图刷新、Compaction 调度 | 定时任务 + 事件驱动 |

**FE 高可用机制**：
- Leader: 唯一写入节点，负责元数据变更和事务协调
- Follower: 同步复制 Leader 的元数据日志，Leader 故障时自动选举新 Leader
- Observer: 异步复制，不参与选举，用于扩展查询能力
- 建议生产部署：1 Leader + 2 Follower + N Observer

### 1.3 BE 核心职责

| 组件 | 功能 | 关键特性 |
|------|------|---------|
| Storage Engine | 数据存储与管理 | Segment V2 格式，列式存储，ZoneMap/BloomFilter |
| Vectorized Engine | 向量化查询执行 | SIMD 指令，Batch 处理（4096 rows/batch） |
| Compaction Engine | 后台数据合并 | Base/Cumulative Compaction，Delete Vector 清理 |
| Import Engine | 数据导入执行 | Stream Load / Broker Load / Routine Load |

**BE 数据组织**：
```
Tablet（数据分片）
  ├── Rowset（一次导入产生一个 Rowset）
  │   ├── Segment File（列式存储文件，最大 1GB）
  │   │   ├── Column Data（压缩编码后的列数据）
  │   │   ├── ZoneMap Index（每列的最大最小值）
  │   │   ├── BloomFilter Index（等值过滤）
  │   │   ├── Bitmap Index（低基数列过滤）
  │   │   └── Prefix Index（前缀索引，加速排序键查找）
  │   └── Delete Vector（标记已删除/更新的行）
  └── ...
```

## 二、MPP 执行模型

### 2.1 MPP 基本原理

MPP（Massively Parallel Processing）将一个大查询拆分成多个子任务，分发到所有 BE 节点并行执行，最后汇总结果。

```
                  FE (Coordinator)
                       │
          ┌────────────┼────────────┐
          ▼            ▼            ▼
      BE-1           BE-2         BE-3
   ┌─────────┐   ┌─────────┐   ┌─────────┐
   │Scan Tab │   │Scan Tab │   │Scan Tab │
   │(Shard 1)│   │(Shard 2)│   │(Shard 3)│
   │Filter   │   │Filter   │   │Filter   │
   │Local Agg│   │Local Agg│   │Local Agg│
   └────┬────┘   └────┬────┘   └────┬────┘
        │             │             │
        └─────────────┼─────────────┘
                      ▼
               Global Aggregate
                      │
                      ▼
                   Result
```

### 2.2 Pipeline 执行引擎

StarRocks 3.x 引入 Pipeline 执行引擎，替代传统的 Volcano/Pull 模型：

```
传统 Volcano 模型:
  每个算子一个线程，线程间通过 Blocking Queue 传递数据
  问题：线程数 = 算子数 × 并发度，上下文切换开销大

Pipeline 模型:
  将算子链切分为多个 Pipeline，每个 Pipeline 由 Driver 执行
  Driver 是轻量级工作单元，由全局线程池调度
  优势：充分利用多核 CPU，减少线程切换

┌──────────────────────────────────────────┐
│         Pipeline Execution Model          │
│                                          │
│  Pipeline-1: Scan → Filter → LocalAgg    │
│  Pipeline-2: Exchange → GlobalAgg        │
│  Pipeline-3: Exchange → Sort → Limit     │
│                                          │
│  Global Thread Pool (N threads)          │
│  ┌────┐┌────┐┌────┐┌────┐┌────┐        │
│  │ T1 ││ T2 ││ T3 ││ T4 ││ TN │        │
│  └────┘└────┘└────┘└────┘└────┘        │
│  每个线程可执行任意 Pipeline 的 Driver       │
└──────────────────────────────────────────┘
```

## 三、CBO 优化器

### 3.1 Cascades 框架

StarRocks 使用基于 Cascades 框架的 CBO（Cost-Based Optimizer），相比传统 RBO（Rule-Based Optimizer）的优势：

| 特性 | RBO | CBO (Cascades) |
|------|-----|----------------|
| 优化依据 | 固定规则 | 统计信息 + 代价模型 |
| JOIN 顺序 | 按 SQL 书写顺序 | 枚举 + 剪枝选最优 |
| JOIN 策略选择 | 固定 | 根据数据量动态选择 |
| 聚合下推 | 有限 | 系统性探索 |
| 谓词下推 | 基础 | 深度探索 |
| 适应性 | 差 | 好（随数据分布变化） |

### 3.2 CBO 关键优化策略

```
1. Runtime Filter
   在 Build 侧生成过滤器，下推到 Probe 侧提前过滤
   ┌─────────────────────────────────┐
   │ Hash Join                       │
   │ Build Side (dim_table)          │
   │   → 生成 BloomFilter            │
   │   → 下推到 fact_table 的 Scan    │
   │ Probe Side (fact_table)         │
   │   → Scan 时用 BF 过滤 80%+ 数据  │
   └─────────────────────────────────┘

2. Colocate Join
   相同 Distribution Key + 相同 Bucket 数的表
   数据天然对齐，JOIN 无需 Shuffle

3. Bucket Shuffle Join
   当右表较小但不满足 Broadcast 条件时
   只 Shuffle 左表，右表保持本地

4. 谓词下推到存储层
   WHERE 条件下推到 Scan 算子
   利用 ZoneMap / BloomFilter / Prefix Index 跳过无关数据块

5. 物化视图自动改写
   CBO 识别查询可以命中异步 MV 时自动改写执行计划
```

## 四、四种表模型详解

### 4.1 Duplicate Model（明细模型）

**定义**：最基础的模型，数据原样存储，不做任何去重或聚合。

```sql
CREATE TABLE user_behavior
(
    event_time  DATETIME NOT NULL,
    user_id     BIGINT NOT NULL,
    page_url    VARCHAR(512),
    action_type VARCHAR(32),
    duration_ms INT
)
DUPLICATE KEY(event_time, user_id)
PARTITION BY RANGE(event_time) (...)
DISTRIBUTED BY HASH(user_id) BUCKETS 16;
```

**内部机制**：
- 数据按 DUPLICATE KEY 排序存储
- 相同 Key 的多条记录全部保留
- 支持所有类型的查询
- 写入性能最好（无额外处理）

**适用场景**：
- 日志/事件流等只追加不更新的数据
- 需要保留全部明细的 Ad-hoc 分析
- 作为其他模型的底层数据源

### 4.2 Aggregate Model（聚合模型）

**定义**：写入时自动按 Key 聚合，类似 ClickHouse 的 AggregatingMergeTree。

```sql
CREATE TABLE page_metrics_agg
(
    stat_date   DATE NOT NULL,
    page_id     BIGINT NOT NULL,
    pv          SUM REPLACE DEFAULT "0",
    uv          BITMAP_UNION BITMAP_DEFAULT "",
    avg_dur     AVG REPLACE DEFAULT "0"
)
AGGREGATE KEY(stat_date, page_id)
PARTITION BY RANGE(stat_date) (...)
DISTRIBUTED BY HASH(page_id) BUCKETS 8;
```

**内部机制**：
```
写入: (2025-06-01, page_1, pv=5, uv={a,b}, avg_dur=100)
      (2025-06-01, page_1, pv=3, uv={c},   avg_dur=200)
                            ↓ Merge
存储: (2025-06-01, page_1, pv=8, uv={a,b,c}, avg_dur=?)

注意: AVG 的聚合语义是加权平均，不是简单平均
```

**支持的聚合函数**：SUM, MIN, MAX, COUNT, BITMAP_UNION, HLL_UNION, REPLACE, REPLACE_IF_NOT_NULL

**与 ClickHouse AggregatingMergeTree 对比**：
| 维度 | SR Aggregate Model | CH AggregatingMergeTree |
|------|-------------------|------------------------|
| 聚合时机 | 写入时 + Compaction 时 | Compaction 时 |
| 可见性 | 写入即可见（预聚合后） | 需 -Merge 函数读取 |
| 灵活性 | 较低（聚合方式建表时固定） | 较高（-State 函数灵活组合） |
| 查询体验 | 直接 SELECT，无需特殊函数 | 必须用 -Merge 函数 |

**适用场景**：
- 固定维度的预聚合报表
- 写入量大但查询维度固定的场景

### 4.3 Unique Model（唯一键模型）

**定义**：保证 Key 唯一，新数据覆盖旧数据。

```sql
CREATE TABLE user_profile
(
    user_id     BIGINT NOT NULL,
    nickname    VARCHAR(128),
    age         INT,
    city        VARCHAR(64),
    updated_at  DATETIME
)
UNIQUE KEY(user_id)
DISTRIBUTED BY HASH(user_id) BUCKETS 16;
```

**内部机制**：
- Merge-on-Read 模式（早期版本）：写入时追加新版本，查询时实时合并取最新
- Merge-on-Write 模式（3.x）：写入时立即处理去重，查询无需额外合并
- 不支持部分列更新（整行覆盖）

**适用场景**：
- 用户画像等需要更新的维度表
- CDC 入仓的目标表
- 不需要部分列更新的 Upsert 场景

### 4.4 Primary Key Model（主键模型）⭐

**定义**：StarRocks 最先进的表模型，支持实时 Upsert + Delete + 部分列更新。

```sql
CREATE TABLE order_realtime
(
    order_id    BIGINT NOT NULL,
    user_id     BIGINT NOT NULL,
    status      VARCHAR(32),
    amount      DECIMAL(18,2),
    pay_time    DATETIME,
    updated_at  DATETIME DEFAULT CURRENT_TIMESTAMP
)
PRIMARY KEY(order_id)
DISTRIBUTED BY HASH(order_id) BUCKETS 16
PROPERTIES (
    "enable_persistent_index" = "true"   -- 持久化主键索引，降低内存占用
);
```

**内部机制**：
```
┌──────────────────────────────────────────────────────┐
│          Primary Key Model 内部流程                    │
│                                                      │
│  UPSERT (order_id=1001, status='paid', amount=99.9)  │
│       │                                              │
│       ▼                                              │
│  ┌────────────────┐                                  │
│  │ 查主键索引      │ ← 找到 order_id=1001 已有行       │
│  │ (内存/持久化)   │                                  │
│  └───────┬────────┘                                  │
│          ▼                                           │
│  ┌────────────────┐                                  │
│  │ 写 Delete Vector│ ← 标记旧行为已删除                │
│  │ 标记旧版本      │                                  │
│  └───────┬────────┘                                  │
│          ▼                                           │
│  ┌────────────────┐                                  │
│  │ 写入新版本数据   │ ← 新行立即可查                    │
│  └────────────────┘                                  │
│                                                      │
│  DELETE FROM order_realtime WHERE order_id = 1001;   │
│       │                                              │
│       ▼                                              │
│  ┌────────────────┐                                  │
│  │ 写 Delete Vector│ ← 标记该行为已删除                 │
│  │ 该行不再可见     │                                  │
│  └────────────────┘                                  │
└──────────────────────────────────────────────────────┘
```

**与 Unique Model 的关键区别**：

| 维度 | Unique Model | Primary Key Model |
|------|-------------|-------------------|
| DELETE 支持 | ❌（请用 PK Model） | ✅ 支持 |
| 部分列更新 | ❌ 整行覆盖 | ✅ 支持 |
| 写入性能 | Merge-on-Read 较好 | Merge-on-Write 略低 |
| 查询性能 | MoR 需合并，MoW 无需 | 始终无需合并 |
| 内存开销 | 较低 | 较高（主键索引常驻） |
| 适用场景 | 低频更新 | 高频实时更新 |

**适用场景**：
- 实时数仓事实表（CDC 实时入仓）
- 需要 DELETE 的场景（数据修正/GDPR）
- 部分列更新（多个上游分别更新同一张表的不同列）
- 对查询一致性要求高的实时应用

## 五、四种模型选择决策树

```
需要什么表模型？
  │
  ├── 只追加不更新？
  │   ├── 是 → Duplicate Model ✅
  │   └── 否 ↓
  │
  ├── 需要预聚合？
  │   ├── 是 → Aggregate Model ✅
  │   └── 否 ↓
  │
  ├── 需要 DELETE / 部分列更新？
  │   ├── 是 → Primary Key Model ✅
  │   └── 否 ↓
  │
  └── 只需要简单的 Key 去重/Upsert？
      └── Unique Model ✅
```

## 六、面试话术要点

### Q: StarRocks 有几种表模型？各自适用什么场景？

> StarRocks 有四种表模型。Duplicate Model 适合只追加的日志和事件数据，写入性能最好。Aggregate Model 适合固定维度的预聚合，写入时自动聚合，查询时直接读结果。Unique Model 支持 Key 级别的 Upsert 去重，适合用户画像等维度表。Primary Key Model 是最强大的模型，支持实时 Upsert、Delete 和部分列更新，适合 CDC 实时入仓和需要强一致性的实时数仓场景。实际项目中我们事实表用 Primary Key Model 对接 CDC，维度表用 Unique Model，日志表用 Duplicate Model，报表层用 Aggregate Model。

### Q: Primary Key Model 为什么能支持实时更新？

> PK Model 维护了一个主键到行位置的映射索引。Upsert 时先查索引找到旧行，写入 Delete Vector 标记旧行删除，再写入新行。整个过程在写入阶段完成（Merge-on-Write），所以查询时一定能读到最新版本，不需要像 ClickHouse ReplacingMergeTree 那样等后台 Merge 或加 FINAL。代价是写入吞吐比 Duplicate Model 低约 60%，且主键索引需要额外内存。可以通过 enable_persistent_index 将索引持久化到磁盘来降低内存压力。

### Q: StarRocks 的 CBO 和 ClickHouse 有什么区别？

> ClickHouse 基本没有 CBO，只有简单的 RBO，JOIN 顺序和策略主要靠人工指定。StarRocks 基于 Cascades 框架实现了完整的 CBO，会根据表的统计信息（行数、NDV、数据大小等）自动选择最优的 JOIN 顺序、JOIN 策略（Broadcast/Shuffle/Colocate）、是否生成 Runtime Filter 等。这意味着同样的 SQL 在 StarRocks 上通常能获得更好的执行计划，尤其在多表 JOIN 场景下优势非常明显。
