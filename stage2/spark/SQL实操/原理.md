# Spark SQL 原理：DataFrame/Dataset API、Catalyst 优化器与 Hive 集成

## 一、DataFrame / Dataset API 核心概念

### 1.1 DataFrame = Dataset[Row]

在 Spark 内部，DataFrame 就是 `Dataset[Row]`。Row 是一个通用的、无类型的行对象。

```scala
// 这两者在 JVM 层面完全等价
val df: DataFrame = spark.read.parquet("data")
val ds: Dataset[Row] = df  // 无需转换
```

Python 中没有 Dataset 概念（因为 Python 没有编译期类型），DataFrame 直接对应内部的 `Dataset[Row]`。

### 1.2 核心 API 分类

| 类别 | 操作示例 | 是否触发执行 |
|------|---------|-------------|
| **Transformation** | select, filter, groupBy, join, withColumn, union | ❌ 惰性 |
| **Action** | show, collect, count, write, foreach | ✅ 触发 Job |
| **DDL** | createOrReplaceTempView, cache, repartition | 部分立即生效 |

### 1.3 Column 表达式体系

```python
from pyspark.sql import functions as F

# 列引用
F.col("name")               # 推荐
df.name                      # 语法糖
df["name"]                   # 字典风格

# 条件表达式
F.when(F.col("age") > 30, "senior").otherwise("junior")
F.col("amount").between(100, 500)

# 聚合函数
F.count("*"), F.sum("amt"), F.avg("score")
F.collect_list("tag"), F.collect_set("category")

# 窗口函数
F.row_number().over(Window.partitionBy("dept").orderBy(F.desc("salary")))
F.lag("value", 1).over(w)

# UDF（应尽量避免，优先用内置函数）
my_udf = F.udf(lambda x: x.upper(), StringType())
```

> **面试话术**：优先使用 Spark 内置函数而非 UDF，因为内置函数可以被 Catalyst 优化（谓词下推、常量折叠等），而 UDF 是黑盒，Catalyst 无法穿透优化。如果必须用 UDF，Scala UDF 比 Python UDF 性能好一个数量级（避免 Python ↔ JVM 序列化开销）。Spark 3.x 的 Pandas UDF (Arrow-based) 是 Python UDF 的性能替代方案。

## 二、Catalyst 优化器详解

Catalyst 是 Spark SQL 的核心引擎，它将用户的声明式查询自动转化为高效的执行计划。

### 2.1 四阶段优化流程

```
┌─────────────────────┐
│   SQL / DF API 代码   │
└──────────┬──────────┘
           │ parse
           ▼
┌─────────────────────┐
│   Unresolved         │ ← 未解析的逻辑计划
│   Logical Plan       │   (AST, 未绑定表/列)
└──────────┬──────────┘
           │ resolve (Analysis)
           ▼
┌─────────────────────┐
│   Analyzed           │ ← 已解析的逻辑计划
│   Logical Plan       │   (绑定了 Catalog 中的表/列/类型)
└──────────┬──────────┘
           │ optimize rules
           ▼
┌─────────────────────┐
│   Optimized          │ ← 优化后的逻辑计划
│   Logical Plan       │   (谓词下推、列裁剪、常量折叠...)
└──────────┬──────────┘
           │ planning strategies
           ▼
┌─────────────────────┐
│   Physical Plans     │ ← 多个候选物理计划
│   (candidates)       │   (HashJoin vs SortMergeJoin vs BroadcastJoin)
└──────────┬──────────┘
           │ cost-based selection
           ▼
┌─────────────────────┐
│   Selected           │ ← 最终选定的物理计划
│   Physical Plan      │   (带运行时指标采集点)
└──────────┬──────────┘
           │ execute
           ▼
┌─────────────────────┐
│   RDD / Whole-Stage  │ ← 实际执行的代码
│   Generated Code     │
└─────────────────────┘
```

### 2.2 各阶段详解

#### Phase 1: Parsing → Unresolved Logical Plan

将 SQL 字符串或 DataFrame API 调用转为 AST（抽象语法树）。此时表和列只是字符串名称，尚未验证存在性。

```sql
SELECT name, age FROM users WHERE age > 25
```
→
```
'Project ['name, 'age]
  'Filter ('age > 25)
    'UnresolvedRelation [users]
```

#### Phase 2: Analysis → Resolved Logical Plan

通过 Catalog（Hive Metastore / In-Memory Catalog）解析所有表和列引用：
- 验证表是否存在
- 解析列名到具体字段
- 类型检查和隐式转换
- 解析函数调用

```
Project [name#12, age#13]
  Filter (age#13 > 25)
    SubqueryAlias users
      Relation[name#12, age#13, city#14] parquet
```

#### Phase 3: Optimization → Optimized Logical Plan

应用一系列基于规则的优化（Rule-Based Optimization）：

| 优化规则 | 说明 | 示例 |
|---------|------|------|
| **谓词下推 (Predicate Pushdown)** | 将 filter 尽可能推到数据源层 | filter 移到 scan 之前 |
| **列裁剪 (Column Pruning)** | 只读取需要的列 | SELECT a,b 不读 c |
| **常量折叠 (Constant Folding)** | 编译期计算常量表达式 | `1+2+col` → `3+col` |
| **NULL 传播** | 简化含 NULL 的表达式 | `null AND x` → `null` |
| **布尔简化** | 化简布尔表达式 | `x=true` → `x` |
| **子查询去关联** | 将相关子查询转为 join | EXISTS → LEFT SEMI JOIN |
| **Join Reorder** | 调整 join 顺序（CBO） | 小表在前 |
| **Union 合并** | 合并相邻的 union | 减少中间结果 |

```
# 优化前
Project [name, age]
  Filter (age > 25)
    Scan parquet [name, age, city, email, phone]

# 优化后（谓词下推 + 列裁剪）
Project [name, age]
  Filter (age > 25)        ← filter 下推到 scan
    Scan parquet [name, age]  ← 只读两列, 且带 filter
```

#### Phase 4: Physical Planning → Selected Physical Plan

将逻辑算子映射为物理算子，生成多个候选计划，选择代价最低的：

| 逻辑算子 | 候选物理算子 | 选择依据 |
|---------|-------------|---------|
| Join | BroadcastHashJoin / SortMergeJoin / ShuffleHashJoin | 表大小、排序状态 |
| Aggregate | HashAggregate / SortAggregate | 分组数、内存 |
| Exchange | ShuffleExchange / BroadcastExchange | 下游需求 |
| Scan | FileSourceScan / InMemoryTableScan | 数据来源 |

**CBO（Cost-Based Optimization）**：
- 基于统计信息（行数、列基数、NDV、直方图）估算代价
- `spark.sql.cbo.enabled=true`（Spark 3.x 默认开启）
- 需要 `ANALYZE TABLE ... COMPUTE STATISTICS` 提供统计信息

### 2.3 Whole-Stage Code Generation

Catalyst 最终会将物理计划编译为 Java 字节码，避免虚函数调用和序列化开销：

```
*(1) Project [name#12, age#13]
   *(1) Filter (isnotnull(age#13) AND (age#13 > 25))
      *(1) ColumnarToRow
         *(1) FileScan parquet [name#12,age#13]
```

`*(1)` 表示这些算子被融合为一个函数（Whole-Stage），在一个 Java 方法内完成 scan + filter + project，避免了逐算子的物化和反序列化。

> **面试话术**：Catalyst 的四阶段优化是 Spark SQL 高性能的核心。开发者只需写声明式查询，优化器自动做谓词下推、列裁剪、Join 选择等。这也是为什么推荐用 DataFrame/SQL 而不是 RDD——RDD 绕过了整个 Catalyst 优化链。

## 三、Hive 集成

### 3.1 Spark on Hive vs Hive on Spark

| 模式 | 含义 | 推荐度 |
|------|------|--------|
| **Spark on Hive** | Spark 作为执行引擎，读取 Hive Metastore 元数据 | ✅ 主流 |
| **Hive on Spark** | Hive 提交 SQL 给 Spark 执行 | ❌ 已废弃 |

### 3.2 连接 Hive Metastore

```python
spark = SparkSession.builder \
    .appName("hive-demo") \
    .enableHiveSupport() \                          # 关键！
    .config("hive.metastore.uris", "thrift://metastore-host:9083") \
    .config("spark.sql.warehouse.dir", "/user/hive/warehouse") \
    .getOrCreate()

# 现在可以像操作本地表一样操作 Hive 表
spark.sql("SHOW DATABASES").show()
spark.sql("SELECT * FROM dw.dwd_orders WHERE dt='2024-06-01'").show()
```

### 3.3 Hive 兼容性

| 特性 | 支持情况 |
|------|---------|
| HiveQL 语法 | 大部分兼容 |
| UDF/UDAF/UDTF | 支持注册 Hive UDF |
| SerDe | 支持自定义 SerDe |
| Bucket Map Join | 有限支持 |
| ACID 表 | Spark 3.x 开始支持读 |
| 事务表写入 | 需额外配置 |

## 四、Schema 推断与管理

### 4.1 Schema 推断

```python
# CSV 自动推断（生产慎用！）
df = spark.read.option("inferSchema", "true").csv("data/")
# 问题：扫描全量数据推断类型，大文件很慢；类型可能不准

# 推荐：显式指定 schema
from pyspark.sql.types import *
schema = StructType([
    StructField("order_id", LongType()),
    StructField("user_id", LongType()),
    StructField("amount", DecimalType(10, 2)),
    StructField("order_date", DateType())
])
df = spark.read.schema(schema).csv("data/")
```

### 4.2 Schema Evolution

Parquet / ORC 支持 schema evolution：
- 新增列：新文件有、旧文件无 → 旧文件该列为 null
- 删除列：新文件无、旧文件有 → 忽略旧列
- 类型变更：需兼容（如 int→long OK，string→int 不行）

```python
spark.conf.set("spark.sql.parquet.mergeSchema", "true")  # 合并多文件 schema
```

## 五、分区管理

### 5.1 分区发现

Spark 自动识别 Hive 风格的目录分区：

```
/user/hive/warehouse/orders/
├── dt=2024-06-01/
│   ├── part-00000.parquet
│   └── part-00001.parquet
├── dt=2024-06-02/
│   └── part-00000.parquet
└── dt=2024-06-03/
    └── part-00000.parquet
```

```python
# 自动发现分区列
df = spark.read.parquet("/user/hive/warehouse/orders/")
# df 自动包含 dt 列

# 手动注册分区（Hive 外部表场景）
spark.sql("ALTER TABLE orders ADD PARTITION (dt='2024-06-04')")
# 或
spark.catalog.recoverPartitions("orders")
```

### 5.2 分区裁剪（Partition Pruning）

```python
# ✅ 会触发分区裁剪（只扫描 dt=2024-06-01 的文件）
df.filter(F.col("dt") == "2024-06-01")

# ❌ 不会触发分区裁剪（函数包裹了分区列）
df.filter(F.substring(F.col("dt"), 1, 7) == "2024-06")

# ✅ 改写为范围过滤即可触发
df.filter((F.col("dt") >= "2024-06-01") & (F.col("dt") <= "2024-06-30"))
```

> **面试话术**：分区裁剪是 Spark SQL 最重要的性能优化之一。使用时注意不要在分区列上套函数，否则 Catalyst 无法识别分区过滤条件。另外，动态分区写入时要设置 `spark.sql.sources.partitionOverwriteMode=dynamic`，否则会覆盖整个表而非只覆盖涉及的分区。

## 六、总结与面试要点

### 核心记忆点

1. **Catalyst 四阶段**：Parse → Analyze → Optimize → Physical Planning
2. **关键优化**：谓词下推、列裁剪、常量折叠、Join 选择、Code Generation
3. **Hive 集成**：`enableHiveSupport()` + Metastore URI
4. **Schema**：生产环境显式指定，避免 inferSchema
5. **分区**：裁剪是大杀器，别在分区列上套函数

### 常见面试追问

- **Q: Catalyst 做了什么你手动做不到的优化？**
  A: 最典型的是谓词下推和列裁剪。手写 RDD 时你需要自己决定先 filter 再 map 还是反过来，而 Catalyst 会自动把 filter 推到离数据源最近的位置。另外 Whole-Stage CodeGen 把多个算子编译成一个 Java 方法，消除虚函数调用，这是手工无法实现的。

- **Q: DataFrame 和 RDD 在执行层面有什么区别？**
  A: DataFrame 走 Catalyst 优化链，最终生成优化的 Java 字节码（Tungsten 二进制格式）。RDD 直接执行用户提供的 lambda/closure，走 Java Serialization，无法被优化器感知。同样的逻辑，DataFrame 通常比 RDD 快 2-10 倍。

- **Q: 为什么有时候 explain 看到的计划和预期不同？**
  A: Catalyst 做了大量自动优化。比如你以为的 nested loop join 可能被优化成了 broadcast hash join；你以为的全表扫描可能因为分区裁剪只读了几个分区。用 `explain(mode="formatted")` 可以看到完整的优化过程和每个阶段的计划变化。
