# Spark 调优入门：数据倾斜、内存模型、分区与 AQE

## 一、调优思维框架

调优不是背参数，而是**定位瓶颈 → 选择策略 → 验证效果**的闭环。

```
性能问题
    │
    ├── 1. 定位瓶颈在哪里？
    │     ├── CPU bound? (Task 计算密集)
    │     ├── IO bound? (Shuffle Read/Write, HDFS)
    │     ├── Memory bound? (GC, Spill, OOM)
    │     └── Skew? (个别 Task 远慢于其他)
    │
    ├── 2. 选择对应策略
    │     ├── 数据倾斜 → Salting / Broadcast / AQE
    │     ├── 内存不足 → 调 memory fraction / 增大分区
    │     ├── Shuffle 过多 → Broadcast Join / 预分区
    │     └── 小文件 → Coalesce / AQE 合并
    │
    └── 3. 验证效果
          ├── Spark UI 对比前后指标
          ├── 运行时间对比
          └── 资源消耗对比
```

## 二、数据倾斜处理

### 2.1 什么是数据倾斜

某些 key 的数据量远超其他 key，导致对应的 Task 处理时间远超平均水平。

```
正常分布:                    倾斜分布:
Task0: ████                  Task0: ████████████████████████ ← 热点
Task1: ████                  Task1: ██
Task2: ████                  Task2: █
Task3: ████                  Task3: ██
Task4: ████                  Task4: █
总耗时 ≈ max(Task)           总耗时 = 热点Task（长尾效应）
```

### 2.2 Salting（加盐打散）

**核心思想**：给热点 key 添加随机前缀，将其分散到多个分区；聚合后再去掉前缀合并结果。

```python
from pyspark.sql import functions as F

# ===== 场景：groupBy countDistinct 倾斜 =====

# Step 1: 加盐 — 给 user_id 加随机前缀
NUM_SALTS = 10
salted_df = df.withColumn(
    "salted_key",
    F.concat(F.lit("_"), (F.rand() * NUM_SALTS).cast("int").cast("string"), 
             F.col("user_id"))
)

# Step 2: 局部聚合（按 salted_key 分组）
partial_agg = salted_df.groupBy("salted_key").agg(
    F.countDistinct("item_id").alias("partial_cnt")
)

# Step 3: 去盐 — 还原原始 key
un_salted = partial_agg.withColumn(
    "user_id",
    F.regexp_extract("salted_key", r"^_\d+(.+)$", 1)
)

# Step 4: 全局聚合
final_result = un_salted.groupBy("user_id").agg(
    F.sum("partial_cnt").alias("distinct_cnt")  # 注意：countDistinct 不能直接 sum
)
```

> ⚠️ **注意**：`countDistinct` 的两阶段拆分需要特殊处理。局部用 `collect_set`，全局用 `size(collect_set)` 或 HyperLogLog。上面代码仅作示意，实际需要用 `approx_count_distinct` 或手动 collect_set。

**Salting 适用条件**：
- ✅ 聚合操作（groupBy + agg）
- ✅ 热点 key 可枚举或可识别
- ❌ Join 操作（用 Broadcast 更好）
- ❌ 全表均匀倾斜（没有明显热点）

### 2.3 Broadcast Join

**核心思想**：将小表广播到所有 Executor，避免 shuffle。

```python
# 自动 broadcast（小表 < threshold）
spark.conf.set("spark.sql.autoBroadcastJoinThreshold", "50MB")

# 手动指定 broadcast
from pyspark.sql.functions import broadcast
result = large_df.join(broadcast(small_df), on="key", how="inner")
```

**限制**：
- 小表必须能放进单节点内存（建议 < Driver 内存的 1/3）
- 不支持 full outer join
- 广播本身也有开销（序列化 + 网络传输）

### 2.4 两阶段聚合

```python
# 第一阶段：加随机后缀做局部聚合
df.withColumn("rand_suffix", (F.rand() * 10).cast("int")) \
  .groupBy("key", "rand_suffix") \
  .agg(F.sum("value").alias("partial_sum")) \
  # 第二阶段：去掉后缀做全局聚合
  .groupBy("key") \
  .agg(F.sum("partial_sum").alias("total_sum"))
```

### 2.5 AQE Skew Join（Spark 3.x）

```python
spark.conf.set("spark.sql.adaptive.enabled", "true")
spark.conf.set("spark.sql.adaptive.skewJoin.enabled", "true")
spark.conf.set("spark.sql.adaptive.skewJoin.skewedPartitionFactor", "5")   # 超过中位数5倍视为倾斜
spark.conf.set("spark.sql.adaptive.skewJoin.skewedPartitionThresholdInBytes", "256MB")
```

AQE 在运行时检测到倾斜分区后，自动将其拆分为多个子分区并行处理。**无需修改业务代码**。

## 三、内存模型详解

### 3.1 Executor 内存区域

```
┌─────────────────────────────────────────────────────────────┐
│                    JVM Heap                                  │
│                                                              │
│  ┌────────────────────────────────────────────────────────┐ │
│  │              spark.executor.memory                       │ │
│  │                                                         │ │
│  │  ┌──────────────────────┬───────────────────────────┐  │ │
│  │  │                      │                           │  │ │
│  │  │  Execution Memory    │    Storage Memory          │  │ │
│  │  │  (动态共享)           │    (动态共享)               │  │ │
│  │  │                      │                           │  │ │
│  │  │  • Shuffle Buffer    │  • RDD Cache/Persist      │  │ │
│  │  │  • Join Hash Table   │  • Broadcast Variables    │  │ │
│  │  │  • Sort Buffer       │  • Unroll Buffer          │  │ │
│  │  │  • Aggregation Buf   │                           │  │ │
│  │  │                      │                           │  │ │
│  │  └──────────────────────┴───────────────────────────┘  │ │
│  │            ↑ memory.fraction (默认 0.6)                 │ │
│  │                                                         │ │
│  │  ┌─────────────────────────────────────────────────┐   │ │
│  │  │         User Memory (用户自定义对象)               │   │ │
│  │  │         ↑ 1 - memory.fraction = 0.4              │   │ │
│  │  └─────────────────────────────────────────────────┘   │ │
│  │                                                         │ │
│  │  ┌─────────────────────────────────────────────────┐   │ │
│  │  │         Reserved Memory (固定 300MB)              │   │ │
│  │  └─────────────────────────────────────────────────┘   │ │
│  └────────────────────────────────────────────────────────┘ │
│                                                              │
│  ┌────────────────────────────────────────────────────────┐ │
│  │         Off-Heap Memory                                 │ │
│  │         spark.memory.offHeap.size                       │ │
│  │         • Tungsten 二进制缓冲                            │ │
│  │         • Netty Direct Buffer                           │ │
│  └────────────────────────────────────────────────────────┘ │
│                                                              │
│  ┌────────────────────────────────────────────────────────┐ │
│  │         Memory Overhead                                 │ │
│  │         spark.executor.memoryOverhead                   │ │
│  │         • JVM Metaspace / Code Cache                    │ │
│  │         • NIO Direct Buffer                             │ │
│  │         • Thread Stack                                  │ │
│  └────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 关键参数

| 参数 | 默认值 | 说明 | 调优建议 |
|------|--------|------|---------|
| `spark.executor.memory` | 1g | Executor 堆内内存 | 根据数据量和 Task 复杂度调整 |
| `spark.memory.fraction` | 0.6 | Exec+Storage 占堆的比例 | 通常不改，除非 GC 严重 |
| `spark.memory.storageFraction` | 0.5 | Storage 保底比例 | Cache 多时可提高 |
| `spark.memory.offHeap.enabled` | false | 是否启用堆外内存 | Tungsten 场景建议开启 |
| `spark.memory.offHeap.size` | 0 | 堆外内存大小 | 设为 executor-memory 的 20-30% |
| `spark.executor.memoryOverhead` | max(384MB, 10%×mem) | JVM 额外内存 | OOM 但不是 heap OOM 时加大 |

### 3.3 内存调优决策树

```
Executor OOM?
    │
    ├── java.lang.OutOfMemoryError: Java heap space
    │     → 增大 spark.executor.memory
    │     → 或增加分区数减少单 Task 数据量
    │     → 或减少 cache 比例
    │
    ├── java.lang.OutOfMemoryError: Metaspace
    │     → 增大 spark.driver/executor.extraJavaOptions=-XX:MaxMetaspaceSize=512m
    │
    ├── Container killed by YARN for exceeding memory limits
    │     → 增大 spark.executor.memoryOverhead
    │     → 或检查是否有内存泄漏
    │
    └── GC overhead limit exceeded
          → 增大内存 + 增大分区数
          → 开启 off-heap 内存
          → 使用 G1GC: -XX:+UseG1GC
```

## 四、分区调优

### 4.1 spark.sql.shuffle.partitions

控制 shuffle 后的分区数，直接影响并行度和输出文件数。

| 数据量 | 推荐值 | 理由 |
|--------|--------|------|
| < 1GB | 10-50 | 避免过多小文件 |
| 1-10GB | 50-200 | 平衡并行度和开销 |
| 10-100GB | 200-500 | 充分利用集群 |
| > 100GB | 500-2000 | 保证每个分区 ~128MB |

```python
# 经验公式：目标分区大小 ≈ 128MB
partitions = total_data_size_bytes / (128 * 1024 * 1024)
```

### 4.2 repartition vs coalesce

| 操作 | 是否 Shuffle | 能否增加分区 | 性能 |
|------|-------------|-------------|------|
| `repartition(n)` | ✅ 全量 shuffle | ✅ | 较慢 |
| `coalesce(n)` | ❌ 仅合并 | ❌ 只能减少 | 较快 |

```python
# 减少分区（写文件前）→ 用 coalesce
df.coalesce(10).write.parquet("output")

# 增加分区（解决单分区过大）→ 用 repartition
df.repartition(200).groupBy("key").agg(...)

# 按列重分区（为后续 join/groupBy 准备）
df.repartition(100, "join_key")
```

## 五、AQE（Adaptive Query Execution）

### 5.1 AQE 三大能力

```
┌────────────────────────────────────────────────────────┐
│                    AQE 三大特性                         │
├────────────────────┬───────────────────┬───────────────┤
│ 动态合并小分区       │ 自适应 Join 策略   │ 倾斜 Join 处理 │
│                    │                   │               │
│ Shuffle 后发现很多  │ 运行时根据实际数据  │ 检测倾斜分区   │
│ 分区很小，合并为更  │ 大小选择最优 Join  │ 自动拆分为子   │
│ 少的分区以减少 Task │ 类型              │ 分区并行处理   │
│ 启动开销            │                   │               │
│                    │ SortMerge→Broadcst│ 无需改代码     │
│                    │ Hash→SortMerge    │               │
└────────────────────┴───────────────────┴───────────────┘
```

### 5.2 关键参数

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `spark.sql.adaptive.enabled` | true (3.2+) | 总开关 |
| `spark.sql.adaptive.coalescePartitions.enabled` | true | 合并小分区 |
| `spark.sql.adaptive.coalescePartitions.minPartitionSize` | 64MB | 最小目标分区大小 |
| `spark.sql.adaptive.skewJoin.enabled` | true | 倾斜 Join 处理 |
| `spark.sql.adaptive.skewJoin.skewedPartitionFactor` | 5 | 倾斜倍数阈值 |
| `spark.sql.adaptive.adviceJoin.enabled` | false | Join 策略建议 |

### 5.3 AQE 工作原理

```
传统模式:
  编译期确定物理计划 → 执行 → 无法根据运行时数据调整

AQE 模式:
  编译期生成初始物理计划
       │
       ▼
  执行 Stage 0 → 收集 Shuffle 统计信息
       │
       ▼
  AQE 分析统计数据
       ├── 发现小分区 → 合并
       ├── 发现表比预估小 → 切换 Broadcast Join
       └── 发现倾斜分区 → 拆分
       │
       ▼
  动态调整后续 Stage 的物理计划
       │
       ▼
  继续执行...
```

> **面试话术**：AQE 是 Spark 3.x 最重要的改进之一。它把"编译期静态优化"升级为"运行时动态优化"。最大的价值在于：开发者不需要精确预估数据分布，AQE 会根据实际 shuffle 数据自动调整分区数和 Join 策略。在生产环境中，开启 AQE 后很多之前需要手动调参的问题都能自动解决。

## 七、总结与面试要点

### 调优优先级

```
1. 数据模型层：合理分区、文件格式、预聚合
2. SQL/API 层：避免 UDF、利用内置函数、合理 Join 顺序
3. 配置层：shuffle.partitions、memory、broadcast threshold
4. 代码层：Salting、两阶段聚合、手动 broadcast
5. 架构层：湖表统一、预计算、物化视图
```

### 常见面试追问

- **Q: 你做过哪些 Spark 调优？**
  A: （举例）① 发现 groupBy 倾斜，用 salting 打散热点 key，Stage 耗时从 20min 降到 3min；② 大小表 join 开启 broadcast，消除了不必要的 shuffle；③ 开启 AQE 后小文件问题自动缓解；④ 将 Python UDF 替换为内置函数，整体提速 3 倍。

- **Q: AQE 和手动调优的关系是什么？**
  A: AQE 解决的是"编译期无法准确预估运行时数据分布"的问题。对于已知的、固定的数据特征，手动调优仍然有效且更精确。AQE 是兜底保障，不是万能药。比如你知道某张表一定小于 100MB，直接手动 broadcast 比等 AQE 判断更可靠。
