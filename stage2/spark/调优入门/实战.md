# Spark 调优实战：倾斜构造、优化对比与 Flink 映射

## 一、构造数据倾斜场景

### 1.1 创建倾斜数据集

```python
from pyspark.sql import SparkSession
from pyspark.sql import functions as F
import time

spark = SparkSession.builder \
    .appName("tuning-demo") \
    .master("local[4]") \
    .config("spark.ui.port", "4040") \
    .config("spark.sql.shuffle.partitions", "8") \
    .config("spark.sql.adaptive.enabled", "false")  # 先关闭 AQE，手动体验
    .getOrCreate()

spark.sparkContext.setLogLevel("WARN")

# === 构造倾斜的订单表 ===
# user_id=0 占 60% 数据，其余 999 个用户平分 40%
from pyspark.sql.types import *
import random

hot_data = [(i, 0, round(random.uniform(10, 500), 2)) for i in range(600000)]
normal_data = [(i + 600000, (i % 999) + 1, round(random.uniform(10, 500), 2)) 
               for i in range(400000)]

skewed_orders = spark.createDataFrame(
    hot_data + normal_data,
    ["order_id", "user_id", "amount"]
).repartition(8)
skewed_orders.cache()
print(f"Skewed orders count: {skewed_orders.count()}")

# 验证倾斜程度
skewed_orders.groupBy("user_id").count() \
    .orderBy(F.desc("count")).show(5)
# +-------+------+
# |user_id| count|
# +-------+------+
# |      0|600000|  ← 热点！
# |      1|   401|
# |      2|   401|
# ...

# === 用户维表（小表）===
users = spark.createDataFrame(
    [(uid, f"user_{uid}") for uid in range(1000)],
    ["user_id", "user_name"]
)
users.cache()
users.count()
```

## 二、调优技术 1：Broadcast Join

### 2.1 未优化的 Shuffle Join

```python
# 强制关闭 broadcast，走 shuffle join
spark.conf.set("spark.sql.autoBroadcastJoinThreshold", "-1")

start = time.time()
result_shuffle = skewed_orders.join(users, on="user_id", how="inner")
cnt = result_shuffle.count()
shuffle_time = time.time() - start
print(f"Shuffle Join: {shuffle_time:.2f}s, rows={cnt}")
```

在 Spark UI 中观察：
- Stage 数：3（scan orders + scan users + join）
- Shuffle Read/Write 较大
- 某个 Task 因热点 key 明显慢

### 2.2 Broadcast Join 优化

```python
# 开启 broadcast
spark.conf.set("spark.sql.autoBroadcastJoinThreshold", "10MB")

start = time.time()
result_broadcast = skewed_orders.join(users, on="user_id", how="inner")
cnt = result_broadcast.count()
broadcast_time = time.time() - start
print(f"Broadcast Join: {broadcast_time:.2f}s, rows={cnt}")

# 或手动指定
from pyspark.sql.functions import broadcast
result_manual = skewed_orders.join(broadcast(users), on="user_id", how="inner")
```

**预期效果**：
```
Shuffle Join:   8.5s    ← 有 shuffle
Broadcast Join: 3.2s    ← 无 shuffle，快 2-3x
```

### 2.3 原理说明

```
Shuffle Join:                          Broadcast Join:
┌──────────┐    Shuffle    ┌────────┐  ┌──────────┐  Broadcast  ┌────────┐
│ Orders   │══════════════►│ Join   │  │ Orders   │────────────►│ Join   │
│ (大表)    │               │ Task   │  │ (大表)    │  (无需shuffle)│ Task   │
└──────────┘               └────────┘  └──────────┘             └────────┘
┌──────────┐    Shuffle         ▲       ┌──────────┐                 ▲
│ Users    │════════════════════┘       │ Users    │ 广播到每个Executor│
│ (小表)    │                            │ (小表)    │ 内存中Hash Table │
└──────────┘                            └──────────┘
```

> **面试话术**：Broadcast Join 是大小表关联的首选优化。将小表（通常 < 100MB）广播到所有 Executor 的内存中，大表侧无需 shuffle 即可本地完成 Join。代价是广播本身的网络传输和小表的内存占用。阈值通过 `autoBroadcastJoinThreshold` 控制，生产环境可适当提高到 50-100MB。

## 三、调优技术 2：Salting 加盐打散

### 3.1 倾斜的 groupBy

```python
# 未优化的 groupBy
start = time.time()
result_naive = skewed_orders.groupBy("user_id").agg(
    F.sum("amount").alias("total_amount"),
    F.count("*").alias("order_cnt")
)
naive_rows = result_naive.collect()
naive_time = time.time() - start
print(f"Naive groupBy: {naive_time:.2f}s")
# 在 UI 中可以看到一个 Task 耗时远超其他
```

### 3.2 Salting 优化

```python
NUM_SALTS = 10

# Step 1: 加随机盐
salted = skewed_orders.withColumn(
    "salt",
    (F.rand() * NUM_SALTS).cast("int")
).withColumn(
    "salted_key",
    F.concat_ws("_", F.col("user_id").cast("string"), F.col("salt").cast("string"))
)

# Step 2: 按 salted_key 局部聚合
partial = salted.groupBy("salted_key").agg(
    F.sum("amount").alias("partial_sum"),
    F.count("*").alias("partial_cnt")
)

# Step 3: 去盐 + 全局聚合
final = partial.withColumn(
    "user_id",
    F.split("salted_key", "_")[0].cast("long")
).groupBy("user_id").agg(
    F.sum("partial_sum").alias("total_amount"),
    F.sum("partial_cnt").alias("order_cnt")
)

start = time.time()
salted_rows = final.collect()
salted_time = time.time() - start
print(f"Salted groupBy: {salted_time:.2f}s")
```

**预期效果**：
```
Naive groupBy:   6.8s    ← 一个 Task 处理 60万条
Salted groupBy:  3.1s    ← 分散到 10 个分区并行
```

### 3.3 Salting 原理图

```
原始数据:                           加盐后:
user_0: ████████████████ (60%)     user_0_0: ████
user_1: █                           user_0_1: ████
user_2: █                           user_0_2: ████
                                    ...
                                    user_0_9: ████
                                    user_1_3: █
                                    user_2_7: █

单 Task 瓶颈                       均匀分散到多个 Task
```

## 四、调优技术 3：两阶段聚合

### 4.1 适用场景

当 `countDistinct` 等不可分解聚合遇到倾斜时，salting 不能简单 sum 局部结果。

```python
# countDistinct 倾斜场景
# 直接 countDistinct 会在 shuffle 后把所有 item 聚到一个 Task
start = time.time()
naive_distinct = skewed_orders.groupBy("user_id").agg(
    F.countDistinct("order_id").alias("distinct_orders")
)
naive_distinct.show(5)
naive_dt = time.time() - start
print(f"Naive countDistinct: {naive_dt:.2f}s")
```

### 4.2 两阶段实现

```python
NUM_SALTS = 10

# Phase 1: 加盐 + collect_set（局部去重）
salted = skewed_orders.withColumn(
    "salt", (F.rand() * NUM_SALTS).cast("int")
).withColumn(
    "salted_key",
    F.concat_ws("_", F.col("user_id").cast("string"), F.col("salt").cast("string"))
)

partial = salted.groupBy("salted_key").agg(
    F.collect_set("order_id").alias("partial_items")
)

# Phase 2: 去盐 + 合并 set + size
from pyspark.sql.types import ArrayType, LongType
final = partial.withColumn(
    "user_id", F.split("salted_key", "_")[0].cast("long")
).groupBy("user_id").agg(
    F.size(F.flatten(F.collect_list("partial_items"))).alias("approx_distinct")
)

start = time.time()
final.show(5)
two_phase_time = time.time() - start
print(f"Two-phase countDistinct: {two_phase_time:.2f}s")
```

> ⚠️ `collect_set` + `flatten` 在大基数下可能 OOM。生产中大基数 countDistinct 推荐用 `approx_count_distinct`（HyperLogLog）。

## 五、调优技术 4：AQE 自动优化

### 5.1 开启 AQE 对比

```python
# 重新创建 session 开启 AQE
spark_aqe = SparkSession.builder \
    .appName("aqe-demo") \
    .master("local[4]") \
    .config("spark.sql.shuffle.partitions", "200")  # 故意设大
    .config("spark.sql.adaptive.enabled", "true") \
    .config("spark.sql.adaptive.coalescePartitions.enabled", "true") \
    .config("spark.sql.adaptive.skewJoin.enabled", "true") \
    .getOrCreate()

# 同样的倾斜 join
start = time.time()
result = skewed_orders.join(users, on="user_id").groupBy("user_id").agg(
    F.sum("amount")
)
result.count()
aqe_time = time.time() - start
print(f"AQE enabled: {aqe_time:.2f}s")
```

### 5.2 AQE 效果观察

在 Spark UI → SQL Tab 中可以看到：
- ✅ "Adaptive Query Execution" 标记
- ✅ 初始 200 个分区被合并为实际需要的数量
- ✅ 倾斜分区被拆分处理
- ✅ Join 策略可能被动态调整

## 六、四种调优技术对比总结

| 技术 | 解决问题 | 改动量 | 效果 | 适用条件 |
|------|---------|--------|------|---------|
| **Broadcast Join** | 大小表 Join Shuffle | ⭐ | 2-5x | 小表 < 内存阈值 |
| **Salting** | GroupBy/Join 倾斜 | ⭐⭐⭐ | 2-10x | 热点 key 可识别 |
| **两阶段聚合** | countDistinct 倾斜 | ⭐⭐⭐⭐ | 3-10x | 不可分解聚合 |
| **AQE** | 通用倾斜+小分区 | ⭐ | 1.5-5x | Spark ≥ 3.0 |

### 调优组合策略

```
实际问题往往需要组合使用：

1. 先开 AQE → 解决大部分通用问题
2. 大小表 Join → 确认 broadcast threshold 合理
3. 仍有倾斜 → 定位热点 key → salting / 两阶段
4. 小文件 → coalesce + AQE 合并
5. GC 频繁 → 增大内存 / off-heap / G1GC
```

## 七、Flink 两阶段聚合 → Spark Salting 映射

### 7.1 Flink 的两阶段聚合

```java
// Flink 中的两阶段聚合（解决 keyBy 后的数据倾斜）
DataStream<Result> result = stream
    // Phase 1: 加随机前缀做局部预聚合
    .keyBy(e -> e.key + "_" + random.nextInt(10))
    .window(TumblingEventTimeWindows.of(Time.minutes(5)))
    .aggregate(new MyAggFunction())
    
    // Phase 2: 去掉前缀做全局聚合
    .keyBy(e -> extractOriginalKey(e.key))
    .window(TumblingEventTimeWindows.of(Time.minutes(5)))
    .aggregate(new MyFinalAggFunction());
```

### 7.2 对应 Spark 实现

```python
# Spark 中等价的 salting 两阶段
NUM_SALTS = 10

# Phase 1: 局部聚合
partial = df.withColumn("salt", (F.rand() * NUM_SALTS).cast("int")) \
    .groupBy("key", "salt") \
    .agg(F.sum("value").alias("partial_sum"))

# Phase 2: 全局聚合
final = partial.groupBy("key") \
    .agg(F.sum("partial_sum").alias("total_sum"))
```

### 7.3 对比分析

| 维度 | Flink 两阶段聚合 | Spark Salting |
|------|-----------------|---------------|
| **触发方式** | 开发者手动实现 | 手动或 AQE 自动 |
| **窗口对齐** | 需保证两阶段窗口一致 | 不涉及窗口概念 |
| **状态管理** | 每阶段独立状态 | 中间结果走 shuffle |
| **增量 vs 全量** | 增量计算 | 全量批次计算 |
| **复杂度** | 较高（窗口语义） | 较低（纯数据处理） |
| **自动化** | Flink 无内置自动解倾斜 | AQE 可自动处理 |

> **面试话术**：Flink 和 Spark 解决数据倾斜的核心思想是一样的——"分而治之"。Flink 叫两阶段聚合，Spark 叫 salting。本质都是给热点 key 加随机前缀打散到多个并行实例，先做局部聚合再做全局聚合。区别在于 Flink 是流式的增量计算，需要维护两个阶段的状态；Spark 是批式的全量计算，中间结果通过 shuffle 传递。Spark 3.x 的 AQE 可以自动检测和处理倾斜，比 Flink 的手动两阶段更方便，但在流式场景下 Flink 的方案更精细。
