# Spark 原理：架构、抽象演进与生态定位

## 一、Spark 在大数据体系中的位置

Spark 是 Apache 基金会下的**分布式计算引擎**，核心定位是"基于内存的批处理优先（batch-first）计算框架"。它并非存储系统，也不是消息队列，而是一个纯粹的**计算层**。

```
┌─────────────────────────────────────────────────────┐
│                   应用层 (Application)                │
│   ETL / Ad-hoc Query / ML / Graph / Streaming        │
├─────────────────────────────────────────────────────┤
│              Spark 计算引擎                           │
│  ┌──────────┬──────────┬──────────┬────────┬───────┐│
│  │Spark Core│Spark SQL │Structured│ MLlib  │GraphX ││
│  │          │          │Streaming │        │       ││
│  ├──────────┴──────────┴──────────┴────────┴───────┤│
│  │         Catalyst / Tungsten / AQE               ││
│  ├─────────────────────────────────────────────────┤│
│  │      RDD / DataFrame / Dataset 抽象层            ││
│  ├─────────────────────────────────────────────────┤│
│  │    Scheduler (DAG + Task) + Memory Manager      ││
│  └─────────────────────────────────────────────────┘│
├─────────────────────────────────────────────────────┤
│           Cluster Manager                            │
│     YARN / Standalone / K8s / Mesos                  │
├─────────────────────────────────────────────────────┤
│           Storage Layer                              │
│     HDFS / S3 / OSS / Hive Metastore / Kafka        │
└─────────────────────────────────────────────────────┘
```

> **面试话术**：Spark 是一个纯计算引擎，不做存储也不做调度资源分配（由 Cluster Manager 负责）。它的核心价值在于：利用内存缓存中间结果，减少磁盘 IO；提供统一的高层抽象（DataFrame/Dataset），让开发者用声明式 API 表达计算逻辑，由 Catalyst 优化器自动选择最优执行计划。

## 二、核心架构：Driver / Executor / Cluster Manager

### 2.1 三大角色

| 角色 | 职责 | 进程数 | 典型配置 |
|------|------|--------|----------|
| **Driver** | 运行 main()，构建 DAG，调度 Task，收集结果 | 1 | spark.driver.memory=4g |
| **Executor** | 执行 Task，缓存数据到内存/磁盘，汇报状态 | N（动态/静态） | spark.executor.memory=8g, cores=4 |
| **Cluster Manager** | 为 Driver 分配 Executor 容器，管理生命周期 | 独立组件 | YARN RM / K8s API Server |

### 2.2 交互流程

```
用户提交 spark-submit
        │
        ▼
   ┌─────────┐    申请资源     ┌──────────────────┐
   │  Driver  │◄──────────────►│  Cluster Manager  │
   │ (main()) │                │  (YARN/K8s/...)   │
   └────┬─────┘                └────────┬─────────┘
        │                               │
        │  分发 Task                     │ 启动 Executor
        ▼                               ▼
   ┌──────────────────────────────────────────┐
   │            Executors (N个)                 │
   │  ┌──────────┐ ┌──────────┐ ┌──────────┐ │
   │  │Executor-1│ │Executor-2│ │Executor-N│ │
   │  │ Cache+Task│ │Cache+Task│ │Cache+Task│ │
   │  └──────────┘ └──────────┘ └──────────┘ │
   └──────────────────────────────────────────┘
        │
        │ 结果汇报 / Shuffle
        ▼
   ┌─────────┐
   │  Driver  │ ← 收集最终结果、更新 DAG 状态
   └─────────┘
```

### 2.3 Deploy Mode

| 模式 | Driver 位置 | 适用场景 | 注意事项 |
|------|------------|---------|---------|
| **Client** | 提交机本地 | 交互式调试、spark-shell | Driver 挂了任务就没了 |
| **Cluster** | Cluster Manager 内部某个节点 | 生产作业 | Driver 有 HA，但日志需从集群拉取 |

> **面试话术**：生产环境推荐 Cluster 模式，因为 Driver 跑在集群内，提交机器断开不影响任务。Client 模式适合开发调试，因为能直接看到 stdout。

## 三、核心抽象演进：RDD → DataFrame → Dataset

### 3.1 RDD（Resilient Distributed Dataset）

- **定义**：不可变、分区、可并行计算的弹性数据集
- **特点**：类型安全（泛型）、惰性求值、血统容错
- **局限**：无 schema 信息，Catalyst 无法优化；序列化开销大（Java Serialization）

```scala
// RDD 风格 WordCount
sc.textFile("hdfs:///data/words.txt")
  .flatMap(_.split(" "))
  .map((_, 1))
  .reduceByKey(_ + _)
  .collect()
```

### 3.2 DataFrame

- **定义**：带 schema 信息的分布式数据集合，等价于关系表
- **优势**：Catalyst 优化器可介入；Tungsten 二进制序列化；支持多种数据源
- **局限**：编译期不检查列名/类型（运行时才报错）

```scala
// DataFrame 风格
val df = spark.read.parquet("hdfs:///data/words.parquet")
df.groupBy("word").count().show()
```

### 3.3 Dataset

- **定义**：兼具 RDD 的类型安全 + DataFrame 的查询优化
- **仅 Scala/Java 可用**（Python 的 DataFrame 本质上就是 Dataset[Row]）
- **Encoder**：将 JVM 对象 ↔ Tungsten 内部格式互相转换

```scala
case class Word(w: String)
val ds: Dataset[Word] = spark.read.parquet("...").as[Word]
ds.groupBy(_.w).count().show()  // 编译期类型检查 + Catalyst 优化
```

### 3.4 三者对比

| 维度 | RDD | DataFrame | Dataset |
|------|-----|-----------|---------|
| Schema | ❌ | ✅ | ✅ |
| 类型安全 | ✅ | ❌ | ✅ |
| Catalyst 优化 | ❌ | ✅ | ✅ |
| Tungsten 编码 | ❌ | ✅ | ✅ |
| Python 支持 | ✅ | ✅ | ❌（等同 DF） |
| 推荐程度 | 底层/自定义算子 | ETL/SQL | Scala 业务代码 |

> **面试话术**：实际项目中优先使用 DataFrame/Dataset API，因为 Catalyst 可以做谓词下推、列裁剪、Join Reorder 等优化。RDD 仅在需要自定义 partitioner、复杂 mapPartitions 逻辑时使用。

## 四、Spark vs Flink：Micro-batch-first vs Stream-first

这是 L2 级别必须讲清楚的核心取舍。

### 4.1 计算模型本质差异

```
Spark（Micro-batch）:
  数据被切成一个个小批次，每个批次当作一个 mini-job 处理
  ──[batch1]──[batch2]──[batch3]──►
  延迟 = batch interval + 处理时间（秒~分钟级）

Flink（Native Streaming / Event-by-event）:
  每条数据到达即处理，watermark 驱动窗口
  ──e1──e2──e3──e4──e5──►
  延迟 = 处理时间（毫秒~亚秒级）
```

### 4.2 核心维度对比

| 维度 | Spark | Flink |
|------|-------|-------|
| **计算模型** | Micro-batch（Structured Streaming）/ Batch | Native event-driven streaming + Batch |
| **延迟** | 秒~分钟级 | 毫秒~亚秒级 |
| **状态管理** | Checkpoint + WAL（基于 RDD lineage） | RocksDB/Heap State + Incremental Checkpoint |
| **Exactly-once** | Structured Streaming 端到端（依赖 sink 幂等/事务） | Source→Sink 全链路 exactly-once（两阶段提交） |
| **SQL** | Spark SQL（成熟、Hive 兼容好） | Flink SQL（流批统一语法，CDC 原生支持） |
| **生态** | MLlib、GraphX、海量 connector | CDC、CEP、丰富 sink/source connector |
| **运维复杂度** | 中等（YARN/K8s 成熟） | 较高（State 管理、Savepoint 升级） |
| **适用场景** | 离线 ETL、数仓分析、ML | 实时指标、CDC、风控、事件驱动 |

### 4.3 "怎么选"的决策树

```
你的需求是什么？
    │
    ├── 纯离线 ETL / 数仓报表 / ML训练
    │       → Spark（成熟、吞吐高、生态完善）
    │
    ├── 实时指标 / 实时监控 / CDC入仓
    │       → Flink（低延迟、精确一次、状态强大）
    │
    ├── 既要离线又要实时，且数据量不大
    │       → Structured Streaming（一套代码兼顾）
    │
    ├── 既要离线又要实时，数据量大且要求严格一致
    │       → Flink 实时 + Spark 离线 + 湖表统一
    │         （或 Paimon/Iceberg 流批一体方案）
    │
    └── 图计算 / 迭代算法
            → Spark GraphX 或专用图引擎
```

> **面试话术**：选型不是"谁更好"，而是"谁更适合当前场景"。Spark 的优势在于批处理的吞吐和成熟的 SQL/Hive 生态；Flink 的优势在于真正的逐条处理和强大的状态管理。如果团队同时有离线和实时需求，现在趋势是用湖表（Paimon/Iceberg）做统一存储层，上面分别用 Spark 做批分析、Flink 做实时写入，实现"存储统一、计算分离"。

## 五、Spark 生态全景

### 5.1 六大组件

```
┌──────────────────────────────────────────────────────┐
│                    Spark 生态                         │
├────────────┬────────────┬────────────┬───────────────┤
│ Spark Core │ Spark SQL  │ Structured │   MLlib       │
│ RDD/API    │ Catalyst   │ Streaming  │  ML Pipeline  │
│ Shuffle    │ Hive Meta  │ Micro-batch│  Estimator/   │
│ Cache      │ Parquet/ORC│ Watermark  │  Transformer  │
├────────────┼────────────┼────────────┼───────────────┤
│ Spark      │ GraphX     │            │               │
│ Streaming  │ Pregel/BSP │            │               │
│ (DStream,  │ VertexRDD  │            │               │
│ 已弃用)    │            │            │               │
└────────────┴────────────┴────────────┴───────────────┘
```

| 组件 | 核心能力 | 现状 |
|------|---------|------|
| **Spark Core** | RDD 抽象、Shuffle、Cache、Accumulator | 基石，所有上层组件的基础 |
| **Spark SQL** | DataFrame/Dataset、Catalyst、Hive 兼容 | 最常用组件，ETL 主力 |
| **Structured Streaming** | 基于 DataFrame 的流处理、Watermark | 替代 DStream，生产推荐 |
| **Spark Streaming (DStream)** | 老版微批流 | 已弃用，仅维护 |
| **MLlib** | 分布式 ML 算法库、Pipeline API | 特征工程常用，深度学习转 PyTorch/TF |
| **GraphX** | 图计算、Pregel 模型 | 小众，社区活跃度低 |

### 5.2 关键概念速查

| 概念 | 说明 |
|------|------|
| **Lazy Evaluation** | Transformation 不触发计算，Action 才真正执行 |
| **Lineage** | RDD 的血统链，用于故障恢复而非 checkpoint |
| **Wide/Narrow Dependency** | 决定 Stage 划分的关键 |
| **Catalyst** | Spark SQL 的查询优化器框架 |
| **Tungsten** | 二进制内存布局和代码生成项目 |
| **AQE** | Adaptive Query Execution，运行时动态优化 |
| **Shuffle** | 跨分区数据重分配，性能瓶颈所在 |

## 六、总结与面试要点

### 核心记忆点

1. **架构三角**：Driver（大脑）+ Executor（手脚）+ Cluster Manager（后勤）
2. **抽象进化**：RDD → DataFrame → Dataset，越高层优化空间越大
3. **定位清晰**：Spark = batch-first 计算引擎，不是存储、不是调度器
4. **vs Flink**：不是优劣之分，是 micro-batch vs event-driven 的模型差异
5. **生态完整**：Core + SQL + Streaming + MLlib 覆盖大部分数据工程场景

### 常见面试追问

- **Q: Spark 为什么比 MapReduce 快？**
  A: 两个原因：① 中间结果缓存在内存而非落盘；② DAG 调度粒度更细，pipeline 化执行减少了不必要的 barrier。

- **Q: RDD 的容错机制是什么？**
  A: Lineage 重算。每个 RDD 记录父 RDD 及变换操作，分区丢失时沿血统链重新计算。对于 shuffle 后的 RDD，会回退到上一个 shuffle boundary 重算。

- **Q: Structured Streaming 和 Flink 的本质区别？**
  A: SS 本质是把流数据按 trigger interval 切成微批次，每个批次走一遍完整的 Spark job 流程。Flink 是逐条处理，状态常驻内存/RocksDB。所以 SS 延迟下限受限于批次间隔，而 Flink 可以做到毫秒级。
