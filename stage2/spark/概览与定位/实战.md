## 一、本地 Spark 环境搭建

### 1.1 前置依赖

| 依赖 | 版本建议 | 安装方式 |
|------|---------|---------|
| Java JDK | 8 或 11（Spark 3.x 推荐 8/11） | `brew install openjdk@11` / sdkman |

| Scala | 2.12（Spark 3.x 默认） | sdkman |
| Spark | 3.4.x / 3.5.x | 手动下载或 brew |

### 1.2 macOS 安装步骤

```bash
# 1. 安装 Java
brew install openjdk@11
export JAVA_HOME=$(/usr/libexec/java_home -v 11)

# 2. 下载 Spark（以 3.5.1 为例）
wget https://dlcdn.apache.org/spark/spark-3.5.1/spark-3.5.1-bin-hadoop3.tgz
tar xzf spark-3.5.1-bin-hadoop3.tgz
export SPARK_HOME=/path/to/spark-3.5.1-bin-hadoop3
export PATH=$SPARK_HOME/bin:$SPARK_HOME/sbin:$PATH

# 3. 验证
spark-shell --version   # Scala REPL
pyspark --version       # Python REPL

# 4.（可选）pip 安装 pyspark 用于 IDE 开发
pip install pyspark==3.5.1
```

## 三、WordCount 全流程详解

### 3.1 PySpark 版本

```python
from pyspark.sql import SparkSession
from pyspark.sql import functions as F

spark = SparkSession.builder \
    .appName("WordCount-Demo") \
    .master("local[2]") \
    .getOrCreate()

# ========== 方式一：DataFrame API（推荐）==========
df = spark.read.text("data/words.txt")          # 每行一个 value 列
result = (df
    .select(F.explode(F.split("value", "\\s+")).alias("word"))
    .filter(F.col("word") != "")
    .groupBy("word")
    .count()
    .orderBy(F.desc("count"))
)
result.show(20, truncate=False)

# ========== 方式二：RDD API（理解原理用）==========
rdd_result = (spark.sparkContext
    .textFile("data/words.txt")
    .flatMap(lambda line: line.split())
    .map(lambda word: (word, 1))
    .reduceByKey(lambda a, b: a + b)
    .sortBy(lambda x: x[1], ascending=False)
)
for item in rdd_result.take(20):
    print(item)

# ========== 方式三：Spark SQL ==========
df.createOrReplaceTempView("words")
spark.sql("""
    SELECT word, COUNT(*) as cnt
    FROM (
        SELECT explode(split(value, '\\s+')) as word
        FROM words
    ) t
    WHERE word != ''
    GROUP BY word
    ORDER BY cnt DESC
""").show(20, truncate=False)

spark.stop()
```

### 3.2 执行过程拆解

```
textFile / read.text
       │
       ▼
┌──────────────┐
│  flatMap /   │ ← Narrow Dependency（同分区内操作）
│  split       │
└──────┬───────┘
       │
       ▼
┌──────────────┐
│  map(w, 1)   │ ← Narrow Dependency
└──────┬───────┘
       │
       ▼
┌──────────────┐
│ reduceByKey  │ ← ★ Wide Dependency → Shuffle Boundary
│              │   Stage 切分点！
└──────┬───────┘
       │
       ▼
┌──────────────┐
│  sortBy      │ ← Wide Dependency → 又一次 Shuffle
└──────┬───────┘
       │
       ▼
   collect/take ← Action 触发整个 DAG 执行
```

**关键观察**：
- `flatMap`、`map` 是窄依赖，可以 pipeline 在同一个 Task 里执行
- `reduceByKey` 是宽依赖，必须 shuffle，这里产生一个 Stage 边界
- `sortBy` 又是宽依赖，再产生一个 Stage
- 整个 Job 被分成 **3 个 Stage**

## 五、Spark vs Flink 详细对比表

| 维度 | Spark | Flink | 选型启示 |
|------|-------|-------|---------|
| **计算模型** | Micro-batch（批优先） | Event-driven（流优先） | 延迟敏感选 Flink |
| **最低延迟** | ~100ms（SS trigger） | ~ms 级 | 风控/实时告警选 Flink |
| **吞吐量** | 极高（批量优化） | 高（但单条处理有开销） | 纯离线大吞吐选 Spark |
| **状态管理** | RDD Lineage + Checkpoint | KeyedState + RocksDB + Incremental CP | 大状态场景 Flink 更稳 |
| **Exactly-once** | Sink 端保证（幂等/事务） | Source→Sink 全链路 2PC | 金融级一致性选 Flink |
| **SQL 成熟度** | 非常成熟，Hive 兼容极好 | 快速追赶，CDC 原生支持 | 数仓 ETL 选 Spark SQL |
| **ML 能力** | MLlib 内置 | 无（需外部对接） | ML Pipeline 选 Spark |
| **图计算** | GraphX | 无（Gelly 已废弃） | 图算法选 Spark 或专用引擎 |
| **CDC 支持** | 有限（需第三方） | 原生 CDC Connector | 数据库同步选 Flink |
| **CEP** | 无 | 强大（Pattern API + SQL MATCH_RECOGNIZE） | 复杂事件处理选 Flink |
| **运维** | YARN/K8s 成熟 | StatefulSet + Savepoint 较复杂 | 团队经验也是考量因素 |
| **社区/文档** | 极其丰富 | 活跃但中文资源偏少 | 新人上手 Spark 更快 |

## 六、面试话术："Spark vs Flink 怎么选"

### 6.1 标准回答框架

> **面试官**：你们项目中 Spark 和 Flink 是怎么选的？

**参考回答**：

> 我们选型的核心原则是 **"按场景匹配计算模型"**，而不是简单比较谁更好。
>
> **第一，看延迟要求**。如果是秒级以上的统计报表、离线 ETL，我们用 Spark，因为它的 micro-batch 模型吞吐更高，SQL 生态也更成熟。如果是毫秒级的实时告警、风控规则引擎，我们用 Flink，因为它是真正的逐条处理，延迟下限远低于 Spark。
>
> **第二，看状态复杂度**。Flink 的 KeyedState + RocksDB 后端能支撑 TB 级状态，增量 checkpoint 也比 Spark 的全量 checkpoint 更适合长窗口聚合。如果业务涉及超长会话窗口、大量用户画像状态，优先 Flink。
>
> **第三，看一致性要求**。Flink 的端到端 exactly-once 通过两阶段提交实现，source 到 sink 全链路保证。Spark Structured Streaming 的 exactly-once 更多依赖 sink 端的幂等写入或事务机制。金融级对账场景我们倾向 Flink。
>
> **第四，看团队能力和运维成本**。Spark 的运维体系更成熟，YARN 上跑了很多年，问题排查资料也多。Flink 的状态管理和 savepoint 升级需要更多经验积累。如果团队刚起步，先用 Spark Structured Streaming 过渡也是合理选择。
>
> **最后，我们现在也在推进湖表方案**（Paimon/Iceberg），让 Flink 负责实时写入湖表，Spark 负责离线分析同一张表，这样在存储层就实现了统一，减少了 Lambda 架构下两套管道的维护成本。

### 6.2 加分项话术

- **关于 Structured Streaming**："SS 本质还是微批，只是把批次间隔缩到了 trigger interval。对于延迟要求在秒级、状态不大的场景，它够用且开发成本低。但如果窗口超过小时级、状态达到 GB 级，SS 的 checkpoint 恢复会很慢，这时候就该换 Flink。"

- **关于统一趋势**："现在行业趋势是'存储统一、计算分离'。用 Paimon 或 Iceberg 做统一存储层，Flink 写、Spark 读，避免维护两套数据管道。这不是说 Spark 和 Flink 要合并，而是让它们各自发挥优势，通过存储层解耦。"

- **避免的说法**：❌ "Spark 比 Flink 快" ❌ "Flink 比 Spark 先进" ✅ "它们解决不同层次的问题，选型要看具体约束条件"
