# Spark 执行原理：Job→Stage→Task、Shuffle 与容错

## 一、执行模型总览

Spark 的执行模型可以用一句话概括：**一个 Action 触发一个 Job，Job 按 Shuffle 边界切分为多个 Stage，每个 Stage 内包含若干并行 Task**。

```
Action (collect/count/save...)
        │
        ▼
     ┌─────┐
     │ Job │ ← 一次 Action = 一个 Job
     └──┬──┘
        │ DAGScheduler 根据宽依赖切分
        ▼
   ┌─────────┐  ┌─────────┐  ┌─────────┐
   │ Stage 0 │→ │ Stage 1 │→ │ Stage 2 │ ← ShuffleMapStage / ResultStage
   └────┬────┘  └────┬────┘  └────┬────┘
        │            │            │
   ┌────┴────┐  ┌────┴────┐  ┌────┴────┐
   │Task│Task│  │Task│Task│  │Task│Task│ ← 每个分区一个 Task
   └─────────┘  └─────────┘  └─────────┘
```

## 二、Job → Stage → Task 划分详解

### 2.1 Job

- **触发条件**：任何一个 Action 操作（`collect`, `count`, `saveAsTextFile`, `foreach`, `take`...）
- **含义**：从最终 RDD 回溯到源头 RDD 的完整计算图
- **注意**：Transformation（`map`, `filter`, `flatMap`...）不会触发 Job，它们是惰性的

### 2.2 Stage

Stage 是 Spark 调度的核心单元，划分依据是**宽窄依赖**。

#### 窄依赖（Narrow Dependency）

父 RDD 的一个分区最多被子 RDD 的一个分区消费。

```
Parent RDD          Child RDD
┌─────────┐        ┌─────────┐
│Partition│───────►│Partition│   map / filter / flatMap
│   0     │        │   0     │   mapPartitions
├─────────┤        ├─────────┤
│Partition│───────►│Partition│
│   1     │        │   1     │
├─────────┤        ├─────────┤
│Partition│───────►│Partition│
│   2     │        │   2     │
└─────────┘        └─────────┘
```

特点：可以在同一个 Task 中 pipeline 执行，无需 shuffle。

#### 宽依赖（Wide Dependency / Shuffle Dependency）

父 RDD 的一个分区被子 RDD 的**多个**分区消费。

```
Parent RDD          Child RDD
┌─────────┐        ┌─────────┐
│Partition│──┬────►│Partition│
│   0     │  │     │   0     │
├─────────┤  │     ├─────────┤
│Partition│──┼────►│Partition│   groupByKey / reduceByKey
│   1     │  │     │   1     │   join (非 broadcast)
├─────────┤  │     ├─────────┤   sortByKey
│Partition│──┘     │Partition│
│   2     │───────►│   2     │
└─────────┘        └─────────┘
```

特点：必须经过 Shuffle，是 Stage 的切分边界。

### 2.3 Stage 类型

| 类型 | 说明 | 输出 |
|------|------|------|
| **ShuffleMapStage** | 有下游 Stage 依赖它，输出 Shuffle 文件 | MapOutputTracker 注册的 shuffle output |
| **ResultStage** | 最终的 Stage，直接产出 Action 结果 | 返回给 Driver |

### 2.4 Task

- 每个 Stage 中的 Task 数量 = 该 Stage 输出 RDD 的**分区数**
- Task 是 Executor 上实际执行的最小工作单元
- 两种 Task 类型：
  - **ShuffleMapTask**：执行 ShuffleMapStage，写 shuffle 文件
  - **ResultTask**：执行 ResultStage，返回结果给 Driver

## 三、调度器：DAGScheduler 与 TaskScheduler

### 3.1 DAGScheduler

```
Action 调用
    │
    ▼
DAGScheduler.submitJob()
    │
    ├── 1. 从目标 RDD 反向遍历依赖链
    ├── 2. 遇到 ShuffleDependency → 切出新 Stage
    ├── 3. 构建 Stage DAG（父子关系）
    ├── 4. 提交最上游的 ShuffleMapStage
    └── 5. 监听 Stage 完成事件，依次提交下游 Stage
```

核心数据结构：
- `stageIdToStage`: Map[StageId, Stage]
- `waitingStages`: 等待父 Stage 完成的 Stage 集合
- `runningStages`: 正在执行的 Stage 集合

### 3.2 TaskScheduler

- 接收 DAGScheduler 提交的 TaskSet（一个 Stage 的所有 Task）
- 通过 SchedulerBackend 将 Task 分发到各 Executor
- 处理 Task 失败重试、推测执行（Speculative Execution）
- 资源分配策略：FIFO / Fair

```
DAGScheduler                    TaskScheduler
    │                                │
    │ submitMissingTasks(stage)      │
    ├───────────────────────────────►│
    │                                │ resourceOffer(executorId)
    │                                ├────────────────► Executor
    │                                │
    │ statusUpdate(taskId, SUCCESS)  │
    │◄───────────────────────────────┤
    │                                │
    │ handleTaskCompletion()         │
    │ → 检查 Stage 是否全部完成       │
    │ → 完成则提交下游 Stage          │
```

### 3.3 推测执行（Speculative Execution）

当某个 Task 明显慢于同 Stage 其他 Task 的中位数时，TaskScheduler 会在另一个 Executor 上启动一个副本，谁先完成就用谁的结果。

```
spark.speculation=true
spark.speculation.multiplier=1.5   # 超过中位数 1.5 倍才触发
spark.speculation.quantile=0.75    # 以 75 分位数为基准
```

> **面试话术**：推测执行能缓解数据倾斜或个别节点慢的问题，但会增加资源消耗。生产环境建议开启，但要配合合理的 multiplier 避免误触发。

## 四、RDD Lineage 与容错

### 4.1 Lineage 机制

每个 RDD 记录：
1. **父 RDD 引用**（dependencies）
2. **变换函数**（compute function）
3. **分区器**（partitioner）
4. **首选位置**（preferredLocations）

当某个分区数据丢失时：

```
丢失: RDD-C Partition-2
    │
    ▼ 回溯 lineage
RDD-B Partition-2 → compute(partition2) 
    │
    ▼ 如果 B 也丢了，继续回溯
RDD-A Partition-2 → compute(partition2)
    │
    ▼ 直到找到可用数据源或 checkpoint
```

### 4.2 Checkpoint vs Cache

| 维度 | Cache/Persist | Checkpoint |
|------|--------------|------------|
| 存储位置 | 内存/磁盘（Executor 本地） | HDFS/S3（可靠存储） |
| 触发时机 | 惰性，首次使用时 | 显式调用 `rdd.checkpoint()` |
| 生命周期 | Job 结束自动清除 | 需手动清理 |
| 用途 | 加速重复使用 | 截断 lineage，防止栈溢出 |
| 容错 | Executor 挂了需重算 | 直接从 HDFS 读取恢复 |

> **面试话术**：Cache 是为了性能，Checkpoint 是为了可靠性。长链路迭代算法（如 PageRank）一定要做 checkpoint，否则 lineage 太长会导致 StackOverflowError 或恢复时间过长。

### 4.3 Shuffle 文件的容错

Shuffle 输出文件由 ShuffleMapTask 写入本地磁盘。如果该 Executor 挂了：
- MapOutputTracker 检测到 loss
- DAGScheduler 重新提交对应的 ShuffleMapStage
- 下游 Stage 等待新的 shuffle 文件就绪后才继续

## 五、Shuffle 机制详解

Shuffle 是 Spark 性能的**核心瓶颈**，理解它是调优的基础。

### 5.1 Shuffle 流程

```
              Shuffle Write                    Shuffle Read
         ┌──────────────────┐            ┌──────────────────┐
Stage N  │                  │            │                  │  Stage N+1
         │  Map Output      │  Network   │  Shuffle Reader  │
         │  File per reducer│◄═══════════►│  Merge + Sort    │
         │                  │            │                  │
         └──────────────────┘            └──────────────────┘
         
         每个 Map Task 为每个               每个 Reduce Task
         Reducer 生成一个                   拉取所有上游 Map
         block 文件                         对应自己的 block
```

### 5.2 三种 Shuffle Manager

| 管理器 | Spark 版本 | 特点 | 状态 |
|--------|-----------|------|------|
| **HashShuffleManager** | < 1.2 | 每个 Task 为每个 Reducer 创建一个文件 | ❌ 已废弃 |
| **SortShuffleManager** | ≥ 1.2 | 先排序再写文件，减少文件数 | ✅ 默认 |
| **Tungsten-Sort** | ≥ 1.5 | 在序列化二进制上排序，避免反序列化 | ✅ 自动启用 |

### 5.3 Sort-Based Shuffle 详细流程

```
Map Task 输出记录
       │
       ▼
┌──────────────────┐
│ AppendOnlyMap    │ ← 内存中聚合（如果有 combine）
│ (in-memory sort) │
└────────┬─────────┘
         │ 内存不足时 spill
         ▼
┌──────────────────┐
│ Spill to Disk    │ ← 每次 spill 生成一个临时文件
│ (sorted runs)    │
└────────┬─────────┘
         │ 所有数据处理完
         ▼
┌──────────────────┐
│ External Sorter  │ ← 多路归并所有 spill 文件
│ (merge sort)     │
└────────┬─────────┘
         │
         ▼
┌──────────────────┐
│ Single Data File │ ← 按 reducer partition 顺序排列
│ + Index File     │   Index 记录每个 partition 的偏移量
└──────────────────┘
```

### 5.4 Bypass Merge Sort Shuffle

当满足以下条件时跳过排序，直接写 Hash 分区文件：
- `reduceByKey` 等无聚合操作的 shuffle
- 分区数 ≤ `spark.shuffle.sort.bypassMergeThreshold`（默认 200）
- 没有 map-side combine

优势：省去排序开销，写入更快。劣势：产生更多小文件。

### 5.5 Tungsten-Sort 优化

- 直接在序列化的二进制字节上比较 key（不反序列化为 JVM 对象）
- 利用 off-heap 内存，减少 GC 压力
- 缓存友好的内存布局

## 六、内存模型

### 6.1 Executor 内存布局

```
┌─────────────────────────────────────────────────┐
│              Executor Memory                     │
├────────────────────┬────────────────────────────┤
│                    │                            │
│   Execution Memory │      Storage Memory        │
│   (动态共享)        │      (动态共享)             │
│                    │                            │
│   • Shuffle Sort   │   • RDD Cache              │
│   • Join Buffer    │   • Broadcast Variable     │
│   • Aggregation    │   • Unroll Buffer          │
│                    │                            │
├────────────────────┴────────────────────────────┤
│              User Memory (自定义数据结构)          │
├─────────────────────────────────────────────────┤
│              Reserved Memory (300MB)             │
└─────────────────────────────────────────────────┘
```

### 6.2 统一内存管理（Unified Memory Management）

自 Spark 1.6 起，Execution 和 Storage 内存**动态共享**：
- `spark.memory.fraction`：堆内可用于 exec+storage 的比例（默认 0.6）
- `spark.memory.storageFraction`：storage 的保底比例（默认 0.5）
- 当一方空闲时，另一方可借用；当被借方需要时，逐步回收

```
初始状态:
Exec: [====30%====]
Stor: [====30%====]

Storage 空闲，Execution 借用:
Exec: [========50%========]
Stor: [==10%==]

Storage 需要回收:
Exec: [======40%======]
Stor: [====20%====]
```

### 6.3 OOM 常见原因

| 场景 | 原因 | 解决方案 |
|------|------|---------|
| Driver OOM | collect 大量数据、广播变量过大 | 避免 collect，用 take/sample |
| Executor OOM | 单分区数据过大、cache 过多 | 增加分区数、调整内存比例 |
| Metaspace OOM | 类加载过多（代码生成） | `-XX:MaxMetaspaceSize` |
| Overhead OOM | 网络缓冲/NIO 直接内存 | `spark.executor.memoryOverhead` |

## 七、总结与面试要点

### 核心记忆点

1. **三级划分**：Action→Job→Stage(按宽依赖切)→Task(按分区数)
2. **两类依赖**：窄依赖=pipeline，宽依赖=shuffle boundary
3. **两个调度器**：DAGScheduler(逻辑划分) + TaskScheduler(物理分发)
4. **容错双保险**：Lineage 重算 + Checkpoint 截断
5. **Shuffle 是瓶颈**：Sort-based 是默认，Tungsten 优化序列化，Bypass 省排序
6. **内存动态共享**：Execution ↔ Storage 弹性借用

### 常见面试追问

- **Q: 为什么 reduceByKey 比 groupByKey 好？**
  A: reduceByKey 有 map-side combine，在 shuffle 前先局部聚合，大幅减少 shuffle 数据量。groupByKey 不做预聚合，所有原始 KV 都要过网络。

- **Q: Shuffle 文件存在哪里？Executor 挂了怎么办？**
  A: 存在 Executor 本地磁盘（`spark.local.dir`）。Executor 挂了，DAGScheduler 会重新提交该 ShuffleMapStage，下游 Task 等待新文件就绪。这就是为什么 Shuffle 文件丢失不需要从头算，只需回退到上一个 Shuffle 边界。

- **Q: Spark 的内存管理和 Flink 有什么区别？**
  A: Spark 基于 JVM Heap 的动态共享模型，受 GC 影响大。Flink 有自己的 MemorySegment 管理机制，可以精细控制堆内/堆外/网络/Managed 内存，GC 压力更小，状态管理更高效。这也是 Flink 在大状态场景下优于 Spark 的底层原因之一。
