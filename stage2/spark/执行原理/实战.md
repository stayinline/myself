# Spark 执行原理实战：UI 分析、数据倾斜与 Shuffle 调优

## 一、运行 groupBy / Join 作业并观察 Stage

### 1.1 构造测试数据

```python
from pyspark.sql import SparkSession
from pyspark.sql import functions as F

spark = SparkSession.builder \
    .appName("execution-demo") \
    .master("local[4]") \
    .config("spark.ui.port", "4040") \
    .config("spark.sql.shuffle.partitions", "8")  # 本地演示用小分区数
    .getOrCreate()

spark.sparkContext.setLogLevel("WARN")

# 生成订单表（100万行）
orders = spark.range(0, 1000000).select(
    F.col("id").alias("order_id"),
    (F.col("id") % 1000).alias("user_id"),
    (F.rand() * 1000).cast("decimal(10,2)").alias("amount"),
    F.date_add(F.lit("2024-01-01"), (F.col("id") % 365)).alias("order_date")
)

# 生成用户表（1000行）
users = spark.range(0, 1000).select(
    F.col("id").alias("user_id"),
    F.concat(F.lit("user_"), F.col("id")).alias("user_name")
)

# 缓存以便重复实验
orders.cache()
users.cache()
orders.count()  # 触发缓存
users.count()
```

### 1.2 groupBy 作业

```python
# groupBy 聚合 → 必然触发 shuffle
result = orders.groupBy("user_id").agg(
    F.count("*").alias("order_cnt"),
    F.sum("amount").alias("total_amount"),
    F.avg("amount").alias("avg_amount")
)
result.show()
```

**预期 Stage 划分**：

```
Stage 0: Scan + 部分聚合（map-side combine，如果启用）
   ↓ Shuffle
Stage 1: 最终聚合 + 结果输出
```

### 1.3 Join 作业

```python
# Shuffle Hash Join（默认大表 join）
joined = orders.join(users, on="user_id", how="inner")
joined.filter(F.col("total_amount") > 5000).show()

# Broadcast Join（小表 < spark.sql.autoBroadcastJoinThreshold）
# users 只有 1000 行，会自动 broadcast
spark.conf.set("spark.sql.autoBroadcastJoinThreshold", "10MB")
joined_bc = orders.join(users, on="user_id", how="inner")
joined_bc.explain(mode="formatted")  # 查看物理计划确认 BroadcastHashJoin
```

**核心指标解读**：

| 指标 | 含义 | 异常信号 |
|------|------|---------|
| Duration Min/Max | Task 耗时范围 | Max >> Min → 数据倾斜 |
| Shuffle Read Size | 从上游拉取的数据量 | 某 Task 远大于其他 → 热点 key |
| Shuffle Write Size | 写入下游的数据量 | 不均匀 → 上游分区不均 |
| GC Time | 垃圾回收时间 | GC 占比 > 30% → 内存不足 |
| Spill (Memory/Disk) | 内存溢出到磁盘 | 有 spill → 考虑增大内存或分区数 |

## 三、宽依赖 vs 窄依赖的性能影响实测

### 3.1 对比实验

```python
import time

# ===== 窄依赖链（无 shuffle）=====
start = time.time()
narrow_result = (orders
    .filter(F.col("amount") > 100)       # narrow
    .withColumn("tax", F.col("amount") * 0.1)  # narrow
    .withColumn("total", F.col("amount") + F.col("tax"))  # narrow
    .select("order_id", "total")          # narrow
)
narrow_count = narrow_result.count()      # Action 触发
narrow_time = time.time() - start
print(f"Narrow chain: {narrow_time:.2f}s, rows={narrow_count}")

# ===== 含宽依赖的操作 =====
start = time.time()
wide_result = (orders
    .groupBy("user_id")                   # wide → shuffle
    .agg(F.sum("amount").alias("total"))
)
wide_count = wide_result.count()
wide_time = time.time() - start
print(f"Wide (groupBy): {wide_time:.2f}s, rows={wide_count}")
```

**典型结果**（本地 4 核）：

```
Narrow chain: 1.2s, rows=850321
Wide (groupBy): 4.8s, rows=1000
```

> 结论：窄依赖操作可以 pipeline 化在一个 Stage 内完成；一旦引入宽依赖（shuffle），耗时会显著增加。

### 3.2 减少 Shuffle 的技巧

| 技巧 | 说明 | 适用场景 |
|------|------|---------|
| `reduceByKey` 替代 `groupByKey` | map-side combine | 聚合操作 |
| Broadcast Join | 避免 shuffle join | 大小表关联 |
| `repartition` 提前规划 | 避免多次隐式 shuffle | 后续多次 groupBy |
| AQE 自适应 | 运行时合并小分区/处理倾斜 | Spark 3.x 通用 |
| 预排序/预分区 | 数据已按 key 分布 | ETL 管道设计阶段 |

## 四、数据倾斜诊断与处理

### 4.1 诊断方法

```python
# 方法一：检查 key 分布
orders.groupBy("user_id").count().orderBy(F.desc("count")).show(10)
# 如果 top key 的 count 远超平均值 → 倾斜

# 方法二：Spark UI 看 Task 耗时分布
# Stage 详情中 Max Duration / Median Duration > 3x → 疑似倾斜

# 方法三：查看 Shuffle Read 分布
# 某个 Task 的 Shuffle Read >> 其他 Task → 热点 partition
```

### 4.2 构造倾斜数据

```python
# 人为制造倾斜：让 user_id=0 占 50% 的数据
skewed_orders = spark.createDataFrame(
    [(i, 0 if i < 500000 else (i % 999) + 1, float(i % 100)) 
     for i in range(1000000)],
    ["order_id", "user_id", "amount"]
)
skewed_orders.cache()
skewed_orders.count()

# 这个 groupBy 会严重倾斜
skewed_orders.groupBy("user_id").agg(F.sum("amount")).show()
# 在 UI 中观察：一个 Task 耗时远超其他
```

### 4.3 解决方案速查

| 方案 | 原理 | 适用条件 | 复杂度 |
|------|------|---------|--------|
| **Salting（加盐）** | 给热点 key 加随机前缀打散 | 聚合类操作 | ⭐⭐⭐ |
| **Broadcast Join** | 小表广播到所有节点 | 一大一小表 join | ⭐ |
| **两阶段聚合** | 先局部聚合再全局聚合 | groupBy/countDistinct | ⭐⭐⭐ |
| **过滤热点 key** | 单独处理热点，其余正常聚合 | 热点 key 可枚举 | ⭐⭐ |
| **AQE Skew Join** | Spark 3.x 自动检测并拆分倾斜分区 | Spark ≥ 3.0 | ⭐ |
| **调整分区数** | 增大 shuffle partitions | 轻度倾斜 | ⭐ |

## 五、Spark Shuffle vs Flink Network/Backpressure

### 5.1 架构差异

```
Spark Shuffle:
  Map Task 全部完成 → 写本地文件 → Reduce Task 拉取 → 处理
  ──────────────────────────────────────────────────►
  Batch 模式：上游写完，下游才读（barrier 同步）

Flink Network:
  上游 Subtask 持续发送 → Network Buffer → 下游 Subtask 持续消费
  ◄═══════ Pipeline 模式：上下游同时运行 ═══════►
  Credit-based flow control：下游向上游发 credit，控制发送速率
```

### 5.2 对比表

| 维度 | Spark Shuffle | Flink Network |
|------|--------------|---------------|
| **数据传输时机** | Barrier：上游完成后下游才开始 | Pipeline：上下游并行 |
| **中间存储** | 本地磁盘文件 | 内存 Buffer（溢写到磁盘） |
| **流控机制** | 拉取模式（pull），按需请求 | Credit-based push-pull 混合 |
| **背压** | 不存在（batch barrier 天然隔离） | 原生支持，反压到 source |
| **容错** | 重算上游 Stage | Checkpoint barrier + replay |
| **性能瓶颈** | 磁盘 IO + 序列化 | 网络带宽 + Buffer 大小 |
| **调优参数** | shuffle.partitions, memory.fraction | network.memory.*, buffer-size |

### 5.3 面试话术

> **Q: Spark 和 Flink 的数据传输有什么区别？**
>
> A: Spark 是 batch barrier 模型，Shuffle 时上游 Stage 必须全部完成后，下游才能开始拉取数据。这意味着集群资源在 Stage 切换时有空闲等待。Flink 是 pipeline 模型，上下游 subtask 同时运行，通过 credit-based 流控实现背压。当消费端变慢时，反压信号会沿着数据流一直传递到 source，source 降速读取外部数据。这种机制让 Flink 的资源利用率更高，但也增加了调试复杂度——背压问题可能出现在链路的任何环节。

### 附：快速排查 Checklist

```
□ Spark UI → Jobs Tab → 找到最慢的 Job
□ 进入该 Job 的 Stages → 找到最慢的 Stage
□ 查看 Summary Metrics → Min/Max 差距是否 > 3x
□ 如果有 → 数据倾斜 → 检查 Shuffle Read 分布
□ 如果 GC Time 占比高 → 内存不足 → 调 executor-memory 或分区数

□ 如果 Shuffle 次数过多 → 检查是否可以合并操作/broadcast join
```
