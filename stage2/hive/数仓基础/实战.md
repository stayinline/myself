## 二、创建分区表实操

### 2.1 创建外部表（ODS 层）

```sql
-- ODS 订单原始表
CREATE EXTERNAL TABLE IF NOT EXISTS ods_orders (
    order_id     BIGINT    COMMENT '订单ID',
    user_id      BIGINT    COMMENT '用户ID',
    amount       DECIMAL(10,2) COMMENT '订单金额',
    status       STRING    COMMENT '订单状态',
    create_time  TIMESTAMP COMMENT '创建时间'
)
COMMENT 'ODS-订单原始表'
PARTITIONED BY (dt STRING COMMENT '业务日期 yyyy-MM-dd')
ROW FORMAT DELIMITED
FIELDS TERMINATED BY ','
STORED AS TEXTFILE
LOCATION '/data/warehouse/ods/ods_orders';
```

### 2.2 创建内部表（DWD 层，Parquet 格式）

```sql
-- DWD 订单明细清洗表
CREATE TABLE IF NOT EXISTS dwd_orders_cleaned (
    order_id     BIGINT    COMMENT '订单ID',
    user_id      BIGINT    COMMENT '用户ID',
    amount       DECIMAL(10,2) COMMENT '订单金额',
    status       STRING    COMMENT '订单状态',
    create_time  TIMESTAMP COMMENT '创建时间'
)
COMMENT 'DWD-订单明细清洗表'
PARTITIONED BY (dt STRING COMMENT '业务日期')
STORED AS PARQUET
TBLPROPERTIES (
    'parquet.compression' = 'SNAPPY',
    'orc.bloom.filter.columns' = 'order_id,user_id'
);
```

### 2.3 创建分桶表

```sql
-- 按 user_id 分桶，优化后续 Join
CREATE TABLE IF NOT EXISTS dwd_orders_bucketed (
    order_id     BIGINT,
    user_id      BIGINT,
    amount       DECIMAL(10,2),
    status       STRING
)
PARTITIONED BY (dt STRING)
CLUSTERED BY (user_id) SORTED BY (create_time) INTO 16 BUCKETS
STORED AS ORC
TBLPROPERTIES ('transactional' = 'false');
```

> **注意**：分桶表的写入需要通过 `INSERT INTO` 或 Spark 的 bucket-aware writer，直接 COPY 文件不会自动分桶。

## 三、数据插入与查询

### 3.1 静态分区插入

```sql
-- 明确指定分区值
INSERT OVERWRITE TABLE dwd_orders_cleaned PARTITION (dt = '2024-06-01')
SELECT 
    order_id,
    user_id,
    amount,
    status,
    create_time
FROM ods_orders
WHERE dt = '2024-06-01'
  AND amount > 0
  AND status IN ('paid', 'pending');
```

### 3.2 动态分区插入

```sql
-- 开启动态分区
SET hive.exec.dynamic.partition = true;
SET hive.exec.dynamic.partition.mode = nonstrict;
SET hive.exec.max.dynamic.partitions = 1000;
SET hive.exec.max.dynamic.partitions.pernode = 500;

-- 分区列必须在 SELECT 的最后
INSERT OVERWRITE TABLE dwd_orders_cleaned PARTITION (dt)
SELECT 
    order_id,
    user_id,
    amount,
    status,
    create_time,
    dt          -- ← 分区列放最后！
FROM ods_orders
WHERE dt BETWEEN '2024-06-01' AND '2024-06-30';
```

> ⚠️ **常见坑**：
> 1. 分区列必须放在 SELECT 列表的**最后**
> 2. `nonstrict` 模式允许所有分区都是动态的；`strict` 模式要求至少一个静态分区
> 3. 动态分区过多可能触发 `max.dynamic.partitions` 限制

### 3.3 Spark 中写入 Hive 分区表

```python
# DataFrame 写入 Hive 分区表
df.write \
    .mode("overwrite") \
    .insertInto("dwd_orders_cleaned")

# 关键配置：动态分区覆盖模式
spark.conf.set("spark.sql.sources.partitionOverwriteMode", "dynamic")
# 不设这个的话，overwrite 会删除整个表再重写！
```

### 3.4 查询示例

```sql
-- ✅ 触发分区裁剪
SELECT user_id, SUM(amount) as total
FROM dwd_orders_cleaned
WHERE dt = '2024-06-01'
GROUP BY user_id
ORDER BY total DESC
LIMIT 10;

-- ✅ 范围裁剪
SELECT dt, COUNT(*) as cnt
FROM dwd_orders_cleaned
WHERE dt >= '2024-06-01' AND dt <= '2024-06-30'
GROUP BY dt;

-- ❌ 不触发裁剪（函数包裹分区列）
SELECT * FROM dwd_orders_cleaned
WHERE date_format(create_time, 'yyyy-MM-dd') = '2024-06-01';
-- 应改为 WHERE dt = '2024-06-01'
```

## 五、动态分区深入

### 5.1 工作原理

```
INSERT OVERWRITE ... PARTITION(dt) SELECT ..., dt FROM source

1. 执行 SELECT，得到结果集（含 dt 列）
2. 按 dt 值分组
3. 对每个 dt 值：
   a. 如果分区不存在 → 自动创建
   b. 写入该分区的数据文件
4. 更新 Metastore 分区信息
```

### 5.2 生产注意事项

| 注意点 | 说明 |
|--------|------|
| **分区数限制** | `max.dynamic.partitions` 默认 1000，超限报错 |
| **小文件问题** | 每个 Task × 每个分区 = 大量小文件 |
| **覆盖行为** | Hive 默认覆盖涉及的分区；Spark 需设 dynamic overwrite |
| **排序写入** | 配合 `DISTRIBUTE BY dt` 减少文件数 |
| **监控** | 关注动态分区创建数量，防止意外生成大量分区 |

```sql
-- 减少动态分区产生的小文件
INSERT OVERWRITE TABLE target PARTITION(dt)
SELECT col1, col2, dt
FROM source
DISTRIBUTE BY dt   -- 让相同分区的数据到同一个 Reducer
SORT BY dt, col1;  -- 可选：区内排序
```

## 六、统计信息管理

### 6.1 为什么需要统计信息

统计信息帮助查询优化器做出更好的决策：
- Join 顺序选择（小表在前）
- Join 策略选择（Broadcast vs SortMerge）
- 聚合策略选择（Hash vs Sort）

### 6.2 收集统计信息

```sql
-- 表级统计
ANALYZE TABLE dwd_orders_cleaned COMPUTE STATISTICS;

-- 分区级统计
ANALYZE TABLE dwd_orders_cleaned PARTITION(dt='2024-06-01') COMPUTE STATISTICS;

-- 列级统计（用于 CBO）
ANALYZE TABLE dwd_orders_cleaned COMPUTE STATISTICS FOR COLUMNS user_id, amount;

-- Spark 中等价操作
spark.sql("ANALYZE TABLE dwd_orders_cleaned COMPUTE STATISTICS FOR ALL COLUMNS")
```

## 七、面试话术集锦

### 7.1 "分区和分桶有什么区别？怎么选？"

> 分区和分桶是 Hive 两种不同粒度的数据组织方式。
>
> **分区**是按某个列的值将数据存放到不同的子目录中，物理上是目录级别的隔离。它的核心价值是**分区裁剪**——查询时根据 WHERE 条件只扫描相关分区，跳过无关数据。适合低基数字段如日期、地域。
>
> **分桶**是对某列做 hash，将数据分散到固定数量的文件中，物理上是文件级别的组织。它的核心价值是**优化 Join**——当两个表按相同的列和桶数分桶时，可以做 Bucket Map Join，避免全量 shuffle。适合高基数字段如 user_id。
>
> 生产中通常**组合使用**：按日期分区保证查询效率，在分区内按业务主键分桶优化 Join。需要注意的是，分区不宜过多（建议 < 万级），否则 NameNode 压力大；分桶数通常是 2 的幂次，方便后续扩展。

### 7.2 "ORC 和 Parquet 怎么选？"

> 两者都是优秀的列式存储格式，性能差异在大多数场景下不超过 10%。选型主要看**生态兼容性**。
>
> 如果团队同时使用 Spark、Presto、Flink，推荐 **Parquet**，因为它是跨引擎的事实标准，也是 Iceberg/Paimon 等湖表的默认格式。如果是纯 Hive 生态的老系统，**ORC** 的索引能力更强（内置 Bloom Filter + Row Index Group），在某些过滤密集的查询上更快。
>
> 我们项目用的是 Parquet + Snappy 压缩，因为下游有 Presto 做即席查询、Flink 做实时入仓，Parquet 的兼容性最好。

### 7.3 "Hive Metastore 的作用是什么？"

> Hive Metastore 是整个大数据生态的**元数据中心**。它用关系数据库（MySQL/PG）存储所有表的 schema、分区信息、存储位置和属性。Spark、Flink、Presto、Trino 等引擎都通过 Thrift RPC 访问同一个 Metastore 来获取表的元数据。
>
> 这意味着：① 用 Spark 创建的表，Flink 和 Presto 都能直接查；② 修改表结构只需改一处，所有引擎同步感知；③ 数仓的血缘管理和权限控制可以基于统一的元数据层实现。
>
> 所以 Hive 在现代架构中的核心价值不是"计算"（MapReduce 已经被 Spark 替代），而是"元数据管理"和"存储规范定义"。

### 7.4 "什么是分区裁剪？什么时候会失效？"

> 分区裁剪是查询优化器根据 WHERE 条件中的分区列过滤条件，在执行前就确定只需要扫描哪些分区目录，从而跳过无关数据的机制。这是 Hive/Spark SQL 最重要的性能优化之一。
>
> 失效的常见情况：① 在分区列上套了函数，如 `substr(dt,1,7)`；② 隐式类型转换，如 dt 是 STRING 但 WHERE 用了 INT；③ 分区列参与了复杂表达式。解决方法是把过滤条件改写为分区列的直接比较，如用 `dt >= '2024-06-01' AND dt <= '2024-06-30'` 替代 `substr(dt,1,7) = '2024-06'`。
