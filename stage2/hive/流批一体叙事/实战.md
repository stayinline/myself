### 1.3 关键差异对照

| 维度 | Lambda 架构 | 流批一体架构 |
|------|-----------|-------------|
| 数据源采集 | Sqoop(批) + CDC(流) 分开 | Flink CDC 统一 |
| 存储层 | Hive + Kafka 双份 | Paimon/Iceberg 一份 |
| ETL 代码 | Spark SQL + Flink SQL 两套 | 可统一或各司其职 |
| 结果表 | Hive + Redis 两份 | 湖表统一 |
| 数据一致性 | 需要对账 | 天然一致 |
| 历史回溯 | 重跑离线链路 | Time Travel |
| 运维复杂度 | 高 | 中 |
| 成熟度 | 非常成熟 | 快速成熟中 |

## 二、面试话术："什么是流批一体"

### 2.1 标准回答（60秒版）

> 流批一体是大数据架构的演进方向，核心目标是**用统一的抽象解决实时和离线两套系统带来的重复开发、口径不一致和运维复杂等问题**。
>
> 它的演进经历了三个阶段：
> - **Lambda 架构**：实时 Flink+Kafka 和离线 Spark+Hive 完全独立，两套代码两套存储，是最常见的痛点来源。
> - **Kappa 架构**：试图用 Flink 统一所有计算，但 Kafka 作为存储层的查询能力和成本都不理想，实践中难以落地。
> - **Lakehouse 架构**：当前主流方案。用 Iceberg/Paimon 等 Table Format 在 HDFS/S3 上构建统一存储层，支持 ACID、Upsert、增量读取等能力。Flink 负责实时写入和流计算，Spark 负责离线分析，两者读写同一张表。
>
> 需要注意的是，"流批一体"是分层次的。**存储层的一体已经基本实现**，但计算层和运维层仍有差距。当前的最佳实践是"存储统一 + 计算分离"，让每个引擎做最擅长的事。

### 2.2 深入追问应对

**Q: 为什么 Kappa 没有成功？**

> Kappa 的核心假设是"Flink 可以替代所有批处理"且"Kafka 可以作为长期存储"。但现实中：① Kafka 不支持高效的随机读取和索引，TB 级历史回放极慢；② Flink 的批模式在复杂 Join、Hive 兼容性和执行效率上还不如 Spark；③ Kafka 的存储成本远高于 HDFS/S3。所以 Kappa 在理论上优美，但在工程上不经济。Lakehouse 吸收了 Kappa "统一"的思想，但把存储层从 Kafka 换成了更适合分析的 Table Format。

**Q: 你们用的什么 Table Format？为什么选它？**

> （以 Paimon 为例）我们选 Paimon，因为核心场景是 CDC 入仓。Paimon 的 LSM-Tree 主键表对高频更新非常友好，写入性能优于 Iceberg 的 Merge-on-Read。同时它原生支持 changelog 产出，下游 Flink 任务可以直接消费变更流做实时聚合。当然如果团队以 OLAP 分析为主、更新频率低，Iceberg 会是更好的选择，它的跨引擎兼容性和社区生态更强。

**Q: 流批一体后还需要 Kafka 吗？**

> 取决于场景。如果数据源到湖表的链路用 Flink CDC 直连，中间确实可以省掉 Kafka。但如果多个下游需要消费同一份变更流，或者需要做流量削峰、解耦，Kafka 仍然有价值。我们的做法是：CDC 入仓走 Flink→Paimon 直连；跨域数据分发和事件驱动场景仍保留 Kafka。

**Q: 流批一体的最大挑战是什么？**

> 我认为有三个：① **语义一致性**——同一套 SQL 在流模式和批模式下结果是否完全相同，目前还没有完全保证；② **运维体系**——湖表的 compaction、snapshot 过期清理、小文件治理需要额外的运维工具链；③ **团队认知切换**——从"两套系统各自为政"到"统一存储层上的协作"，需要团队建立新的开发规范和心智模型。

## 三、简历改写指南

### 3.1 常见低质量描述 ❌

```
❌ "使用 Spark 和 Flink 进行数据处理"
❌ "搭建了流批一体架构"
❌ "使用 Hive 进行数仓建设"
❌ "优化了 Spark 任务性能"
❌ "实现了数据的实时和离线处理"
```

**问题**：太笼统，没有技术深度，无法区分 L1 和 L2。

### 3.2 L2 级别的高质量描述 ✅

#### 示例 1：流批一体项目

```
✅ 基于 Paimon 构建流批一体数仓，将原有 Lambda 架构（Flink+Kafka 实时链路 + 
   Spark+Hive 离线链路）统一为 Paimon 湖表存储层。Flink CDC 实时写入 Paimon 
   DWD 层，Spark 定时读取同一张表产出 T+1 报表，消除了双套 ETL 代码和口径
   不一致问题。上线后 ETL 开发效率提升 40%，数据对账差异率从 2% 降至 0。
```

**拆解要点**：
- 具体技术栈（Paimon、Flink CDC、Spark）
- 前后对比（Lambda → Lakehouse）
- 量化收益（效率提升 40%，差异率 2%→0）

#### 示例 2：Spark 调优

```
✅ 针对订单汇总表 groupBy user_id 的数据倾斜问题（热点 key 占比 60%），采用
   salting 两阶段聚合优化，将 Stage 最大 Task 耗时从 25min 降至 3min，整体
   Job 耗时缩短 70%。同时开启 AQE 自适应查询执行，自动合并小分区，输出文
   件数从 2000+ 降至 200，解决了 NameNode 元数据压力问题。
```

**拆解要点**：
- 具体问题描述（热点 key 占比 60%）
- 具体解决方案（salting + AQE）
- 量化效果（25min→3min，2000→200 文件）

#### 示例 3：Hive 数仓建设

```
✅ 设计并实现 DWD/DWS 分层数仓，ODS 层使用外部表防误删，DWD 层按日期分区
   + Parquet 格式存储，DWS 层按业务主键分桶优化 Join。通过分区裁剪和列式存
   储，核心报表查询耗时从 15min 降至 2min。建立统一的 Hive Metastore 元数
   据管理，支撑 Spark/Flink/Presto 三引擎共享访问。
```

#### 示例 4：Spark vs Flink 选型

```
✅ 主导实时风控系统的技术选型，对比 Spark Structured Streaming 和 Flink 在
   延迟、状态管理和 exactly-once 三个维度的差异。最终选择 Flink 作为实时计
   算引擎（毫秒级延迟 + RocksDB 大状态 + 端到端精确一次），Spark 保留用于
   离线特征工程和模型训练，通过 Paimon 湖表统一存储层实现数据互通。
```

## 四、综合面试模拟

### 4.1 自我介绍中的流批一体叙事

> 我目前主要做数据平台方向的开发，核心工作是围绕 Spark 和 Flink 构建离线和实时的数据处理管道。最近一年参与了公司的流批一体改造项目，将原来的 Lambda 架构迁移到基于 Paimon 的 Lakehouse 架构，统一了存储层，消除了双套 ETL 的维护成本。在这个过程中我对 Spark 的执行原理、Shuffle 调优以及 Hive Metastore 的元数据管理都有了比较深入的实践理解。

### 4.2 项目深挖应对

**面试官**：你说做了流批一体改造，具体怎么做的？遇到了什么问题？

> **回答框架**：
> 1. **背景**：原来是什么架构，有什么痛点（双套代码、口径不一致）
> 2. **方案选型**：对比了 Iceberg/Paimon/Hudi，选了 XX 因为 YY
> 3. **实施过程**：Flink CDC 写入 → Paimon DWD → Spark/Flink 分别消费
> 4. **遇到的问题**：比如小文件合并策略、schema evolution 兼容、历史数据迁移
> 5. **结果**：量化收益 + 遗留问题 + 下一步计划

**面试官**：Paimon 的小文件问题怎么解决的？

> Paimon 有内置的 compaction 机制。我们在 Flink 写入任务中开启了 `write-only` 模式，然后单独启动一个 compaction 任务定期合并小文件。这样写入和合并解耦，不影响实时写入的延迟。同时配置了 `snapshot.num-retained.min/max` 控制快照数量，避免元数据膨胀。

**面试官**：如果现在让你重新选型，你会改什么？

> 我会更早地引入数据质量监控。当时上线后发现 Paimon 表的 compaction 偶尔失败导致查询变慢，但没有及时发现。现在回想应该在 Day 1 就建立 compaction lag、snapshot 数量、文件大小分布等监控指标。另外历史数据迁移那步做得比较粗糙，应该先做 shadow read 验证数据一致性再切流。
