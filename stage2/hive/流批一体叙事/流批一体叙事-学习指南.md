## 读前扫盲：流批一体解决的是「两套系统算同一个数」

在线教育平台普遍面临一个困境：实时大屏用 Flink+Kafka 跑一套逻辑，T+1 报表用 Spark+Hive 再跑一套逻辑。两套代码、两份存储、两个结果——**口径永远对不上**。流批一体的目标不是消灭实时或消灭离线，而是用**一份统一存储 + 一套业务语义**根治重复建设和数据信任危机。

入门先记住三个结论：

| 结论 | 为什么重要 |
|------|------------|
| Lambda 架构的根本问题是"两套独立系统解决同一个业务问题" | 双套代码 ×2、口径不一致、运维复杂度指数级增长 |
| Kappa 理论优美但 Kafka 做不了长期存储，Lakehouse 才是当前答案 | Table Format（Paimon/Iceberg）在 HDFS/S3 上补齐了查询+更新能力 |
| "一体"是分层的：存储已一体，计算和运维仍在路上 | 务实做法是 **存储统一 + 计算分离**，各引擎发挥所长 |

本指南用 **Lambda→Kappa→Lakehouse 演进线** 串起原理，用 **架构对比图** 串起实操，用 **在线教育三场景** 说明选型与陷阱。

## Step 1 原理：传统割裂的痛点与流批一体的根治逻辑

### ① Lambda 架构：最常见的痛

```
┌─────────────────────────────────────────────────────────────┐
│                      应用 / BI / API                         │
│              ⚠️ 需要合并两套数据源！                          │
└──────────┬──────────────────────────────┬───────────────────┘
           │                              │
     ┌─────▼──────┐               ┌──────▼───────┐
     │ Speed Layer │               │ Batch Layer   │
     │ Flink+Kafka │               │ Spark+Hive    │
     │ • 实时指标   │               │ • T+1 报表    │
     │ • 实时告警   │               │ • 全量回刷     │
     └──────▲──────┘               └──────▲────────┘
            │                             │
       ┌────┴─────────────────────────────┴────┐
       │            Kafka / MQ                  │
       └──────────────────▲────────────────────┘
                          │
               ┌──────────┴──────────┐
               │      Data Source     │
               └─────────────────────┘
```

**六大痛点速记表**：

| 痛点 | 具体表现 | 业务影响 |
|------|---------|---------|
| **双套代码** | 同一逻辑 Flink 写一遍、Spark 写一遍 | 开发维护成本 ×2 |
| **口径不一致** | 窗口/过滤/Join 总有微妙差异 | 数据对不上，业务不信任 |
| **双套存储** | Kafka 存一份、Hive 存一份 | 存储成本翻倍 |
| **修正困难** | 实时出错需重跑 Flink + 修复 Kafka | 运维复杂度高 |
| **Schema 演化难** | 两套 schema 变更要同步 | 上线效率低 |
| **资源浪费** | 两套集群独立部署 | 无法弹性共享 |

> **记忆口诀**：**码、径、存、修、演、资** — 六字对应 Lambda 六大痛点。

### ② Kappa 架构：理想很丰满

核心思想：去掉 Batch Layer，所有计算都基于流（Flink + Kafka）。

```
Serving Layer ← Flink (All Computation) ← Kafka（既传输又存储）
```

**Kappa 为什么没成功？**

| 问题 | 说明 |
|------|------|
| Kafka 不适合长期存储 | 成本高、不支持索引、查询能力弱 |
| 历史回放极慢 | TB 级数据从 Kafka 头消费，耗时不可接受 |
| 批处理性能差 | Flink 批模式不如 Spark 成熟 |
| 状态膨胀 | 长窗口/大状态导致 checkpoint 巨大 |

> **关键洞察**：Kappa 启发了 Lakehouse——需要**既有流的特性又有批的能力**的统一存储层，但绝不是 Kafka。

### ③ Lakehouse 架构：当前答案

在对象存储/HDFS 上构建 Table Format，让流和批都能高效读写同一张表。

```
┌─────────────────────────────────────────────────────────┐
│         Spark / Flink / Presto / Trino / StarRocks       │
├─────────────────────────────────────────────────────────┤
│          Iceberg / Paimon / Hudi / Delta                  │
│  ACID · Time Travel · Schema Evo · Upsert · 增量读        │
├─────────────────────────────────────────────────────────┤
│         HDFS / S3 / OSS / MinIO (Parquet / ORC)          │
└─────────────────────────────────────────────────────────┘
```

**流批一体如何根治 Lambda 痛点？**

| Lambda 痛点 | Lakehouse 根治方式 |
|------------|-------------------|
| 双套代码 | Flink 写 + Spark 读同一张表，ETL 逻辑可复用 |
| 口径不一致 | 同源同表，天然一致 |
| 双套存储 | Paimon/Iceberg 一份存储替代 Hive+Kafka |
| 修正困难 | Time Travel 回溯历史快照，无需重跑全链路 |
| Schema 演化 | Table Format 原生支持 Schema Evolution |
| 资源浪费 | 存储统一后集群可弹性共享 |

## Step 2 实操：画一张"割裂架构 → 流批一体架构"对比图

### ① Lambda 架构（传统分离）

```
数据源(MySQL/Log)
    ├── Sqoop(T+1) → Hive(ODS) → Spark(离线ETL) → Hive DWS ──┐
    └── Flink CDC  → Kafka      → Flink(实时ETL) → Redis     ──┤
                                                                ▼
                                                    应用/BI ⚠️ 合并两套!

❌ 两套ETL · 两套存储 · 两套结果 · 口径可能不一致
```

### ② 流批一体架构（Lakehouse）

```
数据源(MySQL/Log)
    └── Flink CDC → Paimon/Iceberg(ODS→DWD→DWS) ← ★ 统一存储层 ★
                       ├── Flink 流读 → 实时指标/告警
                       └── Spark 批读 → T+1报表/Ad-hoc/ML
                              ↓
                    Paimon/Iceberg(DWS/ADS) → 应用/BI ✅ 单一数据源

✅ 一份存储 · 口径一致 · 各引擎发挥所长 · Time Travel
```

### ③ 关键差异对照

| 维度 | Lambda 架构 | 流批一体架构 |
|------|-----------|-------------|
| 数据采集 | Sqoop(批) + CDC(流) 分开 | Flink CDC 统一 |
| 存储层 | Hive + Kafka 双份 | Paimon/Iceberg 一份 |
| ETL 代码 | Spark SQL + Flink SQL 两套 | 可统一或各司其职 |
| 结果表 | Hive + Redis 两份 | 湖表统一 |
| 数据一致性 | 需要对账 | 天然一致 |
| 历史回溯 | 重跑离线链路 | Time Travel |
| 运维复杂度 | 高 | 中 |
| 成熟度 | 非常成熟 | 快速成熟中 |

> **画图练习**：闭卷画出以上两张图，标注 ❌ 和 ✅ 的差异点。这是面试白板题的高频考点。

## Step 3 对比：Lambda → Kappa → 湖仓流批一体的演进逻辑

### ① 演进时间线与驱动力

```
2014          2016           2019-2020          2022-now
  │             │               │                  │
  ▼             ▼               ▼                  ▼
Lambda       Kappa          Table Format        Streaming Lake
(Flink+      (Flink Only)   (Delta/Iceberg/     (Paimon/Iceberg)
 Spark)                      Hudi)               流批读写+Upsert
              │                                    +Schema Evolution
    Lambda 痛 → "只留一条链路?" → Kappa
                                      │
                            Kafka 存储不行 → "换个存储?"
                                              │
                                              ▼
                                        Table Format
                                     "存储统一+计算分离"
```

### ② 三种架构本质对比

| 维度 | Lambda | Kappa | Lakehouse |
|------|--------|-------|-----------|
| **核心思想** | 批流分离，各自最优 | 一切皆流，去掉批 | 统一存储，计算分离 |
| **存储层** | Hive + Kafka | Kafka only | Table Format (HDFS/S3) |
| **计算层** | Spark + Flink | Flink only | Spark + Flink（各取所长） |
| **口径一致性** | ❌ 需要对账 | ✅ 天然一致 | ✅ 天然一致 |
| **Ad-hoc 查询** | ✅ Hive/Presto | ❌ Kafka 不行 | ✅ 多引擎 |
| **历史回溯** | 重跑批链路 | Replay Kafka（慢） | Time Travel |
| **工程落地难度** | 低（成熟） | 高（受限多） | 中（快速成熟） |
| **适用时代** | 2014-2018 | 2016-2019（理想主义） | 2020-now（务实主义） |

> **记忆口诀**：**Lambda 痛、Kappa 梦、Lakehouse 落地用** — 三句话概括演进。

### ③ "一体"到底指什么（分层理解）

| 层次 | 含义 | 现状 |
|------|------|------|
| **API 一体** | 同一套 SQL/API 既跑批又跑流 | Flink SQL ✅ / Spark SS 部分 ✅ |
| **存储一体** | 同一张表既能流读又能批读 | Iceberg/Paimon ✅ |
| **计算一体** | 同一个引擎既做批又做流 | Flink ✅ / Spark 部分 ✅ |
| **语义一体** | 批和流的计算结果完全一致 | 仍在推进中 ⚠️ |
| **运维一体** | 统一的监控、调度、权限 | 尚不成熟 ❌ |

> **诚实判断**：截至 2025-2026 年，存储层已一体，计算层和运维层仍有差距。**最佳实践 = 存储统一 + 计算分离**。

## Step 4 陷阱：流批一体不是银弹

### ① 实时更新 vs 大批扫描的存储权衡

```
┌─────────────────────────────────────────────────────────┐
│                   CAP-like Trade-offs                     │
│   写入延迟 ◄──────────────────────► 读取性能             │
│   更新能力 ◄──────────────────────► 查询性能             │
│   实时性   ◄──────────────────────► 存储成本             │
│   通用性   ◄──────────────────────► 特定场景优化          │
└─────────────────────────────────────────────────────────┘
```

### ② 四大 Table Format 选型

| 格式 | 偏向 | 牺牲 | 适合场景 |
|------|------|------|---------|
| **Iceberg** | 通用性、读取性能 | 高频更新性能 | OLAP 分析为主 |
| **Paimon** | 流式写入、更新 | 纯批读性能略逊 | CDC 入仓、实时更新 |
| **Hudi** | Upsert、增量 | 复杂度较高 | 数据同步、CDC |
| **Delta** | Spark 集成 | 非 Spark 引擎支持弱 | Databricks 生态 |

**选型决策树**：

```
你的主要场景是什么？
    ├── Flink 实时写入 + Spark 离线分析 → Paimon
    ├── 多引擎通用分析（Spark+Presto+Flink） → Iceberg
    ├── Spark 为主 + 需要 Upsert → Delta Lake
    ├── 已有 Hudi 存量 → 继续用，逐步评估迁移
    └── 新项目，不确定 → Iceberg（最安全的选择）
```

### ③ Paimon LSM-Tree 设计要点

```
新数据 → Memory Buffer → flush → Sorted Run L0
                                   Sorted Run L1 ← 后台 Compaction
                                   Sorted Run L2

• 写入快（append to buffer + sort）  • 更新快（LSM merge）
• 读取时需 merge 多个 sorted run     • 天然支持 CDC changelog
```

### ④ 常见认知陷阱

| 陷阱 | 正确认知 |
|------|---------|
| "流批一体 = 只用 Flink" | Flink 批模式还不成熟，存储统一+计算分离才是务实做法 |
| "上了湖表就不需要 Kafka" | 多下游分发、削峰解耦场景 Kafka 仍有价值 |
| "Table Format 选了就不用管" | compaction、snapshot 清理、小文件治理需要额外运维 |
| "流批一体后口径自动一致" | 语义一体仍在推进，SQL 写法差异仍可能导致结果不同 |

> **记忆口诀**：**写读反、更查反、实储反、通专反** — 四组权衡，没有银弹。

## Step 5 面试话术

### 问：什么是流批一体，它解决了什么问题？（60 秒标准版）

> 流批一体是大数据架构的演进方向，核心目标是**用统一的抽象解决实时和离线两套系统带来的重复开发、口径不一致和运维复杂等问题**。
>
> 它的演进经历了三个阶段：
> - **Lambda 架构**：实时 Flink+Kafka 和离线 Spark+Hive 完全独立，两套代码两套存储，是最常见的痛点来源。
> - **Kappa 架构**：试图用 Flink 统一所有计算，但 Kafka 作为存储层的查询能力和成本都不理想，实践中难以落地。
> - **Lakehouse 架构**：当前主流方案。用 Iceberg/Paimon 等 Table Format 在 HDFS/S3 上构建统一存储层，支持 ACID、Upsert、增量读取。Flink 负责实时写入和流计算，Spark 负责离线分析，两者读写同一张表。
>
> 需要注意的是，"一体"是分层次的。**存储层的一体已经基本实现**，但计算层和运维层仍有差距。当前的最佳实践是"存储统一 + 计算分离"，让每个引擎做最擅长的事。

### 追问应对

**Q: 为什么 Kappa 没有成功？**

> Kappa 假设"Flink 可以替代所有批处理"且"Kafka 可以作为长期存储"。但：① Kafka 不支持高效随机读取和索引，TB 级回放极慢；② Flink 批模式在复杂 Join、Hive 兼容性上还不如 Spark；③ Kafka 存储成本远高于 HDFS/S3。Lakehouse 吸收了"统一"思想，但把存储换成了 Table Format。

**Q: 你们用的什么 Table Format？为什么？**

> （以 Paimon 为例）核心场景是 CDC 入仓。Paimon 的 LSM-Tree 主键表对高频更新友好，写入性能优于 Iceberg 的 Merge-on-Read，且原生支持 changelog 产出。如果以 OLAP 分析为主、更新频率低，Iceberg 跨引擎兼容性更强。

**Q: 流批一体后还需要 Kafka 吗？**

> 取决于场景。CDC 入仓走 Flink→Paimon 直连可省 Kafka。但多下游分发、流量削峰场景 Kafka 仍有价值。我们 CDC 入仓直连湖表；跨域分发和事件驱动仍保留 Kafka。

**Q: 流批一体的最大挑战？**

> 三个：① **语义一致性**——同一 SQL 在流/批模式下结果是否相同，还没完全保证；② **运维体系**——compaction、snapshot 清理、小文件治理需额外工具链；③ **团队认知切换**——从两套系统到统一存储层协作，需建立新规范。

## 在线教育典型业务案例（流批一体三角）

> 以下三个场景对应：**Lambda 痛点复现** / **湖表统一落地** / **选型权衡决策**。

---

### 案例一：学员学习时长报表 — Lambda 双链路口径不一致

#### 业务背景

「学员今日学习时长」同时出现在实时大屏（Flink+Kafka，秒级）和家长日报（Spark+Hive，T+1）。两条链路过滤规则、窗口对齐略有不同，导致**大屏 45min vs 日报 52min**，家长投诉。

#### 三要素

| 维度 | 分析 |
|------|------|
| **根因** | Lambda 双链路：两套 ETL + 两套存储 + 口径微差 |
| **信号** | 对账差异率 > 2%；业务方反复质疑数据可信度 |
| **对策** | Paimon 统一存储 → Flink 写入 + Spark 批读同一张表 → 口径天然一致 |

```
Before: Kafka→Flink→Redis(实时) + Hive→Spark→Hive(日报)  ← 两套！
After:  Flink CDC→Paimon(DWD)→ Flink流读(大屏) + Spark批读(日报) ← 同表！
```

---

### 案例二：课程推荐特征工程 — 流批一体存储落地

#### 业务背景

算法团队需要学员近 30 天行为特征做模型训练（批），在线推理需要最近 1 小时实时特征（流）。传统做法离线一份特征表、实时一份 Redis 缓存，字段定义经常不同步，导致**模型离线 AUC 正常、上线效果下降**。

#### 三要素

| 维度 | 分析 |
|------|------|
| **根因** | 特征定义的流批分裂：Hive 表和 Redis 缓存字段不一致 |
| **信号** | 模型上线后效果下降（特征分布偏移） |
| **对策** | Paimon 统一特征表 → Flink 写入+流读(推理) / Spark 批读(训练) |

**踩坑**：Paimon 小文件 → `write-only` + 单独 compaction 任务；Schema Evolution → 下游 Spark 需刷新缓存。

---

### 案例三：教务数据入仓 — Table Format 选型权衡

#### 业务背景

教务 MySQL 200+ 表实时同步数仓，分析师需用 Presto Ad-hoc 查询。团队在 Paimon 和 Iceberg 之间犹豫。

#### 三要素

| 维度 | 分析 |
|------|------|
| **根因** | CDC 高频更新 + 多引擎查询的双重需求 |
| **信号** | CDC QPS 峰值 5k/s；Presto P99 < 10s |
| **对策** | 选 Paimon：LSM-Tree 对高频 CDC 友好 + 原生 changelog |

| 维度 | Paimon | Iceberg |
|------|--------|---------|
| CDC 高频更新 | ✅ LSM-Tree 原生 | ⚠️ MoR 有开销 |
| Presto 兼容 | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ |
| Flink 原生 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ |
| **决策** | **✅ 选 Paimon** | CDC 场景占比大 |
