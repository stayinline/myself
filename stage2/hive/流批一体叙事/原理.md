# 流批一体叙事：从 Lambda 到 Lakehouse 的演进

> 📚 本模块：[学习指南](流批一体叙事-学习指南.md) · **原理（本文）** · [实战](实战.md)

## 一、传统架构的痛点

### 1.1 Lambda 架构的典型形态

```
                    ┌─────────────────────────────┐
                    │        Serving Layer         │
                    │   (Redis / MySQL / HBase)    │
                    └──────┬──────────────┬────────┘
                           │              │
              ┌────────────▼──┐    ┌──────▼───────────┐
              │ Speed Layer   │    │   Batch Layer     │
              │ Flink+Kafka   │    │  Spark+Hive       │
              │               │    │                    │
              │ • 实时指标     │    │ • T+1 离线报表     │
              │ • 实时告警     │    │ • 全量回刷          │
              │ • 秒级延迟     │    │ • 精确对账          │
              └────────▲──────┘    └──────▲────────────┘
                       │                  │
                  ┌────┴──────────────────┴────┐
                  │        Kafka / MQ           │
                  └────────────▲────────────────┘
                               │
                  ┌────────────┴────────────────┐
                  │      Data Source             │
                  │  (DB / Log / API / IoT)      │
                  └─────────────────────────────┘
```

### 1.2 Lambda 的核心痛点

| 痛点 | 具体表现 | 业务影响 |
|------|---------|---------|
| **双套代码** | 同一业务逻辑用 Flink 写一遍、Spark 写一遍 | 开发成本 ×2，维护成本 ×2 |
| **口径不一致** | 实时和离线的计算逻辑总有微妙差异 | 数据对不上，业务不信任 |
| **双套存储** | Kafka 存一份、Hive 存一份 | 存储成本翻倍 |
| **数据修正困难** | 实时链路出错需要重跑 Flink + 修复 Kafka | 运维复杂度高 |
| **Schema 演化难** | 两套系统的 schema 变更要同步 | 上线效率低 |
| **资源浪费** | 两套集群独立部署 | 无法弹性共享资源 |

> **面试话术**：Lambda 架构的根本问题是"用两套独立的系统解决同一个业务问题"。这导致了代码重复、口径不一致、运维复杂等一系列工程问题。流批一体的目标不是消灭实时或消灭离线，而是用一套统一的抽象来同时满足两种需求。

## 二、架构演进路线

### 2.1 Lambda → Kappa → Lakehouse

```
时间线:
2014        2016           2019-2020          2022-now
  │           │               │                  │
  ▼           ▼               ▼                  ▼
Lambda     Kappa          Delta/Iceberg/Hudi   Paimon/Iceberg
(Flink+    (Flink Only)   (Table Format)       (Streaming Lake)
 Spark)                     │                   │
                            │                   │
                          统一存储层            流批一体存储
                          但写入仍以            + 流批读写
                          批为主                + Upsert/Delete
                                                + Schema Evolution
```

### 2.2 Kappa 架构

**核心思想**：去掉 Batch Layer，所有计算都基于流（Kafka/Flink）。

```
┌─────────────────────────────────────┐
│           Serving Layer              │
└──────────────┬──────────────────────┘
               │
┌──────────────▼──────────────────────┐
│      Flink (All Computation)         │
│  • 实时 ETL                          │
│  • 实时聚合                          │
│  • 历史回放（replay Kafka）           │
└──────────────┬──────────────────────┘
               │
┌──────────────▼──────────────────────┐
│           Kafka                      │
│  （既是传输层又是存储层）              │
└─────────────────────────────────────┘
```

**Kappa 的问题**：

| 问题 | 说明 |
|------|------|
| Kafka 不适合做长期存储 | 成本高、不支持索引、查询能力弱 |
| 历史回放慢 | 要从 Kafka 头开始消费，TB 级数据回放慢 |
| 批处理性能差 | Flink 批模式不如 Spark 成熟 |
| 状态膨胀 | 长窗口/大状态导致 checkpoint 巨大 |
| 不适合 Ad-hoc | 无法像 Hive/Presto 那样灵活查询 |

> **结论**：Kappa 在理论上优美，但在实践中因为 Kafka 的存储局限性而难以落地。它启发了后来的 Lakehouse 思路——需要一个既有流的特性又有批的能力的统一存储层。

### 2.3 Lakehouse 架构

**核心思想**：在对象存储/HDFS 上构建具有事务能力的表格式（Table Format），让流和批都能高效读写同一张表。

```
┌─────────────────────────────────────────────────────────┐
│                    Query Engines                         │
│      Spark / Flink / Presto / Trino / StarRocks          │
└────────────────────────┬────────────────────────────────┘
                         │
┌────────────────────────▼────────────────────────────────┐
│                 Table Format Layer                        │
│          Iceberg / Paimon / Hudi / Delta                  │
│                                                          │
│  • ACID Transaction                                      │
│  • Time Travel                                           │
│  • Schema Evolution                                      │
│  • Partition Evolution                                   │
│  • Upsert / Delete                                       │
│  • Incremental Read                                      │
└────────────────────────┬────────────────────────────────┘
                         │
┌────────────────────────▼────────────────────────────────┐
│              Storage Layer                                │
│         HDFS / S3 / OSS / MinIO                          │
│         (Parquet / ORC Files)                            │
└─────────────────────────────────────────────────────────┘
```

### 3.2 选型决策树

```
你的主要场景是什么？
    │
    ├── Flink 实时写入 + Spark 离线分析
    │     → Paimon（Flink 原生支持最好，LSM-Tree 适合高频更新）
    │
    ├── 多引擎通用分析（Spark + Presto + Flink）
    │     → Iceberg（跨引擎兼容性最强，社区最活跃）

    │
    └── 新项目，不确定
          → Iceberg（最安全的选择）
```

## 四、流批一体的存储权衡

### 4.1 没有银弹

流批一体存储需要在以下维度做取舍：

```
┌─────────────────────────────────────────────────────────┐
│                   CAP-like Trade-offs                     │
│                                                          │
│   写入延迟 ◄────────────────────────► 读取性能           │
│   (Append-only快)                   (Sorted/Indexed快)   │
│                                                          │
│   更新能力 ◄────────────────────────► 查询性能           │
│   (LSM-Tree/COW有开销)              (Immutable最快)      │
│                                                          │
│   实时性   ◄────────────────────────► 存储成本           │
│   (频繁commit/snapshot)             (合并后更紧凑)        │
│                                                          │
│   通用性   ◄────────────────────────► 特定场景优化        │
│   (Iceberg通用)                     (Paimon流式优化)      │
└─────────────────────────────────────────────────────────┘
```

### 4.2 各格式的权衡选择

| 格式 | 偏向 | 牺牲 | 适合场景 |
|------|------|------|---------|
| **Iceberg** | 通用性、读取性能 | 高频更新性能 | OLAP 分析为主 |
| **Paimon** | 流式写入、更新 | 纯批读性能略逊 | CDC 入仓、实时更新 |
| **Hudi** | Upsert、增量 | 复杂度较高 | 数据同步、CDC |
| **Delta** | Spark 集成 | 非 Spark 引擎支持弱 | Databricks 生态 |

### 4.3 Paimon 的 LSM-Tree 设计

```
Paimon 主键表的写入路径：

新数据到达
    │
    ▼
┌─────────────────┐
│  Memory Buffer   │ ← 内存中缓冲
└────────┬────────┘
         │ flush
         ▼
┌─────────────────┐
│  Sorted Run L0   │ ← 有序的 Parquet 文件
├─────────────────┤
│  Sorted Run L1   │ ← 后台 Compaction 合并
├─────────────────┤
│  Sorted Run L2   │
└─────────────────┘

特点：
• 写入快（append to buffer + sort）
• 更新快（LSM merge semantics）
• 读取时需 merge 多个 sorted run（compaction 后变好）
• 天然支持 CDC changelog 产出
```

> **面试话术**：Paimon 选择了 LSM-Tree 作为主键表的存储结构，这和传统数据库的写入优化思路一致。代价是读取时需要合并多个 sorted run，但通过后台 compaction 可以控制读放大。对于 CDC 入仓这种"写入密集 + 需要更新"的场景，LSM-Tree 比 Iceberg 的 Merge-on-Read 更高效。但如果主要是大批量分析查询、很少更新，Iceberg 的不可变文件模型读取更快。

## 五、流批一体的真正含义

### 5.1 "一体"到底指什么

| 层次 | 含义 | 现状 |
|------|------|------|
| **API 一体** | 同一套 SQL/API 既跑批又跑流 | Flink SQL ✅ / Spark SS 部分 ✅ |
| **存储一体** | 同一张表既能流读又能批读 | Iceberg/Paimon ✅ |
| **计算一体** | 同一个引擎既做批又做流 | Flink ✅ / Spark 部分 ✅ |
| **语义一体** | 批和流的计算结果完全一致 | 仍在推进中 ⚠️ |
| **运维一体** | 统一的监控、调度、权限 | 尚不成熟 ❌ |

### 5.2 当前现实

> **诚实的判断**：截至 2025-2026 年，"流批一体"在**存储层**已经基本实现（Iceberg/Paimon 可以让 Spark 和 Flink 读写同一张表），但在**计算层**和**运维层**仍有差距。Flink 的批模式在性能和稳定性上还在追赶 Spark；Spark Structured Streaming 的状态管理能力仍弱于 Flink。
>
> 所以当前的最佳实践是：**存储统一 + 计算分离**。用 Paimon/Iceberg 做统一存储层，Flink 负责实时写入和流计算，Spark 负责离线分析和批量回填，各取所长。

## 六、总结与面试要点

### 核心记忆点

1. **Lambda 痛点**：双套代码、口径不一致、存储翻倍
2. **Kappa 理想与现实**：理论优美但 Kafka 存储受限
3. **Lakehouse 是当前答案**：Table Format 统一存储 + 多引擎计算
4. **格式选型**：Paimon 偏流、Iceberg 偏通用、看场景选
5. **"一体"是分层的**：存储已一体，计算和运维仍在路上
6. **务实态度**：存储统一 + 计算分离是当前最佳实践

### 常见面试追问

- **Q: 你们项目是怎么做流批一体的？**
  A: 我们用 Paimon 做统一存储层。Flink CDC 实时写入 Paimon 表，下游 Flink 任务做实时聚合产出实时指标；同时 Spark 定时读同一张 Paimon 表做离线分析和 T+1 报表。这样保证了数据源统一、口径一致，又让 Flink 和 Spark 各自发挥优势。

- **Q: 为什么不全部用 Flink 做批处理？**
  A: Flink 的批模式在 SQL 兼容性、Hive 生态对接、大规模 shuffle 性能上还不如 Spark 成熟。我们的离线 ETL 涉及大量复杂 Join 和 UDF，Spark 的执行效率和稳定性更好。等 Flink 批模式成熟了可以考虑统一，但现在"存储统一 + 计算分离"是更务实的选择。

- **Q: Paimon 和 Iceberg 怎么选？**
  A: 如果你的核心场景是 CDC 入仓 + 实时更新 + Flink 为主，选 Paimon，它的 LSM-Tree 对高频更新更友好，且原生支持 changelog 产出。如果是多引擎 OLAP 分析为主、更新频率不高，选 Iceberg，它的跨引擎兼容性和社区生态更强。我们选 Paimon 是因为 CDC 场景占比大。
