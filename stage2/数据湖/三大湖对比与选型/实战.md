# 三大湖对比与选型：实战篇

## 一、Schema Evolution 实操

### 1.1 Iceberg Schema Evolution 实验

#### 建表并写入初始数据

```sql
-- Iceberg 建表
CREATE TABLE iceberg_catalog.test_db.users (
    user_id     BIGINT,
    name        STRING,
    age         INT,
    created_at  TIMESTAMP
) USING iceberg
PARTITIONED BY (days(created_at));

-- 写入数据
INSERT INTO iceberg_catalog.test_db.users VALUES 
    (1, 'Alice', 25, TIMESTAMP '2024-01-15 10:00:00'),
    (2, 'Bob', 30, TIMESTAMP '2024-01-15 11:00:00');
```

#### 执行 Schema Evolution

```sql
-- 1. 添加列
ALTER TABLE iceberg_catalog.test_db.users ADD COLUMN email STRING;

-- 2. 修改列类型（安全 widening）
ALTER TABLE iceberg_catalog.test_db.users ALTER COLUMN age TYPE BIGINT;

-- 3. 重命名列
ALTER TABLE iceberg_catalog.test_db.users RENAME COLUMN name TO username;

-- 4. 删除列
ALTER TABLE iceberg_catalog.test_db.users DROP COLUMN email;
```

#### 验证旧数据兼容性

```sql
-- 查询当前 schema 的数据（已完成加列、改类型、重命名、删列）
SELECT * FROM iceberg_catalog.test_db.users;
-- user_id | username | age  | created_at
-- 1       | Alice    | 25   | 2024-01-15 10:00:00
-- 2       | Bob      | 30   | 2024-01-15 11:00:00
-- 注: email 列已在步骤 4 删除，不再出现

-- Time Travel 查询旧快照（schema evolution 之前）
SELECT * FROM iceberg_catalog.test_db.users 
FOR VERSION AS OF <old_snapshot_id>;
-- user_id | name   | age | created_at
-- 1       | Alice  | 25  | 2024-01-15 10:00:00    ← 旧列名仍然可用
-- 2       | Bob    | 30  | 2024-01-15 11:00:00
```

### 1.2 Paimon Schema Evolution 实验

```sql
-- Paimon 建表
CREATE TABLE paimon_catalog.test_db.users_pk (
    user_id     BIGINT,
    name        STRING,
    age         INT,
    created_at  TIMESTAMP(3),
    PRIMARY KEY (user_id) NOT ENFORCED
) WITH ('bucket' = '2');

-- 写入数据
INSERT INTO paimon_catalog.test_db.users_pk VALUES 
    (1, 'Alice', 25, TIMESTAMP '2024-01-15 10:00:00'),
    (2, 'Bob', 30, TIMESTAMP '2024-01-15 11:00:00');

-- 添加列
ALTER TABLE paimon_catalog.test_db.users_pk ADD COLUMN email STRING;

-- 修改列类型
ALTER TABLE paimon_catalog.test_db.users_pk MODIFY age BIGINT;

-- 查询：旧数据自动适配新 schema
SELECT * FROM paimon_catalog.test_db.users_pk;
-- user_id | name  | age | created_at          | email
-- 1       | Alice | 25  | 2024-01-15 10:00:00 | NULL
-- 2       | Bob   | 30  | 2024-01-15 11:00:00 | NULL
```

### 1.3 Schema Evolution 兼容性约束

| 操作 | Iceberg | Paimon | Hudi | 安全性 |
|------|---------|--------|------|--------|
| 添加列 | ✅ | ✅ | ✅ | 安全，旧数据填 NULL |
| 删除列 | ✅ | ✅ | ⚠️ | 安全，但字段仍在文件中 |
| 重命名 | ✅ | ✅ | ⚠️ | 安全（基于 field-id） |
| INT → BIGINT | ✅ | ✅ | ✅ | 安全 widening |
| FLOAT → DOUBLE | ✅ | ✅ | ✅ | 安全 widening |
| STRING → INT | ❌ | ❌ | ❌ | 不安全 narrowing |
| DECIMAL(p,s) → DECIMAL(p',s') | p'≥p, s'≥s | p'≥p, s'≥s | 类似 | 精度扩大安全 |
| 改变主键列 | N/A | ❌ | ❌ | 不允许 |
| 改变分区列 | Partition Evolution | ✅ | ⚠️ | 各有机制 |

### 1.4 面试话术："Schema Evolution 是怎么保证旧数据还能读的？"

> 核心机制是 **field-id**。Iceberg 和 Paimon 都不依赖列名来匹配数据，而是给每个列分配一个唯一的数字 ID。当你重命名一个列时，只是改了名字到 ID 的映射，底层数据文件里的 field-id 没变，所以旧数据照样能读出来。
>
> 加列的时候，新列有一个新的 field-id，旧数据文件里没有这个 ID 的数据，读取时引擎自动填充 NULL。删列只是标记该 field-id 为 inactive，不影响已有文件。
>
> 类型变更只允许安全的 widening conversion，比如 INT→BIGINT、FLOAT→DOUBLE，保证旧值在新类型下不会丢失精度。不安全的 narrowing（如 STRING→INT）会被拒绝。这和 Protobuf / Avro 的 schema evolution 规则是类似的思路。

## 二、选型决策表构建

### 2.1 评估维度与权重

在实际项目中，建议用打分表来辅助选型：

```
评分标准: 1=差, 2=一般, 3=良好, 4=优秀, 5=卓越
权重根据项目实际需求调整
```

| 维度 | 权重 | Iceberg | Hudi | Paimon | 备注 |
|------|------|---------|------|--------|------|
| Upsert 性能 | 25% | 3 | 4 | 5 | Paimon LSM 优势明显 |
| 流读 Changelog | 20% | 2 | 3 | 5 | Paimon 完整 +I/-D/-U/+U |
| 读取性能 | 15% | 5 | 4 | 4 | Iceberg Data Skipping 最优 |
| 引擎兼容性 | 15% | 5 | 4 | 3 | Iceberg 生态最广 |
| 社区/文档 | 10% | 5 | 4 | 4 | Iceberg 最成熟 |
| Flink 集成 | 10% | 4 | 3 | 5 | Paimon Flink 原生 |
| 运维复杂度 | 5% | 4 | 3 | 3 | Iceberg 相对简单 |
| **加权总分** | **100%** | **3.75** | **3.65** | **4.35** | 示例：CDC实时入湖场景 |

> 注意：以上分数仅为 CDC 实时入湖场景的示例。不同场景下权重和分数会显著变化。

### 2.2 场景化推荐速查

| 业务场景 | 首选 | 备选 | 理由 |
|----------|------|------|------|
| CDC 实时入湖 + 下游 OLAP | Paimon PK | Hudi MOR | 高频 upsert + 完整 changelog |
| 离线数仓替代 Hive | Iceberg COW | Paimon Append | 读优化 + 广兼容 |
| 日志/事件分析 | Paimon Append / Iceberg | Hudi COW | Append-only，无需 upsert |
| 多流拼接宽表 | Paimon partial-update | Flink State JOIN | 避免 State 膨胀 |
| 跨引擎共享 (Spark+Trino+SR) | Iceberg | Hudi | 生态最广 |
| Databricks 平台 | Delta Lake | Iceberg | 平台绑定 |
| 已有 Hudi 生产集群 | Hudi | 渐进迁移 | 迁移成本高 |
| 算法回溯 + 实时更新 | Paimon PK | Iceberg MOR | 一份存储两种消费 |

## 三、COW vs MOR Trade-off 实验

### 3.1 实验设计

```
数据集: 100万行 Parquet，单文件 100MB
操作: 随机更新 1% 的行（1万行 UPDATE）
分别测试 Iceberg COW、Iceberg MOR、Paimon PK Table
```

### 3.2 预期测量结果

| 指标 | Iceberg COW | Iceberg MOR | Paimon PK |
|------|------------|-------------|-----------|
| 写入耗时 | ~8s (重写100MB文件) | ~0.5s (写delete file) | ~1s (MemTable flush) |
| 写入IO量 | ~100MB | ~50KB | ~1MB |
| 首次读取耗时 | ~1.5s | ~3s (merge deletes) | ~2s (merge sorted runs) |
| Compaction后读取 | ~1.5s | ~1.5s | ~1.5s |
| Compaction耗时 | N/A | ~5s | ~3s |
| 存储空间增量 | +100MB (新文件) | +50KB (delete file) | +1MB (L0 run) |

### 3.3 面试话术："COW 和 MOR 怎么选？"

> COW 和 MOR 本质上是**读写开销的时间分配问题**。
>
> COW 把合并开销放在写入时——改一行就要重写整个文件，写入放大可能达到几十倍甚至上百倍。但好处是读取时直接扫描，没有任何合并逻辑，性能最优。适合读多写少的场景，比如维表、每天跑一次的批量 ETL。
>
> MOR 把合并开销推迟到读取时——写入只需要记录变更（delete file 或 delta log），非常快。但读取时需要将 base file 和变更文件做合并，有额外开销。适合写多读少的场景，比如 CDC 实时入湖、频繁小量更新。
>
> Paimon 的 PK Table 本质上也是 MOR，但它用的是 LSM-Tree 的结构化 merge（有序归并），比 Iceberg/Hudi 的 delete file 逐行匹配更高效。而且 LSM Compaction 持续在后台合并，读取时的 merge 层数通常可控。
>
> 实际选择还要考虑 Compaction 策略。MOR 如果不及时 compaction，delete file 堆积会导致读性能持续恶化。所以 MOR 模式下一定要配好 compaction 定时任务。

## 四、综合对比演练

### 4.1 模拟面试题："你们项目中数据湖是怎么选型的？"

> 我们的核心场景是考试业务的 CDC 实时入湖，MySQL binlog 通过 Flink CDC 接入，下游既要实时同步到 Doris 做运营看板，又要支持算法团队用 Spark 做历史数据回溯。
>
> 我们评估了 Iceberg、Hudi 和 Paimon 三个方案：
>
> **Iceberg** 的读性能和生态兼容性最好，但在高频 Upsert 场景下，COW 写放大太严重（改一行重写整文件），MOR 的 delete file 堆积又影响读性能，而且它的流读只能产出 +I，无法直接驱动下游增量聚合。
>
> **Hudi** 是最早的方案，功能完善，但我们团队以 Flink 为主，Hudi 和 Flink 的集成不如 Paimon 紧密，而且 Hudi 的版本升级历史上有些 breaking change 让我们顾虑。
>
> **Paimon** 最终胜出，原因有三：一是 LSM-Tree 架构天然适合我们的高频 CDC Upsert；二是 input 模式的 changelog-producer 可以直接透传 Flink CDC 的完整 changelog，下游 Doris 同步零改造；三是 Partial-Update merge engine 解决了我们多流拼接宽表的痛点，不再需要 Flink State JOIN。
>
> 同时我们也保留了 Iceberg 用于纯离线分析的场景，比如用户行为日志的分析表，不需要更新，用 Iceberg 的读优化更合适。

### 4.2 追问应对速查

| 追问 | 回答要点 |
|------|----------|
| Paimon 这么新，生产稳定吗？ | 阿里内部大规模验证，我们 PoC 跑了 X 个月无重大问题 |
| 如果 Paimon 不行能切回 Iceberg 吗？ | 可以，两者都支持 Parquet/ORC，数据可迁移；XTable 项目也在做格式互转 |
| Compaction 资源怎么规划的？ | 独立 compaction job，占写入 job 约 30% 的资源，异步执行不影响写入 |
| 为什么不用 Hudi？ | Flink 集成紧密度 + changelog 完整性 + partial-update 能力 |
| 未来会不会统一到某一个？ | 短期不会，各有所长；长期看 XTable/UniForm 可能让互操作更容易 |
| Delta Lake 考虑过吗？ | 开源版功能受限，绑定 Spark/Databricks，我们是 Flink 技术栈 |
