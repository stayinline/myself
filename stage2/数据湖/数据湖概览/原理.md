# 数据湖概览：原理篇

## 一、为什么需要数据湖？——传统数仓的四大痛点

### 1.1 Hive 表不支持行级更新 / ACID

Hive 表本质上是 HDFS 上的目录+文件映射，写入是"追加式"的：

```
-- Hive 无法直接 UPDATE / DELETE
UPDATE ods_orders SET status = 'paid' WHERE order_id = 'ORD-20240101-001';
-- ERROR: Hive does not support UPDATE on non-ACID tables
```

**根本原因**：Hive 的元数据只记录"分区目录下有哪些文件"，没有主键索引、没有事务日志、没有 MVCC。要实现更新只能：
- **INSERT OVERWRITE 整个分区**：即使只改一行也要重写整个分区文件
- **启用 Hive ACID（ORC + transactional=true）**：引入 delta 文件，但查询时需合并 delta，性能差且生态兼容性差
- **业务层绕过**：用 `ROW_NUMBER() OVER(PARTITION BY id ORDER BY ts DESC)` 取最新一条，每次查询都要去重

> **面试话术**：Hive 表的存储模型决定了它是 append-only 的，没有行级 CRUD 能力。在 CDC 入湖场景下，binlog 包含 INSERT/UPDATE/DELETE，Hive 只能做 append，下游必须额外去重，这导致 ETL 链路复杂且资源浪费。

### 1.2 实时与离线双链路（Lambda 架构之痛）

典型 Lambda 架构：

```
                    ┌─────────────┐
  CDC/IoT ────────►│  Kafka      │
                    └──────┬──────┘
                           │
              ┌────────────┼────────────┐
              ▼                         ▼
     Flink 实时 ETL              Spark 离线 ETL
              │                         │
              ▼                         ▼
     Redis/Doris/HBase            Hive/HDFS
     (实时服务层)                 (离线分析层)
              │                         │
              └────────────┬────────────┘
                           ▼
                   应用层合并两套结果
```

**痛点**：
| 问题 | 说明 |
|------|------|
| 口径不一致 | 实时和离线两套 ETL 逻辑，指标计算容易出现偏差 |
| 运维成本翻倍 | 两套代码、两套调度、两套监控 |
| 数据冗余 | 同一份原始数据存两份甚至多份 |
| 回溯困难 | 实时链路出了问题，需要用离线链路重新跑，但两者逻辑可能不同 |

### 1.3 小文件问题

当 Flink 以分钟级 checkpoint 写入 HDFS/Hive 时，每个 checkpoint 产生一批文件：

```
/data/orders/dt=2024-01-01/
├── part-00000-chk001.parquet   (2MB)
├── part-00001-chk001.parquet   (3MB)
├── part-00000-chk002.parquet   (2MB)
├── part-00001-chk002.parquet   (3MB)
... (一天 1440 个 checkpoint × N 个并行度 = 数千个小文件)
```

**影响**：
- NameNode 内存压力（每个文件占 ~150 bytes 元数据）
- 查询引擎打开大量文件的 overhead
- List 操作变慢
- 传统解决方式：额外的 compaction 任务，增加运维复杂度

### 1.4 Schema Evolution 困难

Hive 修改列类型或添加列后，旧数据读取可能出错：

```sql
ALTER TABLE orders CHANGE amount amount DECIMAL(18,6);
-- 旧的 INT 文件被新 schema 读取时可能精度丢失或直接报错
```

Hive 的 schema-on-read 依赖 SerDe，缺乏版本化的 schema 管理。

## 二、数据湖的本质定义

### 2.1 核心认知：数据湖 ≠ "往 HDFS 上扔文件"

> **关键洞察**：数据湖的核心不是存储介质（HDFS/S3/OSS），而是 **Table Format（表格式）**。

Table Format 是在对象存储/HDFS 之上的一层**元数据管理层**，它带来：

```
┌──────────────────────────────────────────────────┐
│              Table Format Layer                   │
│  ┌─────────┐ ┌──────────┐ ┌────────┐ ┌────────┐ │
│  │  ACID   │ │ Snapshot │ │Upsert  │ │Schema  │ │
│  │ 事务    │ │ 快照隔离  │ │行级更新 │ │Evolution│ │
│  └─────────┘ └──────────┘ └────────┘ └────────┘ │
│  ┌──────────────┐ ┌─────────────────────────────┐│
│  │Hidden Partition│ │  Streaming Read / ChangeLog ││
│  │隐藏分区       │ │  流式读取 / 变更日志         ││
│  └──────────────┘ └─────────────────────────────┘│
├──────────────────────────────────────────────────┤
│           Storage Layer                           │
│     HDFS / S3 / OSS / GCS / MinIO                │
└──────────────────────────────────────────────────┘
```

### 2.2 Table Format 带来的核心能力

| 能力 | 说明 | 对比 Hive |
|------|------|-----------|
| **ACID 事务** | 写入原子性，读者不受写入中数据影响 | Hive 无事务隔离 |
| **Snapshot 快照** | 每次提交生成不可变快照，支持 Time Travel | Hive 只能看当前状态 |
| **行级 Upsert/Delete** | 基于主键的 MERGE INTO | Hive 只能 INSERT OVERWRITE |
| **Schema Evolution** | 安全的加列、删列、改类型，旧数据兼容读取 | Hive 改了就可能读错 |
| **Hidden Partitioning** | 用户无需写 partition 表达式，引擎自动处理 | Hive 需手动指定分区字段 |
| **Streaming Read** | 增量消费变更，类似 Kafka Consumer | Hive 不支持流读 |
| **小文件治理** | 内置 Compaction 机制 | 需外部脚本 |

### 2.3 主流 Table Format

| 项目 | 发起方 | 首次发布 | 核心定位 |
|------|--------|----------|----------|
| Apache Iceberg | Netflix | 2017 | 通用型，读优化，大社区 |
| Apache Hudi | Uber | 2016 | 最早，upsert 优先，COW/MOR |
| Apache Paimon | Alibaba (Flink Table Store) | 2022 | 流原生，LSM-Tree，Flink 生态 |
| Delta Lake | Databricks | 2017 | Spark 生态绑定，商业版功能更多 |

## 三、数据仓库 vs 数据湖 vs Lakehouse

### 3.1 概念演进

```
数据仓库 (Data Warehouse)
  ├── 结构化数据
  ├── Schema-on-Write
  ├── 强治理、高质量
  └── 不支持非结构化、灵活性差

        ↓ 需求驱动演进

数据湖 (Data Lake)  ← 早期理解：原始数据dump
  ├── 任意格式数据
  ├── Schema-on-Read
  ├── 灵活但易变成"数据沼泽"
  └── 缺乏治理能力

        ↓ Table Format 赋能

Lakehouse (湖仓一体)
  ├── 兼具湖的灵活性和仓的治理能力
  ├── ACID + Schema Evolution + Upsert
  ├── 统一存储，流批一体
  └── 当前行业主流方向
```

### 3.2 详细对比

| 维度 | 传统数仓 (Hive) | 早期数据湖 | Lakehouse (Iceberg/Paimon) |
|------|-----------------|-----------|---------------------------|
| 数据类型 | 结构化 | 任意 | 结构化 + 半结构化 |
| 事务 | ❌ | ❌ | ✅ ACID |
| 更新 | INSERT OVERWRITE | 不支持 | MERGE INTO / Upsert |
| Schema 管理 | 弱 | 无 | 版本化 Schema Evolution |
| 流式读取 | ❌ | ❌ | ✅ Incremental Read |
| 查询性能 | 中等 | 差 | 优化后接近数仓 |
| 治理 | 强 | 弱 | 中-强 |
| 存储成本 | 高 | 低 | 低（对象存储） |
| 实时性 | T+1 | N/A | 分钟级 |

### 3.3 面试话术："数据湖和数据仓库有什么区别？"

> 传统数据仓库以 Hive 为代表，核心问题是：不支持行级更新、没有 ACID 事务、schema evolution 不安全、实时和离线要维护两套链路。早期的数据湖只是把原始文件扔到 HDFS 上，缺乏元数据管理，容易变成数据沼泽。
>
> 现在说的数据湖其实是指 **Lakehouse 架构**，核心是 Table Format（如 Iceberg、Paimon），它在对象存储之上提供了 ACID 事务、快照隔离、行级 Upsert、Schema Evolution、流式读取等能力。本质上是用开放的表格式把"湖的灵活低成本"和"仓的治理高性能"统一起来了。

## 四、数据湖的技术栈全景

### 4.1 读写引擎兼容

```
          ┌─────────┐  ┌───────┐  ┌────────┐  ┌──────────┐
          │  Spark   │  │ Flink │  │ Trino  │  │ StarRocks│
          └────┬─────┘  └───┬───┘  └───┬────┘  └────┬─────┘
               │            │          │             │
          ┌────▼────────────▼──────────▼─────────────▼────┐
          │           Table Format API                     │
          │    Iceberg / Paimon / Hudi / Delta             │
          ├────────────────────────────────────────────────┤
          │        Catalog (Hive Metastore / REST / JDBC)  │
          ├────────────────────────────────────────────────┤
          │     Storage (HDFS / S3 / OSS / MinIO)          │
          └────────────────────────────────────────────────┘
```

### 4.2 Catalog 的作用

Catalog 是数据湖的"电话簿"，负责：
- 注册/发现表
- 指向最新的 metadata pointer
- 提供 namespace/database 层级管理
- 对接权限控制（Ranger、Grants）

常见 Catalog 实现：
- Hive Metastore Catalog（兼容已有 HMS）
- REST Catalog（云原生，Iceberg 推荐）
- JDBC Catalog（轻量级）
- Nessie Catalog（Git-like 版本控制）

## 五、关键认知总结

### 5.1 五个核心要点

1. **数据湖的核心价值不是存储，而是 Table Format 带来的元数据管理能力**
2. **ACID + Snapshot 是区分"真数据湖"和"文件堆"的分水岭**
3. **Lakehouse 是当前行业共识方向，取代传统 Lambda 架构**
4. **选型不是选存储引擎，而是选 Table Format + 计算引擎的组合**
5. **流批一体的前提是 Table Format 同时支持 streaming read 和 batch read**

### 5.2 常见误区澄清

| 误区 | 正解 |
|------|------|
| 数据湖就是 HDFS | 数据湖的核心是 Table Format，存储可以是 S3/OSS |
| 上了数据湖就不需要数仓了 | Lakehouse 替代的是传统 Hive 数仓，但 OLAP 服务层仍需 Doris/StarRocks |
| 数据湖支持实时就等于低延迟 | 数据湖的实时是"分钟级可见"，不是毫秒级 KV 查询 |
| 所有 Table Format 都一样 | Iceberg 偏读优化，Paimon 偏流原生，Hudi 偏 upsert，各有侧重 |
| Schema Evolution 随便改 | 有兼容性约束，如不能直接把 STRING 改成 INT |

完整阅读顺序见 [数据湖 README](README.md)。概要如下：

