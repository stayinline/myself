# 数据湖概览：实战篇

## 一、场景驱动：用具体业务论证"为什么上数据湖"

### 1.1 场景一：考试业务实时更新 + 离线回溯

#### 背景描述

某在线教育平台，核心业务是考试系统。数据需求包括：

- **实时**：考生答题过程中，需要实时更新答题状态（开始→进行中→已提交→已批改），运营看板需秒级刷新
- **离线**：算法团队需要对历史考试数据进行回溯分析，训练推荐模型
- **变更频繁**：成绩可能被复议修改，订单可能退款，状态经常回退

#### 传统 Hive 架构下的痛点

```
痛点 1：无法行级更新
─────────────────────
binlog: UPDATE exam_record SET score=95 WHERE record_id='R001';

Hive 处理方式：
  方案A: INSERT OVERWRITE 整个 dt=2024-01-01 分区
         → 该分区有 500万行，只为改 1 行重写全部
         → 耗时 30min，资源浪费严重
  
  方案B: Append 所有 binlog，查询时 ROW_NUMBER() 去重
         → 存储膨胀 3-5x
         → 每次查询都要全表扫描+窗口函数
         → 下游 ETL 复杂度高

痛点 2：实时离线双链路
─────────────────────
实时链路: CDC → Flink → Redis (最新状态)
离线链路: CDC → Kafka → Hive (T+1 全量)

问题:
  - Redis 里的数据和 Hive 里的数据口径不一致
  - 算法团队说"昨天的通过率是 72%"，运营说"实时看是 68%"
  - 排查发现：实时链路过滤了"测试账号"，离线链路没过滤
  - 两套 ETL 代码维护成本极高

痛点 3：小文件灾难
─────────────────────
Flink 每 60s checkpoint → 每天 1440 批次 × 8 并行度 = 11,520 个文件
单个文件平均 1-5MB

后果:
  - NameNode 内存占用激增
  - Spark 读取该分区时需要 open 11,520 个文件，task 调度开销巨大
  - 需要额外的 compaction 定时任务，又增加一层运维
```

#### 数据湖方案如何解决

```
CDC → Flink → Paimon/Iceberg 表
                │
        ┌───────┴───────┐
        ▼               ▼
   Streaming Read    Batch Read
   (实时消费变更)    (离线回溯查询)
        │               │
        ▼               ▼
   Doris/StarRocks   Spark SQL
   (实时 OLAP)      (算法回溯)
```

| 痛点 | 数据湖解法 |
|------|-----------|
| 行级更新 | MERGE INTO / Upsert，只写变更行 |
| 双链路 | 一份存储，流批两种消费模式 |
| 小文件 | 内置 Compaction 自动合并 |
| Schema Evolution | 安全加列改类型，旧数据兼容读 |
| 口径一致 | 同一份数据同一个 ETL，消除偏差 |

### 1.2 场景二：IoT 设备时序数据入湖

#### 痛点

- 设备上报频率高（每秒数万条），状态字段频繁 UPDATE
- 需要保留原始明细用于故障回溯
- 同时需要聚合后的指标供实时监控
- 设备固件升级后新增字段，Schema 频繁变化

#### 数据湖价值

- Paimon Primary Key Table 天然支持高频 Upsert
- Schema Evolution 允许安全加列，不影响历史数据读取
- Compaction 自动合并小文件
- 流读给下游实时告警，批读给算法做异常检测训练

## 二、面试话术："为什么要上数据湖？它比 Hive 数仓强在哪？"

### 2.1 结构化回答框架（STAR 变体）

> **建议用时**：2-3 分钟

**开场（定位问题）**：
> 我们之前用的是典型的 Hive 数仓 + Flink 实时链路的 Lambda 架构，在实际运营中遇到了几个核心痛点。

**展开（三个痛点 + 对应解法）**：

> 第一，**Hive 不支持行级更新**。我们的业务有大量 CDC 场景，比如订单状态变更、成绩复议等，binlog 里有大量 UPDATE/DELETE。Hive 只能 append 或者整分区覆写，要么存储膨胀 3-5 倍加上去重开销，要么为改一行重写几百万行的分区。上了数据湖之后，Table Format 原生支持 MERGE INTO 和 Upsert，只处理变更行，写入效率和存储效率都大幅提升。
>
> 第二，**实时和离线两套链路导致指标不一致**。同样的数据源，Flink 实时 ETL 和 Spark 离线 ETL 逻辑分别维护，经常出现"实时看板一个数、离线报表另一个数"的情况，排查成本很高。数据湖作为统一存储层，一份数据同时支持流式读取和批量读取，ETL 逻辑只写一套，从根本上保证了口径一致性。
>
> 第三，**小文件和 Schema Evolution 的运维负担**。Flink 分钟级 checkpoint 产生大量小文件，NameNode 压力大，查询慢，还要额外跑 compaction 任务。而且业务迭代快，加字段很频繁，Hive 改了 schema 旧数据就可能读错。数据湖内置了 Compaction 机制和版本化的 Schema Evolution，这两个问题都得到了系统性解决。

**收尾（总结价值）**：
> 总结来说，数据湖的核心价值是通过 Table Format 在开放存储之上补齐了 ACID、Upsert、快照隔离、流式读取这些能力，把原来 Lambda 架构下分散的实时和离线存储统一起来，降低了运维复杂度和数据不一致风险。

### 2.2 追问应对

**Q: 数据湖能完全替代 Hive 吗？**
> 不是替代 Hive 的所有场景，而是替代 Hive 作为"可更新数据底座"的场景。对于纯离线、不需要更新的日志分析，Hive/Parquet 仍然够用。但对于 CDC 入湖、需要频繁更新、需要流批一体的场景，Table Format 的优势非常明显。另外，数据湖替代的是存储和元数据层，OLAP 服务层还是需要 Doris/StarRocks 这类引擎来保证亚秒级查询。

**Q: 你们选了哪个 Table Format？为什么？**
> （根据实际情况回答，以下为参考）
> 我们选了 Paimon，主要原因是我们的核心场景是 CDC 实时入湖，Paimon 基于 LSM-Tree 的设计天然适合高频 Upsert，而且它是 Flink 生态原生的，与我们的计算引擎集成最紧密。如果是以离线分析为主、偶尔更新的场景，Iceberg 可能是更好的选择，因为它的读性能优化更成熟，社区生态也更广。

**Q: 上了数据湖之后有什么代价或挑战？**
> 主要有三点：一是学习成本，Table Format 的概念（snapshot、manifest、compaction）对团队是新知识；二是 Compaction 的资源消耗，特别是 Paimon 的 LSM merge 在数据量大时需要单独规划资源；三是生态兼容性，虽然主流引擎都支持了 Iceberg/Paimon，但某些 BI 工具或小众引擎的适配还不够完善。

## 三、连接"流批一体"叙事

### 3.1 从数据湖到流批一体的逻辑链

```
数据湖 (Table Format)
    │
    ├── 支持 Upsert → 实时写入不再是问题
    │
    ├── 支持 Streaming Read → 下游可以增量消费
    │
    ├── 支持 Batch Read → 离线回溯无需额外存储
    │
    └── 三者合一 = 流批一体存储底座
         │
         ▼
    一份存储，两种消费模式
    → 消灭 Lambda 双链路
    → 指标口径天然一致
    → 运维成本大幅降低
```

### 3.2 面试中的叙事串联

> 数据湖是我们实现流批一体的基础。传统 Lambda 架构的根本问题是实时和离线用了两套存储、两套 ETL，所以才会有口径不一致和运维翻倍的问题。数据湖通过 Table Format 提供了统一的存储层——同一张表既可以被 Flink 流式写入和读取，也可以被 Spark 批量扫描。这就让"一份数据、两种消费"成为可能，从而真正实现了流批一体，而不是仅仅在计算引擎层面做流批统一的 API。

## 五、常见面试追问速查

| 追问 | 回答要点 |
|------|----------|
| 数据湖和数据沼泽有什么区别？ | 有无 Table Format 治理：ACID、Schema 版本化、Catalog 管理 |
| Lakehouse 和传统 MPP 数仓什么关系？ | 互补而非替代，Lakehouse 做统一底座，MPP 做高性能服务层 |
| 为什么不用 Delta Lake？ | Delta 开源版功能受限，深度绑定 Spark/Databricks，跨引擎兼容不如 Iceberg/Paimon |
| 数据湖的事务是怎么实现的？ | 原子替换 metadata pointer，详见 Iceberg/Paimon 原理篇 |
| 小文件问题数据湖怎么解决的？ | 内置 Compaction，Paimon 在 LSM merge 时自动合并，Iceberg 有 rewrite_data_files action |
| 你们的流批一体是怎么做到的？ | Paimon 统一存储 + Flink 流写流读 + Spark 批读，详见流批一体落地架构篇 |
