# Apache Iceberg 原理篇

## 一、Iceberg 架构总览

### 1.1 核心设计理念

Apache Iceberg 由 Netflix 于 2017 年开源，设计目标是：
- **可序列化的隔离级别**（Serializable Isolation）
- **高效的表扫描**（Scan Planning 在元数据层完成，无需 list 文件）
- **安全的 Schema/Partition Evolution**
- **与存储系统解耦**（不依赖 HDFS rename 等文件系统操作）

### 1.2 四层元数据结构

```
┌─────────────────────────────────────────────────────────────┐
│                     Catalog Layer                            │
│  (Hive Metastore / REST / JDBC / Nessie / Glue / DynamoDB) │
│  存储: table metadata location pointer                       │
└──────────────────────────┬──────────────────────────────────┘
                           │ points to
                           ▼
┌─────────────────────────────────────────────────────────────┐
│               Metadata File (JSON)                           │
│  包含: schema, partition-spec, sort-order,                   │
│        current-snapshot-id, snapshot-log,                    │
│        properties, format-version                            │
│  命名: v1.metadata.json / v2.metadata.json / ...             │
└──────────────────────────┬──────────────────────────────────┘
                           │ references
                           ▼
┌─────────────────────────────────────────────────────────────┐
│              Snapshot → Manifest List (Avro)                 │
│  每个 Snapshot 对应一个 Manifest List                         │
│  记录: 该快照包含哪些 Manifest File                            │
│  命名: snap-<id>-<uuid>.avro                                 │
└──────────────────────────┬──────────────────────────────────┘
                           │ lists
                           ▼
┌─────────────────────────────────────────────────────────────┐
│              Manifest File (Avro)                            │
│  记录: 一组 Data File 的元信息                                 │
│  包括: file_path, file_format, record_count,                 │
│        file_size_in_bytes, column_stats(min/max/null_count), │
│        partition_data                                        │
│  关键: 带有列级统计信息，用于 Data Skipping                    │
└──────────────────────────┬──────────────────────────────────┘
                           │ tracks
                           ▼
┌─────────────────────────────────────────────────────────────┐
│              Data Files (Parquet / ORC / Avro)               │
│  实际的业务数据存储                                           │
└─────────────────────────────────────────────────────────────┘
```

### 1.3 为什么这样设计？

| 设计决策 | 原因 |
|----------|------|
| Metadata File 用 JSON | 人类可读，方便调试 |
| Manifest List / Manifest 用 Avro | 紧凑、高效、支持 schema evolution |
| Manifest 带列统计 | 查询规划时做 Data Skipping，减少读取量 |
| 不依赖文件系统 rename | S3 等对象存储不支持原子 rename，Iceberg 用"写新文件+原子替换指针"代替 |
| Snapshot 不可变 | 保证读一致性，任何时刻的读者看到的都是同一个快照 |

## 二、Snapshot 机制详解

### 2.1 什么是 Snapshot

每次对表的写入（INSERT/UPDATE/DELETE/COMPACT）都会生成一个新的 Snapshot：

```
Snapshot #1 (t=10:00)
  └── Manifest List A
       ├── Manifest M1 → [file1.parquet, file2.parquet]
       └── Manifest M2 → [file3.parquet]

Snapshot #2 (t=10:05) ← INSERT 新数据
  └── Manifest List B
       ├── Manifest M1 → [file1.parquet, file2.parquet]  ← 复用！
       ├── Manifest M2 → [file3.parquet]                  ← 复用！
       └── Manifest M3 → [file4.parquet]                  ← 新增

Snapshot #3 (t=10:10) ← DELETE file2
  └── Manifest List C
       ├── Manifest M1' → [file1.parquet]                 ← 重写(去掉file2)
       ├── Manifest M2 → [file3.parquet]                  ← 复用
       └── Manifest M3 → [file4.parquet]                  ← 复用
```

### 2.2 Snapshot Isolation（快照隔离）

- **读者**始终读取某个确定的 Snapshot，不会看到写入一半的数据
- **写者**创建新的 Snapshot，不影响正在进行的读操作
- 通过原子替换 Metadata File 中的 `current-snapshot-id` 实现提交

```java
// 伪代码：Iceberg 提交流程
while (true) {
    base = table.currentMetadata();
    // 基于 base 构建新的 metadata
    updated = base.newSnapshot(newManifestList);
    // CAS: 原子替换 metadata pointer
    if (table.commit(base, updated)) {
        break; // 提交成功
    }
    // 如果失败（并发冲突），retry
}
```

### 2.3 Time Travel

```sql
-- 查询指定时间点的快照
SELECT * FROM orders FOR TIMESTAMP AS OF '2024-01-15 10:00:00';

-- 查询指定 snapshot id
SELECT * FROM orders FOR VERSION AS OF 389274658;

-- 查看快照历史
SELECT * FROM orders.snapshots;
-- snapshot_id | timestamp_ms | operation | summary
```

## 三、Hidden Partitioning（隐藏分区）

### 3.1 Hive 分区的痛点

```sql
-- Hive 物理分区：用户必须知道分区字段
SELECT * FROM orders WHERE dt = '2024-01-15';
-- 如果写成 WHERE order_time >= '2024-01-15'，不会走分区裁剪！

-- 而且分区值就是目录名，修改分区策略需要迁移数据
```

### 3.2 Iceberg 的解决方案

Iceberg 将分区策略存储在元数据中，对用户透明：

```sql
-- 建表时定义分区转换
CREATE TABLE orders (
    order_id BIGINT,
    user_id BIGINT,
    amount DECIMAL(10,2),
    order_time TIMESTAMP
) PARTITIONED BY (
    days(order_time),      -- 按天分区，但用户不需要知道
    bucket(16, user_id)    -- 按 user_id hash 分桶
);

-- 查询时正常写业务条件，Iceberg 自动做分区裁剪
SELECT * FROM orders 
WHERE order_time >= '2024-01-15' AND order_time < '2024-01-16';
-- Iceberg 内部将 order_time 范围转换为 days 分区范围
```

### 3.3 支持的分区转换

| 转换函数 | 说明 | 示例 |
|----------|------|------|
| `identity` | 原值分区 | `identity(status)` |
| `bucket(N, col)` | Hash 分桶 | `bucket(16, user_id)` |
| `truncate(W, col)` | 截断分区 | `truncate(10, amount)` |
| `years(ts)` | 按年 | `years(created_at)` |
| `months(ts)` | 按月 | `months(created_at)` |
| `days(ts)` | 按天 | `days(order_time)` |
| `hours(ts)` | 按小时 | `hours(event_time)` |

## 四、Schema & Partition Evolution

### 4.1 Schema Evolution

Iceberg 使用 **field-id** 而非字段名来标识列：

```
Schema v1: field-id=1 name=amount type=int
Schema v2: field-id=1 name=amount type=decimal(10,2)  ← 改名改类型，id不变
           field-id=2 name=currency type=string         ← 新增列
```

- 旧数据文件中没有 field-id=2 的列 → 读取时自动填充 NULL
- field-id=1 的类型从 int 改为 decimal → 安全的 widening conversion
- 删除列只是标记为 inactive，不影响旧数据

**安全约束**：
- ✅ int → long, float → double, string widening
- ❌ string → int, double → float（narrowing 不允许）
- ❌ 不能改变 field-id 的含义

### 4.2 Partition Evolution

Iceberg 允许在不迁移数据的情况下修改分区策略：

```sql
-- 初始按天分区
ALTER TABLE orders ADD PARTITION FIELD hours(order_time);
-- 新数据按小时分区，旧数据仍按天分区
-- 两种分区共存，查询时 Iceberg 自动处理
```

这是通过给每个 Partition Spec 分配 spec-id 实现的：
- 旧数据文件的 manifest 记录 spec-id=0（按天）
- 新数据文件的 manifest 记录 spec-id=1（按小时）
- 查询规划时分别处理不同 spec-id 的文件

## 五、ACID 的实现机制

### 5.1 核心原则：原子元数据指针替换

Iceberg 的事务不是传统数据库的 WAL + MVCC，而是：

```
写入流程:
1. 写入新的 Data Files → storage
2. 创建新的 Manifest Files → storage  
3. 创建新的 Manifest List → storage
4. 创建新的 Metadata File → storage
5. 原子替换 Catalog 中的 metadata location pointer

步骤 1-4 都是写新文件，不影响现有读者
步骤 5 是唯一的"切换点"，必须是原子的
```

### 5.2 不同 Catalog 的原子性保证

| Catalog | 原子性机制 |
|---------|-----------|
| Hive Metastore | HMS 的 CAS update（database params 或 table param） |
| REST Catalog | HTTP PUT with ETag / conditional update |
| JDBC Catalog | SQL UPDATE with version check |
| DynamoDB | Conditional update expression |
| GCS/S3 | Object versioning + conditional write |

### 5.3 乐观并发控制

Iceberg 采用 Optimistic Concurrency Control：
- 多个 Writer 可以并行写入数据文件
- 提交时检查 base version 是否变化
- 如果冲突则 retry（rebase on latest snapshot）
- 冲突检测粒度：Snapshot 级别（可配置为更细粒度）

## 六、COW vs MOR

### 6.1 Copy-on-Write (COW)

```
UPDATE orders SET status='paid' WHERE order_id='R001';

COW 行为:
1. 找到包含 R001 的 data file (file_a.parquet, 100万行)
2. 复制整个 file_a → file_a_new.parquet，其中 R001 被修改
3. 在新 Snapshot 中用 file_a_new 替换 file_a

优点: 读取时无需合并，查询性能最优
缺点: 写入放大严重，改一行要重写整个文件
适用: 读多写少，批量更新场景
```

### 6.2 Merge-on-Read (MOR)

```
UPDATE orders SET status='paid' WHERE order_id='R001';

MOR 行为:
1. 将变更写入 delete file (position-delete 或 equality-delete)
2. 新 Snapshot 引用原始 data file + 新的 delete file

读取时:
  data file scan → apply delete files → return results

优点: 写入快，只写变更部分
缺点: 读取时需要合并 delete files，查询有额外开销
适用: 写多读少，频繁小量更新
```

### 6.3 Delete File 类型

| 类型 | 说明 | 效率 |
|------|------|------|
| Position Delete | 记录 (file_path, pos) | 精确，推荐 |
| Equality Delete | 记录 (col=val) 匹配条件 | 灵活但慢，需全文件扫描匹配 |

### 6.4 V2 Format 改进

Iceberg V2 引入了 Row-Level Deletes：
- 支持 Position Delete Files（更高效）
- 引入 Sequence Number 解决 Equality Delete 的顺序问题
- 更好的 MOR 读取性能

## 八、面试话术："Iceberg 的快照和隐藏分区解决了什么？"

### 8.1 关于快照

> Iceberg 的快照机制解决了两个核心问题：
> 
> 第一是**读写隔离**。传统 Hive 写入过程中读者可能读到不完整的数据。Iceberg 每次提交生成一个新的不可变快照，读者始终读取某个完整的快照版本，写入过程对读者完全不可见。这通过原子替换 metadata pointer 来实现，不需要锁。
>
> 第二是**Time Travel**。每个快照都保留了完整的文件清单，所以可以回溯到任意历史时间点的数据。这对数据审计、问题排查、算法回测都非常有价值。比如我们发现今天的数据有问题，可以直接查昨天的快照确认是什么时候出的错。

### 8.2 关于隐藏分区

> 隐藏分区解决了 Hive 分区的两个顽疾：
>
> 第一是**分区感知泄漏**。Hive 要求用户在查询时必须显式指定分区字段才能走分区裁剪，否则就全表扫描。Iceberg 把分区策略存在元数据里，用户只需要写正常的业务过滤条件，引擎自动将条件下推到分区层面进行裁剪。
>
> 第二是**分区演进困难**。Hive 要改分区策略就得迁移所有数据。Iceberg 允许多个分区规格共存，新数据用新分区策略，旧数据保持原样，查询时自动适配。这意味着业务增长后可以无缝调整分区粒度，比如从天级调整为小时级，而不需要停机迁移。

### 8.3 关于 ACID

> Iceberg 的 ACID 不是传统数据库那种基于 WAL 和锁的实现，而是利用了"不可变文件 + 原子指针替换"的思路。所有写入都是创建新文件，不会影响已有读者。只有在最后提交时，才通过 Catalog 提供的原子操作（如 HMS CAS、REST ETag）替换 metadata pointer。这是一种乐观并发控制，适合大数据分析场景中"读多写少、写入批量大"的特点。

## 九、关键概念速查表

| 概念 | 一句话解释 |
|------|-----------|
| Metadata File | 表的完整状态描述（JSON），包含 schema、snapshot 引用等 |
| Snapshot | 某一时刻表的不可变视图，指向一个 Manifest List |
| Manifest List | 列出某个 Snapshot 包含的所有 Manifest File |
| Manifest File | 记录一组 Data File 的路径、大小、列统计等信息 |
| Data File | 实际存储业务数据的 Parquet/ORC 文件 |
| Hidden Partition | 分区策略对用户透明，引擎自动做分区裁剪 |
| Schema Evolution | 基于 field-id 的安全 schema 变更 |
| Partition Evolution | 不改数据即可变更分区策略 |
| COW | 写时复制整文件，读快写慢 |
| MOR | 写时只写 delete file，读时合并，写快读慢 |
| Sequence Number | V2 格式中用于确定 delete file 生效顺序 |
