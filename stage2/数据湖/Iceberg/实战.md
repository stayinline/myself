# Apache Iceberg：实战篇

## 一、环境搭建与基础操作

### 1.1 Flink + Iceberg 快速起步

```sql
-- Flink SQL 中创建 Iceberg Catalog
CREATE CATALOG iceberg_catalog WITH (
    'type' = 'iceberg',
    'catalog-type' = 'hadoop',           -- 或 hive / rest
    'warehouse' = 's3://my-bucket/warehouse',
    'io-impl' = 'org.apache.iceberg.aws.s3.S3FileIO'
);

USE CATALOG iceberg_catalog;

-- 创建数据库
CREATE DATABASE IF NOT EXISTS exam_db;
USE exam_db;

-- 创建 Iceberg 表
CREATE TABLE exam_records (
    record_id   BIGINT,
    student_id  BIGINT,
    exam_id     BIGINT,
    score       DECIMAL(5,2),
    status      STRING,
    created_at  TIMESTAMP(3),
    updated_at  TIMESTAMP(3),
    PRIMARY KEY (record_id) NOT ENFORCED
) PARTITIONED BY (days(created_at))
WITH (
    'format-version' = '2',               -- 启用 V2 格式，支持行级删除
    'write.format.default' = 'parquet',
    'write.parquet.compression-codec' = 'zstd'
);
```

### 1.2 Spark + Iceberg 快速起步

```python
# PySpark 配置
from pyspark.sql import SparkSession

spark = SparkSession.builder \
    .config("spark.sql.extensions", "org.apache.iceberg.spark.extensions.IcebergSparkSessionExtensions") \
    .config("spark.sql.catalog.iceberg_catalog", "org.apache.iceberg.spark.SparkCatalog") \
    .config("spark.sql.catalog.iceberg_catalog.type", "hadoop") \
    .config("spark.sql.catalog.iceberg_catalog.warehouse", "s3://my-bucket/warehouse") \
    .getOrCreate()

# 建表
spark.sql("""
    CREATE TABLE iceberg_catalog.exam_db.exam_records (
        record_id   BIGINT,
        student_id  BIGINT,
        exam_id     BIGINT,
        score       DECIMAL(5,2),
        status      STRING,
        created_at  TIMESTAMP,
        updated_at  TIMESTAMP
    ) USING iceberg
    PARTITIONED BY (days(created_at))
    TBLPROPERTIES ('format-version'='2')
""")
```

## 二、核心操作演练

### 2.1 INSERT / UPDATE / DELETE

```sql
-- 插入数据
INSERT INTO exam_records VALUES 
    (1001, 201, 1, 85.5, 'graded', '2024-01-15 10:00:00', '2024-01-15 10:30:00'),
    (1002, 202, 1, NULL, 'pending', '2024-01-15 10:05:00', '2024-01-15 10:05:00');

-- 行级更新（V2 格式）
UPDATE exam_records 
SET score = 92.0, status = 'graded', updated_at = '2024-01-15 11:00:00'
WHERE record_id = 1002;

-- MERGE INTO（CDC 典型用法）
MERGE INTO exam_records AS target
USING cdc_staging AS source
ON target.record_id = source.record_id
WHEN MATCHED AND source.op = 'D' THEN DELETE
WHEN MATCHED THEN UPDATE SET *
WHEN NOT MATCHED THEN INSERT *;

-- 删除
DELETE FROM exam_records WHERE status = 'cancelled';
```

### 2.3 Schema Evolution 实操

```sql
-- 添加列
ALTER TABLE exam_records ADD COLUMN retry_count INT DEFAULT 0;

-- 修改列类型（安全 widening）
ALTER TABLE exam_records ALTER COLUMN score TYPE DECIMAL(8,4);

-- 重命名列
ALTER TABLE exam_records RENAME COLUMN status TO exam_status;

-- 删除列（标记为 inactive，旧数据不受影响）
ALTER TABLE exam_records DROP COLUMN retry_count;

-- 验证：旧快照仍然可以正常读取
SELECT * FROM exam_records FOR VERSION AS OF 567890123456789;
-- 新增列显示为 NULL，删除的列不再出现
```

Table table = catalog.loadTable(TableIdentifier.of("exam_db", "exam_records"));

// 查看所有快照
for (Snapshot snapshot : table.snapshots()) {
    System.out.printf("Snapshot %d at %s, op=%s%n",
        snapshot.snapshotId(),
        new Date(snapshot.timestampMillis()),
        snapshot.operation());
    
    // 遍历该快照的所有 data files

            .useSnapshot(snapshot.snapshotId())

## 四、小文件问题与 Compaction

### 4.1 小文件产生原因

```
Flink checkpoint interval = 60s
并行度 = 8
每天运行 24h

→ 1440 checkpoints × 8 files = 11,520 files/day
→ 平均每个文件 1-10 MB
→ NameNode 压力 + 查询性能下降
```

### 4.2 Compaction 策略

#### Spark Action（推荐离线场景）

```sql
-- 重写小文件
CALL iceberg_catalog.system.rewrite_data_files(
    table => 'exam_db.exam_records',
    options => map(
        'target-file-size-bytes', '134217728',    -- 128MB
        'min-input-files', '5'                     -- 至少5个小文件才触发合并
    )
);

-- 排序重写（同时做 Z-Order 优化）
CALL iceberg_catalog.system.rewrite_data_files(
    table => 'exam_db.exam_records',
    sort_strategy => 'z-order',
    z_order_col_name => 'student_id'
);

-- 清理过期快照和孤立文件
CALL iceberg_catalog.system.expire_snapshots(
    table => 'exam_db.exam_records',
    older_than => timestamp '2024-01-14 00:00:00',
    retain_last => 5
);

CALL iceberg_catalog.system.remove_orphan_files(
    table => 'exam_db.exam_records',
    older_than => timestamp '2024-01-14 00:00:00'
);
```

#### Flink Action（推荐实时场景）

```bash
# 使用 iceberg-flink-actions JAR 提交 compaction 作业
./bin/flink run /path/to/iceberg-flink-actions.jar \
  --action compact \
  --warehouse s3://my-bucket/warehouse \
  --database exam_db \
  --table exam_records \
  --target-file-size-bytes 134217728
```

### 4.3 Compaction 注意事项

| 要点 | 说明 |
|------|------|
| 不要过于频繁 | Compaction 本身消耗资源，建议 30min-2h 一次 |
| 设置合理目标大小 | 通常 128MB-256MB，太小浪费 IO，太大影响裁剪 |
| 保留足够快照 | expire_snapshots 时保留最近 N 个，避免 Time Travel 失效 |
| 监控文件大小分布 | 通过 metrics 观察 P50/P99 文件大小 |
| MOR 表更需关注 | Delete Files 积累过多会严重影响读性能 |

## 五、面试话术整理

### 5.1 "Iceberg 的快照机制是怎么工作的？"

> Iceberg 每次写入提交都会生成一个新的 Snapshot。Snapshot 是不可变的，它通过一个 Manifest List 引用一组 Manifest File，每个 Manifest File 记录了一批 Data File 的路径、大小和列级统计信息。
>
> 关键的巧妙之处在于**Manifest 复用**——如果某个 Manifest 在新旧快照之间没有变化，新快照直接引用旧的 Manifest File，不需要复制。这使得创建快照的开销非常小。
>
> 提交的原子性通过 Catalog 层的 CAS 操作保证：写入新的 Metadata File 后，原子替换 current-snapshot-id 指针。读者始终看到某个完整的快照，不会读到写了一半的数据。

### 5.2 "Hidden Partition 比 Hive 分区好在哪？"

> 两个核心优势。第一，用户查询时不需要知道底层的分区字段和转换方式，只需要写业务过滤条件，Iceberg 自动将条件下推到分区层进行裁剪。比如表按 `days(order_time)` 分区，用户写 `WHERE order_time >= '2024-01-15'` 就能自动走分区裁剪，而 Hive 必须写 `WHERE dt = '2024-01-15'`。
>
> 第二，支持 Partition Evolution。可以在不迁移数据的情况下改变分区策略，新旧分区共存。比如初期按天分区，后来发现需要更细粒度，可以直接加小时级分区，旧数据保持天级不变。Hive 要改分区就得全量数据迁移。

### 5.3 "COW 和 MOR 怎么选？"

> 简单原则：**读多写少选 COW，写多读少选 MOR**。
>
> COW 在写入时就完成合并，读取时直接扫描数据文件，性能最好。但代价是写入放大——改一行就要重写整个文件。适合批量 ETL、低频更新的场景。
>
> MOR 写入时只生成 delete file，速度快。但读取时需要将 data file 和 delete file 做合并，有额外开销。适合 CDC 实时入湖、高频小量更新的场景。需要注意的是，MOR 表的 delete file 如果不及时 compaction，读性能会持续恶化。

### 5.4 "Iceberg 的小文件怎么治理？"

> Iceberg 提供了内置的 rewrite_data_files Action，可以通过 Spark 或 Flink 触发。它会读取多个小文件，合并成目标大小的新文件，然后生成新快照替换旧文件。还可以结合 Z-Order 排序优化数据局部性。
>
> 在生产中，我们一般配一个定时任务每 30 分钟到 2 小时跑一次 compaction，目标文件大小设 128MB。同时要配合 expire_snapshots 清理过期的快照和孤立文件，否则存储会持续增长。对于 MOR 表，compaction 更加重要，因为 delete file 的积累会直接影响查询性能。

## 七、常见问题排查

| 问题 | 可能原因 | 解决方案 |
|------|---------|---------|
| 查询很慢 | 小文件过多 | 执行 compaction |
| 查询很慢（MOR） | delete files 堆积 | 执行 compaction 合并 deletes |
| Time Travel 报错 | 快照已过期被清理 | 调整 expire_snapshots 保留策略 |
| 并发写入冲突 | 多 Writer 竞争 | 调大 retry 次数，检查写入频率 |
| Schema Evolution 失败 | 不安全的类型转换 | 检查兼容性矩阵 |
| S3 写入慢 | 未开启 multipart upload | 配置 iceberg.s3.multipart.threshold |
| Catalog 不一致 | HMS 版本过低 | 升级 HMS 或切换 REST Catalog |
