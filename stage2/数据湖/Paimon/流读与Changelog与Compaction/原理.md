# Paimon 流读与 Changelog 与 Compaction：原理篇

## 一、为什么需要完整的 Changelog？

### 1.1 下游消费场景的需求

Paimon 作为统一存储层，下游有多种消费模式：

```
Paimon PK Table
      │
      ├── Batch Read → Spark SQL / Trino (离线分析)
      │
      ├── Streaming Read → Flink Job (实时 ETL)
      │       └── 需要完整的 -I/+I/-U/+U changelog
      │           才能正确驱动下游聚合、JOIN、写入 OLAP
      │
      └── Streaming Read → Doris/StarRocks Routine Load
              └── 需要准确的 INSERT/UPDATE/DELETE 语义
```

### 1.2 什么是完整的 Changelog

在 Flink 的 changelog 协议中，完整变更包含四种消息类型：

| 消息类型 | 含义 | 示例 |
|----------|------|------|
| `+I` (INSERT) | 新增一行 | (pk=1, name='Alice', score=85) |
| `-D` (DELETE) | 删除一行 | (pk=1, name='Alice', score=85) |
| `-U` (UPDATE_BEFORE) | 更新前的旧值 | (pk=1, name='Alice', score=85) |
| `+U` (UPDATE_AFTER) | 更新后的新值 | (pk=1, name='Alice', score=92) |

**为什么 -U/+U 很重要？**

```sql
-- 下游做 SUM(score) 聚合
-- 如果没有 -U/+U，只有 +U:
--   收到 +U(92)，不知道旧值是 85
--   无法正确维护 SUM: SUM = SUM - old + new = SUM - 85 + 92

-- 有了完整的 -U/+U:
--   收到 -U(85): SUM = SUM - 85
--   收到 +U(92): SUM = SUM + 92
--   结果正确 ✅
```

### 1.3 Iceberg 的局限

Iceberg 的 streaming read 只能产出 **+I（append）** 类型的 changelog。对于 UPDATE 操作，Iceberg 无法高效地产出对应的 -U 消息，因为：
- COW 模式下，旧文件被替换，但旧值没有单独记录
- MOR 模式下，delete file 只记录了删除位置，不便于提取旧值

这意味着下游如果用 Iceberg 做流式消费，**不能直接做聚合运算**，必须先做一次"changelog 补全"（通常通过 lookup 或 state）。

## 二、Changelog Producer 四种模式详解

### 2.1 模式总览

```
┌─────────────────────────────────────────────────────────────┐
│                  Changelog Producer Modes                     │
│                                                              │
│  none          → 不产出 changelog，仅支持 batch read          │
│  input         → 直接透传上游写入的 changelog                  │
│  lookup        → 写入时 lookup 已有值，补全 -U                │
│  full-compaction → Compaction 时对比新旧值，产出完整 changelog │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 none 模式

```sql
'changelog-producer' = 'none'
```

- **行为**：不生成任何 changelog 文件
- **流读**：不支持（报错）
- **批读**：正常
- **适用**：纯离线分析表，不需要流式消费
- **开销**：零额外开销

### 2.3 input 模式

```sql
'changelog-producer' = 'input'
```

- **行为**：直接将上游写入的 changelog 透传为表的 changelog
- **前提**：上游必须已经提供了完整的 -I/+I/-U/+U changelog
- **典型来源**：CDC Source（MySQL binlog → Flink CDC → Paimon）
- **优点**：零额外计算开销，延迟最低
- **缺点**：如果上游只有 +I（如 append-only 日志），则下游也只能看到 +I

```
CDC Source (-I/+I/-U/+U) → Paimon (input mode) → 下游 Stream Read (-I/+I/-U/+U) ✅
Append Source (+I only) → Paimon (input mode) → 下游 Stream Read (+I only) ⚠️
```

### 2.4 lookup 模式

```sql
'changelog-producer' = 'lookup'
```

- **行为**：每次写入新记录时，先 lookup 该主键的当前值，然后产出 -U(old) + +U(new)
- **实现**：维护一个内存中的 KV 索引（类似 Flink State），用于快速查找当前值
- **优点**：实时性好，写入时即产出完整 changelog
- **缺点**：
  - 额外的 lookup 开销（每次写入多一次读）
  - 内存消耗（需要缓存热数据）
  - 如果数据量很大且热点不明显，lookup 可能成为瓶颈

```
写入 (pk=1, val='new') 
  → lookup 当前值 (pk=1, val='old')
  → 产出 changelog: -U(pk=1, val='old'), +U(pk=1, val='new')
  → 写入 LSM
```

### 2.5 full-compaction 模式

```sql
'changelog-producer' = 'full-compaction'
```

- **行为**：在 Compaction 过程中，对比 merge 前后的值，产出完整的 changelog
- **实现**：Compaction merge 多个 Sorted Run 时，同一个 key 的旧值和新值都在内存中，可以直接比较
- **优点**：
  - 无需额外的 lookup 开销
  - 利用 Compaction 已有的 merge 过程，边际成本低
  - 适合大数据量场景
- **缺点**：
  - Changelog 产出延迟取决于 Compaction 频率
  - 如果 Compaction 间隔长，下游感知变更的延迟也长

```
Compaction:
  L0 Run1: (pk=1, val='v1')
  L0 Run2: (pk=1, val='v2')    ← 最新
  
  Merge 时发现 pk=1 有 v1→v2 的变化
  → 产出 changelog: -U(pk=1, val='v1'), +U(pk=1, val='v2')
  → 合并后写入 L1: (pk=1, val='v2')
```

### 2.6 四种模式对比

| 维度 | none | input | lookup | full-compaction |
|------|------|-------|--------|-----------------|
| Changelog 完整性 | ❌ | 取决于上游 | ✅ | ✅ |
| 额外写入开销 | 无 | 无 | 高（每次 lookup） | 低（compaction 附带） |
| Changelog 延迟 | N/A | 最低（实时） | 低（实时） | 较高（compaction 周期） |
| 内存消耗 | 无 | 无 | 高（KV 缓存） | 中等（compaction buffer） |
| 适用场景 | 纯批读 | CDC 入湖 | 小数据量实时更新 | 大数据量、可接受延迟 |
| 推荐度 | ⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐⭐ |

### 2.7 选型决策树

```
需要流式读取吗？
  ├─ 否 → none
  └─ 是 → 上游是 CDC Source 且提供完整 changelog？
            ├─ 是 → input（最优选择）
            └─ 否 → 数据量大吗？能接受 compaction 级别延迟吗？
                      ├─ 是 → full-compaction
                      └─ 否 → lookup
```

## 三、Streaming Read 机制

### 3.1 增量消费原理

Paimon 的 Streaming Read 基于 Snapshot 增量：

```
Snapshot 1 → Snapshot 2 → Snapshot 3 → Snapshot 4
     │             │             │             │
     ▼             ▼             ▼             ▼
 Consumer 从 snapshot-2 开始消费
   → 读取 delta manifest (snapshot-2 相对 snapshot-1 的增量)
   → 读取 changelog manifest (如果有)
   → 产出 changelog records
   → 记录消费位点 (consumer offset)
   → 下次从 snapshot-3 继续
```

### 3.2 Consumer 管理

```sql
-- 创建带 consumer id 的流读作业
SELECT * FROM exam_records_pk 
/*+ OPTIONS('consumer-id' = 'my-consumer-1') */;

-- Consumer 位点持久化在表中
-- 重启作业后自动从上次位点继续消费
```

Consumer 位点存储在表的 `consumer/` 目录下：
```
table_dir/consumer/
  └── my-consumer-1    ← 记录当前消费的 snapshot id
```

### 3.3 Watermark 对齐

Paimon 支持在 Streaming Read 中传递 Watermark：

```sql
-- 写入时配置 watermark
CREATE TABLE source_table (
    event_time TIMESTAMP(3),
    WATERMARK FOR event_time AS event_time - INTERVAL '5' SECOND,
    ...
);

-- 流读时 watermark 会自动传递给下游
-- 下游可以基于 event time 做窗口聚合
```

## 四、小文件问题深度分析

### 4.1 小文件的产生

Paimon PK Table 的小文件主要来自两个途径：

```
途径 1: Flink Checkpoint Flush
  每个 checkpoint → MemTable flush → 生成 L0 Sorted Run
  checkpoint interval = 10s → 每天 8640 个 L0 文件/bucket
  
途径 2: Partial-Update 多流写入
  多个流各自 checkpoint → 各自 flush → 同一 bucket 产生更多文件
```

### 4.2 小文件的影响

| 影响 | 说明 |
|------|------|
| 读取性能 | 每个 Sorted Run 都需要打开和扫描，层数越多越慢 |
| 元数据膨胀 | Manifest 文件变大，Snapshot 加载变慢 |
| S3 API 费用 | ListObjects 调用次数增加 |
| NameNode 压力 | HDFS 场景下每个文件占 ~150B 元数据 |
| Compaction 效率 | 太多小文件导致单次 compaction 耗时过长 |

### 4.3 缓解策略

```
策略 1: 增大 Checkpoint Interval
  10s → 60s → 文件数减少 6x
  代价: 数据可见延迟增加

策略 2: 增大 Write Buffer
  'write-buffer-size' = '256 mb'
  更大的 MemTable → 更少次 flush → 更大的 L0 文件

策略 3: 异步 Compaction
  'async-compaction' = 'true'
  Compaction 不阻塞写入，后台持续合并

策略 4: 独立 Compaction Job
  将 compaction 从写入 job 中分离出来
  避免 compaction 影响写入吞吐

策略 5: 合理设置 Bucket 数
  Bucket 太少 → 单 bucket 文件过多
  Bucket 太多 → 每个 bucket 文件过小
```

## 五、Compaction 资源消耗分析

### 5.1 Compaction 的工作量

```
Compaction 需要:
  1. 读取多个 Sorted Run 的数据文件 (IO Read)
  2. 在内存中 merge sort (CPU + Memory)
  3. 应用 Merge Engine 逻辑 (CPU)
  4. 产出 changelog (如果需要) (CPU + IO Write)
  5. 写入合并后的新文件 (IO Write)
  6. 提交新 Snapshot (轻量级)
```

### 5.2 资源消耗估算

| 参数 | 典型值 | 说明 |
|------|--------|------|
| 读放大 | 2-10x | 取决于 LSM 层数和重叠度 |
| 写放大 | 2-5x | 每层 compaction 重写一次 |
| CPU 开销 | 中等 | merge sort + merge engine |
| 内存开销 | write-buffer-size × writers | 并发 writer 数影响 |
| IO 带宽 | 通常是瓶颈 | S3 场景尤其明显 |

### 5.3 Compaction 调优建议

```sql
-- 控制 compaction 触发频率
'num-sorted-run.compact-trigger' = '5'     -- 5个 sorted run 才触发
'compaction.min.file-num' = '5'            -- 至少5个文件才合并

-- 控制 compaction 并行度
'compaction.parallelism' = '4'             -- compaction 线程数

-- 独立 compaction job（推荐生产环境）
-- 写入 job 只做 append，compaction 由单独 job 负责
'write-only' = 'true'                      -- 写入 job 不做 compaction

-- Flink Action 触发 compaction
-- CALL sys.compact('exam_db.exam_records_pk');
```

## 六、面试话术整理

### 6.1 "Paimon 的 changelog-producer 有哪几种模式？怎么选？"

> Paimon 有四种 changelog-producer 模式：
>
> **none**：不产出 changelog，只支持批量读取，适用于纯离线分析表。
>
> **input**：直接透传上游写入的 changelog。如果上游是 CDC Source（比如 MySQL binlog 通过 Flink CDC 接入），它本身就带有完整的 -I/+I/-U/+U，直接用 input 模式零额外开销，是最优选择。
>
> **lookup**：每次写入时先查一下该主键当前的值，然后补全 -U 消息。优点是实时性好，缺点是每次写入多一次 lookup，数据量大时开销显著。
>
> **full-compaction**：在 Compaction 过程中对比新旧值来产出 changelog。利用了 merge 本身就要比较同 key 不同版本的事实，边际成本低。缺点是 changelog 的产出延迟取决于 compaction 的频率。
>
> 实际选型：CDC 入湖优先用 input；非 CDC 场景、数据量大且可接受分钟级延迟用 full-compaction；数据量小且要求低延迟用 lookup。

### 6.2 "Paimon 流读的 -U/+U 有什么用？"

> -U/+U 是 Flink changelog 协议中表示更新操作的两种消息。-U 是更新前的旧值，+U 是更新后的新值。
>
> 它们的核心作用是让下游能够**正确维护增量聚合状态**。比如下游在做 SUM(score)，收到一条 UPDATE 把 score 从 85 改成 92，如果没有 -U(85)，下游就不知道要减去旧值，SUM 就会出错。有了 -U/+U，下游先减 85 再加 92，结果一定正确。
>
> 这也是 Paimon 相比 Iceberg 的一个关键优势——Iceberg 的流读只能产出 +I，下游如果要更新聚合就需要自己维护一份 lookup state 来补全 -U，增加了复杂度和资源消耗。Paimon 直接在存储层解决了这个问题。

### 6.3 "Paimon 的小文件问题怎么解决？"

> Paimon 的小文件主要来自 Flink checkpoint 触发的 MemTable flush。解决方案分几个层面：
>
> 第一，**源头控制**：适当增大 checkpoint interval（比如从 10s 调到 60s），增大 write-buffer-size，减少 flush 频率。
>
> 第二，**Compaction 治理**：开启异步 compaction，或者将 compaction 拆成独立的 Flink Job，避免影响写入吞吐。生产环境中我们一般用 write-only 模式让写入 job 不做 compaction，再用单独的 compact action job 持续合并。
>
> 第三，**Bucket 设计**：bucket 数量要和数据量匹配，太少会导致单 bucket 文件堆积，太多会产生大量小 bucket。
>
> 本质上这是 LSM-Tree 架构的固有 trade-off：写入快是因为 append-only，但 append 频繁就会产生小文件，需要通过 compaction 来回收。这和 RocksDB 面临的问题是一样的。

## 七、关键概念速查

| 概念 | 说明 |
|------|------|
| Changelog Producer | 控制 Paimon 如何产出变更日志的模式 |
| -I/+I/-U/+U | Flink changelog 协议的四种消息类型 |
| input 模式 | 透传上游 changelog，CDC 场景最优 |
| lookup 模式 | 写入时查旧值补全 -U，实时但有开销 |
| full-compaction 模式 | Compaction 时对比新旧值，低开销但有延迟 |
| Consumer | 流读消费者，持久化消费位点 |
| Watermark | 事件时间进度标记，支持端到端传递 |
| 读放大 | Compaction 前需要读取多层 Sorted Run |
| 写放大 | Compaction 重写合并后的文件 |
| write-only | 写入 job 不做 compaction，分离关注点 |
