# Paimon 流读与 Changelog 与 Compaction：实战篇

## 一、实验环境准备

### 1.1 前置条件

- Flink 1.18 + Paimon 0.8 环境已搭建（参考存储结构与表类型/实战.md）
- MinIO / HDFS 存储可用
- Flink SQL Client 或 Flink Job 提交能力

### 1.2 创建测试用主键表

```sql
CREATE CATALOG paimon_catalog WITH (
    'type' = 'paimon',
```

## 二、Streaming Read 实操

### 2.1 启动流读作业（终端 1）

```sql
-- 终端 1: 流读 orders_stream
SET 'execution.runtime-mode' = 'streaming';

SELECT 
    order_id,
    user_id,
    amount,
    status,
    updated_at
FROM orders_stream 
/*+ OPTIONS('consumer-id' = 'demo-consumer') */;
```

此时作业进入 RUNNING 状态，等待上游数据。

### 2.2 写入数据并观察 Changelog（终端 2）

```sql
-- 终端 2: 批量写入初始数据
INSERT INTO orders_stream VALUES 
    (1001, 201, 299.00, 'created', TIMESTAMP '2024-01-15 10:00:00'),
    (1002, 202, 199.50, 'created', TIMESTAMP '2024-01-15 10:05:00');
```

**终端 1 输出**：
```
+I(1001, 201, 299.00, created, 2024-01-15 10:00:00)
+I(1002, 202, 199.50, created, 2024-01-15 10:05:00)
```

### 2.3 执行 Upsert 观察 -U/+U

```sql
-- 终端 2: 更新订单状态
INSERT INTO orders_stream VALUES 
    (1001, 201, 299.00, 'paid', TIMESTAMP '2024-01-15 10:30:00');
```

**终端 1 输出**（input 模式下，取决于上游是否发送了 -U）：
```
-U(1001, 201, 299.00, created, 2024-01-15 10:00:00)   ← 旧值
+U(1001, 201, 299.00, paid, 2024-01-15 10:30:00)       ← 新值
```

> **注意**：如果通过 SQL INSERT 写入，Flink 可能只发送 +I/+U 而不发送 -U。要看到完整的 -U/+U，最好使用 Flink CDC Source 作为上游，或者使用 lookup/full-compaction 模式。

### 2.4 验证下游聚合正确性

```sql
-- 终端 1 改为聚合查询
SELECT 
    status,
    COUNT(*) as order_count,
    SUM(amount) as total_amount
FROM orders_stream 
/*+ OPTIONS('consumer-id' = 'agg-consumer') */
GROUP BY status;
```

写入变更后，聚合结果应该自动更新：
```
(created, 1, 199.50)    ← 1001 从 created 变为 paid 后
(paid, 1, 299.00)       ← 正确反映了变更
```

如果没有 -U，created 的 count 不会减少，total_amount 也不会修正。

## 三、不同 Changelog Producer 模式对比实验

### 3.1 实验设计

创建三张结构相同的表，分别使用 input / lookup / full-compaction 模式：

```sql
-- input 模式表
CREATE TABLE orders_input (...) WITH ('changelog-producer' = 'input');

-- lookup 模式表
CREATE TABLE orders_lookup (...) WITH ('changelog-producer' = 'lookup');

-- full-compaction 模式表
CREATE TABLE orders_full_compact (...) WITH (
    'changelog-producer' = 'full-compaction',
    'changelog-producer.compaction-interval' = '10 s'  -- 测试用短间隔
);
```

### 3.2 写入相同数据

```sql
-- 对三张表写入相同的初始数据和更新
INSERT INTO orders_input VALUES (1001, 201, 100.00, 'v1', ...);
INSERT INTO orders_lookup VALUES (1001, 201, 100.00, 'v1', ...);
INSERT INTO orders_full_compact VALUES (1001, 201, 100.00, 'v1', ...);

-- 然后更新
INSERT INTO orders_input VALUES (1001, 201, 200.00, 'v2', ...);
INSERT INTO orders_lookup VALUES (1001, 201, 200.00, 'v2', ...);
INSERT INTO orders_full_compact VALUES (1001, 201, 200.00, 'v2', ...);
```

### 3.3 对比观察

| 观察维度 | input | lookup | full-compaction |
|----------|-------|--------|-----------------|
| 流读收到 -U/+U | 取决于上游 | ✅ 立即 | ✅ compaction 后 |
| 写入延迟增加 | 无 | ~10-50ms (lookup) | 无（compaction 异步） |
| Changelog 产出延迟 | 实时 | 实时 | ~10s (compaction间隔) |
| 内存消耗 | 低 | 高（KV cache） | 中（compaction buffer） |
| CPU 消耗 | 低 | 中（lookup hash） | 中（merge compare） |

### 3.4 面试话术："changelog-producer 几种模式在实际项目中怎么选的？"

> 我们的实际经验是：**CDC 入湖场景一律用 input**。因为 Flink CDC Connector 本身就从 binlog 解析出了完整的 changelog（包含 before image），直接透传零开销。
>
> 对于非 CDC 场景，比如多个 Flink Job 的输出需要汇合后再给下游消费，我们用 **full-compaction**。虽然 changelog 有几十秒到几分钟的延迟，但对于我们的业务完全可以接受，而且不需要额外的 lookup 资源。
>
> **lookup 模式**我们只在数据量小（百万级以下）、且对延迟敏感的维表场景用过。数据量大了之后 lookup 的内存和 IO 开销太明显，不如 full-compaction 稳定。

## 四、Compaction 实操与观察

### 4.1 制造小文件场景

```sql
-- 设置很短的 checkpoint interval 来制造大量小文件
SET 'execution.checkpointing.interval' = '5s';

-- 持续写入数据（模拟高频 CDC）
-- 运行 5 分钟后停止
```

```bash
# 查看产生的文件数量
aws s3 ls s3://my-bucket/paimon-warehouse/exam_db.db/orders_stream/bucket-0/ | wc -l
# 预期: 约 60 个文件（5min × 12 checkpoints/min）
```

### 4.2 触发手动 Compaction

```sql
-- 方式 1: Flink SQL Action
CALL sys.compact('exam_db.orders_stream');

-- 方式 2: Flink Action Job (推荐生产环境)
-- flink run \
--   /opt/flink/lib/paimon-flink-action-0.8.jar \
--   compact \
--   --warehouse s3://my-bucket/paimon-warehouse \
--   --database exam_db \
--   --table orders_stream
```

### 4.3 观察 Compaction 效果

```bash
# Compaction 前
aws s3 ls s3://.../bucket-0/ | wc -l
# → 60 files

# Compaction 后
aws s3 ls s3://.../bucket-0/ | wc -l
# → 3-5 files（合并后的 Sorted Run）

# 查看 snapshot 确认 compaction 提交
aws s3 cat s3://.../snapshot/snapshot-latest
# commit_kind = "COMPACT"
```

### 4.4 独立 Compaction Job 配置

```yaml
# 生产环境推荐架构
# Job 1: 写入 Job（write-only 模式）
flink:
  write-only: true                    # 不做 compaction
  checkpoint.interval: 30s
  
# Job 2: Compaction Job（独立运行）
flink:
  compact.trigger: num-sorted-runs    # 按 sorted run 数量触发
  compact.parallelism: 4              # compaction 并行度
  target-file-size: 128mb
```

## 五、端到端实时 Pipeline 实战

### 5.1 架构：Paimon → OLAP / 向量数据库

```
MySQL (binlog)
      │
      ▼
Flink CDC Source (-I/+I/-U/+U)
      │
      ▼
Paimon PK Table (changelog-producer=input)
      │
      ├── Streaming Read → Flink ETL → Doris/StarRocks (实时 OLAP)
      │
      ├── Streaming Read → Flink ETL → Milvus/Qdrant (向量 DB 实时更新)
      │
      └── Batch Read → Spark SQL (离线分析/算法回溯)
```

### 5.2 完整 Flink SQL 示例

```sql
-- Step 1: CDC Source
CREATE TABLE mysql_orders (
    order_id    BIGINT,
    user_id     BIGINT,
    amount      DECIMAL(10,2),
    status      STRING,
    updated_at  TIMESTAMP(3),
    PRIMARY KEY (order_id) NOT ENFORCED
) WITH (
    'connector' = 'mysql-cdc',
    'hostname' = 'mysql-host',
    'port' = '3306',
    'username' = 'root',
    'password' = 'xxx',
    'database-name' = 'production',
    'table-name' = 'orders'
);

-- Step 2: 写入 Paimon
INSERT INTO orders_stream SELECT * FROM mysql_orders;

-- Step 3: 流读到 Doris（另一个 Flink Job）
CREATE TABLE doris_orders (
    order_id    BIGINT,
    user_id     BIGINT,
    amount      DECIMAL(10,2),
    status      STRING,
    updated_at  TIMESTAMP(3),
    PRIMARY KEY (order_id) NOT ENFORCED
) WITH (
    'connector' = 'doris',
    'fenodes' = 'doris-fe:8030',
    'table.identifier' = 'analytics.orders',
    'username' = 'root',
    'password' = 'xxx',
    'sink.properties.format' = 'json',
    'sink.properties.read_json_by_line' = 'true'
);

INSERT INTO doris_orders 
SELECT * FROM orders_stream 
/*+ OPTIONS('consumer-id' = 'doris-sync') */;
```

### 5.3 流读到向量数据库的场景

```
适用场景: 商品 Embedding 实时更新

Product Catalog (MySQL) → CDC → Paimon PK Table
                                      │
                              Streaming Read (-U/+U)
                                      │
                                      ▼
                            Flink ETL Job:
                              -U → 删除旧 embedding
                              +U → 调用 Embedding API → 写入 Milvus
```

关键点：
- -U 消息告诉下游"这个商品的旧 embedding 需要删除"
- +U 消息携带新的商品信息，触发重新生成 embedding
- 没有 -U 的话，向量数据库中会残留过期的 embedding

## 六、监控与运维

### 6.1 关键监控指标

| 指标 | 说明 | 告警阈值建议 |
|------|------|-------------|
| Snapshot Count | 未清理的快照数 | > 1000 |
| Data File Count per Bucket | 单 bucket 文件数 | > 200 |
| Sorted Run Count | 单 bucket sorted run 层数 | > 10 |
| Compaction Duration | 单次 compaction 耗时 | > 5min |
| Consumer Lag | 流读消费者落后快照数 | > 100 |
| Changelog File Size | changelog 文件大小增速 | 突增需关注 |
| Write Buffer Usage | MemTable 内存使用率 | > 80% |

### 6.2 定期维护任务

```sql
-- 1. 过期快照清理（每天执行）
CALL sys.expire_snapshots(
    'exam_db.orders_stream', 
    TIMESTAMP '2024-01-14 00:00:00',  -- 保留最近 N 天
    24                                   -- 至少保留 24 个快照
);

-- 2. 孤立文件清理（每周执行）
CALL sys.remove_orphan_files(
    'exam_db.orders_stream',
    TIMESTAMP '2024-01-13 00:00:00'
);

-- 3. Consumer 位点清理（不再使用的 consumer）
CALL sys.drop_consumer('exam_db.orders_stream', 'old-consumer-id');
```

## 八、常见问题排查

| 问题 | 可能原因 | 解决方案 |
|------|---------|---------|
| 流读收不到 -U | changelog-producer=none 或 input+上游无-U | 切换为 lookup 或 full-compaction |
| 流读延迟大 | Compaction 间隔太长（full-compaction 模式） | 缩短 compaction interval |
| Consumer lag 持续增长 | 下游处理慢或 compaction 跟不上 | 增加下游并行度或优化 compaction |
| 流读重启后重复消费 | Consumer id 未配置 | 添加 `consumer-id` hint |
| Compaction OOM | 单次 merge 数据量过大 | 调小 compaction.max.file-num |
| Lookup 写入慢 | 数据量大，KV cache miss 率高 | 切换 full-compaction 或增大缓存 |
| Changelog 文件膨胀 | 更新频率过高 | 检查是否有无效更新，考虑 aggregation 预聚合 |
