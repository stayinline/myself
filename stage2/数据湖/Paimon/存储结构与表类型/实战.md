# Paimon 存储结构与表类型：实战篇

## 一、环境准备

### 1.1 Flink + Paimon 本地环境（Docker Compose）

```yaml
# docker-compose.yml
version: '3.8'
services:
  minio:
    image: minio/minio:latest
    command: server /data --console-address ":9001"
    ports:
      - "9000:9000"
      - "9001:9001"
    environment:
      MINIO_ROOT_USER: admin
      MINIO_ROOT_PASSWORD: password123

  jobmanager:
    image: flink:1.18-java11
    command: jobmanager
    ports:
      - "8081:8081"
    volumes:
      - ./paimon-flink-1.18-0.8.jar:/opt/flink/lib/paimon.jar
    environment:
      FLINK_PROPERTIES: |
        jobmanager.rpc.address: jobmanager

  taskmanager:
    image: flink:1.18-java11
    command: taskmanager
    depends_on:
      - jobmanager
    volumes:
      - ./paimon-flink-1.18-0.8.jar:/opt/flink/lib/paimon.jar
    environment:
      FLINK_PROPERTIES: |
        jobmanager.rpc.address: jobmanager
        taskmanager.numberOfTaskSlots: 4
```

### 1.2 Flink SQL Client 连接

```bash
./bin/sql-client.sh \
  -Dexecution.checkpointing.interval=10s \
  -Dexecution.checkpointing.mode=EXACTLY_ONCE
```

## 二、Primary Key Table 实操

### 2.1 创建 Catalog 和主键表

```sql
-- 创建 Paimon Catalog
CREATE CATALOG paimon_catalog WITH (
    'type' = 'paimon',
    'warehouse' = 's3://my-bucket/paimon-warehouse',
    's3.endpoint' = 'http://minio:9000',
    's3.access-key-id' = 'admin',
    's3.secret-access-key' = 'password123',
    's3.path.style.access' = 'true'
);

USE CATALOG paimon_catalog;
CREATE DATABASE IF NOT EXISTS exam_db;
USE exam_db;

-- 创建主键表（CDC 入湖场景）
CREATE TABLE exam_records_pk (
    record_id   BIGINT,
    student_id  BIGINT,
    exam_id     BIGINT,
    score       DECIMAL(5,2),
    status      STRING,
    updated_at  TIMESTAMP(3),
    PRIMARY KEY (record_id) NOT ENFORCED
) WITH (
    'bucket' = '4',
    'merge-engine' = 'deduplicate',
    'changelog-producer' = 'input',
    'write-buffer-size' = '64 mb',
    'target-file-size' = '128 mb'
);
```

### 2.2 执行 Upsert 操作

```sql
-- 插入初始数据
INSERT INTO exam_records_pk VALUES 
    (1001, 201, 1, 85.5, 'graded', TIMESTAMP '2024-01-15 10:00:00'),
    (1002, 202, 1, NULL, 'pending', TIMESTAMP '2024-01-15 10:05:00'),
    (1003, 203, 1, 72.0, 'graded', TIMESTAMP '2024-01-15 10:10:00');

-- 模拟 CDC UPDATE：修改 1002 的成绩
INSERT INTO exam_records_pk VALUES 
    (1002, 202, 1, 92.0, 'graded', TIMESTAMP '2024-01-15 11:00:00');

-- 验证：1002 应该是最新值
SELECT * FROM exam_records_pk WHERE record_id = 1002;
-- 结果: (1002, 202, 1, 92.0, 'graded', 2024-01-15 11:00:00) ✅

-- 模拟 DELETE：通过写入带标记的记录
-- （需要配置 'delete.behavior' 或使用 changelog 模式）
```

### 2.3 观察 LSM 文件结构

```bash
# 查看 bucket 目录下的文件
aws s3 ls s3://my-bucket/paimon-warehouse/exam_db.db/exam_records_pk/bucket-0/ --recursive

# 典型输出:
# bucket-0/data-uuid-0.orc       ← L0 Sorted Run (第一次 flush)
# bucket-0/data-uuid-1.orc       ← L0 Sorted Run (第二次 flush，含 update)
# bucket-0/changelog-uuid-0.orc  ← Changelog 文件

# 查看 snapshot
aws s3 cat s3://my-bucket/paimon-warehouse/exam_db.db/exam_records_pk/snapshot/snapshot-2

# Compaction 后文件减少:
# bucket-0/data-uuid-merged.orc  ← 合并后的 Sorted Run
```

### 2.4 查看 Snapshot 变化

```sql
-- Paimon 系统表查看快照
SELECT * FROM exam_records_pk$snapshots;

-- 输出:
-- snapshot_id | schema_id | commit_user      | commit_kind | time_millis       | base_manifest_list | delta_manifest_list
-- 1           | 0         | flink-writer-0   | APPEND      | 1705312800000     | manifest-list-xxx  | manifest-list-yyy
-- 2           | 0         | flink-writer-0   | APPEND      | 1705316400000     | manifest-list-aaa  | manifest-list-bbb
-- 3           | 0         | compaction-worker| COMPACT     | 1705320000000     | manifest-list-ccc  | manifest-list-ddd
```

注意 `commit_kind` 字段：
- `APPEND`：正常写入
- `COMPACT`：Compaction 提交
- `OVERWRITE`：覆写提交

## 三、Append Table 实操

### 3.1 创建追加表

```sql
-- 创建 Append 表（日志/事件场景）
CREATE TABLE user_events (
    event_time  TIMESTAMP(3),
    user_id     BIGINT,
    event_type  STRING,
    page_url    STRING,
    duration_ms INT
) PARTITIONED BY (days(event_time))
WITH (
    'bucket' = '-1',          -- 动态分桶
    'target-file-size' = '128 mb'
);

-- 写入数据
INSERT INTO user_events VALUES 
    (TIMESTAMP '2024-01-15 10:00:00', 101, 'click', '/exam/list', 200),
    (TIMESTAMP '2024-01-15 10:01:00', 102, 'view', '/exam/detail/1', 5000),
    (TIMESTAMP '2024-01-15 10:02:00', 101, 'submit', '/exam/1', 30000);

-- 查询（和普通表一样）
SELECT event_type, COUNT(*) as cnt 
FROM user_events 
WHERE event_time >= TIMESTAMP '2024-01-15 10:00:00'
GROUP BY event_type;
```

### 3.2 Append Table vs PK Table 文件对比

```
Append Table 文件结构:
  partition=2024-01-15/
    bucket-auto-0/
      data-uuid-0.orc    ← 普通列存文件，无 LSM 层级
      data-uuid-1.orc
    bucket-auto-1/
      data-uuid-2.orc

PK Table 文件结构:
  bucket-0/
    data-uuid-0.orc      ← L0 Sorted Run
    data-uuid-1.orc      ← L0 Sorted Run  
    data-uuid-2.orc      ← L1 Sorted Run (compaction 产出)
  bucket-1/
    ...
```

## 四、Partial-Update Merge Engine 实战

### 4.1 场景：多流拼接宽表

```
订单流: (order_id, user_id, amount, order_status)
支付流: (order_id, pay_method, pay_time)
物流流: (order_id, tracking_no, delivery_status)

目标: 拼接成一张完整的订单宽表
```

### 4.2 建表与写入

```sql
-- 创建 partial-update 表
CREATE TABLE order_wide (
    order_id        BIGINT,
    user_id         BIGINT,
    amount          DECIMAL(10,2),
    order_status    STRING,
    pay_method      STRING,
    pay_time        TIMESTAMP(3),
    tracking_no     STRING,
    delivery_status STRING,
    PRIMARY KEY (order_id) NOT ENFORCED
) WITH (
    'bucket' = '4',
    'merge-engine' = 'partial-update',
    'changelog-producer' = 'lookup'
);

-- 订单流写入（只有订单相关字段有值）
INSERT INTO order_wide VALUES 
    (1001, 201, 299.00, 'created', CAST(NULL AS STRING), CAST(NULL AS TIMESTAMP), CAST(NULL AS STRING), CAST(NULL AS STRING));

-- 支付流写入（只有支付相关字段有值）
INSERT INTO order_wide VALUES 
    (1001, CAST(NULL AS BIGINT), CAST(NULL AS DECIMAL), CAST(NULL AS STRING), 'alipay', TIMESTAMP '2024-01-15 10:30:00', CAST(NULL AS STRING), CAST(NULL AS STRING));

-- 物流流写入（只有物流相关字段有值）
INSERT INTO order_wide VALUES 
    (1001, CAST(NULL AS BIGINT), CAST(NULL AS DECIMAL), CAST(NULL AS STRING), CAST(NULL AS STRING), CAST(NULL AS TIMESTAMP), 'SF1234567890', 'in_transit');

-- 查询结果：三个流的字段已自动拼接
SELECT * FROM order_wide WHERE order_id = 1001;
-- 结果: (1001, 201, 299.00, 'created', 'alipay', 2024-01-15 10:30:00, 'SF1234567890', 'in_transit') ✅
```

### 4.3 与传统 Flink 多流 JOIN 的对比

| 维度 | Flink 多流 JOIN | Paimon Partial-Update |
|------|----------------|----------------------|
| 状态管理 | Flink State（内存/RocksDB），可能膨胀 | Paimon 外部存储，不受 Flink State 限制 |
| 容错恢复 | 从 checkpoint 重建 state | 直接从 lake 读取最新状态 |
| 回溯能力 | 有限（受 state TTL 限制） | 完整历史快照 |
| 下游消费 | 只能接在 Flink Job 后面 | 任意引擎可读 |
| 资源消耗 | State 大 → TM 内存压力大 | Compaction 消耗 CPU/IO |
| 延迟 | 毫秒级 | 秒-分钟级（取决于 checkpoint） |

## 五、Paimon PK Table vs Iceberg 对比实验

### 5.1 实验设计

```
数据集: 100万行，主键均匀分布
操作: 随机更新 10% 的行（10万行 UPDATE）
指标: 写入耗时、读取耗时、存储空间、文件数量
```

### 5.2 预期结果

| 指标 | Iceberg COW | Iceberg MOR | Paimon PK |
|------|------------|-------------|-----------|
| 写入耗时 | ~5min（重写文件） | ~10s（写 delete file） | ~15s（MemTable flush） |
| 读取耗时（首次） | ~3s | ~8s（merge deletes） | ~5s（merge sorted runs） |
| 读取耗时（compaction后） | ~3s | ~3s | ~3s |
| 存储空间 | 较大（COW复制） | 中等（delete files） | 中等（LSM多层） |
| 文件数量 | 少（大文件） | 多（data+delete） | 中（compaction后收敛） |

### 5.3 面试话术："Paimon 主键表和 Iceberg 怎么选？"

> 选择的核心依据是**更新频率和消费模式**。
>
> 如果是 CDC 实时入湖、每秒数千条 Upsert、下游需要流式消费完整 Changelog，选 Paimon。它的 LSM-Tree 架构天然适合高频写入，而且能产出包含 -U/+U 的完整变更日志，可以直接驱动下游 OLAP 或向量数据库的实时更新。
>
> 如果是以离线分析为主、每天跑几次批量 ETL、偶尔有更新需求，选 Iceberg。它的读性能优化更成熟（Data Skipping、Manifest 裁剪），社区生态更广（Trino、DuckDB、StarRocks 都有深度集成），文档和社区支持也更完善。
>
> 在实际项目中，两者并不互斥。我们有些表用 Paimon 做实时入湖和流式加工，有些表用 Iceberg 做离线分析和跨引擎共享。关键是理解各自的 trade-off，按场景选型。

## 七、常见问题排查

| 问题 | 可能原因 | 解决方案 |
|------|---------|---------|
| 写入慢 | Bucket 数太少导致热点 | 增加 bucket 数或检查数据倾斜 |
| 读取慢 | Sorted Run 层数过多未 compaction | 触发 compaction 或调整 compaction 策略 |
| 小文件多 | Checkpoint 间隔太短 | 增大 checkpoint interval 或开启 async compaction |
| Partial-update 丢字段 | NULL 覆盖了已有值 | 检查 ignore-delete 配置 |
| OOM | MemTable 太大 | 调小 write-buffer-size |
| S3 写入失败 | endpoint/path style 配置错误 | 检查 s3.path.style.access=true |
| Changelog 不完整 | changelog-producer 配置不对 | 根据需求选择 input/lookup/full-compaction |

## 八、进阶调优参数

```sql
-- 写入优化
'write-buffer-size' = '256 mb'           -- MemTable 大小，越大 flush 越少
'write-max-writers-to-spill' = '10'      -- 允许 spill 的 writer 数
'target-file-size' = '128 mb'            -- 目标文件大小
'write-buffer-for-append' = 'true'       -- Append 模式也使用 buffer

-- Compaction 优化
'compaction.min.file-num' = '5'          -- 最少文件数才触发 compaction
'compaction.max.file-num' = '50'         -- 单次 compaction 最多处理文件数
'num-sorted-run.compact-trigger' = '5'   -- Sorted Run 达到此数量触发 compaction
'changelog-producer.compaction-interval' = '30 s'  -- full-compaction 模式间隔

-- 读取优化
'scan.parallelism' = '8'                 -- 扫描并行度
'read.batch-size' = '1024'               -- 批量读取行数
'source.split.target-size' = '128 mb'    -- Source split 大小
```
