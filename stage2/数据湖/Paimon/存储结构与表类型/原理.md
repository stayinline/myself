# Paimon 存储结构与表类型：原理篇

## 一、Paimon 定位与背景

### 1.1 从 Flink Table Store 到 Apache Paimon

- **2022年**：阿里巴巴开源 Flink Table Store（FTS），目标是解决 Flink 实时数仓的存储问题
- **2023年**：捐赠给 Apache，更名为 Apache Paimon
- **核心定位**：**Streaming Data Lake** —— 为流式计算原生设计的表存储格式

### 1.2 为什么 Iceberg 不够用？

Iceberg 是优秀的通用 Table Format，但在以下场景存在短板：

| 场景 | Iceberg 的问题 | Paimon 的优势 |
|------|---------------|--------------|
| 高频 Upsert | COW 写放大严重；MOR delete file 堆积 | LSM-Tree merge 天然高效 |
| 完整 Changelog 产出 | 只能产生 append-only changelog | 可产出 -I/+I/-U/+U 完整变更 |
| Partial Update | 不支持 | 内置 partial-update merge engine |
| Aggregation 预聚合 | 不支持 | 内置 aggregation merge engine |
| Flink State 替代 | 无关 | 可直接作为 Flink State Backend 的外部化存储 |

## 二、LSM-Tree 存储结构

### 2.1 LSM-Tree 基础回顾

LSM-Tree（Log-Structured Merge Tree）是 RocksDB、LevelDB 等 KV 引擎的核心数据结构：

```
写入路径:
  Write → MemTable (内存) → Flush → L0 Sorted Run → Compaction → L1+ Sorted Runs

读取路径:
  Read → MemTable → L0 → L1 → ... → Ln (逐层查找，越新越优先)

核心思想:
  - 写入顺序化（append-only），避免随机 IO
  - 通过后台 Compaction 合并多层数据，控制读放大
  - 牺牲一定的读性能换取极高的写性能
```

### 2.2 Paimon 如何将 LSM-Tree 映射到文件系统

Paimon 将 LSM-Tree 的概念映射到数据湖的文件组织上：

```
┌─────────────────────────────────────────────────────────────┐
│                     Paimon Table                             │
│                                                              │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐                  │
│  │ Bucket 0 │  │ Bucket 1 │  │ Bucket N │   ← 按主键 hash  │
│  │          │  │          │  │          │     分桶           │
│  │ ┌──────┐ │  │ ┌──────┐ │  │ ┌──────┐ │                  │
│  │ │MemTab│ │  │ │MemTab│ │  │ │MemTab│ │  ← 内存缓冲     │
│  │ ├──────┤ │  │ ├──────┤ │  │ ├──────┤ │                  │
│  │ │L0 Run│ │  │ │L0 Run│ │  │ │L0 Run│ │  ← Level 0      │
│  │ ├──────┤ │  │ ├──────┤ │  │ ├──────┤ │                  │
│  │ │L1 Run│ │  │ │L1 Run│ │  │ │L1 Run│ │  ← Level 1      │
│  │ ├──────┤ │  │ ├──────┤ │  │ ├──────┤ │                  │
│  │ │L2 Run│ │  │ │L2 Run│ │  │ │L2 Run│ │  ← Level 2      │
│  │ └──────┘ │  │ └──────┘ │  │ └──────┘ │                  │
│  └──────────┘  └──────────┘  └──────────┘                  │
└─────────────────────────────────────────────────────────────┘
```

### 2.3 文件层级详解

```
table_dir/
├── snapshot/                    ← 快照元数据
│   ├── snapshot-1
│   ├── snapshot-2
│   └── snapshot-3
├── manifest/                    ← Manifest 文件
│   ├── manifest-list-xxx        ← 指向一组 manifest
│   └── manifest-xxx             ← 记录 data file 列表
├── bucket-0/                    ← 每个 bucket 一个目录
│   ├── data-xxx.orc             ← Sorted Run 中的数据文件
│   ├── data-yyy.orc
│   └── changelog-xxx.orc        ← Changelog 文件（可选）
├── bucket-1/
│   └── ...
└── schema/                      ← Schema 版本历史
    ├── schema-0
    └── schema-1
```

### 2.4 Snapshot 结构

```json
{
  "version": 3,
  "id": 3,
  "schemaId": 0,
  "baseManifestList": "manifest-list-base-xxx",
  "deltaManifestList": "manifest-list-delta-xxx",
  "changelogManifestList": "manifest-list-changelog-xxx",
  "commitUser": "flink-writer-0",
  "commitIdentifier": 100,
  "commitKind": "APPEND",
  "timeMillis": 1705312805000,
  "totalRecordCount": 50000,
  "deltaRecordCount": 1000,
  "changelogRecordCount": 500
}
```

关键点：
- `baseManifestList`：基线数据文件清单（上次 compaction 后的稳定状态）
- `deltaManifestList`：增量数据文件清单（自上次 compaction 以来的新增）
- `changelogManifestList`：变更日志文件清单（用于下游流读）
- 这种 base + delta 分离设计让读取可以并行扫描两部分

## 三、Sorted Run 与 Compaction

### 3.1 Sorted Run 概念

在 Paimon 中，一个 Sorted Run 是一个有序的、内部无重复键的数据文件集合：

```
Sorted Run (Level 0):
  [key=A, val=1] [key=C, val=3] [key=E, val=5]   ← data-001.orc

Sorted Run (Level 0):  
  [key=B, val=2] [key=D, val=4] [key=F, val=6]   ← data-002.orc

注意: 同一 Level 的不同 Sorted Run 之间 key 范围可能重叠
      同一 Sorted Run 内部 key 有序且不重复
```

### 3.2 Compaction 过程

```
Compaction 前:
  L0: Run1=[A,C,E]  Run2=[B,D,F]  Run3=[A,B,G]
  L1: Run4=[A,B,C,D,E,F,G,H]

Compaction 后 (merge L0 所有 runs + overlapping L1 runs):
  L0: (empty)
  L1: Run5=[A(latest),B(latest),C(latest),D(latest),E(latest),F(latest),G,H]
```

Compaction 的作用：
1. **合并多个 Sorted Run**，减少读取时的 merge 开销
2. **应用 Merge Engine**（去重/部分更新/聚合）
3. **合并小文件**，减少文件数量
4. **产出 Changelog**（如果配置了 changelog-producer）

### 3.3 Compaction 策略

| 策略 | 说明 | 适用场景 |
|------|------|---------|
| Universal | 类似 RocksDB universal compaction | 默认，平衡读写 |
| Leveled | 类似 RocksDB leveled compaction | 读多写少 |
| Force | 手动触发全量 compaction | 维护窗口 |

## 四、Primary Key Table vs Append Table

### 4.1 Primary Key Table（主键表）

#### 定义与特征

```sql
CREATE TABLE orders (
    order_id    BIGINT,
    user_id     BIGINT,
    amount      DECIMAL(10,2),
    status      STRING,
    updated_at  TIMESTAMP(3),
    PRIMARY KEY (order_id) NOT ENFORCED
) WITH (
    'bucket' = '8',
    'merge-engine' = 'deduplicate'
);
```

- **必须有主键**
- **基于 LSM-Tree 存储**
- **支持 Upsert / Delete**
- **支持多种 Merge Engine**
- **支持 Changelog 产出**
- 数据按主键 hash 分配到 Bucket

#### 写入流程

```
INSERT/UPDATE → 分配 Bucket(hash(pk) % num_buckets)
              → 写入对应 Bucket 的 MemTable
              → MemTable 满 → Flush 为 L0 Sorted Run
              → 后台 Compaction 合并 Sorted Runs
              → 应用 Merge Engine 决定最终值
```

#### 读取流程

```
Batch Read:
  扫描所有 Bucket → 每个 Bucket 内 merge 所有 Sorted Runs → 返回最新值

Streaming Read:
  消费 Snapshot 增量 → 根据 changelog-producer 产出变更 → 下游消费
```

### 4.2 Append Table（追加表）

#### 定义与特征

```sql
CREATE TABLE logs (
    event_time  TIMESTAMP(3),
    user_id     BIGINT,
    action      STRING,
    detail      STRING
) PARTITIONED BY (days(event_time))
WITH (
    'bucket' = '-1'    -- 动态分桶
);
```

- **没有主键**
- **Append-only 写入**
- **类似 Iceberg 的使用模式**
- **支持分区**
- **适合日志、事件流等不需要更新的场景**

### 4.3 对比总结

| 维度 | Primary Key Table | Append Table |
|------|-------------------|-------------|
| 主键 | 必须 | 无 |
| 存储结构 | LSM-Tree | Columnar files |
| Upsert/Delete | ✅ | ❌ |
| Merge Engine | deduplicate/partial-update/aggregation | N/A |
| Changelog | 完整 (-I/+I/-U/+U) | 仅 +I |
| 分区 | 可选 | 常用 |
| Bucket | 固定或动态 | 通常动态(-1) |
| 类比 | RocksDB / HBase | Iceberg / Hive |
| 典型场景 | CDC 入湖、维表、宽表拼接 | 日志、埋点、事件流 |

## 五、Bucket 设计

### 5.1 固定 Bucket

```sql
'bucket' = '8'
-- 数据按 hash(primary_key) % 8 分配到 8 个 bucket
-- 每个 bucket 独立维护自己的 LSM-Tree
```

**优点**：数据分布确定，compaction 可控
**缺点**：数据倾斜时某些 bucket 过大；扩容需要重新 shuffle

### 5.2 动态 Bucket

```sql
'bucket' = '-1'
-- Paimon 自动管理 bucket 数量
-- 当某个 bucket 数据量达到阈值时自动创建新 bucket
```

**优点**：自动扩缩容，无需预估
**缺点**：bucket 数量不可控，可能导致过多小 bucket

### 5.3 Bucket 选择建议

| 场景 | 建议 |
|------|------|
| 数据量可预估，均匀分布 | 固定 bucket，大小 ≈ 总数据量 / (bucket数 × 目标文件大小) |
| 数据量不确定或增长快 | 动态 bucket |
| 有热点 key | 增加 bucket 数或使用 salted key |
| Append 表 | 通常用动态 bucket (-1) |

## 六、Merge Engine 详解

### 6.1 Deduplicate（去重引擎，默认）

```
相同主键的多条记录，只保留最新一条（按 sequence field 排序）

Input:  (pk=1, val='a', seq=1)
        (pk=1, val='b', seq=2)
        (pk=1, val='c', seq=3)

Output: (pk=1, val='c', seq=3)   ← 保留最新的
```

- 适用于 CDC 入湖（binlog replay）
- DELETE 操作：如果最后一条是 delete 标记，则删除该行

### 6.2 Partial-Update（部分更新引擎）

```
不同流的字段分别更新，最终拼接成完整行

Stream A: (pk=1, name='Alice', age=NULL, city=NULL)
Stream B: (pk=1, name=NULL, age=25, city=NULL)
Stream C: (pk=1, name=NULL, age=NULL, city='Beijing')

Merged:   (pk=1, name='Alice', age=25, city='Beijing')
```

**关键配置**：
```sql
'merge-engine' = 'partial-update',
'partial-update.ignore-delete' = 'true',  -- 忽略 delete 消息中的 null 字段
'fields.age.ignore-retract' = 'true'      -- 特定字段忽略回撤消息
```

**应用场景**：多流 JOIN 构建宽表（替代 Flink 多流 JOIN 的状态膨胀问题）

### 6.3 Aggregation（聚合引擎）

```
相同主键的记录进行预聚合

Input:  (pk=user_1, pv=1, uv=1)
        (pk=user_1, pv=1, uv=0)
        (pk=user_1, pv=1, uv=1)

Output: (pk=user_1, pv=3, uv=2)   ← SUM 聚合
```

**配置**：
```sql
'merge-engine' = 'aggregation',
'fields.pv.aggregate-function' = 'sum',
'fields.uv.aggregate-function' = 'sum',
'fields.last_login.aggregate-function' = 'last_non_null_value'
```

**支持的聚合函数**：sum, min, max, count, first_value, last_value, last_non_null_value, listagg, collect, nested_update

### 6.4 First-Row（首行引擎）

```
相同主键只保留第一条到达的记录，后续全部丢弃
适用于"只关心首次出现"的场景，如设备首次激活
```

## 七、为什么 Paimon 比 Iceberg 更适合实时更新？

### 7.1 架构层面的差异

```
Iceberg 更新路径:
  UPDATE → 找到目标文件 → COW:复制整个文件 / MOR:写delete file
  → 生成新 Snapshot
  
  问题: 
    COW: 改一行重写整文件，写放大 O(file_size)
    MOR: delete file 积累，读放大 O(delete_files)

Paimon 更新路径:
  UPDATE → hash(pk) → 写入 Bucket MemTable → Flush to L0
  → Compaction 时 LSM merge 自然完成去重
  
  优势:
    写入: O(1) append，无文件级复制
    合并: LSM Compaction 天然处理，增量 merge
    读取: 合并后的 Sorted Run 已经是最新状态
```

### 7.2 量化对比

假设表有 1000万行，每次更新 1000 行：

| 指标 | Iceberg COW | Iceberg MOR | Paimon PK Table |
|------|------------|-------------|-----------------|
| 单次写入量 | ~100MB（重写文件） | ~1KB（delete file） | ~10KB（MemTable flush） |
| 写入延迟 | 高 | 低 | 低 |
| 读取额外开销 | 无 | 合并 delete files | merge sorted runs |
| Compaction 频率 | 低频 | 中频（清理 deletes） | 持续（LSM 固有） |
| Changelog 完整性 | 不完整 | 不完整 | 完整 (-I/+I/-U/+U) |

### 7.3 何时仍然选 Iceberg

- 以离线分析为主，更新频率低
- 需要广泛的引擎兼容性（Trino、DuckDB、Hive 等）
- 社区生态和文档成熟度优先
- 不需要完整的流式 Changelog

## 八、面试话术："Paimon 的存储结构和 Iceberg 有什么本质区别？"

> Paimon 的主键表底层用的是 LSM-Tree，这和 Iceberg 的 columnar file + delete file 是完全不同的存储范式。
>
> Iceberg 本质上是面向批处理的：数据文件是不可变的列存文件，更新要么重写整个文件（COW），要么额外写 delete file 在读取时合并（MOR）。这在批量 ETL 场景下很好，但在高频小量更新时，COW 写放大严重，MOR 的 delete file 堆积又影响读性能。
>
> Paimon 把 LSM-Tree 搬到了数据湖的文件系统上。写入先攒到 MemTable，满了 flush 成一个 Sorted Run 文件，后台 Compaction 不断合并各层的 Sorted Run。这个过程天然就是增量 merge，不需要重写大文件，也不需要额外的 delete file。而且 LSM merge 的过程中可以直接产出完整的 Changelog（包含 -U/+U），这是 Iceberg 做不到的。
>
> 所以如果你的核心场景是 CDC 实时入湖、高频 Upsert、或者需要完整 Changelog 驱动下游，Paimon 的 LSM 架构比 Iceberg 更合适。反过来，如果是以离线分析为主、偶尔批量更新，Iceberg 的读优化设计和更广的生态兼容性更有优势。

## 九、关键概念速查

| 概念 | 说明 |
|------|------|
| LSM-Tree | Log-Structured Merge Tree，Paimon PK 表的底层存储结构 |
| Sorted Run | LSM 中一层有序的、内部无重复键的数据文件集合 |
| Bucket | 按主键 hash 分片，每个 Bucket 独立维护 LSM-Tree |
| MemTable | 内存缓冲区，满后 flush 为 L0 Sorted Run |
| Compaction | 合并多层 Sorted Run，减少读放大，应用 Merge Engine |
| Merge Engine | 定义同主键记录的合并策略（dedup/partial-update/aggregation） |
| Primary Key Table | 有主键，LSM 存储，支持 Upsert |
| Append Table | 无主键，Append-only，类似 Iceberg |
| Base/Delta Manifest | 快照中基线和增量文件的分离管理 |
| Changelog Manifest | 专用于流读的变更日志文件 |
