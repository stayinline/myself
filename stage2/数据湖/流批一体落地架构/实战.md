## 二、面试话术："工业 IoT 流批一体平台"

### 2.1 项目概述（30秒电梯演讲）

> 我们搭建了一个基于 Paimon 的工业 IoT 流批一体数据平台。采集产线设备的遥测数据、MES 工单和 ERP 订单，通过 Flink CDC/Kafka 实时写入 Paimon 湖表。下游既可以流式读取驱动 Doris 实时看板和告警，也可以批量读取供算法团队做异常检测模型的回测训练。用一套存储替代了之前 Kafka+Hive 的双链路架构，指标一致性事故归零，ETL 维护人力减少约 60%。

### 2.2 技术细节展开（2-3分钟）

**数据采集与入湖**：
> 设备遥测通过 MQTT Gateway 接入 Kafka，MES 和 ERP 通过 Flink CDC 直连数据库。ODS 层全部用 Paimon Primary Key Table，changelog-producer 设为 input，因为上游 CDC 已经带了完整的 changelog。Checkpoint interval 设 30 秒，写入 job 开启 write-only 模式不做 compaction。

**DWD 宽表拼接**：
> 这是最有技术含量的部分。传统做法是用 Flink 多流 JOIN，但设备遥测量太大，State 膨胀到几百 GB，checkpoint 经常超时。我们用 Paimon 的 partial-update merge engine 替代了 Flink State JOIN——三个流各自独立写入同一张 Paimon 表的对应字段，Compaction 时自动按主键合并成完整宽表。这样 Flink Job 变成无状态的了，稳定性大幅提升。

**DWS 预聚合**：
> 设备小时级统计用 aggregation merge engine，写入时直接做 SUM/AVG/MAX 预聚合。下游查小时级趋势不需要再扫描明细数据，响应时间从几十秒降到亚秒级。

**OLAP 同步**：
> DWD/DWS 表通过 Streaming Read 增量同步到 Doris。因为 Paimon 产出了完整的 -U/+U changelog，Doris 的 Unique Key 模型可以正确处理更新。运营看板的数据延迟从之前的 T+1 缩短到 1-2 分钟。

**算法回溯**：
> 算法团队之前抱怨最多的是"实时数据和离线数据对不上"。现在他们直接用 Spark SQL 读 Paimon 表，和实时链路用的是同一份数据，口径天然一致。而且 Paimon 保留了完整的快照历史，他们可以 Time Travel 到任意时间点做回测，不再受限于 Kafka 7 天的 retention。

### 2.3 关键数字（面试官最爱听）

| 指标 | Lambda 架构 | 流批一体后 | 改善 |
|------|-----------|-----------|------|
| ETL Job 数量 | 100+ (双链路) | 55 (单链路+compaction) | -45% |
| 存储成本 | 2x (双副本) | 1.2x | -40% |
| 指标不一致工单 | 3-5起/月 | 0 | -100% |
| 算法回溯时效 | T+1 | 准实时(1-2min) | 质变 |
| 数据可见延迟 | 实时秒级/离线T+1 | 统一1-2min | 可接受trade-off |
| Flink State 大小(DWD) | 200GB+ | ~0 (无状态) | 质变 |

## 三、算法团队直接回溯 Paimon 的实践路径

### 3.1 需求分析

算法团队的典型需求：
- "给我过去 3 个月的设备振动数据，按设备 ID 和时间排序"
- "我需要上周某次故障前后的完整上下文数据"
- "帮我跑一下这个模型在历史数据上的效果"

### 3.2 接入方案

```python
# 方案 1: Spark SQL 直读 Paimon（推荐）
from pyspark.sql import SparkSession

spark = SparkSession.builder \
    .config("spark.sql.catalog.paimon", "org.apache.paimon.spark.SparkCatalog") \
    .config("spark.sql.catalog.paimon.warehouse", "s3://my-bucket/paimon-warehouse") \
    .getOrCreate()

# 读取最近 3 个月数据
df = spark.sql("""
    SELECT device_id, ts, vibration_x, vibration_y, temperature
    FROM paimon.iot_db.dwd_device_runtime_wide
    WHERE ts >= '2024-01-01' AND ts < '2024-04-01'
    ORDER BY device_id, ts
""")

# Time Travel 到故障时刻
df_fault = spark.sql("""
    SELECT * FROM paimon.iot_db.dwd_device_runtime_wide
    FOR TIMESTAMP AS OF '2024-03-15 14:30:00'
    WHERE device_id = 'DEV-0042'
""")

# 方案 2: Paimon Python Reader（轻量级，无需 Spark）
# pip install pypaimon
import pypaimon

table = pypaimon.Table("s3://my-bucket/paimon-warehouse/iot_db.db/dwd_device_runtime_wide")
reader = table.new_read_builder().where("device_id = 'DEV-0042'").build()
for batch in reader:
    process(batch)
```

### 3.4 性能优化建议

| 优化项 | 说明 |
|--------|------|
| 分区裁剪 | 确保查询带时间范围条件，触发分区裁剪 |
| Z-Order | 对常用过滤列做 Z-Order 排序，提升 Data Skipping |
| 文件格式 | 使用 Parquet + ZSTD 压缩，兼顾大小和读取速度 |
| Snapshot 管理 | 保留足够快照支持回溯，但定期 expire 避免元数据膨胀 |
| Cache | 频繁回溯的表可开启本地缓存 |

## 四、简历 Paimon 段落改写指南

### 4.1 反面教材（空洞/夸大）

> ❌ "负责数据湖平台建设，使用 Paimon 实现流批一体，大幅提升数据处理效率"
> 
> 问题：没有具体场景、没有技术细节、没有量化结果、"大幅提升"不可验证

### 4.2 改写示范（技术扎实、诚实可信）

#### 版本 A：偏架构设计

> **工业 IoT 流批一体数据平台**
> 基于 Apache Paimon + Flink 构建统一存储层，替代原有 Kafka+Hive Lambda 双链路架构。设计 ODS/DWD/DWS 三层湖表体系：ODS 层使用 PK Table + input changelog-producer 承接 CDC 实时入湖；DWD 层采用 partial-update merge engine 实现多流无状态宽表拼接，消除 Flink State 200GB+ 膨胀问题；DWS 层利用 aggregation merge engine 做写入时预聚合。下游通过 Streaming Read 同步至 Doris 支撑实时看板，同时支持算法团队 Spark SQL 批量回溯。上线后 ETL Job 从 100+ 精简至 55 个，指标一致性事故归零，存储成本降低 40%。

#### 版本 B：偏问题解决

> **CDC 实时入湖与多流宽表优化**
> 解决考试业务 CDC 入湖场景中 Hive 不支持行级更新、实时/离线双链路指标不一致的问题。选型 Apache Paimon 作为湖存储，利用 LSM-Tree 架构承载高频 Upsert（峰值 5000 TPS）。针对多流拼接宽表导致的 Flink State 膨胀问题，引入 Paimon partial-update merge engine 将三流 JOIN 转为无状态独立写入，Checkpoint 耗时从 120s 降至 15s。配置 full-compaction changelog-producer 保证下游增量聚合正确性。配合独立 Compaction Job 治理小文件，单 bucket 文件数从日均 8000+ 控制在 50 以内。

#### 版本 C：偏端到端交付

> **Paimon 流批一体平台落地**
> 主导从 Lambda 架构向 Lakehouse 迁移的全流程：PoC 验证（2周）→ 非核心表迁移（4周）→ 核心链路切换（8周）→ 旧链路退役。技术上解决了三个关键问题：① CDC 入湖选用 input changelog-producer 零开销透传 binlog changelog；② write-only + 独立 compaction job 分离写入与合并，保障写入 SLA；③ 建立 snapshot 生命周期管理和 consumer lag 监控体系。最终 50 张核心表完成迁移，算法团队可直接 Spark SQL 回溯 Paimon 历史数据，回溯时效从 T+1 提升至准实时。

### 4.3 简历写作原则

| 原则 | 说明 |
|------|------|
| **有场景** | 不要说"使用了 Paimon"，要说"在什么场景下为什么选了 Paimon" |
| **有细节** | 提到具体的技术点（partial-update、changelog-producer、write-only） |
| **有数字** | 量化改善（Job 数量、State 大小、延迟、成本），不要说"大幅提升" |
| **有 trade-off** | 承认代价（如实时性从秒级降到分钟级），体现深度思考 |
| **诚实** | 不夸大自己的角色，不编造没做过的优化 |
| **可追问** | 写的每个技术点都要能展开讲 3 分钟 |

## 六、常见面试追问速查

| 追问 | 回答要点 |
|------|----------|
| 为什么选 Paimon 不选 Iceberg？ | CDC 高频 upsert + 完整 changelog + partial-update，详见三大湖对比 |
| Compaction 资源怎么规划的？ | 独立 job，占写入约 30% 资源，异步不影响 SLA |
| 数据延迟从秒级降到分钟级，业务能接受吗？ | 看板类场景 1-2min 完全够用；毫秒级需求仍走 Redis/Doris |
| Partial-update 怎么处理字段冲突？ | 按 sequence field 排序，最新值覆盖；可配 ignore-delete |
| 如果 Paimon 出问题怎么兜底？ | 保留 Kafka 原始 topic 7天；可临时切回老链路 |
| 算法团队反馈怎么样？ | 回溯时效 T+1→准实时；口径一致不再扯皮；Spark SQL 上手快 |
| 有没有考虑过 Delta Lake？ | 开源版功能受限，绑定 Spark，我们是 Flink 技术栈 |
| 未来规划是什么？ | ADS 层探索 Paimon 直接服务（减少 Doris 同步链路）；探索 XTable 互操作 |

## 七、总结：流批一体的核心价值主张

```
一句话总结:
  数据湖不是目的，流批一体才是目标。
  Paimon/Iceberg 提供了统一的存储底座，
  让"一份数据、两种消费"从口号变成现实。

核心价值:
  ✅ 指标口径天然一致（消灭双链路根因）
  ✅ 运维成本大幅降低（一套 ETL、一套存储）
  ✅ 回溯能力质变（完整历史快照 vs Kafka 7天）
  ✅ 开发效率提升（新需求只需一套代码）

核心 Trade-off:
  ⚠️ 实时性: 秒级 → 分钟级
  ⚠️ 资源: Compaction 额外开销
  ⚠️ 学习曲线: 新概念（snapshot/manifest/changelog-producer）
  ⚠️ 生态成熟度: 不如传统 Hive/DW 完善

务实态度:
  流批一体不是银弹，而是工程 trade-off。
  关键是理解它的适用边界，在合适的场景发挥价值。
```
