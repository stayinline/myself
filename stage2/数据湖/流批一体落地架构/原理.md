# 流批一体落地架构：原理篇

## 一、什么是真正的流批一体

### 1.1 三个层次的"流批一体"

```
Level 1: API 统一（计算层）
  Flink DataStream/Table API 同时支持流和批
  Spark Structured Streaming 统一 DataFrame API
  → 开发者写一套代码，切换执行模式即可
  → 但底层存储仍然是两套！

Level 2: 存储统一（存储层）★ 数据湖的核心价值
  Paimon/Iceberg 同一张表既支持 streaming read 也支持 batch read
  → 一份存储，两种消费模式
  → 消灭 Lambda 双链路的数据冗余和口径不一致

Level 3: 全链路统一（架构层）
  从数据采集到 ETL 到服务层，全链路一套系统
  → 理想目标，实际中仍需 OLAP 服务层做加速
```

### 1.2 为什么 Level 2 是关键

Lambda 架构的根本问题不是"代码写了两套"，而是**数据存储了两份**：

```
Lambda 架构的根因:
  实时存储 (Kafka/Redis/HBase) ≠ 离线存储 (Hive/HDFS)
  ↓
  即使 API 统一了，数据还是要写两份
  ↓
  两份数据的 ETL 逻辑必然有差异
  ↓
  指标不一致 + 运维翻倍
```

数据湖通过 Table Format 提供了统一的存储底座，从根本上解决了这个问题。

### 2.2 与传统 Lambda 架构对比

| 维度 | Lambda 架构 | Paimon 流批一体 |
|------|-----------|----------------|
| 存储 | 实时(Kafka/Redis) + 离线(Hive) 两套 | Paimon 一套 |
| ETL | 实时 Flink + 离线 Spark 两套 | Flink 一套（或 Flink+Spark 读同一表） |
| 指标一致性 | 容易不一致 | 天然一致（同一数据源） |
| 数据冗余 | 至少 2x | 1x（compaction 前短暂 2x） |
| 运维复杂度 | 高（两套调度/监控/告警） | 低（一套管理） |
| 回溯能力 | 受限于 Kafka retention | 完整历史快照 |
| 实时性 | 秒级 | 分钟级（checkpoint 粒度） |
| 查询延迟 | Redis ms级 / Hive min级 | Doris/SR ms级 / Spark min级 |

## 三、数据分层在湖上的实现

### 3.1 经典数仓分层适配

```
ODS (Operational Data Store)
  ├── 来源: CDC / 日志 / API
  ├── Paimon 表类型: PK Table (CDC) / Append Table (日志)
  ├── 写入方式: Flink CDC → Paimon Upsert
  ├── 保留策略: 全量保留，支持 Time Travel
  └── 示例: ods_orders, ods_user_events

DWD (Data Warehouse Detail)
  ├── 来源: ODS 层流读 + 维表关联
  ├── Paimon 表类型: PK Table (宽表拼接用 partial-update)
  ├── 写入方式: Flink Stream Read ODS → Transform → Paimon Upsert
  ├── 关键操作: 多流 JOIN → partial-update merge engine
  └── 示例: dwd_order_wide, dwd_user_behavior

DWS (Data Warehouse Summary)
  ├── 来源: DWD 层流读 + 预聚合
  ├── Paimon 表类型: PK Table (aggregation merge engine)
  ├── 写入方式: Flink Stream Read DWD → Aggregate → Paimon Upsert
  ├── 优势: 预聚合减少下游计算量
  └── 示例: dws_user_daily_stats, dws_exam_summary

ADS (Application Data Store)
  ├── 来源: DWS/DWD 层
  ├── 存储: Doris/StarRocks (OLAP 服务层)
  ├── 写入方式: Paimon Streaming Read → Flink → Doris
  ├── 用途: BI 看板、API 查询、实时告警
  └── 说明: ADS 通常不在湖内，因为需要亚秒级响应
```

### 3.2 各层 Changelog Producer 配置建议

| 层级 | changelog-producer | 理由 |
|------|-------------------|------|
| ODS | input | CDC 上游自带完整 changelog |
| DWD | lookup 或 full-compaction | Flink ETL 产出可能不含 -U |
| DWS | full-compaction | 聚合结果变更频率较低 |
| ADS | N/A (OLAP 引擎) | 不需要 |

### 3.3 各层 Compaction 策略

| 层级 | Compaction 频率 | 策略 |
|------|----------------|------|
| ODS | 高频（10-30min） | CDC 写入量大，需要及时合并 |
| DWD | 中频（30-60min） | 宽表拼接后数据量适中 |
| DWS | 低频（1-2h） | 预聚合后数据量小 |

## 四、Trade-off 分析

### 4.1 实时更新频率 vs 批量扫描性能

```
矛盾点:
  高频更新 → 更多 L0 Sorted Runs → 读取时 merge 开销增大
  频繁 Compaction → 解决读性能 → 消耗 IO/CPU 资源
  
平衡策略:
  ┌─────────────────────────────────────────────────┐
  │ 写入 Job (write-only=true)                       │
  │   → 只做 MemTable flush，不做 compaction          │
  │   → 保证写入吞吐不受 compaction 影响               │
  │                                                   │
  │ Compaction Job (独立运行)                          │
  │   → 持续合并 Sorted Runs                          │
  │   → 可独立扩缩容                                  │
  │   → 不影响写入 SLA                                │
  └─────────────────────────────────────────────────┘
```

### 4.2 实时性 vs 资源成本

| Checkpoint Interval | 数据可见延迟 | 小文件数量 | Compaction 压力 | 推荐场景 |
|---------------------|-------------|-----------|----------------|---------|
| 5s | ~5s | 极高 | 极大 | 仅超高实时需求 |
| 30s | ~30s | 高 | 大 | 实时交易/风控 |
| 60s | ~1min | 中等 | 中等 | 一般实时场景（推荐） |
| 300s | ~5min | 低 | 小 | 准实时/近线分析 |
| 3600s | ~1h | 极低 | 极小 | 离线分析 |

### 4.3 一致性保证

Paimon + Flink 端到端 Exactly-Once：

```
Flink Checkpoint + Paimon Snapshot = 两阶段提交

1. Flink checkpoint barrier 到达
2. Paimon Writer flush MemTable，准备 snapshot
3. Checkpoint coordinator 确认所有算子完成
4. Paimon 原子提交 snapshot
5. 如果任何步骤失败 → rollback，不产生不完整快照

结果: 下游消费者看到的永远是完整的、一致的快照
```

## 五、与纯 Lambda 架构的详细对比

### 5.1 Lambda 架构的问题量化

假设一个中型互联网公司：
- 50 个业务表需要同步
- 每个表需要实时+离线两套 ETL
- 平均每套 ETL 开发维护耗时 2 人天/月

```
Lambda 架构月度成本:
  ETL 开发维护: 50 × 2套 × 2人天 = 200 人天/月
  存储冗余: 实时副本 + 离线副本 ≈ 2x 存储
  数据事故: 平均每月 3-5 起口径不一致问题
  排查耗时: 每起 2-4 小时 → 约 10-20 小时/月

流批一体架构月度成本:
  ETL 开发维护: 50 × 1套 × 1.5人天 = 75 人天/月 (节省 62.5%)
  存储: 1x + compaction 临时 ≈ 1.2x 存储 (节省 40%)
  数据事故: 趋近于 0
  排查耗时: 大幅减少
```

### 5.2 迁移路径

```
Phase 1: PoC 验证 (2周)
  ├── 选 2-3 个非核心表
  ├── 搭建 Paimon 测试环境
  ├── 验证 CDC 入湖 + 流读 + 批读
  └── 输出: PoC 报告 + 性能基准

Phase 2: 非核心表迁移 (4周)
  ├── 迁移 10-20 个非核心表
  ├── 建立 Compaction / 监控 / 告警体系
  ├── 培训团队成员
  └── 输出: 运维手册 + 最佳实践

Phase 3: 核心链路切换 (8周)
  ├── 核心业务表逐步迁移
  ├── 双跑对比验证（Lambda + Lakehouse 并行）
  ├── 确认指标一致性后切流量
  └── 输出: 迁移完成报告

Phase 4: Lambda 退役 (4周)
  ├── 下线旧的实时/离线双链路
  ├── 清理冗余存储
  ├── 更新文档和监控
  └── 输出: 项目总结
```

## 六、面试话术整理

### 6.1 "你们的流批一体是怎么做的？"

> 我们用 Paimon 作为统一存储底座来实现流批一体。核心思路是：**一份存储，两种消费**。
>
> 具体来说，MySQL 的 binlog 通过 Flink CDC 实时写入 Paimon 的 ODS 层主键表。这张表基于 LSM-Tree，天然支持高频 Upsert。然后下游有两种消费方式：
>
> **实时消费**：另一个 Flink Job 通过 Streaming Read 增量消费 ODS 表的 changelog，做 ETL 加工后写入 DWD/DWS 层，最终同步到 Doris 供运营看板使用。因为 Paimon 能产出完整的 -U/+U changelog，下游的增量聚合可以正确维护状态。
>
> **批量消费**：算法团队直接用 Spark SQL 读取 Paimon 表的历史数据做模型训练和回测。因为是同一张表，数据和实时链路完全一致，不存在口径偏差。
>
> 相比之前的 Lambda 架构，我们省掉了实时和离线两套 ETL、两套存储，指标一致性问题的工单从每月 3-5 起降到接近零，ETL 开发维护的人力投入减少了约 60%。

### 6.2 "流批一体的代价是什么？"

> 主要有三个 trade-off：
>
> 第一，**实时性从秒级降到分钟级**。Paimon 的数据可见性取决于 Flink checkpoint interval，通常 30s-60s。如果需要毫秒级响应，还是需要 Redis/Doris 这类专用引擎做服务层。
>
> 第二，**Compaction 的资源开销**。LSM-Tree 的写入快是以后台 merge 为代价的。在生产环境中，Compaction Job 通常需要写入 Job 约 30% 的额外资源。我们通过 write-only 模式将 compaction 独立出来，避免影响写入 SLA。
>
> 第三，**生态成熟度**。Paimon 相比 Iceberg 还年轻一些，某些 BI 工具或小众引擎的适配还不够完善。不过核心链路（Flink + Spark + Doris/StarRocks）已经足够覆盖我们的需求。

## 七、关键概念速查

| 概念 | 说明 |
|------|------|
| 流批一体 | 一份存储同时支持流式读取和批量读取 |
| Level 2 统一 | 存储层的统一，数据湖的核心价值 |
| Write-Only | 写入 job 不做 compaction，分离关注点 |
| 两阶段提交 | Flink checkpoint + Paimon snapshot 保证端到端一致性 |
| ODS/DWD/DWS/ADS | 数仓分层在湖上的适配 |
| Partial-Update | 多流拼接宽表，替代 Flink State JOIN |
| Aggregation Engine | 预聚合写入，减少下游计算 |
| Consumer | 流读消费者，持久化消费位点 |
| Exactly-Once | Flink + Paimon 端到端精确一次语义 |
| Compaction Job | 独立的 compaction 任务，不影响写入 |
