## 二、各组件技术选型与理由标注

### 2.1 选型决策矩阵

| 组件 | 选择 | 备选 | 核心理由 | 面试一句话解释 |
|------|------|------|----------|---------------|
| CDC | Flink CDC | Debezium+Kafka | 全在Flink内完成，减少一跳延迟 | "CDC直接在Flink里做，省掉中间Kafka的一跳" |
| MQ | Kafka | Pulsar | 生态最成熟，团队经验最丰富 | "选Kafka因为团队最熟、社区最大" |
| 计算 | Flink | Spark Streaming | 真正的流处理，非微批 | "Flink是event-driven真流处理，延迟更低" |
| 湖仓 | Paimon | Iceberg/Hudi | Flink原生集成+Upsert支持 | "Paimon和Flink同宗同源，流式写入最顺" |
| OLAP | StarRocks | ClickHouse/Doris | Join强+Paimon Catalog+运维简单 | "StarRocks的Join性能和Paimon集成是最好的" |
| 向量库 | Milvus | pgvector | 百万级规模+分布式+生产验证 | "百万级以上向量检索Milvus是业界标准" |
| Embedding | bge-m3 | text2vec/OpenAI | 中文效果好+可自部署+免费 | "bge-m3中文效果Top级别且可私有化部署" |
| Agent框架 | LangGraph | OpenAI Agents SDK/自研 | Graph编排灵活+不绑定厂商 | "LangGraph的状态图模型最适合有流程的Agent" |
| 工具协议 | MCP | 自定义Function | 社区标准+工具可复用 | "MCP是Agent工具连接的事实标准协议" |
| 调度 | DolphinScheduler | Airflow | 国内生态好+可视化+轻量 | "DS对国内团队更友好，UI直观" |
| 监控 | Prometheus+Grafana | Datadog | 开源+灵活+无厂商锁定 | "Prometheus生态是云原生监控事实标准" |

### 2.2 每个组件的关键配置要点

```yaml
# Flink 关键配置
flink:
  state.backend: rocksdb
  checkpoint.interval: 30s
  checkpoint.mode: EXACTLY_ONCE
  parallelism.default: 4
  watermark.delay: 5s
  async.capacity: 100  # Embedding异步调用并发

# Paimon 表配置
paimon:
  write-buffer-size: 256mb
  compaction.min-file-num: 5
  snapshot.retention: 7d
  changelog-producer: lookup  # 支持CDC upsert

# StarRocks 关键配置
starrocks:
  refresh_interval: 5s  # Paimon catalog刷新
  materialized_view_refresh: 1min
  query_timeout: 30s

# Milvus 索引配置
milvus:
  index_type: HNSW
  metric_type: COSINE
  params:
    M: 16
    efConstruction: 256
  search_ef: 128
```

### 4.1 项目概述（简历用）

> **实时 RAG 数据基础设施** | Flink + Paimon + StarRocks + Milvus

## 五、STAR 项目故事准备

### 故事 1: Flink 窗口聚合 + 容错

> **Situation**: 业务需要实时统计每小时的订单金额分布，但源数据存在乱序和迟到问题，早期使用 Tumbling Window 导致约 8% 的数据被错误归入下一个窗口。
> 
> **Task**: 设计一个既能正确处理乱序数据，又能在故障恢复后保证结果一致性的窗口聚合方案。
> 
> **Action**: 
> 1. 将 Tumbling Window 替换为 Event Time + Watermark（允许 5s 乱序），配合 Allowed Lateness(1min) + Side Output 处理迟到数据；
> 2. 状态后端从 HashMap 切换为 RocksDB + Incremental Checkpoint，解决大状态下的 CP 超时问题；
> 3. 开启 Exactly-Once 语义，配合 Paimon Sink 的两阶段提交保证端到端一致性；
> 4. 建立窗口结果的自动化对账机制，每小时与离线 Hive 结果比对。
> 
> **Result**: 窗口数据准确率从 92% 提升到 99.7%，Checkpoint 耗时从 45s 降至 8s，迟到数据处理覆盖率 99.5%。对账差异率 <0.1%。

### 故事 2: CDC + SCD 维度表

> **Situation**: 用户画像表频繁变更（地址、会员等级等），下游分析需要保留历史快照来做趋势分析，原方案用每日全量快照，存储成本高且无法精确到变更时间点。
> 
> **Task**: 实现基于 CDC 的 SCD Type 2 维度表，支持精确到秒级的历史版本追溯。
> 
> **Action**:
> 1. 使用 Flink CDC 捕获 MySQL binlog 变更事件；
> 2. 在 Flink 中实现 SCD Type 2 逻辑：收到 UPDATE 时将旧记录标记失效（设置 end_time），插入新记录（start_time=now, end_time=null）；
> 3. 写入 Paimon 主键表，利用其 Upsert 能力高效维护有效/失效记录；
> 4. 处理 Schema Evolution：上游加字段时 Flink 自动适配，Paimon 支持 Schema 演进无需停服。
> 
> **Result**: 存储成本降低 70%（增量 vs 全量快照），历史版本查询精度从"天级"提升到"秒级"，Schema 变更零停机。

### 故事 3: RAG 准确率优化

> **Situation**: 初始 RAG 系统基于简单的向量相似度检索，回答准确率仅 65%，主要问题是检索到的文档不够精准、缺少关键上下文、以及 LLM 幻觉。
> 
> **Task**: 在不更换大模型的前提下，通过数据工程和检索策略优化将准确率提升到 85% 以上。
> 
> **Action**:
> 1. 文档预处理优化：改进切片策略（按段落语义切分而非固定长度），增加 metadata 标注（类别、时间、来源）；
> 2. 混合检索：向量检索 + BM25 关键词检索 + RRF 融合排序，解决纯语义检索遗漏精确匹配的问题；
> 3. Rerank 精排：用 cross-encoder 对 Top-20 候选重排序取 Top-5，显著提升相关性；
> 4. Prompt Engineering：System Prompt 强制要求引用来源，Few-shot 示例引导格式，CoT 引导推理；
> 5. 评估体系：构建 200 条 golden test set，自动化评测 Faithfulness/Relevance/Completeness。
> 
> **Result**: RAG 准确率从 65% 提升到 89%，幻觉率从 18% 降至 4%，平均检索延迟增加 <50ms（rerank 开销）。
