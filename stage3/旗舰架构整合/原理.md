### 1.2 架构设计哲学

| 原则 | 体现 | 为什么重要 |
|------|------|-----------|
| **可信 (Trustworthy)** | 数据血缘、Schema 演进、Time Travel | AI 应用的数据必须可追溯、可审计 |
| **实时 (Real-time)** | CDC → Flink → StarRocks 全链路秒级延迟 | RAG 知识库需要近实时更新 |
| **可观测 (Observable)** | 全链路 Tracing、质量指标、异常告警 | 生产系统必须能被理解和诊断 |
| **流批一体** | Paimon 统一存储，避免 Lambda 架构 | 降低运维复杂度和数据不一致风险 |
| **渐进增强** | 每层独立可用，逐步叠加能力 | 降低项目风险，支持敏捷迭代 |

## 二、各阶段关键技术决策

### 2.1 数据采集层：CDC vs 轮询 vs 日志

| 方案 | 延迟 | 完整性 | 对源库影响 | 复杂度 | 推荐场景 |
|------|------|--------|-----------|--------|----------|
| Binlog CDC | 秒级 | ✅ 完整 | 极低 | 中 | 首选方案 |
| 定时轮询 | 分钟级 | ❌ 可能漏删改 | 中 | 低 | 简单同步 |
| 应用双写 | 毫秒级 | ❌ 不一致风险 | 无 | 高 | 不推荐 |
| Trigger | 秒级 | ✅ 完整 | 高 | 中 | 小表/低频 |

**决策**: 采用 Debezium/Flink CDC 做 Binlog 采集
**理由**: 秒级延迟 + 完整变更捕获 + 对源库几乎零侵入

### 2.2 Flink 处理层：核心技术选型

```
┌────────────────────────────────────────────────────────────┐
│                 Flink Processing Decisions                   │
│                                                              │
│  1. 窗口策略                                                │
│     选择: Event Time + Watermark (非 Processing Time)        │
│     原因: 保证乱序数据的正确性，支持迟到数据处理               │
│                                                              │
│  2. Join 策略                                               │
│     选择: Interval Join (流-流) + Lookup Join (流-维表)      │
│     原因: Interval Join 控制状态大小；Lookup Join 适合维表    │
│                                                              │
│  3. 状态后端                                                │
│     选择: RocksDB + Incremental Checkpoint                   │
│     原因: 支持超大状态，增量 CP 减少 IO                       │
│                                                              │
│  4. Embedding 生成                                          │
│     选择: Async I/O + 批量调用 Embedding API                 │
│     原因: Embedding API 是外部瓶颈，异步+批量提升吞吐         │
│                                                              │
│  5. Exactly-Once                                            │
│     选择: Source(CDC) + Sink(Paimon) 都支持 EO               │
│     原因: 端到端一致性保证数据不丢不重                         │
└────────────────────────────────────────────────────────────┘
```

### 2.3 Paimon 存储层：为什么选 Paimon

| 特性 | Iceberg | Hudi | Paimon | 我们的需求 |
|------|---------|------|--------|-----------|
| Flink 集成 | ⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | Flink 为主引擎 |
| 流式写入 | ⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | CDC 实时入湖 |
| 主键更新 | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | SCD/Upset |
| 读取性能 | ⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐⭐ | OLAP 查询 |
| 社区活跃度 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐⭐ | 长期维护 |
| 与 StarRocks 集成 | ⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐⭐ | 分析查询 |

**决策**: 主存储选 Paimon，兼顾流式写入和 Flink 生态
**备选**: 如果 Spark 为主则选 Iceberg；如果需要大量 Upsert 且用 Spark 则选 Hudi

### 2.4 OLAP 引擎选型

| 维度 | ClickHouse | Doris | StarRocks |
|------|-----------|-------|-----------|
| 单表查询 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ |
| Join 性能 | ⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ |
| 实时更新 | ⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ |
| Paimon Catalog | ❌ | ✅ | ✅ |
| 运维复杂度 | 高 | 中 | 中低 |
| 物化视图 | ⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ |
| 适用规模 | PB级 | TB-PB级 | TB-PB级 |

**决策**: StarRocks 作为主要 OLAP 引擎
**理由**: Join 性能优秀（多表关联分析）、原生支持 Paimon External Catalog、物化视图加速高频查询、运维相对简单

### 2.5 向量数据库选型

| 维度 | Milvus | pgvector | Weaviate | ChromaDB |
|------|--------|----------|----------|----------|
| 大规模 (>1M) | ⭐⭐⭐⭐⭐ | ⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐ |
| 索引类型 | HNSW/IVF/DiskANN | HNSW/IVF | HNSW | HNSW |
| 混合检索 | ✅ 标量过滤 | ✅ SQL | ✅ GraphQL | 🟡 基础 |
| 分布式 | ✅ | ❌ | ✅ | ❌ |
| 运维 | 中高 | 低(复用PG) | 中 | 极低 |
| 生产就绪 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐ |

**决策**: 生产用 Milvus，开发/小规模用 pgvector
**理由**: 百万级以上向量首选 Milvus；如果已有 PG 基础设施且数据量 <50万，pgvector 性价比更高

## 三、三大关键词深度解读

### 3.1 可信 (Trustworthy)

```
数据可信的三个层次:

Level 1: 数据血缘 (Lineage)
  → 每个字段从哪个源表经过哪些转换而来
  → Paimon 自带 lineage + DataHub 做可视化
  → "这个 RAG 回答引用的数据是什么时候更新的？" → 可回答

Level 2: 数据质量 (Quality)
  → 完整性、准确性、及时性、一致性
  → Flink 中嵌入质量检查 UDF
  → 异常数据打标而非丢弃（保留审计痕迹）

Level 3: 模型可信 (Model Trust)
  → RAG 引用溯源（每条回答标注来源文档）
  → Agent 执行审计日志（每步 Thought/Action/Observation）
  → 输出置信度评分（低于阈值转人工）
```

### 3.2 实时 (Real-time)

```
全链路延迟预算分配:

CDC 采集:          ≤ 1s     (Debezium binlog tailing)
Kafka 传输:        ≤ 0.5s   (broker replication)
Flink 处理:        ≤ 2s     (window + embedding async)
Paimon 写入:       ≤ 1s     (compaction interval)
StarRocks 可见:    ≤ 3s     (refresh interval)
Milvus 可见:       ≤ 2s     (index build)
─────────────────────────
端到端总延迟:      ≤ 10s    (从数据变更到可被查询)

对比传统方案:
  T+1 ETL: 24小时
  微批处理: 5-15分钟
  本架构:   ~10秒 ← 质的飞跃
```

### 3.3 可观测 (Observable)

```
四大支柱:

1. Metrics (指标)
   → Flink: throughput, latency, checkpoint duration, backpressure
   → StarRocks: QPS, query latency, resource usage
   → LLM: token consumption, success rate, avg latency
   → 业务: 任务完成率, 用户满意度

2. Traces (链路追踪)
   → OpenTelemetry 贯穿 CDC→Flink→Storage→Application
   → 每次 RAG 查询的完整 trace: query→embedding→search→rerank→generate
   → 每次 Agent 执行的 step-by-step trace

3. Logs (日志)
   → 结构化日志 (JSON format)
   → Correlation ID 跨组件关联
   → LLM input/output 采样记录

4. Alerts (告警)
   → 数据延迟 > 30s → P1
   → LLM 错误率 > 5% → P2
   → Checkpoint 失败 → P1
   → 向量检索 P99 > 500ms → P2
```

## 四、关键 Trade-offs 记录

| 决策点 | 选项 A | 选项 B | 最终选择 | 理由 |
|--------|--------|--------|----------|------|
| 消息队列 | Kafka | Pulsar | Kafka | 生态成熟，团队熟悉 |
| 流批存储 | Iceberg | Paimon | Paimon | Flink 原生集成更好 |
| OLAP | ClickHouse | StarRocks | StarRocks | Join 性能 + Paimon Catalog |
| 向量库 | pgvector | Milvus | 两者都用 | 小规模PG，大规模Milvus |
| Embedding | OpenAI API | 自部署bge | bge自部署 | 成本可控 + 数据安全 |
| Agent框架 | LangGraph | OpenAI Agents SDK | LangGraph | 灵活编排 + 不绑定厂商 |
| 工具协议 | 自定义 | MCP | MCP | 社区标准、工具可复用 |

## 五、面试核心问答

### Q: 请介绍一下你的 AI 数据基础设施架构

> "我设计的是一套基于 Flink + Paimon 的实时 RAG 数据基础设施。数据从 MySQL CDC 和 IoT 设备采集后经 Kafka 进入 Flink 做实时清洗、窗口聚合和 Embedding 生成，然后写入 Paimon 做流批一体的统一存储。下游分两路：结构化数据通过 StarRocks 做实时 OLAP 分析，向量数据入 Milvus 支撑 RAG 检索。上层应用包括 RAG 问答、Agent 助手和 BI Dashboard。整套架构的核心设计理念是三个词：可信——全链路数据血缘和质量监控；实时——从数据变更到可查询端到端 10 秒内；可观测——Metrics/Traces/Logs/Alerts 四大支柱全覆盖。"

### Q: 为什么选 Paimon 而不是 Iceberg/Hudi？

> "三个原因：第一，我们以 Flink 为核心计算引擎，Paimon 是 Flink 社区原生项目，流式写入和读取的集成最顺畅；第二，我们需要 CDC 数据的 Upsert 能力来维护 SCD 维度表，Paimon 的主键表天然支持；第三，Paimon 的 Compaction 机制对流式场景优化得更好，写入放大比 Hudi 小。当然如果团队以 Spark 为主我会选 Iceberg，它的 Spark 集成更成熟。技术选型没有绝对好坏，要和团队技术栈和业务需求匹配。"

### Q: 这个架构最大的风险是什么？

> "两个主要风险：一是 Flink 状态的膨胀，CDC 数据量大时 RocksDB 状态可能超过预期，我们通过 TTL + 增量 Checkpoint + 状态分区来控制；二是 LLM 调用的不确定性，Embedding 生成和 RAG 推理都有延迟波动和质量波动，我们通过异步 IO + 缓存 + 质量监控来缓解。另外整体架构组件较多，运维复杂度是一个持续挑战，我们通过容器化部署 + 完善的可观测体系来应对。"
