## 一、Flink 核心（Window / Join / 容错 / State）

### Q1: Flink 的 Window 有哪些类型？Event Time 和 Processing Time 有什么区别？

**答题要点**:
- **Window 类型**: Tumbling(滚动)、Sliding(滑动)、Session(会话)、Global(全局)
- **Processing Time**: 以机器系统时间为基准，简单但不确定（重启/延迟导致结果不一致）
- **Event Time**: 以数据自带的时间戳为基准，需要 Watermark 推进时间，结果确定性可重现
- **选型**: 生产环境几乎都用 Event Time，只有对延迟极度敏感且允许不精确时才用 Processing Time
- **追问准备**: Watermark 怎么生成？→ BoundedOutOfOrderness + 自定义提取器；Watermark 设太大/太小会怎样？→ 太大会丢迟到数据，太小会增加延迟

### Q2: 如何处理乱序数据和迟到数据？

**答题要点**:
- **三层防线**: Watermark(delay) → AllowedLateness → SideOutput
- Watermark 容忍一定范围内的乱序（如 5s）
- AllowedLateness 允许窗口在 Watermark 通过后仍接收迟到数据并更新结果
- SideOutput 收集超过 AllowedLateness 的极端迟到数据到单独流
- **实战数据**: 分析源数据乱序分布 P99=4.2s，所以 Watermark delay 设 5s
- **追问准备**: 迟到数据更新的结果下游怎么感知？→ Retracted Stream / Upsert Sink

### Q3: Flink 的 Join 有哪些类型？各自适用什么场景？

**答题要点**:

| Join 类型 | 适用场景 | 状态大小 | 特点 |
|-----------|----------|----------|------|
| Regular Join | 双流等值关联 | 无限增长⚠️ | 保留所有匹配，需手动管理状态 |
| Interval Join | 双流时间范围关联 | 有界✅ | timeA ∈ [timeB-a, timeB+b] |
| Lookup Join | 流关联维表 | 外部存储 | 实时查外部DB/KV，不存状态 |
| Temporal Join | 流关联时态维表 | 有界 | 按事件时间匹配历史版本 |

- **关键经验**: Regular Join 在生产中要慎用，状态可能无限膨胀；优先用 Interval Join 或 Lookup Join
- **追问准备**: Lookup Join 的性能瓶颈在哪？→ 外部存储查询延迟，解决方案是 Async I/O + 本地缓存

### Q4: Flink Checkpoint 的原理是什么？怎么优化大状态的 CP 性能？

**答题要点**:
- **原理**: Chandy-Lamport 分布式快照算法，Barrier 对齐 + 异步快照
- **Exactly-Once 保证**: Source 重放 + Operator 快照 + Sink 两阶段提交
- **大状态优化**:
  1. HashMap → RocksDB（支持增量、不受 JVM 堆限制）
  2. Full CP → Incremental CP（只上传变化的 SST 文件）
  3. Unaligned CP（反压时不对齐 Barrier，减少 CP 延迟）
  4. State TTL（自动清理过期状态，减小状态体积）
  5. 调整并行度和 Slot Sharing Group
- **追问准备**: Unaligned CP 的代价？→ 状态更大（包含 in-flight 数据）、恢复更慢

### Q5: Flink 的状态后端有哪些？怎么选？

**答题要点**:
- **HashMapStateBackend**: 状态存在 JVM 堆内存，速度快但受 GC 和内存限制，适合小状态开发测试
- **EmbeddedRocksDBStateBackend**: 状态存在 RocksDB (本地磁盘+内存缓存)，支持超大状态和增量 CP，生产首选
- **选型标准**: 状态 >1GB 或需要增量 CP → RocksDB；状态很小且追求极致延迟 → HashMap
- **追问准备**: RocksDB 的 Compaction 策略影响？→ Level vs Universal，写入密集选 Universal

## 二、数据建模 / SCD

### Q6: SCD Type 1/2/3 分别是什么？怎么选？

**答题要点**:
- **Type 1**: 直接覆盖，不保留历史。适合修正错误或不关心历史的字段
- **Type 2**: 新增行 + 有效时间区间(start_time, end_time, is_current)。保留完整历史，适合需要趋势分析的维度
- **Type 3**: 增加"前值"列，只保留一层历史。折中方案，较少使用
- **选型**: 分析需求决定——需要"某个时间点的数据长什么样"→ Type 2；只需要最新值 → Type 1
- **追问准备**: Type 2 的查询怎么写？→ WHERE event_time BETWEEN start_time AND COALESCE(end_time, '9999-12-31')

### Q7: 事实表和维度表的设计原则？

**答题要点**:
- **事实表**: 记录业务事件（订单、点击），通常是 append-only，粒度要细
- **维度表**: 描述业务实体（用户、产品），会变更，需要做 SCD
- **星型模型 vs 雪花模型**: 星型查询性能好但冗余，雪花规范化但 Join 多。OLAP 场景倾向星型
- **宽表 vs 规范表**: 宽表查询快但更新难，规范表灵活但 Join 多。根据查询模式权衡

## 三、CDC (Change Data Capture)

### Q8: CDC 的实现方式有哪些？各有什么优缺点？

**答题要点**:

| 方式 | 延迟 | 完整性 | 侵入性 | 复杂度 |
|------|------|--------|--------|--------|
| Binlog CDC (Debezium/Flink CDC) | 秒级 | ✅ INSERT/UPDATE/DELETE | 低 | 中 |
| 定时轮询 | 分钟级 | ❌ 漏删改 | 中 | 低 |
| Trigger | 秒级 | ✅ | 高(性能) | 中 |
| 应用双写 | 毫秒级 | ❌ 不一致 | 无 | 高 |
| 日志解析 | 秒级 | ✅ | 低 | 高 |

- **推荐**: Binlog CDC 是最均衡的方案
- **追问准备**: Debezium vs Flink CDC？→ Debezium 独立部署经 Kafka 中转，Flink CDC 集成在 Flink 内少一跳

### Q9: CDC 怎么处理 Schema Evolution？

**答题要点**:
- Flink CDC 开启 `includeSchemaChanges=true` 捕获 DDL 事件
- Paimon/Iceberg 支持 Schema Evolution：新增列自动追加，删除列标记废弃
- 类型不兼容变更（如 INT→VARCHAR）需要告警 + 人工介入
- 字段重命名 CDC 无法识别，视为删旧增新，需应用层映射
- **追问准备**: 上游频繁 DDL 怎么办？→ Schema Registry + 变更审批流程

## 四、OLAP 引擎选型

### Q10: ClickHouse / Doris / StarRocks 怎么选？

**答题要点**:

| 维度 | ClickHouse | Doris | StarRocks |
|------|-----------|-------|-----------|
| 单表扫描 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ |
| 多表 Join | ⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ |
| 实时更新 | ⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ |
| 运维复杂度 | 高 | 中 | 中低 |
| 数据湖集成 | 差 | 好 | 好 |
| 物化视图 | ⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ |

- **选型逻辑**: 日志分析/单表→CH；通用 OLAP/Join 多→SR/Doris；已有 PG 生态→Doris
- **追问准备**: SR 的 CBO 为什么比 CH 好？→ CH 早期没有 CBO，Join 靠手工优化；SR 有完整的 Cost-Based Optimizer

### Q11: StarRocks 的物化视图怎么用？什么时候该用？

**答题要点**:
- **适用场景**: 高频重复查询、预聚合加速、Join 结果缓存
- **创建方式**: CREATE MATERIALIZED VIEW ... REFRESH ASYNC EVERY (INTERVAL 1 MINUTE)
- **注意事项**: 刷新有资源消耗，不要建太多；TTL 控制存储空间；监控刷新是否成功
- **效果**: 实测复杂 Join 查询 P99 从 8s 降至 1.2s

## 五、数据湖 (Iceberg / Paimon / Hudi)

### Q12: Iceberg / Paimon / Hudi 三大框架怎么选？

**答题要点**:

| 特性 | Iceberg | Paimon | Hudi |
|------|---------|--------|------|
| 计算引擎偏好 | Spark | Flink | Spark/Flink |
| 流式写入 | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ |
| Upsert/Delete | MOR/Merge-on-Read | LSM Primary Key | MOR/COW |
| 读取性能 | ⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐ |
| 社区成熟度 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐⭐ |
| Schema Evolution | ✅ | ✅ | ✅ |
| Time Travel | ✅ | ✅ | ✅ |

- **选型决策树**: Flink 为主→Paimon; Spark 为主→Iceberg; 大量 Upsert+Spark→Hudi
- **追问准备**: Paimon 的 LSM-Tree 写入放大问题？→ Compaction 策略调优 + 分离 Compaction 线程

### Q13: Paimon 的 Compaction 机制是怎样的？怎么调优？

**答题要点**:
- LSM-Tree 结构：数据先写 MemTable → Flush 到 Sorted Run → Compaction 合并
- Compaction 触发条件：Sorted Run 数量达到阈值 / 文件大小达到阈值
- 对流式写入的影响：Compaction 期间写入可能阻塞或延迟增加
- **调优参数**: compaction.min-file-num(默认5→调大到10), num-sorted-run.compact-trigger, 分离 Compaction 线程池
- **追问准备**: Full Compaction vs Minor Compaction？→ Minor 合并少量文件，Full 重写整个分区

## 六、向量数据库

### Q14: HNSW 和 IVF 索引的原理和区别？

**答题要点**:
- **HNSW (Hierarchical Navigable Small World)**: 多层跳表图结构，贪心搜索。查询 O(logN)，召回率高，但内存占用大（需存图结构）
- **IVF (Inverted File Index)**: 聚类分桶 + 桶内暴力搜索。查询 O(N/k)，可通过 nprobe 调节精度/速度权衡，内存占用相对小
- **选型**: <1000万向量追求高召回→HNSW；>1000万向量内存受限→IVF；超大规模→DiskANN/ScaNN
- **追问准备**: HNSW 的 M 和 efConstruction 参数含义？→ M=每层连接数(越大召回越好内存越高)，efConstruction=构建时搜索宽度

### Q15: Milvus 的架构是怎样的？怎么做混合检索？

**答题要点**:
- **架构**: Proxy(coordinator) + QueryNode(search) + DataNode(write) + IndexNode(build) + etcd(metadata) + MinIO(storage)
- **混合检索**: 向量相似度 + 标量过滤表达式(expr="category=='tech' AND created_at > 1700000000")
- **执行顺序**: 先标量过滤缩小范围 → 再向量搜索（Pre-filter）或先向量搜索再标量过滤（Post-filter），Milvus 自动选择最优策略
- **追问准备**: Partition Key 的作用？→ 数据物理隔离，过滤时只扫描相关分区

## 七、RAG

### Q16: RAG 幻觉问题的根因和解法？

**答题要点**:
- **根因分析**:
  1. 检索不准：召回了无关文档，LLM 基于错误上下文编造
  2. 知识缺失：正确答案不在知识库中，LLM 用参数知识填补
  3. Prompt 不当：没有明确要求"仅基于提供的上下文回答"
  4. 模型局限：LLM 本身的 sycophancy（迎合倾向）
- **解法**:
  1. 提升检索质量（混合检索+Rerank）
  2. System Prompt 强制引用来源 + "不确定时说不知道"
  3. Faithfulness 检测（NLI 模型判断回答是否有上下文支撑）
  4. 置信度评分 + 低置信度转人工
- **追问准备**: NLI 检测的成本？→ 可以用小模型(如 bge-reranker)做轻量级检测

### Q17: 怎么评估 RAG 系统的效果？

**答题要点**:
- **评估维度**:
  - Retrieval: Recall@K, MRR, NDCG
  - Generation: Faithfulness(忠实度), Relevance(相关性), Completeness(完整性)
  - End-to-End: 用户满意度、任务完成率
- **评估方法**: Golden Test Set + LLM-as-Judge + 人工抽检
- **自动化框架**: RAGAS / TruLens / Aries
- **追问准备**: LLM-as-Judge 可靠吗？→ 与人工标注一致性约 80%，适合趋势监控不适合绝对评判

### Q18: 文档切片策略怎么选？

**答题要点**:
- **固定长度**: 简单但可能切断语义（512 tokens + overlap 50）
- **语义切分**: 按段落/标题/章节自然边界切分，保持语义完整
- **递归切分**: 先按大标题分，再按小标题分，最后按段落分
- **Metadata 增强**: 每个 chunk 附带来源、类别、时间等元数据用于过滤
- **最佳实践**: 语义切分 + Metadata + Chunk Size 300-800 tokens + Overlap 10-20%

## 八、Agent

### Q19: Agent 在生产为什么不稳定？你怎么兜底？

**答题要点**:
- **根因**: 误差累积指数效应（单步 90% × 5步 = 59%）+ LLM 非确定性 + 工具不可靠
- **四层兜底**:
  1. 执行约束: Max Steps(3-5) + Timeout(30s/step) + Token Budget
  2. 容错机制: 重试2次 + 区分瞬时/确定性故障 + Loop Detection
  3. 输出校验: Pydantic/JSON Schema Validation
  4. 人工兜底: 置信度 <0.5 或异常 → 转人工
- **量化效果**: 任务完成率 48%→87%，P99 延迟 <15s

### Q20: Agent 的记忆怎么设计？

**答题要点**:
- **三层记忆**: 工作记忆(Context Window) + 短期记忆(摘要压缩) + 长期记忆(Vector DB)
- **长期记忆 = RAG**: 历史交互 Embedding → Vector DB → 按需检索回填
- **关键挑战**: 时效性衰减、意图区分、记忆污染、检索延迟
- **工程手段**: 时间衰减权重 + memory_type 过滤 + 置信度评分 + TTL 过期

### Q21: Multi-Agent 什么时候该用？

**答题要点**:
- **该用**: 角色天然分离、需要对抗验证、并行处理独立子任务、单 Agent 工具过多(>15)
- **不该用**: 简单问答链、线性顺序任务、单一领域深度推理、原型验证阶段
- **成本**: 通常 1.5-3x 单 Agent，加上通信一致性和调试复杂度
- **原则**: 能单 Agent 解决的绝不拆多 Agent

## 九、Prompt Engineering / 结构化输出

### Q22: 怎么让 LLM 稳定输出可被程序解析的结果？

**答题要点**:
- **三层保障**: API 格式约束(JSON Mode/Structured Outputs) + Prompt 示例(Few-shot) + 运行时校验(Pydantic + 重试)
- **实测数据**: 裸调用可用率 68% → 三层保障 98%，平均重试 1.24 次，额外成本 +28%
- **降级策略**: 重试失败 → 正则 best-effort 提取 → 入待处理队列

### Q23: CoT 和 Few-shot 怎么选？

**答题要点**:
- 需要"想"(多步推理) → CoT
- 需要"模仿"(格式/风格) → Few-shot
- 两者可组合: Few-shot CoT = 带推理过程的示例
- CoT 的代价: Token 3-10x, 延迟增加, 简单任务可能过度思考
