# Agent 原理与稳定性 — 原理篇

> **核心论点**：Agent 的可靠性来自工程约束，而非模型本身。模型提供"可能性"，工程提供"确定性"。

---

## 一、ReAct 范式：Agent 的核心推理循环

### 1.1 什么是 ReAct

ReAct = **Re**asoning + **Act**ing，由 Yao et al. (2022) 提出。其核心思想是让 LLM 在生成动作之前先进行显式推理，形成如下循环：

```
┌─────────────────────────────────────────────┐
│              ReAct Loop                      │
│                                              │
│   Thought → Action → Observation → Thought   │
│      ↑                            │          │
│      └────────────────────────────┘          │
│                                              │
│   终止条件: Final Answer / Max Steps          │
└─────────────────────────────────────────────┘
```

### 1.2 三要素详解

| 要素 | 说明 | 示例 |
|------|------|------|
| **Thought** | 模型的显式推理过程，解释"为什么做这个动作" | "用户问的是订单状态，我需要先查询订单系统" |
| **Action** | 调用外部工具或 API，格式化为结构化指令 | `search_orders(user_id="U123", status="all")` |
| **Observation** | 工具返回的结果，作为下一轮推理的输入 | `{"order_id": "O456", "status": "shipped", ...}` |

### 1.3 ReAct vs 纯 CoT vs 纯 Acting

```
纯 CoT:    Thought → Thought → Thought → Answer     (无外部交互，易幻觉)
纯 Acting: Action → Observation → Action → Answer   (无推理，盲目试错)
ReAct:     Thought → Action → Observation → Thought  (推理+行动交替)
```

**面试话术**：
> "ReAct 的本质是把 LLM 的隐式推理变成显式的 Thought-Action-Observation 循环。相比纯 CoT，它能通过工具获取实时信息减少幻觉；相比纯工具调用，它有推理环节保证动作的目的性。但 ReAct 的稳定性高度依赖每一步的质量——任何一步出错都会累积放大。"

### 1.4 ReAct 的工程实现要点

```python
# 伪代码：ReAct 核心循环
def react_loop(query, tools, max_steps=5):
    history = [{"role": "user", "content": query}]

    for step in range(max_steps):
        # 1. LLM 生成 Thought + Action
        response = llm.chat(
            messages=history,
            system_prompt=REACT_SYSTEM_PROMPT,
            tools=tool_schemas
        )

        # 2. 检查是否产生最终答案
        if response.is_final_answer:
            return response.content

        # 3. 解析并执行 Action
        action = parse_action(response)
        observation = execute_tool(action, tools)

        # 4. 将结果追加到历史
        history.append({"role": "assistant", "content": response.content})
        history.append({"role": "tool", "content": str(observation)})

    return "达到最大步数限制，未能得出最终答案"
```

## 二、Agent 四要素架构

### 2.1 总览

```
┌──────────────────────────────────────────────────────┐
│                  Agent Architecture                   │
│                                                       │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────┐ │
│  │ Planning │  │  Tools   │  │  Memory  │  │ Exec │ │
│  │  规划     │  │  工具     │  │  记忆     │  │ 执行  │ │
│  └────┬─────┘  └────┬─────┘  └────┬─────┘  └──┬───┘ │
│       │             │             │            │      │
│       └─────────────┴──────┬──────┴────────────┘      │
│                            │                          │
│                    ┌───────▼────────┐                 │
│                    │   LLM Core     │                 │
│                    │  (推理引擎)     │                 │
│                    └────────────────┘                 │
└──────────────────────────────────────────────────────┘
```

### 2.2 Planning（规划）

**职责**：将复杂任务分解为可执行的子任务序列。

| 规划策略 | 描述 | 适用场景 |
|----------|------|----------|
| Task Decomposition | 一次性拆解为子任务列表 | 步骤明确的任务 |
| ReAct Loop | 边执行边规划，每步动态决策 | 探索性任务 |
| Plan-and-Solve | 先生成完整计划，再逐步执行 | 需要全局视角的复杂任务 |
| Reflection | 执行后回顾评估，修正后续计划 | 需要自我纠错的场景 |

**关键问题**：
- LLM 的规划能力受限于上下文窗口和推理深度
- 过长的计划容易在执行中偏离
- 实际工程中常用**混合策略**：粗粒度预规划 + 细粒度 ReAct

### 2.3 Tools（工具）

**职责**：扩展 Agent 的能力边界，使其能操作外部世界。

**工具定义的关键要素**：
```yaml
tool_name: search_orders
description: |
  根据用户ID和状态搜索订单信息。
  当用户询问订单相关问题时使用此工具。
parameters:
  user_id:
    type: string
    required: true
    description: 用户唯一标识符
  status:
    type: enum["pending", "shipped", "delivered", "all"]
    required: false
    description: 订单状态过滤，默认 "all"
returns:
  type: array<Order>
  description: 匹配的订单列表
```

**工具设计的工程原则**：
1. **单一职责**：每个工具只做一件事
2. **明确契约**：参数类型、返回值、错误码都要清晰
3. **幂等安全**：读操作天然幂等，写操作需要幂等键
4. **错误友好**：返回人类可读的错误信息，而非原始异常堆栈
5. **数量克制**：工具太多会导致选择困难，建议 ≤15 个

### 2.4 Memory（记忆）

**职责**：维持上下文连贯性和长期知识积累。

| 记忆类型 | 实现方式 | 容量 | 延迟 |
|----------|----------|------|------|
| 工作记忆 | Context Window | 有限(4K-128K tokens) | 零 |
| 短期记忆 | 对话历史缓冲区 | 中等(最近N轮) | 极低 |
| 长期记忆 | Vector DB / KV Store | 理论无限 | 检索延迟 |
| 程序化记忆 | 硬编码规则/SOP | 固定 | 零 |

> ⚠️ 记忆系统设计是 Agent 稳定性的关键因素之一，详见《记忆与多智能体》章节。

### 2.5 Execution（执行）

**职责**：将规划转化为实际行动，处理工具调用的生命周期。

**执行层需要处理的工程问题**：
- 工具调用的序列化/反序列化
- 并发 vs 串行执行策略
- 超时控制与重试机制
- 执行结果的校验与清洗
- 异常捕获与降级处理

## 三、Agent 不稳定的根因分析

### 3.1 根因全景图

```
Agent 不稳定
├── 1. 误差累积 (Error Accumulation)
│   ├── 单步成功率 p → N步成功率 p^N
│   ├── Thought 偏差导致后续 Action 全部偏离
│   └── Observation 噪声被放大
├── 2. 无限循环 (Infinite Loops)
│   ├── 相同 Thought-Action 反复出现
│   ├── 工具返回空结果但不被识别
│   └── 缺乏有效的终止条件判断
├── 3. 工具失败 (Tool Failures)
│   ├── 参数格式错误 (LLM 生成了非法参数)
│   ├── 工具本身不可用 (API 宕机/限流)
│   ├── 返回结果超出预期 (过大/格式变化)
│   └── 权限/认证问题
├── 4. 幻觉工具调用 (Hallucinated Tool Calls)
│   ├── 调用不存在的工具
│   ├── 编造工具参数
│   ├── 误判工具使用场景
│   └── 虚构工具返回结果
└── 5. 上下文溢出 (Context Overflow)
    ├── 长对话超出窗口限制
    ├── 重要信息被截断
    └── Token 成本失控
```

### 3.2 误差累积的数学本质

这是理解 Agent 不稳定最核心的视角：

```
假设每步成功率为 p = 0.9（已经很高了）

步骤数 N    整体成功率 p^N    含义
─────────────────────────────────────────
1           90.0%            基本可用
2           81.0%            勉强接受
3           72.9%            开始出问题
5           59.0%            不可靠
10          34.9%            几乎不可用
20          12.2%            完全不可信
```

**结论**：即使单步准确率高达 90%，10 步任务的整体成功率也只有 35%。这就是为什么 Agent 在生产环境中必须被严格约束步骤数。

### 3.3 无限循环的典型模式

```python
# 典型死循环模式 1：重复调用
while True:
    result = agent.step()
    # LLM 每次都认为需要再次搜索，因为上次的结果"不够好"
    # 但实际上它在用相同的参数重复调用同一个工具

# 典型死循环模式 2：错误恢复失败
while True:
    result = agent.step()
    # 工具返回错误 → LLM 尝试修复参数 → 仍然错误 → 再次尝试...
    # 没有最大重试次数限制

# 典型死循环模式 3：目标漂移
while True:
    result = agent.step()
    # 初始目标是查订单，但中间发现了优惠券信息
    # LLM 决定先查优惠券详情，然后忘了原始目标
```

### 3.4 幻觉工具调用的危害

| 幻觉类型 | 示例 | 危害等级 |
|----------|------|----------|
| 虚构工具名 | 调用 `delete_all_users()` 当该工具不存在 | 🟡 低（通常会被拦截） |
| 编造参数值 | 传入不存在的 user_id | 🔴 高（可能触发意外行为） |
| 误用工具 | 用 `send_email` 回答查询类问题 | 🟡 中（浪费资源+用户体验差） |
| 虚构返回值 | 不调用工具直接编造结果 | 🔴🔴 极高（静默数据污染） |

## 四、工程视角：从"模型能力"到"工程约束"

### 4.1 核心理念转变

```
❌ 错误认知："换个更强的模型 Agent 就稳了"
✅ 正确认知："Agent 的可靠性 = f(模型能力, 工程约束, 任务复杂度)"

模型能力提升 → 天花板提高
工程约束加强 → 地板提高
任务复杂度降低 → 成功率提高
```

### 4.2 工程约束清单

| 约束维度 | 具体措施 | 效果 |
|----------|----------|------|
| **步数限制** | max_steps=3~5 | 防止无限循环和误差累积 |
| **超时控制** | 单步 timeout=30s, 全局 timeout=120s | 防止卡死 |
| **工具白名单** | 只暴露当前任务需要的工具 | 减少误选概率 |
| **输出校验** | JSON Schema / Pydantic 验证 | 拦截格式错误的输出 |
| **重试+降级** | 工具失败自动重试2次，仍失败则降级 | 提升容错性 |
| **人工兜底** | 置信度低时转人工 | 最后防线 |
| **审计日志** | 记录每步 Thought/Action/Observation | 可追溯、可调试 |
| **成本控制** | Token 预算上限 | 防止成本爆炸 |

### 4.3 可靠性金字塔

```
                    ▲
                   ╱ ╲
                  ╱ LLM╲           ← 模型能力（最难控制）
                 ╱______╲
                ╱  Prompt ╲        ← 提示词工程（中等可控）
               ╱____________╲
              ╱   Workflow    ╲     ← 工作流编排（高度可控）
             ╱__________________╲
            ╱    Guardrails      ╲  ← 安全护栏（完全可控）
           ╱________________________╲
          ╱      Human Fallback       ╲ ← 人工兜底（绝对可靠）
         ╱______________________________╲
```

> **面试话术**：
> "我在做 Agent 时的核心理念是'可靠性金字塔'——不能把所有希望寄托在 LLM 上。底层是人工兜底保证不出大事，往上是 Guardrails 做安全校验，再往上是 Workflow 做流程控制，然后是 Prompt Engineering 引导模型行为，最顶层才是模型自身能力。每一层都是对上一层的保险。生产环境中，下面几层的投入往往比换模型更有效。"

### 4.4 评估 Agent 稳定性的指标体系

| 指标 | 定义 | 目标值 |
|------|------|--------|
| 任务完成率 | 成功完成任务的比例 | ≥85% |
| 平均步数 | 完成任务的平均推理步数 | ≤4 |
| 工具调用准确率 | 正确调用工具的比例 | ≥95% |
| 平均耗时 | 端到端响应时间 | ≤10s |
| 异常率 | 触发兜底/报错的比例 | ≤5% |
| Token 消耗 | 每次任务的平均 Token 用量 | 有预算上限 |
| 幻觉率 | 输出包含虚假信息的比例 | ≤2% |

## 六、面试核心问答

### Q1: 什么是 ReAct？和 CoT 有什么区别？

> ReAct 是 Reasoning + Acting 的结合，让 LLM 在 Thought→Action→Observation 循环中交替推理和行动。CoT 只有推理链没有外部交互，ReAct 通过工具调用弥补了 CoT 无法获取实时信息的缺陷。但 ReAct 也引入了工具调用带来的额外不稳定因素。

### Q2: Agent 为什么在生产环境不稳定？

> 根本原因是误差累积的指数效应。即使每步 90% 准确，10 步任务整体只有 35% 成功率。加上 LLM 的非确定性、工具的不可靠性、上下文的有限性，多重因素叠加导致裸 Agent 在生产中几乎不可用。解决方案不是换更强的模型，而是加工程约束：限制步数、超时控制、输出校验、重试降级、人工兜底。

### Q3: 你怎么设计一个可靠的 Agent？

> 我会按"可靠性金字塔"分层设计：最底层是人工兜底，确保极端情况有人接管；第二层是 Guardrails，包括输出格式校验、敏感操作审批、Token 预算；第三层是 Workflow 编排，把复杂任务拆成确定性子流程，只在需要灵活决策的节点用 LLM；第四层才是 Prompt 优化和模型选择。同时做好可观测性，每步都记日志，出问题能快速定位。

### Q4: Agent 的工具越多越好吗？

> 不是。工具过多会导致三个问题：一是 LLM 选择困难，准确率下降；二是上下文占用增加，挤压推理空间；三是维护成本上升。实践中建议单次任务暴露的工具不超过 10-15 个，可以通过意图识别先筛选相关工具子集再给 Agent。
