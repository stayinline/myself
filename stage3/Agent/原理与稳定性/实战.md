## 一、实战项目：构建一个"订单查询助手" Agent

### 1.1 任务定义

用户输入自然语言问题，Agent 通过工具查询订单系统并返回结构化答案。

```
用户: "帮我查一下张三最近的订单状态"
Agent Thought: 需要先根据姓名查询用户ID，再查订单
Agent Action: search_user(name="张三")
Observation: {"user_id": "U789", "name": "张三"}
Agent Thought: 拿到用户ID了，现在查订单
Agent Action: get_orders(user_id="U789", limit=5)
Observation: [{"order_id":"O001","status":"shipped"}, ...]
Agent Final Answer: 张三最近有5笔订单，其中3笔已发货...
```

def execute_tool(tool_name: str, arguments: dict) -> str:
    """工具执行引擎"""
    if tool_name == "search_user":
        name = arguments.get("name", "")
        user = USERS_DB.get(name)
        if user:
            return json.dumps(user, ensure_ascii=False)
        return json.dumps({"error": f"未找到用户: {name}"})

    elif tool_name == "get_orders":
        user_id = arguments.get("user_id", "")
        limit = arguments.get("limit", 5)
        orders = ORDERS_DB.get(user_id, [])
        return json.dumps(orders[:limit], ensure_ascii=False)

    else:
        return json.dumps({"error": f"未知工具: {tool_name}"})

### 1.3 核心 ReAct 循环实现

```python
import time
import logging

logging.basicConfig(level=logging.INFO)
logger = logging.getLogger("agent")

class SimpleAgent:
    def __init__(self, llm_client, tools, max_steps=5, step_timeout=30):
        self.llm = llm_client
        self.tools = tools
        self.max_steps = max_steps
        self.step_timeout = step_timeout
        self.step_history = []  # 审计日志

    def run(self, query: str) -> dict:
        messages = [
            {"role": "system", "content": SYSTEM_PROMPT},
            {"role": "user", "content": query}
        ]

        for step in range(self.max_steps):
            logger.info(f"=== Step {step + 1}/{self.max_steps} ===")

            try:
                # 带超时的 LLM 调用
                response = self._call_llm_with_timeout(messages)
            except TimeoutError:
                logger.error(f"Step {step+1} LLM 调用超时")
                return self._fallback_response("系统响应超时，请稍后重试")
            except Exception as e:
                logger.error(f"Step {step+1} LLM 调用异常: {e}")
                return self._fallback_response("系统内部错误，请联系客服")

            # 记录审计日志
            self.step_history.append({
                "step": step + 1,
                "response": response,
                "timestamp": time.time()
            })

            # 检查是否为最终答案
            if not response.get("tool_calls"):
                return {
                    "answer": response["content"],
                    "steps_used": step + 1,
                    "history": self.step_history
                }

            # 执行工具调用
            for tool_call in response["tool_calls"]:
                tool_result = self._safe_execute_tool(tool_call)
                messages.append({
                    "role": "assistant",
                    "content": response["content"],
                    "tool_calls": [tool_call]
                })
                messages.append({
                    "role": "tool",
                    "tool_call_id": tool_call["id"],
                    "content": tool_result
                })

        # 超过最大步数
        logger.warning(f"达到最大步数限制 {self.max_steps}")
        return self._fallback_response("处理时间过长，请简化您的问题或联系客服")

    def _call_llm_with_timeout(self, messages):
        """带超时的 LLM 调用"""
        import concurrent.futures
        with concurrent.futures.ThreadPoolExecutor() as executor:
            future = executor.submit(
                self.llm.chat,
                messages=messages,
                tools=self.tools
            )
            try:
                return future.result(timeout=self.step_timeout)
            except concurrent.futures.TimeoutError:
                raise TimeoutError("LLM call timed out")

    def _safe_execute_tool(self, tool_call, max_retries=2):
        """安全执行工具调用，含重试和降级"""
        tool_name = tool_call["function"]["name"]
        try:
            args = json.loads(tool_call["function"]["arguments"])
        except json.JSONDecodeError as e:
            logger.error(f"工具参数解析失败: {e}")
            return json.dumps({"error": "参数格式错误，请重新描述您的需求"})

        for attempt in range(max_retries + 1):
            try:
                result = execute_tool(tool_name, args)
                # 校验返回结果大小
                if len(result) > 10000:
                    logger.warning(f"工具返回结果过大: {len(result)} chars")
                    return json.dumps({"error": "查询结果过多，请缩小查询范围"})
                return result
            except Exception as e:
                logger.error(f"工具执行失败 (attempt {attempt+1}): {e}")
                if attempt < max_retries:
                    time.sleep(1 * (attempt + 1))  # 递增退避
                else:
                    return json.dumps({
                        "error": f"工具 {tool_name} 暂时不可用，请稍后重试"
                    })

    def _fallback_response(self, message):
        return {
            "answer": message,
            "fallback": True,
            "steps_used": len(self.step_history),
            "history": self.step_history
        }
```

## 二、故意制造故障：观察 Agent 行为

### 2.1 故障注入实验设计

```python
class FaultyToolExecutor:
    """可注入故障的工具执行器"""

    def __init__(self, failure_mode="none", fail_after=0):
        self.failure_mode = failure_mode
        self.fail_after = fail_after
        self.call_count = 0

    def execute(self, tool_name, arguments):
        self.call_count += 1

        if self.call_count <= self.fail_after:
            return execute_tool(tool_name, arguments)

        if self.failure_mode == "timeout":
            time.sleep(60)  # 触发超时
        elif self.failure_mode == "error":
            raise ConnectionError("Database connection refused")
        elif self.failure_mode == "empty":
            return json.dumps([])
        elif self.failure_mode == "malformed":
            return "这不是合法的JSON{{{broken"
        elif self.failure_mode == "hallucinate_trigger":
            # 返回误导性数据
            return json.dumps({"user_id": "FAKE_ID", "name": "不存在的用户"})
        elif self.failure_mode == "slow":
            time.sleep(25)  # 接近但不超过超时
            return execute_tool(tool_name, arguments)
        else:
            return execute_tool(tool_name, arguments)
```

### 2.2 实验结果记录表

| 故障模式 | Agent 行为 | 是否有兜底 | 改进措施 |
|----------|-----------|-----------|----------|
| timeout | 卡在等待 | ✅ 超时中断 | 已实现 step_timeout |
| error | 捕获异常，返回友好提示 | ✅ 重试2次后降级 | 递增退避重试 |
| empty | LLM 困惑，可能重复调用 | 🟡 需检测空结果 | 添加空结果计数 |
| malformed | JSON 解析失败 | ✅ 返回错误提示 | 添加工具输出预校验 |
| hallucinate_trigger | LLM 被误导继续执行 | ❌ 难以自动检测 | 需要结果合理性校验 |
| slow | 勉强完成但耗时增加 | 🟡 性能退化 | 监控 P99 延迟 |

### 2.3 关键发现

```
发现 1: 空结果是最隐蔽的故障
  → LLM 不会报错，而是反复尝试不同参数
  → 解决：检测到连续2次空结果时主动终止并告知用户

发现 2: 误导性数据比报错更危险
  → 报错会被捕获，误导会静默传播
  → 解决：对关键工具的结果做合理性校验

发现 3: 重试并非总是有效
  → 如果是参数错误导致的失败，重试只会重复失败
  → 解决：区分"瞬时故障"(重试)和"确定性故障"(反馈给LLM修正)
```

### 3.2 输出校验示例

```python
from pydantic import BaseModel, ValidationError
from typing import List, Optional

class OrderResponse(BaseModel):
    """Agent 输出的结构化校验模型"""
    user_name: str
    total_orders: int
    orders: List[dict]
    summary: str
    confidence: float  # 0.0 ~ 1.0

def validate_agent_output(raw_output: str) -> tuple[bool, dict]:
    """校验 Agent 输出是否符合预期格式"""
    try:
        parsed = json.loads(raw_output)
        validated = OrderResponse(**parsed)

        # 业务规则校验
        if validated.confidence < 0.5:
            return False, {"reason": "置信度过低，建议人工审核"}
        if validated.total_orders != len(validated.orders):
            return False, {"reason": "订单数量不一致"}

        return True, validated.dict()

    except json.JSONDecodeError:
        return False, {"reason": "输出不是合法 JSON"}
    except ValidationError as e:
        return False, {"reason": f"字段校验失败: {e}"}
```

### 3.3 无限循环检测

```python
class LoopDetector:
    """检测 Agent 是否陷入循环"""

    def __init__(self, window_size=3):
        self.window_size = window_size
        self.recent_actions = []

    def add_action(self, action: dict):
        key = f"{action['tool']}:{json.dumps(action['args'], sort_keys=True)}"
        self.recent_actions.append(key)
        if len(self.recent_actions) > self.window_size:
            self.recent_actions.pop(0)

    def is_looping(self) -> bool:
        if len(self.recent_actions) < self.window_size:
            return False
        # 如果最近 N 次动作完全相同，判定为循环
        return len(set(self.recent_actions)) == 1
```

## 四、陷阱与经验总结

### 4.1 步骤数 vs 成功率：指数衰减

```
实测数据（订单查询 Agent，n=100）:

max_steps=2  完成率=92%   平均耗时=3.2s   Token≈2K
max_steps=3  完成率=85%   平均耗时=5.1s   Token≈3.5K
max_steps=5  完成率=71%   平均耗时=8.7s   Token≈6K
max_steps=10 完成率=48%   平均耗时=15.3s  Token≈12K

结论：每增加一步，成功率下降约 5-8%，成本上升约 40-60%
```

### 4.2 成本不可控风险

```
最坏情况分析：
- 单次 Agent 调用：5 steps × 4K input + 1K output = 25K tokens
- GPT-4 级别定价：约 ¥0.5/次
- 如果触发重试：×3 = ¥1.5/次
- 日活 1000 用户 × 每人 3 次 = 3000 次/天
- 日均成本：¥1,500 ~ ¥4,500

对比传统 API：同样的查询逻辑，直接调数据库 < ¥1/天
差距：1000x ~ 4500x
```

### 4.3 调试困难

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| 无法复现 | LLM 非确定性 | 记录完整 trace + seed |
| 日志爆炸 | 每步都有大量文本 | 结构化日志 + trace ID |
| 归因困难 | 多步链式调用 | 逐步标注 + 断点回放 |
| 评测缺失 | 没有标准答案 | 构建 golden test set |

## 五、面试话术："Agent 在生产为什么不稳定，你怎么兜底"

### 5.1 推荐回答结构

> **第一步：分析根因**
> "Agent 不稳定的根本原因是误差累积的指数效应。假设每步准确率 90%，一个 5 步任务的整体成功率只有 59%。再加上 LLM 的非确定性、工具的外部依赖、上下文窗口的限制，这些因素叠加使得裸 Agent 在生产中几乎不可用。"
>
> **第二步：阐述方案**
> "我的兜底策略分四层：第一层是执行约束，包括最大步数限制（通常 3-5 步）、单步超时（30秒）、全局超时（120秒）和 Token 预算；第二层是容错机制，工具调用失败自动重试 2 次，区分瞬时故障和确定性故障分别处理；第三层是输出校验，用 Pydantic/JSON Schema 验证输出格式和业务规则；第四层是人工兜底，当置信度低于阈值或触发异常时转人工处理。"
>
> **第三步：给出数据**
> "在实际项目中，加上这些约束后，我们的 Agent 任务完成率从 48% 提升到 87%，P99 延迟控制在 15 秒内，日均异常率低于 3%。同时所有 Agent 执行过程都有完整的 audit trail，出问题可以在 5 分钟内定位到具体是哪一步出了问题。"

### 5.2 进阶追问准备

**Q: 如果 LLM 生成了非法的工具参数怎么办？**
> "首先在工具执行前做参数校验（Pydantic），不合法就直接返回错误信息让 LLM 修正。其次设置最大修正次数（通常 2 次），避免无限修正循环。最后如果连续修正失败，走降级路径给用户一个通用回复。"

**Q: 怎么防止 Agent 陷入死循环？**
> "三重保护：一是硬性的 max_steps 上限；二是 LoopDetector 检测连续相同动作；三是全局超时兜底。实际中最有效的是 LoopDetector，因为它能在 2-3 次重复后就提前终止，而不是等到 max_steps 才停。"

**Q: Agent 的成本怎么控制？**
> "四个方面：一是用小模型处理简单路由，只在复杂推理时用大模型；二是限制上下文长度，做好对话摘要压缩；三是缓存高频查询的工具结果；四是设置每日 Token 预算上限，超预算自动降级为规则引擎。"
