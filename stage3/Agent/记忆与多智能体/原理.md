# Agent 记忆与多智能体 — 原理篇

> **核心论点**：Agent 记忆 = 向量检索，回到你的数据/向量优势。多 Agent 不是银弹，单 Agent 能解决的不要拆成多 Agent。

---

## 一、Agent 记忆系统

### 1.1 为什么 Agent 需要记忆

LLM 本身是"无状态"的——每次调用都是独立的。没有记忆的 Agent 就像一个失忆症患者：
- 无法记住用户之前说过的话
- 无法利用历史交互中学到的偏好
- 无法跨会话积累知识
- 每次都要从头解释上下文

### 1.2 记忆分类体系

```
┌─────────────────────────────────────────────────────┐
│              Agent Memory Taxonomy                   │
│                                                      │
│  ┌─────────────────────────────────────────────────┐ │
│  │           Short-term Memory (工作记忆)           │ │
│  │  • Context Window 内的对话历史                   │ │
│  │  • 当前任务的中间状态                             │ │
│  │  • 容量有限，延迟为零                             │ │
│  └─────────────────────────────────────────────────┘ │
│                                                      │
│  ┌─────────────────────────────────────────────────┐ │
│  │           Long-term Memory (长期记忆)            │ │
│  │  • Vector DB 存储的历史交互                      │ │
│  │  • 结构化数据库中的用户画像                       │ │
│  │  • 知识库文档                                     │ │
│  │  • 容量理论无限，有检索延迟                        │ │
│  └─────────────────────────────────────────────────┘ │
│                                                      │
│  ┌─────────────────────────────────────────────────┐ │
│  │          Procedural Memory (程序化记忆)           │ │
│  │  • System Prompt 中的规则和 SOP                  │ │
│  │  • 硬编码的业务逻辑                               │ │
│  │  • 工具使用规范                                   │ │
│  │  • 固定不变，零延迟                               │ │
│  └─────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────┘
```

### 1.3 短期记忆：Context Window 管理

#### 1.3.1 基本策略

| 策略 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| Full History | 保留所有对话 | 信息完整 | Token 爆炸 |
| Sliding Window | 只保留最近 N 轮 | 简单高效 | 丢失早期重要信息 |
| Summarization | 定期压缩历史为摘要 | 平衡信息和成本 | 压缩可能丢细节 |
| Selective Recall | 按相关性选择性加载 | 精准高效 | 需要检索系统 |
| Hybrid | 近期完整 + 远期摘要 + 按需检索 | 最优效果 | 实现复杂 |

#### 1.3.2 滑动窗口 vs 摘要压缩对比

```
原始对话（10轮）: ~8000 tokens

方案 A - 滑动窗口（保留最近3轮）:
  Input: [最近3轮完整] = ~2400 tokens
  问题: 第1轮提到的用户偏好丢失了

方案 B - 摘要压缩:
  Input: [前7轮摘要 + 最近3轮完整] = ~3500 tokens
  优势: 保留了关键信息，成本可控

方案 C - 混合策略:
  Input: [关键事实摘要 + 最近3轮完整 + 按需检索的相关历史]
  效果: 最优但最复杂
```

#### 1.3.3 摘要压缩的工程实现

```python
def compress_history(messages, max_tokens=4000):
    """将过长的对话历史压缩为摘要"""
    total_tokens = count_tokens(messages)

    if total_tokens <= max_tokens:
        return messages  # 不需要压缩

    # 保留最近 N 轮完整对话
    recent_count = 3
    recent_messages = messages[-recent_count * 2:]  # user+assistant 成对

    # 将早期对话发给 LLM 做摘要
    old_messages = messages[:-recent_count * 2]
    summary_prompt = f"""请将以下对话历史压缩为简洁摘要，保留：
    1. 用户明确表达的偏好和需求
    2. 已确认的关键事实和数据
    3. 未完成的任务和承诺
    4. 重要的决策和原因

    对话历史：
    {format_messages(old_messages)}
    """
    summary = llm.generate(summary_prompt)

    return [{"role": "system", "content": f"历史对话摘要：{summary}"}] + recent_messages
```

### 1.4 长期记忆：向量检索回填

> ⚠️ **这里直接复用你在 Stage 2 学的 Vector DB + RAG 技能**

#### 1.4.1 长期记忆架构

```
┌──────────────────────────────────────────────────┐
│           Long-term Memory Architecture           │
│                                                   │
│  User Query ──→ Embedding ──→ Vector Search       │
│                                   │               │
│                            Top-K Results          │
│                                   │               │
│                    ┌──────────────▼──────────┐    │
│                    │   Relevance Filtering    │    │
│                    │   (Score Threshold)      │    │
│                    └──────────────┬──────────┘    │
│                                   │               │
│                    Memory-Augmented Prompt         │
│                                   │               │
│                              LLM Response          │
│                                   │               │
│                    ┌──────────────▼──────────┐    │
│                    │   Memory Update          │    │
│                    │   (Store new interaction) │    │
│                    └─────────────────────────┘    │
└──────────────────────────────────────────────────┘
```

#### 1.4.2 什么值得存入长期记忆

| 类别 | 示例 | 存储时机 |
|------|------|----------|
| 用户偏好 | "我喜欢简洁的回答" | 用户明确表达时 |
| 关键事实 | "我的服务器IP是10.0.1.5" | 首次出现时 |
| 任务结果 | "上次部署失败的根因是内存不足" | 重要任务完成后 |
| 纠正记录 | "不是MySQL，我用的是PostgreSQL" | 用户纠正时 |
| 领域知识 | 业务规则、SOP更新 | 管理员配置时 |

#### 1.4.3 记忆检索的关键挑战

```
挑战 1: 相关性 ≠ 时效性
  → 用户上个月说"用Java"，这个月改用了Go
  → 向量检索可能返回旧的Java相关记忆
  → 解决：加入时间衰减权重

挑战 2: 语义相似 ≠ 意图匹配
  → "订单为什么延迟" vs "怎么让订单延迟" 语义相近但意图相反
  → 解决：结合关键词过滤 + 意图分类

挑战 3: 记忆污染
  → 错误信息被存入记忆后持续影响后续交互
  → 解决：记忆可信度评分 + 人工审核机制

挑战 4: 检索延迟叠加
  → 每次 Agent 步骤都检索记忆会增加延迟
  → 解决：异步预检索 + 缓存热点查询
```

## 二、Multi-Agent 模式

### 2.1 什么时候需要 Multi-Agent

#### ✅ 适合 Multi-Agent 的场景

| 场景 | 原因 | 示例 |
|------|------|------|
| 角色天然分离 | 不同角色需要不同的 system prompt 和工具集 | 客服+技术专家+产品经理 |
| 对抗验证 | 需要一个 Agent 质疑另一个的输出 | 代码生成 + Code Review |
| 并行处理 | 多个独立子任务可以同时进行 | 同时搜索多个数据源 |
| 专业分工 | 单个 Agent 的工具/知识过多导致选择困难 | 数据分析 Agent + 报告撰写 Agent |
| 流程审批 | 需要多层级审核 | 起草 → 审核 → 批准 |

#### ❌ 不适合 Multi-Agent 的场景

| 场景 | 为什么单 Agent 更好 |
|------|---------------------|
| 简单的问答链 | 增加通信开销，无额外收益 |
| 顺序依赖的线性任务 | 串行执行不需要多 Agent |
| 单一领域的深度推理 | 拆分会丢失上下文连贯性 |
| 原型验证阶段 | 先验证单 Agent 可行性再拆分 |

> **面试话术**：
> "Multi-Agent 不是越多越好。我的判断标准是：如果一个 Agent 的 system prompt 超过 3000 tokens 且包含 3 种以上截然不同的角色定义，或者工具列表超过 15 个且明显分为几个不相关的组，那才考虑拆分。否则单 Agent + 好的 Prompt Engineering 就够了。过早引入 Multi-Agent 会带来通信开销、一致性问题和调试困难。"

### 2.2 主流 Multi-Agent 模式

#### 模式 1: Orchestrator-Executor（编排者-执行者）

```
                    ┌──────────────┐
                    │ Orchestrator │ ← 接收用户请求，分配子任务
                    │   编排者      │
                    └──────┬───────┘
                           │
              ┌────────────┼────────────┐
              ▼            ▼            ▼
        ┌──────────┐ ┌──────────┐ ┌──────────┐
        │ Executor │ │ Executor │ │ Executor │
        │ 数据查询  │ │ 分析计算  │ │ 报告生成  │
        └────┬─────┘ └────┬─────┘ └────┬─────┘
             │            │            │
             └────────────┼────────────┘
                          ▼
                    ┌──────────────┐
                    │ Orchestrator │ ← 汇总结果，生成最终回答
                    └──────────────┘
```

**特点**：
- 中心化控制，Orchestrator 决定谁做什么
- Executor 之间不直接通信
- 适合有明确分工的流程型任务
- Orchestrator 是单点瓶颈和故障点

#### 模式 2: Debate（辩论式）

```
        ┌──────────┐     ┌──────────┐
        │ Agent A  │◄───►│ Agent B  │
        │ 正方观点  │     │ 反方观点  │
        └────┬─────┘     └────┬─────┘
             │                │
             └───────┬────────┘
                     ▼
              ┌──────────────┐
              │   Judge      │ ← 综合双方观点做出判断
              │   裁判        │
              └──────────────┘
```

**特点**：
- 通过对抗提升输出质量
- 适合需要批判性思维的场景（代码审查、方案评审）
- 成本高（至少 3x LLM 调用）
- 可能陷入无限辩论（需要设置最大轮次）

#### 模式 3: Role Division（角色分工）

```
        ┌──────────┐   ┌──────────┐   ┌──────────┐
        │ Product  │──►│ Engineer │──►│ Tester   │
        │ Manager  │   │ Developer│   │ QA       │
        │ 需求分析  │   │ 编码实现  │   │ 测试验证  │
        └──────────┘   └──────────┘   └──────────┘
```

**特点**：
- 模拟真实团队协作流程
- 每个角色有独立的 system prompt 和专业知识
- 信息在角色间单向或双向传递
- 适合软件开发、内容创作等流程

### 2.3 Multi-Agent 通信协议

| 通信方式 | 描述 | 适用模式 |
|----------|------|----------|
| Shared State | 共享状态空间，各 Agent 读写 | Orchestrator |
| Message Passing | Agent 间发送消息 | Role Division |
| Blackboard | 共享公告板，订阅感兴趣的内容 | Debate |
| Hierarchical | 上级向下级发指令，下级汇报 | Orchestrator |
| Peer-to-Peer | 平等节点间自由通信 | Debate |

### 2.4 Multi-Agent 的成本模型

```
假设单 Agent 完成任务需要:
  - 5 steps × 2K tokens = 10K tokens
  - 耗时 8s

Multi-Agent (3 Agents + 1 Orchestrator):
  - Orchestrator 路由: 2K tokens
  - Agent A: 4K tokens
  - Agent B: 3K tokens  
  - Agent C: 3K tokens
  - Orchestrator 汇总: 2K tokens
  - Agent 间通信: 4K tokens
  - Total: 18K tokens (1.8x)
  - 耗时: 12-15s (含通信开销)

结论: Multi-Agent 通常带来 1.5-3x 的成本增长
      只有在质量提升 > 成本增长时才值得
```

## 三、记忆系统的工程权衡

### 3.1 记忆 vs 无记忆的对比

| 维度 | 无记忆 | 短期记忆 | 长期记忆 |
|------|--------|----------|----------|
| 用户体验 | 每轮重复自己 | 会话内连贯 | 跨会话个性化 |
| 实现复杂度 | 零 | 低 | 高 |
| 基础设施 | 无 | 内存/Redis | Vector DB + Embedding |
| 延迟影响 | 无 | 极小 | 检索 50-200ms |
| 成本影响 | 基准 | +20-50% | +50-200% |
| 数据治理 | 无需 | 会话结束清理 | GDPR/隐私合规 |
| 故障风险 | 无 | 低 | 检索不准/记忆污染 |

### 3.2 决策树：要不要加记忆

```
用户是否需要跨轮次的上下文？
├── 否 → 不需要记忆，用 stateless API
└── 是 → 会话内还是跨会话？
    ├── 会话内 → 短期记忆（滑动窗口/摘要）
    └── 跨会话 → 信息是否敏感/个人化？
        ├── 否 → 全局知识库（RAG 即可）
        └── 是 → 用户级长期记忆
            ├── 数据量 < 100条/用户 → KV Store / DB
            └── 数据量 > 100条/用户 → Vector DB
```

## 四、面试核心问答

### Q1: Agent 怎么记住历史和长期知识？

> "我把 Agent 记忆分三层：工作记忆就是 Context Window 里的当前对话；短期记忆通过滑动窗口或摘要压缩保留近期上下文；长期记忆本质上是 RAG——把历史交互 embedding 后存入 Vector DB，新请求来时做相似度检索回填到 prompt 里。这其实回到了我们做数据基础设施的老本行：索引、检索、排序、过滤。关键技术挑战不在存储而在检索相关性，比如要处理时效性衰减、意图区分、记忆去重等问题。"

### Q2: 什么时候该用 Multi-Agent？

> "我的原则是'能单 Agent 解决的绝不拆多 Agent'。只有三种情况我会考虑 Multi-Agent：一是角色天然分离且需要不同的 prompt 和工具集；二是需要对抗验证来提升质量；三是工具有明确的分组且单 Agent 选择准确率已经下降。Multi-Agent 的成本通常是单 Agent 的 1.5-3 倍，还要加上通信一致性和调试复杂度的隐性成本。"

### Q3: 长期记忆用 Vector DB 有什么问题？

> "主要三个问题：第一是时效性，向量检索只看语义相似度不看时间，可能召回过时的记忆，需要加时间衰减权重；第二是意图混淆，语义相近但意图相反的查询会召回错误记忆，需要额外的意图分类层；第三是记忆污染，错误信息一旦入库就会持续影响，需要可信度评分和纠错机制。这些问题的解法都偏向传统数据工程：索引设计、过滤策略、数据质量治理。"

## 五、技术选型参考

### 5.1 记忆存储选型

| 方案 | 适用规模 | 检索能力 | 运维复杂度 | 推荐场景 |
|------|----------|----------|-----------|----------|
