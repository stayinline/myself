# Prompt 工程与结构化输出 — 原理篇

> **核心论点**：Prompt Engineering 不是玄学，而是"用自然语言编程"。结构化输出是让 LLM 融入数据管道的关键桥梁。

---

## 一、System Prompt：定义角色与约束

### 1.1 System Prompt 的作用

System Prompt 是 LLM 行为的"宪法"——它定义了模型在整个对话中的基本行为准则。

```
┌─────────────────────────────────────────────────────┐
│              System Prompt Structure                  │
│                                                      │
│  1. Role Definition (你是谁)                          │
│     "你是一个资深数据工程师..."                        │
│                                                      │
│  2. Task Scope (你能做什么)                           │
│     "你的职责是帮助用户分析数据管道问题..."             │
│                                                      │
│  3. Constraints (你不能做什么)                        │
│     "不要编造不确定的数据..."                          │
│                                                      │
│  4. Output Format (输出格式要求)                      │
│     "请以 JSON 格式返回..."                           │
│                                                      │
│  5. Tone & Style (语气风格)                          │
│     "回答简洁专业，避免冗余解释..."                     │
│                                                      │
│  6. Examples (可选的行为示例)                         │
│     "当用户问X时，你应该..."                           │
└─────────────────────────────────────────────────────┘
```

### 1.2 System Prompt 设计原则

| 原则 | ✅ 好的写法 | ❌ 差的写法 |
|------|-----------|-----------|
| 具体明确 | "返回包含 order_id, status, amount 三个字段的 JSON" | "返回结构化数据" |
| 正面表述 | "只讨论数据工程相关问题" | "不要回答和数据工程无关的问题" |
| 分层组织 | 用 markdown 标题分隔不同部分 | 一大段不分段的文字 |
| 有优先级 | "如果信息不足，先说'不确定'再推测" | "要准确但也允许推测" |
| 可测试 | "金额字段保留两位小数" | "金额要精确" |

### 1.3 System Prompt 的长度权衡

```
过短 (<200 tokens):
  → 行为不稳定，每次回复风格差异大
  → 容易被用户 prompt injection 覆盖

过长 (>3000 tokens):
  → 每次请求都消耗大量 Token（成本↑）
  → 重要指令可能被"遗忘"（Lost in the Middle 效应）
  → 挤压用户内容的空间

最佳实践: 500-1500 tokens
  → 足够定义清晰的行为规范
  → 成本可控
  → 关键指令放在开头和结尾（利用 Primacy + Recency Effect）
```

## 二、Few-shot Learning：通过示例引导

### 2.1 Few-shot 的本质

Few-shot 不是"训练"，而是"上下文学习"(In-context Learning)。模型通过示例理解你期望的输入输出映射模式。

```
Zero-shot:
  Input: "分类这条评论的情感"
  Output: ??? (模型猜测你想要的格式和内容)

One-shot:
  Input: "评论: '这个产品太棒了！' → 情感: positive
         分类这条评论的情感: '服务很差'"
  Output: "情感: negative" (有了格式参考)

Few-shot (3-shot):
  Input: "评论: '这个产品太棒了！' → 情感: positive
         评论: '快递很慢但东西还行' → 情感: mixed
         评论: '完全是骗局' → 情感: negative
         分类这条评论的情感: '包装精美物流也快'"
  Output: "情感: positive" (模式更稳定)
```

### 2.2 Few-shot 适用场景

| 场景 | 效果 | 示例数量建议 |
|------|------|-------------|
| 风格一致性 | ⭐⭐⭐⭐⭐ | 3-5 个 |
| 特定格式输出 | ⭐⭐⭐⭐⭐ | 2-3 个 |
| 分类任务 | ⭐⭐⭐⭐ | 每类 1-2 个 |
| 复杂推理 | ⭐⭐ | 不如 CoT |
| 知识注入 | ⭐ | 应该用 RAG |
| 创造性任务 | ⭐⭐⭐ | 2-3 个定基调 |

### 2.3 Few-shot 示例选择策略

```
原则 1: 覆盖边界情况
  ✅ 包含正常案例 + 边缘案例 + 异常情况
  ❌ 全部是理想化的简单案例

原则 2: 多样性 > 数量
  ✅ 3个涵盖不同情况的例子
  ❌ 10个高度相似的例子

原则 3: 从易到难排列
  ✅ 简单→中等→复杂（利用 recency effect）
  ❌ 随机排列

原则 4: 真实数据 > 人造数据
  ✅ 从生产日志中抽取的真实案例
  ❌ 完美但脱离实际的人造案例
```

### 2.4 Few-shot 的成本考量

```
假设每个示例 ~200 tokens，3-shot = 600 tokens
每次请求额外成本: 600 input tokens ≈ ¥0.01

日请求量 10,000 次:
  额外日成本: ¥100
  月成本: ¥3,000

优化手段:
  - 缓存 system prompt + few-shot (API 层面的 prompt caching)
  - 动态选择最相关的示例而非固定示例
  - 将 few-shot 压缩为更紧凑的格式
```

## 三、Chain of Thought (CoT)：引导推理

### 3.1 CoT 的原理

CoT 让模型在给出最终答案前显式展示推理过程，类似于人类"想清楚再说"。

```
无 CoT:
  Q: "一个水池有两个进水管A和B，A管3小时注满，B管5小时注满，同时开几小时注满？"
  A: "1.875小时" ← 可能对可能错，无法验证过程

有 CoT:
  Q: "同上，请一步步思考"
  A: "让我一步步计算：
      1. A管每小时注入 1/3 池
      2. B管每小时注入 1/5 池
      3. 同时开每小时注入 1/3 + 1/5 = 8/15 池
      4. 注满需要 15/8 = 1.875 小时
      答案: 1.875小时" ← 过程可验证，准确率更高
```

### 3.2 CoT 变体对比

| 变体 | 触发方式 | 适用场景 | 效果 |
|------|----------|----------|------|
| Zero-shot CoT | "Let's think step by step" | 通用推理 | ⭐⭐⭐ |
| Few-shot CoT | 提供带推理过程的示例 | 特定领域推理 | ⭐⭐⭐⭐ |
| Self-Consistency | 多次采样取多数投票 | 高准确率需求 | ⭐⭐⭐⭐⭐ |
| Tree of Thoughts | 探索多条推理路径 | 复杂规划 | ⭐⭐⭐⭐⭐ |
| ReAct | Thought-Action-Observation | 需要外部工具 | ⭐⭐⭐⭐ |

### 3.3 CoT 的工程注意事项

```
优点:
  ✅ 提升复杂推理准确率 10-30%
  ✅ 推理过程可审计、可调试
  ✅ 减少"自信的错误答案"

缺点:
  ❌ 输出长度增加 3-10x → Token 成本上升
  ❌ 延迟增加（更多 token 生成）
  ❌ 推理过程本身也可能出错（"忠实推理"问题）
  ❌ 简单任务用 CoT 反而可能降低准确率（过度思考）

何时使用 CoT:
  → 多步数学/逻辑推理
  → 需要解释决策原因
  → 错误代价高的场景
  → 评估模型推理能力

何时不用 CoT:
  → 简单分类/提取
  → 实时性要求极高
  → 任务本身不需要推理
```

## 四、JSON Mode / Schema 约束输出

### 4.1 为什么需要结构化输出

```
LLM 的自由文本输出:
  "张三有3个订单，分别是O001已发货、O002已送达、O003待处理，总金额977.5元"

程序需要的结构化数据:
  {
    "user": "张三",
    "orders": [
      {"id": "O001", "status": "shipped", "amount": 299.0},
      {"id": "O002", "status": "delivered", "amount": 158.5},
      {"id": "O003", "status": "pending", "amount": 520.0}
    ],
    "total_amount": 977.5
  }

问题: 自由文本无法被下游系统直接消费
解决: 强制 LLM 输出结构化格式
```

### 4.2 结构化输出的三种方式

#### 方式 1: Prompt 指令约束

```python
prompt = """请将以下信息提取为 JSON 格式：
{
  "user_name": string,
  "orders": [{"order_id": string, "status": string, "amount": number}],
  "total_amount": number
}

注意：
- 必须是合法的 JSON
- 不要添加任何解释文字
- amount 字段为数字类型，不要加引号
- 如果信息缺失，对应字段填 null

文本：张三有3个订单..."""
```

**优点**: 简单，兼容所有模型
**缺点**: 不保证一定输出合法 JSON，需要校验+重试

#### 方式 2: API 原生 JSON Mode

```python
# OpenAI API
response = client.chat.completions.create(
    model="gpt-4",
    messages=[...],
    response_format={"type": "json_object"}  # 强制 JSON 输出
)

# 或使用 Structured Outputs (严格模式)
response = client.chat.completions.create(
    model="gpt-4o",
    messages=[...],
    response_format={
        "type": "json_schema",
        "json_schema": {
            "name": "order_response",
            "strict": True,
            "schema": {
                "type": "object",
                "properties": {
                    "user_name": {"type": "string"},
                    "orders": {
                        "type": "array",
                        "items": {
                            "type": "object",
                            "properties": {
                                "order_id": {"type": "string"},
                                "status": {"type": "string"},
                                "amount": {"type": "number"}
                            },
                            "required": ["order_id", "status", "amount"]
                        }
                    },
                    "total_amount": {"type": "number"}
                },
                "required": ["user_name", "orders", "total_amount"]
            }
        }
    }
)
```

**优点**: API 层面保证输出合法性
**缺点**: 并非所有模型/API 支持；strict mode 有限制

#### 方式 3: Function Calling / Tool Use

```python
# 通过定义工具的返回格式来约束输出
tools = [{
    "type": "function",
    "function": {
        "name": "extract_order_info",
        "description": "从文本中提取订单信息",
        "parameters": {
            "type": "object",
            "properties": {
                "user_name": {"type": "string"},
                "orders": {...},
                "total_amount": {"type": "number"}
            },
            "required": ["user_name", "orders", "total_amount"]
        }
    }
}]
```

### 4.3 结构化输出的陷阱

```
陷阱 1: JSON Mode ≠ 正确的 JSON
  → 保证语法合法，但不保证语义正确
  → 字段名可能不对、值类型可能错、必填字段可能缺失
  → 必须做 Schema Validation

陷阱 2: Strict Mode 的限制
  → 不支持所有 JSON Schema 特性（如 pattern, minLength）
  → 可能需要简化 Schema
  → 某些模型版本不支持

陷阱 3: 重试策略
  → 第一次输出不合法 → 追加错误信息让模型修正
  → 最多重试 2-3 次 → 仍失败则走降级路径
  → 不要无限重试

陷阱 4: Token 开销
  → JSON Schema 定义本身消耗 tokens
  → 复杂的 Schema 可能占 500-1000 tokens
  → 考虑缓存和复用
```

## 五、场景-技术匹配矩阵

| 业务场景 | 推荐技术 | 原因 |
|----------|----------|------|
| 信息提取 → DB | JSON Schema + Validation | 需要精确的结构化数据 |
| 文本分类 | Few-shot (每类1-2例) | 示例比描述更有效 |
| 复杂数据分析 | CoT + Tool Use | 需要多步推理+计算 |
| 风格统一的回复 | Few-shot + System Prompt | 示例定义风格最直观 |
| 代码生成 | Few-shot CoT | 示例+推理过程双重引导 |
| 客服问答 | RAG + System Prompt | 知识来自知识库而非参数 |
| 报告生成 | CoT + Template | 需要结构化推理+固定格式 |
| 数据清洗 | Few-shot + JSON Mode | 大量相似转换需要一致格式 |

## 六、面试核心问答

### Q1: 怎么让 LLM 稳定输出可被程序解析的结果？

> "我通常三层保障：第一层是用 JSON Schema 或 Function Calling 在 API 层面约束输出格式；第二层是在 prompt 中明确给出期望的 JSON 结构和示例；第三层是用 Pydantic/jsonschema 做运行时校验，不合法就追加错误信息重试，最多重试 2-3 次。实测下来，三层结合后结构化输出的合法率可以从裸调用的 85% 提升到 99.5% 以上。关键是不要把希望寄托在任何单一手段上。"

### Q2: CoT 和 Few-shot 什么时候用哪个？

> "简单判断：需要'想'的用 CoT，需要'模仿'的用 Few-shot。具体来说，多步推理、数学计算、因果分析用 CoT；格式统一、风格一致、分类标注用 Few-shot。两者也可以组合——Few-shot CoT 就是给模型看几个'带推理过程的示例'，既有格式参考又有推理示范，在复杂任务上效果最好。"

### Q3: Prompt 太长会有什么问题？

> "三个问题：一是成本，system prompt 每次请求都要发送，1000 tokens 的 prompt 日均万次请求就是额外的 ¥100/天；二是 Lost in the Middle 效应，中间位置的指令容易被忽略，关键约束要放在开头和结尾；三是挤占了用户内容和检索结果的空间。我的经验是 system prompt 控制在 500-1500 tokens，超过的话考虑精简或拆分为按需加载的模块。"

2. 不确定的字段填 null，不要猜测

### 7.2 动态 Few-shot 选择

```python
def select_few_shot_examples(query, example_pool, top_k=3):
    """根据当前查询动态选择最相关的示例"""
    query_embedding = embed(query)
    similarities = cosine_similarity(query_embedding, example_pool.embeddings)
    top_indices = np.argsort(similarities)[-top_k:]

    # 确保类别多样性
    selected = diverse_selection(top_indices, example_pool, min_categories=2)
    return selected
```
