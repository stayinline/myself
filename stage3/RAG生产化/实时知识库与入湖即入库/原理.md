# 实时知识库与入湖即入库 — 原理篇

## 一、核心问题：知识库如何跟上数据变化

### 1.1 企业数据的动态性

| 数据源 | 更新频率 | 示例 |
|--------|---------|------|
| Wiki/Confluence | 每天数次 | 政策文档、操作手册更新 |
| CRM/工单系统 | 每分钟 | 新客服工单、客户反馈 |
| IoT 传感器 | 每秒 | 设备状态、告警日志 |
| 数据库 CDC | 实时 | 订单变更、库存变动 |
| 代码仓库 | 每次提交 | README、API 文档更新 |

传统 RAG 的**批量重建**模式（每周/每月全量重新 Embedding）已经不够用。用户问的是最新信息，知识库却停留在上周。

### 1.2 "入湖即入库"的定义

**入湖即入库 = 数据进入数据湖的同时，自动完成 Chunking + Embedding + 写入向量库**

```
传统模式:
  数据入湖 → (等待调度) → ETL 抽取 → Chunking → Embedding → 写入向量库
  延迟: 小时~天级

入湖即入库:
  数据入湖 ──同步──→ Chunking → Embedding(Async) → 写入向量库
  延迟: 分钟级
```

### 2.2 各组件职责

| 组件 | 职责 | 技术选型 |

| Flink | 流式处理：清洗、chunking、调用 Embedding | Apache Flink + Async I/O |

| Lineage Store | 记录转换链路 | Paimon 表 / Atlas / DataHub |

## 三、三种变更状态的处理

### 3.1 Add（新增）

```
CDC Event: INSERT
    ↓
Flink: 解析新记录 → Chunking → Embedding
    ↓
Paimon: INSERT chunk records
Milvus: INSERT vectors with metadata
Lineage: INSERT lineage record
```

最简单的场景，直接写入即可。

### 3.2 Modify（修改）

```
CDC Event: UPDATE
    ↓
Flink: 
  1. 根据 doc_id + version 找到旧 chunks
  2. 标记旧 chunks 为 deprecated
  3. 对新内容重新 Chunking → Embedding
  4. 写入新 chunks
    ↓
Paimon: UPDATE old chunks (set deprecated=true), INSERT new chunks
Milvus: DELETE old vectors, INSERT new vectors
Lineage: INSERT update lineage
```

**关键挑战**：向量数据库没有原生的 UPDATE 操作，需要先 DELETE 再 INSERT。必须保证这两步的原子性或至少最终一致性。

### 3.3 Delete（删除）

```
CDC Event: DELETE
    ↓
Flink: 根据 doc_id 找到所有关联 chunks
    ↓
Paimon: UPDATE chunks (set deprecated=true) 或物理删除
Milvus: DELETE vectors by doc_id filter
Lineage: INSERT delete lineage
```

**关键挑战**：Milvus 的 DELETE 是异步的，删除后短时间内可能仍能检索到。需要：
- 在检索时加 `deprecated=false` 过滤条件
- 或接受短暂的最终一致性窗口

### 3.4 一致性保障策略

| 策略 | 做法 | 一致性级别 | 复杂度 |
|------|------|-----------|--------|
| 软删除 + 过滤 | deprecated 标记 + 检索过滤 | 强一致 | 低 |
| 事务补偿 | DELETE 失败则重试 + 告警 | 最终一致 | 中 |
| 版本号机制 | 每次更新递增 version，检索只取最新 | 强一致 | 中 |
| 双写确认 | Paimon + Milvus 都成功后才 ACK | 强一致 | 高 |

**推荐**：大多数场景用**软删除 + 版本过滤**即可，简单且有效。

## 四、Flink Async I/O 调用 Embedding

### 4.1 为什么用 Async I/O

Embedding API 调用是 IO 密集型操作（网络请求），如果用同步调用：
- 每条记录等 100ms → 吞吐量仅 10 条/秒
- 而 Embedding API 通常支持 batch + 并发

Async I/O 允许在等待 IO 响应时继续处理其他记录：

```
Sync:   [req1 ▓▓▓▓] [req2 ▓▓▓▓] [req3 ▓▓▓▓]  → 300ms/3条
Async:  [req1 ▓▓▓▓]
        [req2 ▓▓▓▓]                              → ~120ms/3条
        [req3 ▓▓▓▓]
```

### 4.2 关键参数

| 参数 | 说明 | 建议值 |
|------|------|--------|
| capacity | 最大并发请求数 | 50-200（取决于 API rate limit） |
| timeout | 单次请求超时 | 5-10s |
| ordered | 是否保持输入顺序 | true（保证因果序） |
| retry | 失败重试次数 | 3 |
| backoff | 重试退避策略 | exponential, base=1s |

### 4.3 成本控制

| 策略 | 说明 |
|------|------|
| Batch Embedding | 多条文本合并为一次 API 调用 |
| 缓存 | 相同内容不重复 Embedding（MD5 去重） |
| 限流 | 令牌桶/漏桶控制 API 调用速率 |
| 分级模型 | 重要文档用高质量模型，一般文档用小模型 |
| 本地模型 | 高频场景部署本地 Embedding 服务 |

## 五、增量去重

### 5.1 为什么需要去重

- CDC 可能产生重复事件（at-least-once 语义）
- 同一文档短时间内多次更新
- 重试导致的重复写入

### 5.2 去重策略

| 层级 | 方法 | 说明 |
|------|------|------|
| Flink 层 | KeyedProcessFunction + State | 基于 doc_id+version 去重 |
| 写入层 | Upsert 语义 | Paimon 原生支持 primary key upsert |
| Milvus 层 | chunk_id 作为 primary key | 相同 ID 覆盖写入 |
| 内容层 | Content Hash | MD5(content) 相同则跳过 Embedding |

```python
# 内容哈希去重伪代码
def should_embed(chunk_text: str, existing_hashes: set) -> bool:
    content_hash = md5(chunk_text)
    if content_hash in existing_hashes:
        return False  # 内容未变，跳过 Embedding
    existing_hashes.add(content_hash)
    return True
```

## 七、面试话术

### Q: "你的'入湖即入库'方案是怎么做的？"

> 这是一套端到端的实时知识库更新 pipeline。
>
> **数据接入层**：通过 CDC（Debezium）或 Kafka 消息捕获数据源的变更事件，支持新增、修改、删除三种操作。
>
> **流式处理层**：用 Flink 做实时处理。收到变更事件后，先做清洗和解析，然后按文档结构做 Chunking。接着通过 Async I/O 调用 Embedding API——这里用了异步是因为 Embedding 是 IO 密集操作，同步调用吞吐量太低。我还做了内容哈希去重，相同内容不重复调 Embedding，节省成本。
>
> **存储层**：处理完的结果双写到两个地方——Paimon 存 chunk 明细和血缘信息（用于精确查询和审计），Milvus 存向量和元数据（用于 RAG 语义检索）。
>
> **一致性保障**：对于修改和删除操作，我用软删除加版本号机制。旧 chunk 标记为 deprecated，检索时过滤掉。这样避免了向量数据库 DELETE 操作的异步一致性问题。
>
> **成本控制**：Batch Embedding + 内容缓存 + 令牌桶限流，把 Embedding 调用成本控制在可接受范围内。
>
> 这套方案实现了**分钟级的知识库更新**，用户问到的永远是最新信息。它也是我个人方法论的核心——展示了数据工程师从采集、处理、存储到 AI 集成的全栈能力。

### Q: "向量数据库的增删改一致性你怎么处理的？"

> 向量数据库不像关系型数据库有完善的事务支持，所以我采用了**软删除 + 版本过滤**的策略。
>
> 具体来说：每个 chunk 有 doc_id 和 version 字段。文档更新时，不是直接删除旧向量，而是插入新版本 chunk 并把旧版本标记为 deprecated。检索时在 Milvus 加 expr 过滤 `deprecated == false`，确保只返回最新版本。
>
> 这样做的好处是：第一，避免了 DELETE 操作的异步一致性问题；第二，保留了历史版本可用于审计和回溯；第三，写入操作只有 INSERT，不需要复杂的分布式事务。
>
> 对于真正的物理清理，我设了一个定时任务，每天凌晨删除超过 30 天的 deprecated chunks，控制存储增长。

## 八、关键概念速查

| 术语 | 说明 |
|------|------|
| CDC | Change Data Capture，捕获数据库变更事件 |
| Async I/O | Flink 异步 IO，提高外部服务调用吞吐 |
| Paimon | Apache Paimon，流批一体数据湖格式 |
| Upsert | INSERT or UPDATE ON CONFLICT 语义 |
| Soft Delete | 标记删除而非物理删除 |
| Content Hash | 对内容做哈希用于去重 |
| Rate Limiting | 令牌桶/漏桶控制 API 调用频率 |
| Lineage | 数据血缘，记录数据转换链路 |
