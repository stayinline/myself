# 引用来源与可信供给 — 原理篇

## 一、为什么 RAG 必须有引用溯源

### 1.1 信任是 RAG 落地的前提

在企业场景中，RAG 不是聊天玩具，而是**决策辅助工具**。用户需要：
- **验证**：回答是否来自权威/最新的文档？
- **追溯**：能否点击链接查看原文？
- **审计**：出了问题能否追责到具体数据源？
- **合规**：金融/医疗/法律场景要求答案可溯源。

```
没有引用的 RAG: "您的年假余额为5天" → 用户不敢信
有引用的 RAG:   "您的年假余额为5天 [来源: HR系统-假期管理V3.2, 更新于2024-03-15]" → 可信
```

### 1.2 "可信数据供给"的定义

**可信数据供给 = Chunk 元数据 + 数据血缘 + 引用约束**

这三者缺一不可：
- **Chunk 元数据**：每个向量片段知道自己从哪来
- **数据血缘**：从原始数据源到最终 chunk 的完整转换链路
- **引用约束**：模型被强制标注信息来源，且标注可验证

> 这是 AI Infra / 数据工程在 RAG 领域的**核心卖点**。模型谁都能调，但让数据可信、可追溯，这是数据工程师的专业壁垒。

## 二、数据侧设计：Chunk 元数据 Schema

### 2.1 必备元数据字段

| 字段 | 类型 | 说明 | 示例 |
|------|------|------|------|
| `chunk_id` | string | 全局唯一标识 | `doc_2024_v3_chunk_017` |
| `doc_id` | string | 所属文档标识 | `policy_hr_leave_2024` |
| `source_url` | string | 原始文档 URL/路径 | `https://wiki.internal/hr/leave-policy` |
| `page_number` | int | 页码（PDF/Word） | 12 |
| `section_title` | string | 所在章节标题 | "第三章 年假规则" |
| `doc_version` | string | 文档版本号 | `v3.2` |
| `created_at` | datetime | chunk 创建时间 | `2024-03-15T10:30:00Z` |
| `updated_at` | datetime | chunk 最后更新时间 | `2024-06-01T14:20:00Z` |
| `source_system` | string | 数据来源系统 | `Confluence / SAP HR / MySQL` |
| `author` | string | 内容作者/负责人 | `张三 (HRBP)` |
| `tags` | list[string] | 业务标签 | `["hr", "leave", "policy"]` |
| `access_level` | string | 访问权限级别 | `internal / confidential / public` |

### 2.2 元数据设计原则

1. **入库时写入，而非查询时补**：元数据必须在 chunking 阶段就绑定，事后补录成本高且容易遗漏。
2. **粒度对齐**：元数据粒度要与 chunk 粒度一致。一个 chunk 对应一条元数据记录。
3. **版本化**：同一文档更新后，旧 chunk 不直接覆盖，而是标记 deprecated + 写入新 chunk，保留历史。
4. **可序列化**：元数据要能存入向量数据库的 metadata 字段（通常是 JSON），也要能存入结构化存储。

### 2.3 存储架构

```
┌──────────────┐     ┌──────────────────┐     ┌─────────────────┐
│  原始文档     │────→│  Chunking Pipeline│────→│  向量数据库      │
│ (Wiki/PDF/DB)│     │  (Flink/Batch)   │     │  (Milvus/Qdrant)│
└──────────────┘     └──────────────────┘     │ vector + meta   │
                           │                   └─────────────────┘
                           ↓
                     ┌──────────────────┐
                     │  元数据存储       │
                     │ (Paimon/Hive/MySQL)│
                     │ chunk_id → meta   │
                     └──────────────────┘
```

## 三、生成侧设计：引用标注机制

### 3.1 Prompt 中强制引用

```
System Prompt 模板:

你是一个企业知识库助手。请严格遵循以下规则：
1. 仅基于提供的【参考资料】回答问题，不得使用自身知识补充。
2. 每个事实性陈述必须标注引用编号，格式为 [n]，n 对应参考资料的序号。
3. 如果多个资料支持同一观点，标注所有相关编号，如 [1][3]。
4. 如果资料中没有相关信息，明确回答"根据提供的资料无法回答此问题"。
5. 不得编造引用编号。只使用实际存在的编号。

参考资料:
[1] {chunk_1_content} (来源: {chunk_1_source})
[2] {chunk_2_content} (来源: {chunk_2_source})
...
```

### 3.2 引用格式的标准化

| 格式 | 适用场景 | 示例 |
|------|---------|------|
| `[n]` | 通用学术风格 | "年假为5天 [1]" |
| `(来源: xxx)` | 企业友好风格 | "年假为5天 (来源: HR手册v3)" |
| `<ref id="n"/>` | 结构化输出 | 适合前端渲染超链接 |
| Markdown link | Web 展示 | `[HR手册](url)` |

### 3.3 引用验证层

模型输出的引用**不一定可靠**，需要后处理验证：

```python
def validate_citations(answer: str, retrieved_chunks: list) -> dict:
    """验证模型输出中的引用编号是否合法"""
    import re
    cited_ids = set(re.findall(r'\[(\d+)\]', answer))
    valid_ids = set(str(i+1) for i in range(len(retrieved_chunks)))

    return {
        "valid_citations": cited_ids & valid_ids,
        "fabricated_citations": cited_ids - valid_ids,  # ⚠️ 编造的引用
        "unused_chunks": valid_ids - cited_ids,          # 未被引用的chunk
        "is_trustworthy": len(cited_ids - valid_ids) == 0,
    }
```

## 四、端到端数据血缘

### 4.1 血缘链路设计

```
原始数据源                ETL/Chunking              向量索引             RAG 回答
┌──────────┐           ┌──────────────┐          ┌──────────┐        ┌──────────┐
│ Confluence│──page_id──→│ Flink Job    │─chunk_id─→│ Milvus   │─ref──→│ LLM 输出  │
│ Page #123│           │ job_run_abc  │          │ idx_xyz  │        │ [1][3]   │
└──────────┘           └──────────────┘          └──────────┘        └──────────┘
                              │
                              ↓
                        ┌──────────────┐
                        │ Lineage Store │
                        │ (Paimon/Atlas)│
                        │ page→chunk→idx│
                        └──────────────┘
```

### 4.2 血缘记录的核心字段

| 字段 | 说明 |
|------|------|
| `lineage_id` | 血缘记录唯一ID |
| `source_type` | 数据源类型 (confluence/mysql/pdf/iot) |
| `source_identifier` | 源标识 (page_id/table_name/file_path) |
| `transform_job` | 转换任务标识 (job_name + run_id) |
| `transform_logic` | 转换逻辑描述 (chunking策略/embedding模型) |
| `output_chunk_ids` | 产出的 chunk ID 列表 |
| `execution_timestamp` | 执行时间 |
| `data_quality_score` | 数据质量评分 |

### 4.3 血缘的价值

1. **影响分析**：某文档更新了 → 通过血缘找到所有受影响的 chunks → 触发重新 Embedding
2. **根因定位**：RAG 回答了错误信息 → 追溯到 chunk → 追溯到原始文档 → 发现文档本身过期了
3. **合规审计**：监管问"这条回答依据是什么" → 从引用编号 → chunk → 血缘 → 原始文档全链路证明
4. **数据新鲜度监控**：通过血缘中的 timestamp 判断知识库内容是否过期

## 五、"可信数据供给"作为 AI Infra 核心卖点

### 5.1 市场痛点

| 痛点 | 现状 | 可信供给的解法 |
|------|------|--------------|
| 答案不可信 | 用户不敢用 RAG 做决策 | 引用+溯源让用户可验证 |
| 幻觉难追责 | 出错不知道谁的锅 | 血缘定位到具体文档/版本 |
| 合规过不去 | 金融/医疗要求数据可审计 | 端到端 lineage 满足审计 |
| 数据更新混乱 | 改了文档不知道 RAG 有没有同步 | 血缘驱动增量更新 |
| 权限泄露 | 敏感信息被不该看到的人查到 | metadata 中的 access_level 过滤 |

### 5.2 数据工程师的独特价值

- **模型工程师**关注 prompt engineering、fine-tuning、评测
- **数据工程师**关注数据的完整性、新鲜度、可追溯性、权限控制
- RAG 的"可信"二字，本质上是**数据治理问题**，这正是数据工程师的主场

## 六、面试话术

### Q: "怎么让 RAG 的答案可信、可追溯到原文？"

> 我从三个层面来做：
>
> **第一层，数据侧**。每个 chunk 入库时必须携带完整的元数据——来源文档、URL、页码、版本号、时间戳、作者。这些元数据和向量一起存入向量数据库。没有元数据的 chunk 不允许入库，这是硬性规范。
>
> **第二层，生成侧**。在 prompt 中强制要求模型对每个事实性陈述标注引用编号 [n]，并且明确告诉它"不得编造引用"。同时在参考资料中标注每条的来源信息，让模型有据可引。
>
> **第三层，验证侧**。模型输出的引用不一定靠谱，我会加一层后处理验证——解析回答中的所有引用编号，检查是否在合法的 chunk 范围内。如果发现编造的引用，要么自动删除该引用，要么标记为"未验证"提醒用户。
>
> 此外，我还建了一套数据血缘系统，从原始文档到 chunk 到向量索引到最终回答，全链路可追溯。这样不仅用户可以点引用看原文，出了问题也能精确定位到是哪个文档、哪个版本的哪个段落导致的。
>
> 这套"chunk 元数据 + 血缘 + 引用约束"的组合，我称之为**可信数据供给**，它是 RAG 在企业场景落地的基础设施，也是数据工程师在这个领域最大的差异化价值。

### Q: "模型编造引用怎么办？"

> 这是很常见的问题。我的应对方案分三层：
>
> 第一层是**Prompt 约束**，明确告诉模型只能用给定的编号，不能自己编。这能减少大部分编造行为。
>
> 第二层是**后处理验证**，用正则提取回答中的所有引用编号，和实际传入的 chunk 编号做交集校验。不在范围内的就是编造的，可以自动移除或标红。
>
> 第三层是**结构化输出**，对于高要求场景，让模型以 JSON 格式输出，每个 statement 带 citation 字段，方便程序化验证。
>
> 实测下来，Prompt 约束 + 后处理验证可以把编造引用率从约 15% 降到 1% 以下。

## 七、对比总结

| 维度 | 无引用的 RAG | 有可信供给的 RAG |
|------|-------------|-----------------|
| 用户体验 | 不敢信、不敢用 | 可验证、可追溯 |
| 出错排查 | 大海捞针 | 血缘精准定位 |
| 合规审计 | 无法通过 | 全链路证据链 |
| 数据更新 | 手动/遗漏 | 血缘驱动自动更新 |
| 权限控制 | 粗粒度/泄露风险 | metadata 级精细过滤 |
| 工程复杂度 | 低 | 中高（但一次建设长期受益） |
