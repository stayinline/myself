# 结构化数据入 RAG — 原理篇

## 一、核心问题：表格/数据库数据怎么进 RAG

企业知识不仅有文档，还有大量**结构化数据**：
- 数据库表（订单、用户、交易、库存）
- Excel/CSV 报表
- BI 看板指标
- API 返回的 JSON 数据

这类数据不适合直接做向量检索——它们是**精确的、关系型的、需要聚合计算的**。

### 1.1 两条技术路线

```
┌─────────────────────────────────────────────────────────┐
│              结构化数据入 RAG 的两条路                     │
├──────────────────────┬──────────────────────────────────┤
│ Route 1: Text2SQL    │ Route 2: 向量化                   │
│ NL → SQL → DB → LLM │ 行/记录 → 文本 → Embedding        │
├──────────────────────┼──────────────────────────────────┤
│ ✅ 精确数值查询       │ ✅ 模糊语义搜索                    │
│ ✅ 聚合/统计/JOIN     │ ✅ "类似"/"相关"类查询             │
│ ✅ 实时数据           │ ✅ 跨表语义关联                    │
│ ❌ 不能做模糊语义     │ ❌ 不支持聚合计算                  │
│ ❌ SQL 生成可能出错   │ ❌ 大表全量向量化成本高             │
│ ❌ 安全风险(SQL注入)  │ ❌ 数据更新需重新 Embedding         │
└──────────────────────┴──────────────────────────────────┘
```

## 二、Route 1: Text2SQL

### 2.1 工作原理

```
用户自然语言                LLM                     数据库
┌──────────────┐      ┌──────────────┐        ┌──────────────┐
│ "上季度华东区 │─────→│ Schema + NL  │──SQL──→│ SELECT ...   │
│  销售额多少？"│      │ → SQL 生成   │        │ FROM sales   │
└──────────────┘      └──────────────┘        └──────┬───────┘
                                                      │
                         ┌──────────────┐             │
                         │ LLM 总结结果  │←────────────┘
                         │ → 自然语言回答 │
                         └──────────────┘
```

### 2.2 适用场景

| 查询类型 | 示例 | Text2SQL 适合度 |
|---------|------|----------------|
| 精确数值 | "上月GMV是多少" | ⭐⭐⭐⭐⭐ |
| 聚合统计 | "各区域Q3销售额排名" | ⭐⭐⭐⭐⭐ |
| 多表关联 | "退货率最高的供应商是哪个" | ⭐⭐⭐⭐ |
| 条件筛选 | "列出所有逾期超过30天的订单" | ⭐⭐⭐⭐ |
| 趋势分析 | "近6个月DAU变化趋势" | ⭐⭐⭐⭐ |
| 模糊语义 | "哪些产品口碑不好" | ⭐⭐ |
| 开放问答 | "我们的竞争优势是什么" | ⭐ |

### 2.3 Text2SQL 的核心挑战

#### （1）SQL 生成错误

常见错误类型：
- **Schema 理解错误**：字段名/表名搞错、JOIN 条件不对
- **业务逻辑错误**：GROUP BY 维度选错、WHERE 条件遗漏
- **方言差异**：MySQL vs ClickHouse vs PostgreSQL 语法不同
- **复杂查询失败**：嵌套子查询、窗口函数、CTE 容易出错

#### （2）安全风险

- **SQL 注入**：LLM 生成的 SQL 可能被恶意 query 诱导
- **越权查询**：用户通过自然语言查到不该看的数据
- **写操作风险**：LLM 可能生成 INSERT/UPDATE/DELETE
- **资源耗尽**：无 LIMIT 的全表扫描拖垮数据库

#### （3）结果过大

- 查询返回 10 万行结果，无法全部塞入 LLM context
- 需要在 SQL 层或后处理层做聚合/截断

### 2.4 安全设计原则

```
┌────────────────────────────────────────────────────┐
│            Text2SQL 安全四道防线                      │
├────────────────────────────────────────────────────┤
│ 1. 只读账号：DB 连接用 SELECT-only 权限              │
│ 2. SQL 白名单：禁止 DROP/INSERT/UPDATE/DELETE/ALTER  │
│ 3. 查询改写：自动加 LIMIT / 超时限制 / 资源配额       │
│ 4. 审计日志：所有生成的 SQL + 执行结果留痕            │
└────────────────────────────────────────────────────┘
```

## 三、Route 2: 结构化数据向量化

### 3.1 工作原理

将每一行/每条记录转换为自然语言文本，再做 Embedding 入库。

```python
# 原始数据行
row = {
    "product_name": "无线蓝牙耳机 Pro",
    "category": "消费电子",
    "price": 599,
    "rating": 4.7,
    "review_summary": "音质好，降噪强，续航略短",
    "complaint_count": 12,
}

# 转换为自然语言文本
text = (
    f"产品名称：{row['product_name']}，分类：{row['category']}，"
    f"价格：{row['price']}元，评分：{row['rating']}分，"
    f"用户评价摘要：{row['review_summary']}，投诉数：{row['complaint_count']}次。"
)

# 对 text 做 Embedding → 存入向量库
```

### 3.2 适用场景

| 查询类型 | 示例 | 向量化适合度 |
|---------|------|-------------|
| 相似查找 | "和这个产品类似的有哪些" | ⭐⭐⭐⭐⭐ |
| 语义搜索 | "有没有性价比高的降噪耳机" | ⭐⭐⭐⭐⭐ |
| 模糊匹配 | "之前有个客户反馈过电池问题" | ⭐⭐⭐⭐ |
| 跨表关联 | "差评多的产品的供应商是谁" | ⭐⭐⭐ |
| 精确数值 | "上个月销售额" | ⭐ |
| 聚合统计 | "各品类平均评分" | ⭐ |

### 3.3 注意事项

- **模板设计很重要**：转换后的文本质量直接影响检索效果
- **选择性字段**：不是所有字段都需要入文本，ID/时间戳等通常无用
- **大表采样**：百万级表不可能全量向量化，需要按业务需求筛选
- **增量更新**：数据变更时需要重新 Embedding 对应记录

## 四、两路结合的策略

### 4.1 路由决策

```
用户 Query
    │
    ├── 意图识别/分类
    │     ├── 包含数值/统计/排名/趋势 → Text2SQL
    │     ├── 包含"类似"/"相关"/"推荐" → 向量检索
    │     ├── 两者都有 → 并行执行 + 结果合并
    │     └── 不确定 → 先尝试 Text2SQL，失败 fallback 向量
    │
    └── 也可以让 LLM 自己决定
          Prompt: "判断以下问题应该查数据库还是搜知识库..."
```

### 4.2 组合示例

```
Query: "Q3 华东区卖得最好的产品和它的用户反馈"

Step 1 (Text2SQL): 
  SELECT product_name, SUM(amount) as total_sales 
  FROM sales WHERE region='华东' AND quarter='Q3' 
  GROUP BY product_name ORDER BY total_sales DESC LIMIT 5
  
Step 2 (向量检索):
  用 Step 1 返回的产品名作为 query，在评论向量库中检索相关反馈

Step 3 (LLM 整合):
  将 SQL 结果 + 向量检索结果一起送入 LLM，生成综合回答
```

## 五、数据工程师在 Text2SQL 中的天然优势

### 5.1 为什么数据工程师做 Text2SQL 更有优势

| 能力 | 模型工程师 | 数据工程师 |
|------|-----------|-----------|
| 理解 Schema | 一般 | ⭐⭐⭐ 日常建模 |
| SQL 能力 | 基础 | ⭐⭐⭐ 日常查询 |
| 数据质量感知 | 弱 | ⭐⭐⭐ 知道哪些字段可靠 |
| 性能优化 | 不了解 | ⭐⭐⭐ 索引/分区/物化视图 |
| 权限管控 | 不熟悉 | ⭐⭐⭐ 数据安全是本职 |
| 业务知识 | 从 prompt 获取 | ⭐⭐⭐ 长期积累 |

### 5.2 具体贡献点

1. **Schema 描述优化**：写好 table/column comment，这是 Text2SQL 准确率的基础
2. **Few-shot 样例构建**：从历史查询日志中挑选高质量 NL-SQL pair
3. **查询验证**：对 LLM 生成的 SQL 做语法检查、执行计划分析
4. **性能保障**：确保生成的 SQL 不会触发全表扫描
5. **安全加固**：只读账号、白名单、审计日志
6. **结果预处理**：大结果集聚合后再喂给 LLM

## 六、面试话术

### Q: "结构化数据怎么接入 RAG？"

> 我有两条路线，根据查询意图选择：
>
> **Text2SQL** 用于精确查询和统计分析。比如"上季度华东销售额"这种，LLM 把自然语言转成 SQL，去 ClickHouse/MySQL 查完再让 LLM 总结结果。这条路我做了四层安全防护：只读账号、SQL 关键词白名单、自动加 LIMIT 和超时、全量审计日志。数据工程师在这条路上有天然优势——我们懂 schema、懂建模、懂性能、懂权限，这些都是 Text2SQL 准确和安全的关键。
>
> **向量化** 用于模糊语义搜索。比如"和这个产品类似的反馈"，我把每行记录转成自然语言文本做 embedding，然后用向量检索找相似的。这条路适合非精确匹配的场景。
>
> 实际项目中我做了**意图路由**：先让 LLM 判断查询类型，精确走 Text2SQL，模糊走向量，混合的就并行执行再合并结果。这样既保证了数值查询的准确性，又保留了语义搜索的灵活性。

### Q: "Text2SQL 的安全你怎么做的？"

> 四道防线：第一，数据库连接使用专门的只读账号，物理上杜绝写操作。第二，SQL 解析层做关键词白名单，DROP/INSERT/UPDATE/DELETE/ALTER/TRUNCATE 一律拦截。第三，查询改写层自动加 LIMIT 1000 和 30 秒超时，防止全表扫描拖垮库。第四，所有生成的 SQL 和执行结果都写入审计日志，可追溯可复盘。
>
> 另外我在 schema 层面也做了防护：敏感字段（如手机号、身份证）在 schema description 中标记为脱敏，LLM 生成的 SQL 如果涉及这些字段会自动触发告警。

## 七、对比总结

| 维度 | Text2SQL | 向量化 | 组合方案 |
|------|----------|--------|---------|
| 精确查询 | ✅ | ❌ | ✅ |
| 模糊语义 | ❌ | ✅ | ✅ |
| 聚合统计 | ✅ | ❌ | ✅ |
| 实现复杂度 | 中高 | 中 | 高 |
| 安全风险 | 高(需防护) | 低 | 中 |
| 数据时效性 | 实时 | 需同步 | 混合 |
| 适用数据规模 | 不限 | 万~十万级 | 不限 |
| 数据工程师优势 | ⭐⭐⭐ | ⭐⭐ | ⭐⭐⭐ |
