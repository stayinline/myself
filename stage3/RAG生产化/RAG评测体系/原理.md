# RAG 评测体系 — 原理篇

## 一、为什么评测是 RAG 工程化的基石

### 1.1 没有评测 = 玄学调优

```
❌ "感觉换了 chunk_size=512 好像好一点"
✅ "chunk_size 从 256→512，Recall@5 从 0.62→0.78，Faithfulness 从 0.71→0.83"
```

RAG 系统有多个可调组件（chunking / embedding / retrieval / rerank / prompt / model），每次改动都需要**量化验证**。没有评测集的调优就是盲调。

### 1.2 评测驱动迭代闭环

```
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│ 建立评测集 │──→│ 跑基线指标 │──→│ 调整组件  │──→│ 对比指标  │
└──────────┘    └──────────┘    └──────────┘    └─────┬────┘
       ↑                                               │
       └─────────────── 持续迭代 ←──────────────────────┘
```

## 二、检索侧评测指标

### 2.1 核心指标详解

#### Hit Rate（命中率）

$$
\text{HitRate@K} = \frac{1}{N}\sum_{i=1}^{N} \mathbb{1}(\text{relevant}_i \in \text{topK}_i)
$$

- 含义：Top-K 结果中**至少包含一个**相关文档的查询比例
- 优点：简单直观
- 缺点：不考虑排名位置，top-1 和 top-K 命中同等对待

#### Recall@K（召回率）

$$
\text{Recall@K} = \frac{1}{N}\sum_{i=1}^{N} \frac{|\text{relevant}_i \cap \text{retrieved\_topK}_i|}{|\text{relevant}_i|}
$$

- 含义：Top-K 中召回的相关文档占全部相关文档的比例
- 适合：多个相关文档的场景
- 注意：需要标注所有相关文档，标注成本高

#### MRR（Mean Reciprocal Rank）

$$
\text{MRR} = \frac{1}{N}\sum_{i=1}^{N} \frac{1}{\text{rank of first relevant doc}_i}
$$

- 含义：第一个相关文档排名的倒数的均值
- 适合：只有一个正确答案的场景（如事实问答）
- 值域：[0, 1]，越接近 1 越好

#### NDCG@K（Normalized Discounted Cumulative Gain）

$$
\text{DCG@K} = \sum_{i=1}^{K} \frac{\text{rel}_i}{\log_2(i+1)}, \quad
\text{NDCG@K} = \frac{\text{DCG@K}}{\text{IDCG@K}}
$$

- 含义：考虑排名位置的检索质量，排名越高权重越大
- 适合：有分级相关性标注的场景（0/1/2/3 级）
- 最全面的检索指标，但标注成本也最高

### 2.2 指标选择指南

| 场景 | 推荐指标 | 原因 |
|------|---------|------|
| 快速评估 | Hit Rate@5 | 标注简单，计算快 |
| 事实问答 | MRR | 关注第一个正确答案的位置 |
| 多文档问答 | Recall@K | 衡量覆盖度 |
| 精细评估 | NDCG@K | 考虑排名和相关性级别 |
| 线上监控 | Hit Rate@3 | 轻量、实时 |

## 三、生成侧评测指标

### 3.1 三大核心维度

```
┌─────────────────────────────────────────────────────────┐
│                 生成质量三维模型                          │
├──────────────┬──────────────────────────────────────────┤
│ Faithfulness │ 回答是否忠实于给定上下文？有无幻觉？        │
│ (忠实度)     │ 例: 上下文说5天年假，回答说10天 → 不忠实   │
├──────────────┼──────────────────────────────────────────┤
│ Answer       │ 回答是否与用户问题相关？有没有答非所问？     │
│ Relevance    │ 例: 问退货政策，回答了配送时间 → 不相关    │
│ (答案相关性) │                                            │
├──────────────┼──────────────────────────────────────────┤
│ Context      │ 送入模型的上下文中有多少是真正有用的？       │
│ Precision    │ 例: top-5 中只有 1 条相关 → precision=0.2  │
│ (上下文精度) │                                            │
└──────────────┴──────────────────────────────────────────┘
```

### 3.2 LLM as Judge

用大模型来评估另一个大模型的输出质量：

```python
FAITHFULNESS_PROMPT = """
请评估以下回答是否忠实于给定的上下文资料。

【上下文】
{context}

【问题】
{question}

【回答】
{answer}

请从以下维度评分(1-5):
- 忠实度: 回答中的每个事实陈述是否都能在上下文中找到依据？
- 无幻觉: 回答是否包含了上下文中不存在的信息？

输出 JSON: {{"faithfulness": <score>, "hallucination_count": <count>, "reason": "<说明>"}}
"""
```

### 3.3 LLM as Judge 的偏差与应对

| 偏差类型 | 表现 | 应对 |
|---------|------|------|
| 自我偏好 | 倾向给自己生成的内容高分 | 用不同模型做 judge |
| 长度偏好 | 长回答得分偏高 | Prompt 明确说"不以长度论质量" |
| 格式偏好 | 结构化输出得分偏高 | 分离内容评估和格式评估 |
| 位置偏好 | 先出现的答案得分高 | 多次评估取均值 |
| 一致性差 | 同一输入多次评估分数波动 | 设 temperature=0 + 多次采样 |

## 四、评测工具生态

### 4.1 RAGAS 框架

[RAGAS](https://github.com/explodinggradients/ragas) 是目前最流行的 RAG 评测框架：

```python
from ragas import evaluate
from ragas.metrics import (
    faithfulness,
    answer_relevancy,
    context_precision,
    context_recall,
)
from datasets import Dataset

eval_dataset = Dataset.from_dict({
    "question": ["退货政策是什么？", "HTTP 502怎么处理？"],
    "answer": ["根据资料，7天内可退...", "502错误通常是..."],
    "contexts": [["自签收之日起7天内...", "..."], ["HTTP 502 Bad Gateway..."]],
    "ground_truth": ["7天无理由退货，运费买家承担", "检查上游服务和nginx配置"],
})

results = evaluate(
    eval_dataset,
    metrics=[faithfulness, answer_relevancy, context_precision, context_recall],
)
print(results)
# {'faithfulness': 0.85, 'answer_relevancy': 0.91, 
#  'context_precision': 0.78, 'context_recall': 0.82}
```

## 五、评测驱动的迭代方法论

### 5.1 建立 Baseline

```
Day 1: 搭建最小 RAG → 跑评测 → 记录 baseline
       Recall@5=0.62, Faithfulness=0.71, AnswerRelevance=0.80
```

### 5.2 单变量实验

每次只改一个变量，对比指标变化：

| 实验 | 变更 | Recall@5 | Faithfulness | 结论 |
|------|------|----------|-------------|------|
| E1 | chunk_size 256→512 | 0.62→0.74 | 0.71→0.78 | ✅ 采纳 |
| E2 | overlap 0→64 | 0.74→0.78 | 0.78→0.80 | ✅ 采纳 |
| E3 | 加 BM25 混合 | 0.78→0.85 | 0.80→0.81 | ✅ 采纳 |
| E4 | 加 Rerank | 0.85→0.89 | 0.81→0.85 | ✅ 采纳 |
| E5 | GPT-4o-mini→4o | 0.89→0.89 | 0.85→0.90 | ⚠️ 仅生成提升 |
| E6 | temperature 0→0.7 | 0.89→0.89 | 0.90→0.72 | ❌ 回滚 |

### 5.3 常见误区

| 误区 | 后果 | 正确做法 |
|------|------|---------|
| 没有评测集就调参 | 凭感觉、无法复现 | 先建评测集再调优 |
| 只看生成不看检索 | 漏掉 80% 的问题 | 检索+生成分开评 |
| LLM judge 不做校验 | 评分不准 | 人工抽检 10-20% |
| 评测集太小 | 统计不显著 | 至少 30 条，建议 50+ |
| 评测集和训练集重叠 | 过拟合假象 | 严格划分 train/eval |
| 一次改多个变量 | 归因不清 | 单变量控制实验 |

## 六、面试话术

### Q: "你怎么衡量一个 RAG 系统好不好？"

> 我把 RAG 评测分为检索侧和生成侧两个维度：
>
> **检索侧**主要看 Recall@K 和 MRR。我有一个 50 条标注数据的评测集，每条都标注了相关的 chunk ID。每次改完 chunking、embedding 或检索策略，都跑一遍 Recall@5，确认有提升才上线。
>
> **生成侧**看三个指标：Faithfulness（忠实度）、Answer Relevancy（答案相关性）、Context Precision（上下文精度）。我用 RAGAS 框架 + LLM-as-Judge 来自动化评估，同时定期人工抽检 10-20% 的结果来校准 judge 的准确性。
>
> 最重要的是**评测驱动迭代**：我维护了一张实验记录表，每次只改一个变量，记录指标变化。比如 chunk_size 从 256 改到 512，Recall@5 从 0.62 提升到 0.74；加混合检索后提升到 0.85；加 Rerank 后到 0.89。每一步都有数据支撑，不是凭感觉。
>
> 我还发现一个关键原则：**检索和生成要分开评**。很多人只看最终回答质量，但 80% 的问题出在检索侧。分开评才能精准定位瓶颈在哪。

### Q: "LLM as Judge 靠谱吗？"

> 有局限性但在实践中可用。主要问题是自我偏好、长度偏好和一致性差。我的应对方法是：第一，用不同于生成模型的模型做 judge，避免自我偏好；第二，temperature 设为 0 并多次采样取均值提高一致性；第三，定期人工标注一批 golden set 来校准 judge 的打分——如果人工分和 judge 分的相关性低于 0.7，就调整 judge prompt。
>
> 实际上在我们的项目中，经过校准后的 LLM judge 和人工评分的相关性能达到 0.82，对于日常迭代够用了。关键是不要把它当绝对真理，而是当作一个高效的筛选工具——异常低分的 case 优先人工复核。

## 七、指标速查表

| 指标 | 维度 | 值域 | 越高越好 | 计算依赖 |
|------|------|------|---------|---------|
| Hit Rate@K | 检索 | [0,1] | ✅ | 相关性标注 |
| Recall@K | 检索 | [0,1] | ✅ | 完整相关性标注 |
| MRR | 检索 | [0,1] | ✅ | 首个相关排名 |
| NDCG@K | 检索 | [0,1] | ✅ | 分级相关性 |
| Faithfulness | 生成 | [0,1] | ✅ | LLM judge |
| Answer Relevancy | 生成 | [0,1] | ✅ | LLM judge |
| Context Precision | 生成 | [0,1] | ✅ | LLM judge |
| Context Recall | 生成 | [0,1] | ✅ | ground truth |
