# RAG 幻觉与答非所问排查 — 实战篇

## 一、构建最小 RAG 系统用于排查实验

### 1.1 最小可复现代码

```python
"""minimal_rag.py - 用于排查实验的最小 RAG 系统"""
import numpy as np
from openai import OpenAI
from typing import List, Dict, Tuple

client = OpenAI()  # 或兼容 API

class MinimalRAG:
    def __init__(self, chunks: List[str], embedding_model="text-embedding-3-small"):
        self.chunks = chunks
        self.embedding_model = embedding_model
        # 预计算所有 chunk 的 embedding
        resp = client.embeddings.create(input=chunks, model=embedding_model)
        self.chunk_embeddings = np.array([d.embedding for d in resp.data])

    def retrieve(self, query: str, top_k: int = 5) -> List[Dict]:
        """检索 top-k 相关 chunk"""
        q_resp = client.embeddings.create(input=[query], model=self.embedding_model)
        q_emb = np.array(q_resp.data[0].embedding)
        # 余弦相似度
        scores = self.chunk_embeddings @ q_emb / (
            np.linalg.norm(self.chunk_embeddings, axis=1) * np.linalg.norm(q_emb)
        )
        top_indices = np.argsort(scores)[::-1][:top_k]
        return [
            {"index": i, "score": float(scores[i]), "content": self.chunks[i]}
            for i in top_indices
        ]

    def generate(self, query: str, contexts: List[Dict], temperature: float = 0.0) -> str:
        """基于召回内容生成回答"""
        context_text = "\n\n---\n\n".join(
            f"[{i+1}] {c['content']}" for i, c in enumerate(contexts)
        )
        prompt = f"""请仅基于以下参考资料回答问题。如果资料中没有相关信息，请明确说"根据提供的资料无法回答"。

参考资料:
{context_text}

问题: {query}

回答:"""
        resp = client.chat.completions.create(
            model="gpt-4o-mini",
            messages=[{"role": "user", "content": prompt}],
            temperature=temperature,
        )
        return resp.choices[0].message.content

    def ask(self, query: str, top_k: int = 5, temperature: float = 0.0):
        """完整问答流程，同时返回中间结果便于排查"""
        retrieved = self.retrieve(query, top_k)
        answer = self.generate(query, retrieved, temperature)
        return {
            "query": query,
            "retrieved": retrieved,
            "answer": answer,
            "debug": {
                "top_k": top_k,
                "temperature": temperature,
                "num_chunks_total": len(self.chunks),
            }
        }
```

## 二、故意制造两类错误

### 2.1 制造检索侧错误

#### 方法一：使用不匹配的 Embedding 模型

```python
# 用中文文档 + 英文专用 Embedding → 语义对齐差
rag_bad_embedding = MinimalRAG(knowledge_base, embedding_model="text-embedding-ada-002")
result = rag_bad_embedding.ask("怎么退货？运费谁出？")
# 观察：top-k 可能不包含 Chunk 0（退货政策）
```

#### 方法二：极端的 Chunking（过细切分）

```python
# 按句号切分，每个 chunk 只有一句话，上下文丢失
bad_chunks = []
for chunk in knowledge_base:
    sentences = chunk.split("。")
    bad_chunks.extend([s + "。" for s in sentences if s.strip()])

rag_bad_chunk = MinimalRAG(bad_chunks)
result = rag_bad_chunk.ask("退货运费谁承担？退款多久到账？")
# 观察：运费和退款信息被分到不同 chunk，单次检索可能只命中其一
```

#### 方法三：模糊 Query

```python
result = rag.ask("这个怎么办？")  # 缺少实体，检索随机
```

## 三、建立分层排查习惯

### 3.1 排查脚本模板

```python
def diagnose_rag(rag: MinimalRAG, query: str, expected_answer_keywords: list):
    """
    RAG 分层诊断工具
    expected_answer_keywords: 期望答案中应包含的关键词列表
    """
    result = rag.ask(query)

    print("=" * 60)
    print(f"🔍 诊断: {query}")
    print("=" * 60)

    # ===== Step 1: 检查召回 =====
    print("\n📋 Step 1: 召回内容检查")
    recall_hit = False
    for i, chunk in enumerate(result["retrieved"]):
        has_keyword = any(kw in chunk["content"] for kw in expected_answer_keywords)
        status = "✅ HIT" if has_keyword else "❌ MISS"
        print(f"  [{i+1}] score={chunk['score']:.4f} {status}")
        print(f"      {chunk['content'][:80]}...")
        if has_keyword:
            recall_hit = True

    if not recall_hit:
        print("\n⚠️  结论: 检索侧问题 — 正确答案未被召回")
        print("   建议: 检查 chunking / embedding / query 改写")
        return "RECALL_FAILURE"

    # ===== Step 2: 检查生成 =====
    print("\n📋 Step 2: 生成结果检查")
    answer = result["answer"]
    answer_has_keyword = any(kw in answer for kw in expected_answer_keywords)

    print(f"  回答: {answer[:200]}...")
    print(f"  包含预期关键词: {'✅ 是' if answer_has_keyword else '❌ 否'}")
    print(f"  Temperature: {result['debug']['temperature']}")
    print(f"  Top-K: {result['debug']['top_k']}")

    if not answer_has_keyword:
        print("\n⚠️  结论: 生成侧问题 — 召回正确但模型未忠实使用")
        print("   建议: 检查 prompt 约束 / temperature / 上下文长度")
        return "GENERATION_FAILURE"

    print("\n✅ 结论: 检索和生成均正常")
    return "OK"


# 使用示例
diagnose_rag(rag, "退货政策是什么？运费谁出？", ["7天", "运费", "买家承担"])
diagnose_rag(rag, "怎么退货？", ["7天", "退货"])
diagnose_rag(rag, "这个怎么处理？", ["退货"])  # 预期检索失败
```

## 五、盲目换大模型的陷阱

### 5.1 成本对比

| 方案 | 准确率提升 | 单次调用成本 | 延迟增加 | 工程复杂度 |
|------|-----------|-------------|---------|-----------|
| GPT-3.5 → GPT-4 | +5~10% | ×10 | ×3 | 低 |
| 优化 Chunking | +10~20% | ×1 | 无 | 中 |
| 加混合检索+Rerank | +15~25% | ×1.2 | +200ms | 中 |
| Query Rewriting | +5~15% | ×1.1 | +500ms | 中 |
| 以上组合 | +25~40% | ×1.3 | +700ms | 中高 |

## 六、面试话术

### Q: "用户说 RAG 答得不对，你怎么定位？"

> 我的排查思路是分层的：**先查召回，再查生成**。
>
> 首先我会把这条 query 对应的 top-k 召回结果拉出来看。如果里面根本没有包含正确答案的 chunk，那就是检索问题。我会进一步看是 chunking 切得不合理导致信息碎片化了，还是 Embedding 模型在当前领域语义对齐不好，还是用户的 query 太模糊需要改写。
>
> 如果召回内容是对的、答案就在某个 chunk 里，那问题在生成侧。我会检查 prompt 有没有明确要求"仅基于给定资料回答"，temperature 是不是太高了，以及是不是塞了太多无关 chunk 导致模型注意力被分散——也就是所谓的 lost in the middle 问题。
>
> 实际工作中我总结了一个经验：**大约 80% 的 RAG 问题是数据和检索的问题**，不是模型的问题。所以不要一上来就想着换更大的模型，先把召回做好，ROI 远高于换模型。
>
> 我还会建一套排查 checklist 和自动化诊断脚本，让团队成员都能按统一流程定位问题，而不是靠感觉猜。

### Q: "你遇到过 Lost in the Middle 吗？怎么解决的？"

> 遇到过。我们把 top-10 的 chunks 全塞给模型，发现排在第 4-7 位的相关信息经常被忽略。做了个对照实验，把同一个正确 chunk 分别放在首位、中间和末位，首尾的回答准确率分别是 90% 和 85%，中间只有 55%。
>
> 解决方案有三层：第一，加了 Rerank 模型做精排，确保最相关的 chunk 排到前面；第二，把 top-k 从 10 降到 5，减少噪音；第三，对于特别重要的场景，用 Map-Reduce 模式对每个 chunk 单独提取信息再汇总，彻底规避位置偏差。这三步做完，整体准确率提升了 18%。

### Q: "你说 80% 是数据问题，有量化依据吗？"

> 有的。我们做过一个内部复盘，统计了三个月内 200+ 条 bad case。其中 162 条（81%）根因在检索侧：chunking 不合理占 35%，Embedding 不匹配占 28%，query 表述问题占 18%。只有 38 条（19%）是纯生成侧问题，主要是 prompt 约束不够和 temperature 设置过高。修复检索侧问题后，不换模型的情况下 bad case 减少了 72%。
