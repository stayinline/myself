# RAG 幻觉与答非所问排查 — 原理篇

## 一、核心认知：RAG 错误的两大根因

RAG（Retrieval-Augmented Generation）系统回答错误，**本质上只有两类根因**：

```
用户提问 → [检索侧] → 召回内容 → [生成侧] → 最终回答
              ↑                          ↑
          根因①                       根因②
       没召回正确片段               召回了但模型没忠实使用
```

> **关键原则**：排查 RAG 问题必须分层——**先查召回内容，再查生成过程**。混在一起看永远定位不准。

---

## 二、根因①：检索侧问题（Recall Failure）

### 2.1 什么是检索侧失败

用户的问题在知识库中**有对应答案**，但检索阶段没有把包含答案的 chunk 召回到 top-k 结果中。模型拿到的上下文里根本没有正确答案，自然无法回答。

### 2.2 检索失败的三大子原因

#### （1）Chunking 策略不当

| 问题类型 | 表现 | 根因 |
|---------|------|------|
| 切分过粗 | 一个 chunk 包含多个主题，语义稀释 | 固定长度切分未考虑段落边界 |
| 切分过细 | 单个 chunk 缺少上下文，语义不完整 | 句子级切分丢失前后文 |
| 跨段信息丢失 | 答案横跨两个 chunk，任一片段都不完整 | 无重叠（overlap）或重叠不足 |
| 结构化信息破坏 | 表格/列表被切断，变成无意义文本 | 未对特殊结构做专门处理 |

**核心要点**：chunk 是 RAG 的最小检索单元，chunk 质量直接决定检索上限。

```python
# 典型的 chunking 参数配置示例
chunk_config = {
    "chunk_size": 512,        # token 数
    "chunk_overlap": 64,      # 重叠 token 数，防止跨段丢失
    "separator": "\n\n",      # 优先按段落切分
    "length_function": len,   # 或 tiktoken 编码计数
}
```

#### （2）Embedding 语义不匹配

- **领域术语差异**：通用 Embedding 模型不理解行业黑话。例如"爆仓"在物流领域指仓库满溢，通用模型可能理解为金融爆仓。
- **查询-文档不对称**：用户问的是口语化短句，文档是书面长句，语义空间距离远。
- **多语言对齐差**：中英文混合场景下，某些 Embedding 模型的跨语言对齐能力不足。
- **模型版本不一致**：入库时用 v1 模型，查询时用 v2 模型，向量空间不对齐。

```
用户查询: "怎么退货"
文档原文: "商品退换流程需在购买之日起七个工作日内通过售后渠道提交申请..."

→ 语义相似但用词完全不同，弱 Embedding 模型可能召回失败
```

#### （3）Query 表述不佳

- **用户查询太模糊**："这个怎么处理？"——缺乏实体和上下文。
- **多意图混合**："A产品的价格和B产品的退货政策"——单次检索难以同时满足。
- **拼写/缩写**：内部系统名、项目代号等不在 Embedding 词表中。
- **隐含前提**：用户省略了对话历史中的上下文，导致独立 query 不可检索。

### 2.3 检索侧问题的诊断信号

| 信号 | 含义 |
|------|------|
| Top-k 内容与问题明显无关 | Chunking 或 Embedding 严重不匹配 |
| Top-k 中有相关内容但排名靠后 | Rerank 缺失或 Recall 阈值过低 |
| Top-k 内容部分相关但不完整 | Chunk overlap 不足或 chunk_size 过小 |
| 不同问法结果差异极大 | Query 鲁棒性差，需要 query rewriting |
| 人工能找到答案但系统找不到 | 确认是检索问题而非数据缺失 |

## 三、根因②：生成侧问题（Generation Failure）

### 3.1 什么是生成侧失败

正确的 chunk **已经被召回**到上下文中，但模型在生成回答时没有忠实使用这些内容，产生了幻觉或偏离。

### 3.2 生成失败的四大子原因

#### （1）模型不忠实（Unfaithful）

- 模型倾向于使用自身预训练知识而非给定上下文。
- 当上下文信息与模型内部知识冲突时，模型可能"自信地"忽略上下文。
- 小模型尤其容易出现此问题，因为指令跟随能力弱。

```
上下文: "公司年假政策为入职满一年后享有5天年假"
模型回答: "根据劳动法规定，员工每年享有10天年假"
→ 模型用了预训练知识，忽略了具体上下文
```

#### （2）被无关上下文干扰（Distraction）

- Top-k 中混入了大量不相关 chunk，模型注意力被分散。
- **"Lost in the Middle"现象**：研究表明 LLM 对上下文中间位置的信息关注度最低，首尾信息利用率高。
- 上下文越长，模型越容易"走神"。

```
Top-5 召回:
[1] ✅ 相关（首位，容易被利用）
[2] ❌ 无关
[3] ❌ 无关 ← Lost in the Middle
[4] ❌ 无关
[5] ✅ 相关（末位，容易被利用）

→ 中间的无关内容稀释了模型对关键信息的注意力
```

#### （3）过度发挥（Over-elaboration）

- 模型在上下文提供的信息基础上自行补充细节，这些细节可能是编造的。
- 常见于模型试图给出"完整"回答时，上下文只提供了部分信息。
- Temperature > 0 时概率更高。

#### （4）Prompt 工程缺陷

- 没有在 system prompt 中明确要求"仅基于给定上下文回答"。
- 没有提供回答格式约束，模型自由发挥空间过大。
- 上下文注入位置不当（如放在 user message 末尾而非专用 context 区域）。

### 3.3 生成侧问题的诊断信号

| 信号 | 含义 |
|------|------|
| 召回内容中有答案但回答不包含 | 模型不忠实或 Prompt 约束不够 |
| 回答包含上下文中没有的细节 | 过度发挥 / 幻觉 |
| 回答引用了错误 chunk 的信息 | 被无关上下文干扰 |
| 同样上下文多次生成结果不一致 | Temperature 过高或模型不稳定 |
| 换个更强的模型就好了 | 原模型指令跟随能力不足 |

## 四、分层排查方法论

### 4.1 核心流程

```
┌─────────────────────────────────────────┐
│         RAG 回答异常排查流程              │
├─────────────────────────────────────────┤
│                                         │
│  Step 1: 查看召回的 Top-K chunks        │
│     ↓                                   │
│  召回内容是否正确？                      │
│     ├── 否 → 🔍 检索侧问题             │
│     │    ├── 检查 chunking 策略         │
│     │    ├── 检查 Embedding 模型        │
│     │    ├── 检查 Query 预处理          │
│     │    └── 检查索引完整性             │
│     │                                   │
│     └── 是 → 🔍 生成侧问题             │
│          ├── 检查 Prompt 约束           │
│          ├── 检查上下文排列顺序          │
│          ├── 检查 Temperature           │
│          ├── 检查上下文长度             │
│          └── 检查模型能力               │
│                                         │
└─────────────────────────────────────────┘
```

### 4.2 排查 Checklist

| 序号 | 检查项 | 方法 | 预期 |
|------|--------|------|------|
| 1 | 召回 Top-K 内容 | 打印/日志输出召回 chunks | 至少 1 个 chunk 含正确答案 |
| 2 | 召回排序 | 检查相关 chunk 的排名位置 | 相关 chunk 应在 Top-3 |
| 3 | Prompt 模板 | Review system/user prompt | 明确要求基于上下文回答 |
| 4 | 上下文总长度 | 计算 token 数 | 不超过模型有效窗口 |
| 5 | Temperature | 检查生成参数 | 事实类问答建议 ≤ 0.1 |
| 6 | 模型版本 | 确认模型标识 | 与评测时一致 |
| 7 | 数据新鲜度 | 检查知识库更新时间 | 数据未过期 |

## 五、80% 是数据/检索问题——数据工程师的价值

### 5.1 为什么强调这个比例

业界实践反复验证：**RAG 系统的瓶颈 80% 在数据和检索侧，而非模型侧**。

原因分析：
1. **当前主流 LLM 的指令跟随能力已经足够强**（GPT-4/Claude/Qwen-Max 级别），只要给对了上下文，基本都能忠实回答。
2. **数据质量是天花板**：垃圾进垃圾出，再强的模型也无法从错误的 chunks 中生成正确答案。
3. **检索是工程问题**：chunking、embedding、indexing、reranking 每一步都有明确的工程优化手段。
4. **模型升级成本高**：换模型意味着重新评测、重新调参、可能还有合规审批；而优化检索链路可以快速迭代。

### 5.2 数据工程师在 RAG 中的核心价值

| 价值维度 | 具体体现 |
|---------|---------|
| 数据治理 | 确保入知识库的数据干净、完整、有元数据 |
| Chunking 设计 | 根据数据类型设计最优切分策略 |
| 检索管线 | 构建多路召回、融合排序、增量更新 |
| 评测体系 | 建立可量化的检索+生成评估闭环 |
| 数据血缘 | 实现答案到原文的全链路追溯 |
| 成本控制 | Embedding 调用优化、缓存策略、批量处理 |

### 5.3 反面案例：盲目换大模型治标不治本

```
场景: 客服 RAG 系统回答准确率 60%
错误做法: GPT-3.5 → GPT-4，准确率提升到 65%，成本翻 10 倍
正确做法: 
  1. 排查发现 Top-5 召回命中率仅 45%
  2. 优化 chunking + 加入 BM25 混合检索
  3. 召回命中率提升到 85%
  4. 不换模型，准确率提升到 82%
  5. 成本不变
```

## 六、面试话术

### Q: "RAG 系统出现幻觉怎么排查？"

> 我把 RAG 的错误分为两类：检索侧和生成侧。排查时一定**先看召回内容再看生成**。
>
> 第一步，拉出这条 query 的 top-k 召回结果，看里面有没有包含正确答案的 chunk。如果没有，就是检索问题——我会依次检查 chunking 粒度是否合理、Embedding 模型是否匹配当前领域、query 是否需要改写或扩展。
>
> 如果召回内容是对的，那就是生成侧问题——我会检查 prompt 有没有明确要求基于上下文回答、temperature 是否设得太高、上下文是不是太长导致 lost in the middle。
>
> 实践中我发现大约 80% 的 RAG 问题出在检索和数据侧。所以我的经验是：**不要急着换模型，先把数据和检索做好**。这也是数据工程师在 RAG 项目中最大的价值点。

### Q: "为什么说 80% 是数据问题？"

> 因为现在主流大模型的指令跟随能力已经很强了，只要你给它的上下文是对的、prompt 约束到位，它基本能忠实回答。但如果召回的内容本身就不对或不全，再强的模型也救不了。而且优化检索链路的 ROI 远高于换模型——改 chunking、加混合检索、上 rerank 这些都是工程手段，可以快速迭代、效果可量化。换模型则成本高、周期长、还要重新评测。
