# 检索增强技巧 — 原理篇

## 一、为什么纯向量召回不够用

### 1.1 向量检索的天然局限

| 局限 | 说明 |
|------|------|
| 精确匹配弱 | "HTTP 502" 这种精确术语，向量可能把 502/503/504 混在一起 |
| 数字/ID 不敏感 | 订单号、版本号等纯数字串在向量空间中区分度低 |
| 短文本退化 | query 太短时 embedding 信息量不足 |
| 领域术语盲区 | 通用模型不理解行业缩写和内部代号 |
| 同义词覆盖不全 | "退货" vs "退换" vs "refund" 未必在同一向量邻域 |

### 1.2 解决方案全景图

```
┌─────────────────────────────────────────────────────────────┐
│                    检索增强技术栈                             │
├──────────────┬──────────────────────────────────────────────┤
│ Query 改写    │ 同义扩展 / 子问题分解 / HyDE                 │
│ 多路召回      │ 向量 + BM25 关键词 + 结构化过滤              │
│ 融合排序      │ RRF (Reciprocal Rank Fusion) / 加权融合      │
│ 精排 Rerank   │ Cross-Encoder / Cohere Rerank               │
│ 后处理        │ 去重 / 过滤 / 截断                           │
└──────────────┴──────────────────────────────────────────────┘
```

## 二、Query Rewriting（查询改写）

### 2.1 核心思路

用户的原始 query 往往不是最优检索形式。通过改写让 query 更"可检索"。

### 2.2 三种改写策略

#### （1）同义词扩展

```
原始: "怎么退货"
扩展: "怎么退货 OR 退换 OR 退款 OR refund OR return"
```

适用：用户口语化表达 vs 文档书面表达的 gap。

实现方式：
- 同义词词典（手工维护或从日志挖掘）
- LLM 生成同义表达
- Embedding 空间中的近义词搜索

#### （2）子问题分解

```
原始: "A产品和B产品在续航和重量上有什么区别？"
分解:
  Q1: "A产品续航时间"
  Q2: "B产品续航时间"
  Q3: "A产品重量"
  Q4: "B产品重量"
→ 分别检索后合并结果
```

适用：复合查询、多维度对比。

#### （3）HyDE（Hypothetical Document Embeddings）

```
原始 Query: "公司年假政策"
Step 1: LLM 生成假设性回答 → "根据公司规定，入职满一年后享有5天年假..."
Step 2: 对假设回答做 Embedding
Step 3: 用假设回答的向量去检索（而非原始 query 的向量）
```

**原理**：假设回答与真实文档在语义空间中更接近（都是陈述句、都用书面语），比短 query 的检索效果更好。

**代价**：
- 额外一次 LLM 调用（延迟 +500ms~2s，成本增加）
- 如果 LLM 生成的假设回答方向错误，检索也会跑偏
- 适合离线/低频场景，不适合实时高并发

### 2.3 何时用什么

| 场景 | 推荐策略 | 原因 |
|------|---------|------|
| 口语化 query | 同义词扩展 | 弥合口语-书面语 gap |
| 复合问题 | 子问题分解 | 单次检索无法覆盖多维度 |
| 长尾/冷门问题 | HyDE | 用生成弥补检索不足 |
| 高并发在线服务 | 同义词扩展(预计算) | HyDE 延迟不可接受 |
| 内部系统代号 | 同义词词典 | LLM 不知道内部代号 |

## 三、多路召回（Multi-Route Recall）

### 3.1 向量 + BM25 混合检索

```
                   ┌─────────────┐
                   │  User Query  │
                   └──────┬──────┘
                          │
                ┌─────────┼─────────┐
                ↓                   ↓
        ┌──────────────┐    ┌──────────────┐
        │ Vector Search │    │ BM25 Search  │
        │ (语义相似度)   │    │ (关键词匹配)  │
        └──────┬───────┘    └──────┬───────┘
               │                    │
               └────────┬───────────┘
                        ↓
               ┌──────────────┐
               │  RRF Fusion   │
               │ (排名融合)    │
               └──────┬───────┘
                      ↓
               ┌──────────────┐
               │  Top-K Results│
               └──────────────┘
```

### 3.2 BM25 的优势

- **精确匹配强**："HTTP 502" 就是 "HTTP 502"，不会和 503 混淆
- **无需训练**：基于词频统计，开箱即用
- **可解释性好**：得分来自 TF-IDF 变体，容易 debug
- **成熟生态**：Elasticsearch / OpenSearch 原生支持

### 3.3 RRF（Reciprocal Rank Fusion）

RRF 是最常用的多路融合算法，不需要对不同路的分数做归一化：

$$
\text{RRF}(d) = \sum_{r \in R} \frac{1}{k + \text{rank}_r(d)}
$$

其中 $k$ 通常取 60，$R$ 是所有检索路，$\text{rank}_r(d)$ 是文档 $d$ 在第 $r$ 路中的排名。

**直觉理解**：一个文档在多路检索中排名都靠前 → RRF 分高；只在一路排名高 → RRF 分中等；都不靠前 → RRF 分低。

```python
def reciprocal_rank_fusion(rankings: list[list[str]], k: int = 60) -> list[tuple[str, float]]:
    """
    rankings: 多路检索结果的排名列表，每路是一个 doc_id 列表（按相关性降序）
    """
    scores = {}
    for ranking in rankings:
        for rank, doc_id in enumerate(ranking):
            if doc_id not in scores:
                scores[doc_id] = 0.0
            scores[doc_id] += 1.0 / (k + rank + 1)

    # 按分数降序排列
    sorted_docs = sorted(scores.items(), key=lambda x: x[1], reverse=True)
    return sorted_docs
```

### 3.4 其他融合方式对比

| 方法 | 优点 | 缺点 |
|------|------|------|
| RRF | 无需归一化，简单鲁棒 | 不考虑分数绝对值 |
| 加权线性组合 | 可调权重 | 需要分数归一化 |
| Learning to Rank | 效果上限高 | 需要标注数据训练 |
| Rerank 替代融合 | 精度最高 | 成本高，见下节 |

## 四、Rerank（重排序）

### 4.1 为什么需要 Rerank

向量检索是**双塔模型**（bi-encoder）：query 和 document 分别编码再算相似度，速度快但交互不充分。

Rerank 是**交叉编码器**（cross-encoder）：query 和 document 拼接后一起输入模型，充分交互注意力，精度高但速度慢。

```
Bi-Encoder (检索):   Q → Enc → q_vec ─┐
                                       ├→ cosine_sim → score
                   D → Enc → d_vec ─┘
                   ✅ 快 (毫秒级)  ❌ 精度有上限

Cross-Encoder (Rerank): [Q; D] → Enc → score
                   ❌ 慢 (百毫秒级)  ✅ 精度高
```

### 4.2 Rerank 的使用模式

**粗召回 + 精排**：
1. 向量/BM25 粗召回 top-100（毫秒级）
2. Rerank 模型对 100 条精排（秒级）
3. 取 top-5 送入 LLM

### 4.3 常用 Rerank 模型

| 模型 | 语言 | 延迟(per pair) | 备注 |
|------|------|---------------|------|
| bge-reranker-v2-m3 | 中英 | ~50ms | BAAI 开源，中文效果好 |
| cohere-rerank-v3 | 多语言 | API ~100ms | SaaS，无需部署 |
| ms-marco-MiniLM | 英文 | ~20ms | 轻量级 |
| GTE-Rerank | 中英 | ~60ms | 阿里开源 |

### 4.4 开销评估

```
假设粗召回 100 条，Rerank 每条 50ms:
总 Rerank 耗时 = 100 × 50ms = 5秒 ← 不可接受！

优化方案:
1. 减少粗召回数量: 100 → 20, 耗时 = 1秒
2. GPU batch 推理: 20条一批, 耗时 ≈ 200ms
3. 使用更小模型: MiniLM, 耗时 ≈ 100ms
4. 异步并行: 与其他步骤 overlap
```

## 五、选型决策树

```
用户 Query 进来
       │
       ├── Query 是否模糊/口语化？
       │     └── 是 → Query Rewriting（同义扩展）
       │
       ├── 是否包含精确术语/ID/数字？
       │     └── 是 → 必须加 BM25 路
       │
       ├── 是否是复合问题？
       │     └── 是 → 子问题分解 + 并行检索
       │
       ├── 是否是长尾/冷门问题？
       │     └── 是 + 延迟可接受 → HyDE
       │
       ├── 纯向量召回效果够吗？
       │     ├── 够 → 只用向量
       │     └── 不够 → 向量 + BM25 + RRF
       │
       └── 融合后精度够吗？
             ├── 够 → 直接取 top-k
             └── 不够 → 加 Rerank 精排
```

## 六、面试话术

### Q: "纯向量召回不准，你有哪些提升手段？"

> 我有四层递进的增强手段：
>
> **第一层是 Query 改写**。用户的原始 query 经常是口语化的、模糊的，我会做同义词扩展或者用 LLM 把复合问题拆成子问题分别检索。对于长尾问题还可以用 HyDE——先让 LLM 生成一个假设性回答，再用这个回答的向量去检索，因为假设回答和文档在语义空间里更接近。不过 HyDE 有额外延迟，高并发场景要慎用。
>
> **第二层是多路召回**。向量检索擅长语义匹配但对精确术语和数字不敏感，所以我一定会加一路 BM25 关键词检索。两路结果用 RRF（Reciprocal Rank Fusion）融合，这个方法不需要对不同路的分数做归一化，简单且鲁棒。BM25 可以直接用 Elasticsearch，和数据工程师的技术栈天然契合。
>
> **第三层是 Rerank 精排**。粗召回 top-50 到 top-100，然后用 cross-encoder 模型做精排，取 top-5 送给 LLM。Rerank 精度远高于 bi-encoder，但要注意控制粗召回数量和 batch 推理来管理延迟。我们用的 bge-reranker-v2-m3，GPU batch 20 条约 200ms。
>
> **第四层是持续调优**。这些组件的参数（top-k、RRF 的 k 值、Rerank 阈值、Query 改写的 prompt）都不是拍脑袋定的，而是通过评测集驱动迭代。每次调整都跑一遍 Recall@K 和 Faithfulness 指标，确认有提升才上线。
>
> 实际项目中，加上混合检索 + Rerank 后，我们的 Recall@5 从 62% 提升到了 89%，效果非常显著。

### Q: "HyDE 的优缺点是什么？什么时候用？"

> HyDE 的优点是对长尾问题和语义 gap 大的场景效果好，因为它用 LLM 生成的假设回答做 embedding，和文档的语义距离比短 query 更近。
>
> 缺点是三个：第一，额外一次 LLM 调用增加 500ms-2s 延迟；第二，如果 LLM 生成了错误方向的假设，检索反而更差；第三，增加了成本和复杂度。
>
> 所以我的建议是：在线高并发服务不用 HyDE，用 Query 改写 + 混合检索就够了；离线分析、客服工单处理这类延迟不敏感的场景可以用；以及作为 fallback——当常规检索置信度低于阈值时再触发 HyDE。

## 七、关键概念速查表

| 术语 | 全称 | 一句话解释 |
|------|------|-----------|
| BM25 | Best Matching 25 | 基于词频的经典关键词检索算法 |
| RRF | Reciprocal Rank Fusion | 多路检索排名融合算法，k=60 |
| HyDE | Hypothetical Document Embeddings | 用 LLM 生成假设回答再做向量检索 |
| Bi-Encoder | - | 双塔模型，query/doc 分别编码，快但不精 |
| Cross-Encoder | - | 交叉编码器，query+doc 联合编码，精但慢 |
| Rerank | Re-ranking | 粗召回后用精排模型重排 |
| Recall@K | - | Top-K 结果中包含正确答案的比例 |
| MRR | Mean Reciprocal Rank | 第一个正确结果排名的倒数均值 |
| NDCG | Normalized Discounted Cumulative Gain | 考虑排名位置的检索质量指标 |
