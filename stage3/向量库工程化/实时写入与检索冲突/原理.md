# 实时写入与检索冲突原理

## 一、核心矛盾

### 1.1 问题描述

在向量数据库中，存在一个根本性的矛盾：

> **索引是高效检索的前提，但索引的构建需要时间和批量数据。新写入的数据在索引构建完成之前，要么不可见，要么只能以暴力扫描的方式低效可见。**

这类似于传统数据库中"写入立即可见"与"B-Tree 索引维护"之间的矛盾，但在向量数据库中更加突出，因为：
- 向量索引（如 HNSW）的构建远比 B-Tree 复杂耗时
- 向量索引通常是批量的，不支持高效的逐条增量更新
- 用户对语义搜索的实时性期望越来越高

### 1.2 具体表现

```
时间线:
t₀: 用户提交文档 → Embedding → 写入向量库
t₁: 数据进入 Growing Segment（内存缓冲，无索引）
    ├── 此时搜索能否找到？取决于实现：
    │   ├── Milvus: ✅ 可找到（暴力扫描 Growing Segment），但慢
    │   ├── Qdrant: ✅ 可找到（有简单的增量索引）
    │   └── 某些系统: ❌ 不可见（只对已建索引的 Segment 搜索）
t₂: Growing Segment Seal → Flush 到存储
t₃: Index Node 开始构建索引
t₄: 索引构建完成，Query Node 加载
    └── 此时搜索高效且准确 ✓

延迟窗口 = t₄ - t₀ ≈ 几十秒到几分钟
```

## 二、Growing Segment 的影响分析

### 2.1 Growing Segment 的搜索机制

当 Query Node 收到搜索请求时，它需要同时搜索两部分数据：

```
Search Request
      │
      ├──→ Sealed Segments (有索引) → ANN 搜索，毫秒级
      │
      └──→ Growing Segments (无索引) → 暴力扫描，线性复杂度
              │
              └── 结果合并排序 → Top-K
```

### 2.2 性能影响量化

假设：
- Sealed Segments: 1000万条，HNSW 索引，搜索延迟 2ms
- Growing Segment: 50万条，无索引，暴力扫描

| Growing 大小 | Growing 搜索延迟 | 总延迟 | 影响 |
|-------------|----------------|--------|------|
| 0 | 0ms | 2ms | 无影响 |
| 10万 | ~4ms | ~6ms | 轻微 |
| 50万 | ~20ms | ~22ms | 明显 |
| 200万 | ~80ms | ~82ms | 严重 |
| 500万 | ~200ms | ~202ms | 灾难性 |

> ⚠️ Growing Segment 的暴力扫描不仅增加了绝对延迟，还引入了**延迟抖动**——因为 Growing Segment 的大小在不断变化，每次查询的延迟可能不同。

### 2.3 Recall 影响

Growing Segment 使用暴力搜索，理论上 Recall=1.0（精确搜索）。所以 Growing Segment 不会降低整体 Recall，反而可能提升（因为它贡献了精确结果）。但问题是：

- 如果 Growing Segment 太大导致超时被跳过 → Recall 下降
- 如果系统对 Growing Segment 不做搜索 → 刚写入的数据完全不可见 → Recall 下降

## 三、"入湖即入库"概念的关联

### 3.1 什么是"入湖即入库"

在数据湖仓架构中，"入湖即入库"指的是数据写入数据湖的同时就建立了索引，可以立即被查询。这个概念与向量数据库的实时写入问题高度相关：

```
传统模式:  数据入湖 → ETL → 建索引 → 可查询     (小时级延迟)
理想模式:  数据入湖 = 建索引 = 可查询            (秒级延迟)
```

### 3.2 映射到向量数据库

| 数据湖仓概念 | 向量数据库对应 |
|------------|-------------|
| 数据入湖 | Embedding 写入向量库 |
| 建索引 | HNSW/IVF 索引构建 |
| 可查询 | 高效 ANN 搜索 |
| Delta Table / Iceberg | Growing + Sealed Segment |
| Compaction | Segment Merge / Compaction |
| Streaming Ingestion | Flink/Kafka → 向量库 Sink |

### 3.3 为什么不能完全"入湖即入库"

向量索引的特殊性决定了完全的实时索引极其困难：
- HNSW 需要在图中建立高质量的邻居关系，单条插入的质量不如批量构建
- IVF 需要全局聚类信息，无法增量更新聚类中心
- PQ 需要训练码本，更不能逐条更新

## 四、批量写入平衡实时性与索引效率

### 4.1 核心思想

> **不要逐条写入向量数据库，而是攒够一批再写。这样既减少了 Growing Segment 的数量和碎片化，又让索引构建更高效。**

### 4.2 与 Flink Batch Sink 的关联

在流式数据处理中，Flink 写入下游存储通常不会逐条发送，而是做微批聚合：

```
Flink Stream Processing:
[Event] → [Embedding Model] → [Batch Buffer] → [Vector DB Sink]
                                      │
                              batch.size = 1000
                              flush.interval = 5s
                              
                              满足任一条件即触发写入:
                              - buffer 满 1000 条
                              - 距上次写入超过 5 秒
```

### 4.3 批量大小的权衡

| Batch Size | 写入吞吐 | 实时性 | 索引效率 | Growing Segment 碎片 |
|-----------|---------|--------|---------|---------------------|
| 1 (逐条) | 低 | 最好 | 最差 | 最多 |
| 100 | 中等 | 好 | 一般 | 较多 |
| 1000 | 高 | 秒级 | 好 | 少 |
| 10000 | 很高 | 十秒级 | 很好 | 很少 |
| 100000 | 极高 | 分钟级 | 最好 | 最少 |

> 💡 **经验法则**：batch_size=1000~5000, flush_interval=3~10s 是大多数场景的最优区间。

### 4.4 Flink Sink 伪代码

```python
class MilvusSinkFunction(SinkFunction):
    def __init__(self, batch_size=1000, flush_interval_ms=5000):
        self.batch_size = batch_size
        self.flush_interval_ms = flush_interval_ms
        self.buffer = []
        self.last_flush_time = current_time_ms()
    
    def invoke(self, record):
        self.buffer.append(record)
        
        should_flush = (
            len(self.buffer) >= self.batch_size or
            current_time_ms() - self.last_flush_time >= self.flush_interval_ms
        )
        
        if should_flush:
            self._flush()
    
    def _flush(self):
        if not self.buffer:
            return
        
        # 批量写入 Milvus
        embeddings = [r.embedding for r in self.buffer]
        texts = [r.text for r in self.buffer]
        collection.insert([embeddings, texts])
        
        self.buffer.clear()
        self.last_flush_time = current_time_ms()
```

## 五、各种缓解策略的原理

### 5.1 策略总览

```
                    实时性 ←─────────────────→ 索引效率
                    
策略A: 调小 flush 阈值          ●────────────────○
策略B: Streaming Upsert         ●●───────────────○
策略C: 定期重建索引             ○────────────────●●
策略D: 读写合并新旧             ●●●──────────────○
策略E: 热数据独立分区           ●●───────────────●
策略F: Streaming-capable索引    ●●●──────────────●●
```

### 5.2 策略详解

#### A. 调小 Flush/Seal 阈值

- 让 Growing Segment 更快变成 Sealed → 更早开始建索引
- 代价：产生更多小 Segment → 搜索时需要合并更多段 → 可能需要频繁 Compaction
- 适用：写入量不大但对实时性要求较高的场景

#### B. Streaming Upsert

- 一些向量数据库（如 Qdrant、Milvus 2.4+）支持流式 upsert
- 新数据先写入一个轻量的增量结构（如小的 HNSW 子图或 Flat buffer）
- 后台异步将增量结构合并到主索引
- 比纯 Growing Segment 暴力扫描快得多

#### C. 定期重建索引

- 累积一段时间后全量重建索引
- 保证索引质量最优
- 代价：重建期间资源消耗大，可能有短暂的服务降级
- 适用：离线/准实时场景

#### D. 读写合并新旧

- 搜索时同时查已建索引的旧数据和未建索引的新数据
- 结果合并排序后返回
- 这就是 Milvus 的默认行为（Growing + Sealed 合并搜索）
- 关键是要控制好新数据的量，避免暴力扫描部分过大

#### E. 热数据独立分区

- 最近 N 小时/天的数据放在单独的 Partition
- 这个 Partition 可以用更适合实时的索引（如 Flat 或小 HNSW）
- 搜索时同时查热分区和冷分区
- 冷分区用大 HNSW/IVF-PQ 保证效率

#### F. Streaming-capable 索引

- DiskANN/FreshDiskANN 等新型索引支持流式更新
- 新数据插入时局部更新图结构，不需要全量重建
- 兼顾了实时性和索引质量
- 目前还在发展中，生产可用度逐步提升

## 六、面试话术

### Q："向量数据库写入后多久能被搜到？你怎么处理实时性问题？"

> 这取决于向量数据库的实现和配置。以 Milvus 为例，写入后数据先进入 Growing Segment，此时已经可以被搜到，但是暴力扫描没有索引，延迟比较高。当 Growing Segment 达到阈值被 Seal 并建完索引后，才能高效检索。默认配置下整个链路大约几十秒到几分钟。
>
> 在实际项目中，我们做了几个层面的优化来处理这个问题。第一，在写入端做微批聚合——我们的 Flink Sink 设置了 batch_size=2000 和 flush_interval=5s，避免逐条写入导致的 Growing Segment 碎片化。第二，调整了 Milvus 的 seal 阈值，让 Segment 在积累约 100 万行时就触发 seal，而不是等到默认的 200 万行，缩短了从写入到建索引的等待时间。第三，对于实时性要求极高的场景（比如用户刚上传的文档要立刻能搜到），我们把最近一小时的数据放在独立的"热分区"，用较小的 HNSW 索引甚至 Flat Index，牺牲一点效率换取秒级可见性；超过一小时的数据自动归入"冷分区"用标准 HNSW。第四，监控 Growing Segment 的大小和数量作为告警指标——如果 Growing Segment 积压过多说明索引构建跟不上写入速度，需要扩容 Index Node 或调整参数。
>
> 本质上这是一个实时性和索引效率的 trade-off。没有银弹，只有根据业务 SLA 选择最合适的平衡点。

### Q："这和'入湖即入库'是什么关系？"

> "入湖即入库"是数据湖仓领域的概念，指数据写入的同时就建好索引可以立即查询。向量数据库面临类似的问题——写入的向量什么时候能被高效检索。但向量索引比传统的 B-Tree/倒排索引复杂得多，HNSW 需要构建多层图结构，IVF-PQ 需要训练聚类和码本，这些都很难做到真正的"即写即索引"。所以更现实的目标不是"入湖即入库"而是"近实时可检索"——通过微批写入、增量索引、热冷分区等手段把延迟控制在秒级到分钟级，满足绝大多数业务需求。

### Q："Flink 写入向量库为什么要做 batching？"

> 三个原因。第一，减少 Growing Segment 碎片——逐条写入会产生大量小的 Growing Segment，每个都要单独 seal、建索引、加载，开销巨大。批量写入让每个 Segment 更饱满，减少碎片。第二，提高写入吞吐——每次 insert API 调用都有网络开销和序列化开销，批量操作摊薄了这些固定成本。第三，配合索引构建节奏——向量索引本身是批量操作的，攒够一批再写入可以让索引构建更高效。我们在 Flink Sink 中设置 batch_size=2000 和 flush_interval=5s，实测写入吞吐提升了 8 倍，同时 Growing Segment 数量从数百降到了个位数，搜索 P99 延迟也更稳定了。
