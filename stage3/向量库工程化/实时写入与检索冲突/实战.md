# 实时写入与检索冲突实战

## 一、实验：写入后可见性延迟测量

### 1.1 实验设计

目标：量化"从写入到可被高效检索"的完整延迟链路。

```python
import numpy as np
import time
from pymilvus import connections, Collection, utility

connections.connect(host="localhost", port="19530")
collection = Collection("realtime_test")
collection.load()

# 准备一个特殊的查询向量（确保我们知道它在数据库中的精确位置）
special_vec = np.random.randn(1, 768).astype(np.float32)
special_vec /= np.linalg.norm(special_vec)

# 记录写入时间
write_time = time.time()
collection.insert([special_vec.tolist(), ["special_document"]])
print(f"[{write_time:.3f}] Inserted special document")

# 轮询搜索，记录首次可见时间和首次高效可见时间
poll_interval = 0.5  # 每0.5秒检查一次
max_wait = 300       # 最多等5分钟

first_visible_time = None
first_efficient_time = None
search_params = {"metric_type": "IP", "params": {"ef": 64}}

start = time.time()
while time.time() - start < max_wait:
    t0 = time.perf_counter()
    results = collection.search(
        data=special_vec.tolist(),
        anns_field="embedding",
        param=search_params,
        limit=5,
        output_fields=["text"],
        consistency_level="Strong"  # 强一致，确保读到最新
    )
    latency = (time.perf_counter() - t0) * 1000
    
    found = any(hit.entity.get("text") == "special_document" for hit in results[0])
    
    if found and first_visible_time is None:
        first_visible_time = time.time()
        print(f"[{first_visible_time:.3f}] ✅ First visible! "
              f"Delay={first_visible_time-write_time:.2f}s, Latency={latency:.1f}ms")
    
    if found and latency < 10 and first_efficient_time is None:
        first_efficient_time = time.time()
        print(f"[{first_efficient_time:.3f}] ⚡ Efficient! "
              f"Delay={first_efficient_time-write_time:.2f}s, Latency={latency:.1f}ms")
        break
    
    time.sleep(poll_interval)

print(f"\n=== Summary ===")
print(f"Write → Visible delay:    {first_visible_time-write_time:.2f}s" if first_visible_time else "Never visible!")
print(f"Write → Efficient delay:  {first_efficient_time-write_time:.2f}s" if first_efficient_time else "Never efficient!")
```

### 1.2 预期输出

```
[1700000000.000] Inserted special document
[1700000000.512] ✅ First visible! Delay=0.51s, Latency=45.3ms
[1700000012.345] ⚡ Efficient! Delay=12.35s, Latency=3.2ms

=== Summary ===
Write → Visible delay:    0.51s
Write → Efficient delay:  12.35s
```

> 解读：写入后 0.5 秒就能搜到（Growing Segment 暴力扫描），但延迟高达 45ms。约 12 秒后索引构建完成，延迟降到 3ms。

## 二、Growing Segment 对延迟的影响实验

### 2.1 模拟不同大小的 Growing Segment

```python
def measure_growing_impact(collection, base_query, growing_sizes=[0, 10000, 50000, 200000]):
    """测量不同 Growing Segment 大小对搜索延迟的影响"""
    search_params = {"metric_type": "IP", "params": {"ef": 64}}
    results = {}
    
    for size in growing_sizes:
        # 插入指定数量的数据（不 flush，保持在 Growing Segment）
        if size > 0:
            vecs = np.random.randn(size, 768).astype(np.float32).tolist()
            texts = [f"growing_doc_{i}" for i in range(size)]
            collection.insert([vecs, texts])
            print(f"Inserted {size} docs into Growing Segment")
        
        # 等待稳定
        time.sleep(2)
        
        # 测量搜索延迟（多次取平均和 P99）
        latencies = []
        for _ in range(100):
            t0 = time.perf_counter()
            collection.search(
                data=base_query,
                anns_field="embedding",
                param=search_params,
                limit=10,
                consistency_level="Eventually"
            )
            latencies.append((time.perf_counter() - t0) * 1000)
        
        avg_lat = np.mean(latencies)
        p99_lat = np.percentile(latencies, 99)
        results[size] = {"avg_ms": round(avg_lat, 2), "p99_ms": round(p99_lat, 2)}
        
        print(f"  Growing={size:>7d}: Avg={avg_lat:.2f}ms, P99={p99_lat:.2f}ms")
    
    return results

query_vec = np.random.randn(1, 768).astype(np.float32).tolist()
impact_results = measure_growing_impact(collection, query_vec)
```

### 2.2 预期结果

```
Growing=      0: Avg=2.15ms, P99=4.32ms
Growing=  10000: Avg=2.87ms, P99=6.15ms
Growing=  50000: Avg=5.43ms, P99=12.87ms
Growing= 200000: Avg=18.76ms, P99=45.23ms
```

| Growing 大小 | Avg Latency | P99 Latency | 影响评估 |
|-------------|------------|------------|---------|
| 0 | 2.15ms | 4.32ms | 基线 |
| 1万 | 2.87ms (+33%) | 6.15ms (+42%) | 轻微 |
| 5万 | 5.43ms (+153%) | 12.87ms (+198%) | 明显 |
| 20万 | 18.76ms (+773%) | 45.23ms (+947%) | 严重 |

> ⚠️ **关键发现**：Growing Segment 对 P99 延迟的影响远大于对平均延迟的影响。这是因为暴力扫描的时间方差大，偶尔会出现特别慢的查询。对于 SLA 敏感的系统，必须控制 Growing Segment 的大小。

### 3.3 阈值选择指南

| 场景 | 推荐 maxSize | 推荐 maxLife | 理由 |
|------|------------|-------------|------|
| 高实时性（<10s可见） | 64~128 MB | 30s | 快速 seal，多小段 |
| 平衡型 | 256~512 MB | 300s | 默认配置 |
| 高吞吐批处理 | 1024~2048 MB | 3600s | 大段少碎片 |
| 写入极少 | 64 MB | 60s | 避免长时间不 seal |

> 💡 调小阈值会让 seal 更频繁 → 索引构建更频繁 → Index Node 压力更大。需要同时关注 Index Node 的资源利用率。

### 4.2 Flink 批量 Sink 实现

```python
"""
Flink Milvus Sink with Batching
实际使用时需要用 PyFlink 或 Java Flink API
这里展示核心逻辑
"""

class BatchedMilvusSink:
    def __init__(self, collection_name, batch_size=2000, 
                 flush_interval_s=5, max_retries=3):
        self.collection_name = collection_name
        self.batch_size = batch_size
        self.flush_interval_s = flush_interval_s
        self.max_retries = max_retries
        
        self.buffer_embeddings = []
        self.buffer_texts = []
        self.buffer_tenants = []
        self.last_flush_time = time.time()
    
    def add(self, record: dict):
        """添加一条记录到缓冲区"""
        self.buffer_embeddings.append(record["embedding"])
        self.buffer_texts.append(record["text"])
        self.buffer_tenants.append(record["tenant_id"])
        
        should_flush = (
            len(self.buffer_embeddings) >= self.batch_size or
            time.time() - self.last_flush_time >= self.flush_interval_s
        )
        
        if should_flush:
            self.flush()
    
    def flush(self):
        """批量写入 Milvus"""
        if not self.buffer_embeddings:
            return
        
        n = len(self.buffer_embeddings)
        
        for retry in range(self.max_retries):
            try:
                collection = Collection(self.collection_name)
                result = collection.insert([
                    self.buffer_embeddings,
                    self.buffer_texts,
                    self.buffer_tenants
                ])
                print(f"Flushed {n} records (attempt {retry+1})")
                
                # 清空缓冲区
                self.buffer_embeddings.clear()
                self.buffer_texts.clear()
                self.buffer_tenants.clear()
                self.last_flush_time = time.time()
                return
                
            except Exception as e:
                print(f"Flush failed (attempt {retry+1}): {e}")
                time.sleep(2 ** retry)  # 指数退避
        
        # 所有重试失败 → 告警 + 丢弃或持久化到死信队列
        print(f"⚠️ Failed to flush {n} records after {self.max_retries} retries!")
        # send_to_dead_letter_queue(buffer)
    
    def close(self):
        """关闭时刷新剩余数据"""
        self.flush()
```

## 六、面试话术

### Q："你怎么处理向量库的实时写入和检索一致性？"

> 这个问题我在实际项目中深入处理过。核心矛盾是新写入的数据在索引构建完成前要么不可见、要么低效可见。我们的解决方案是分层的：
>
> 第一层是写入端的微批聚合。Flink Sink 设置 batch_size=2000、flush_interval=5s，避免逐条写入导致的 Growing Segment 碎片化。实测这让 Growing Segment 数量从数百降到了个位数。
>
> 第二层是 Milvus 侧的 seal 阈值调优。我们把 maxLife 从默认的 3600s 调到 120s，让没有新写入的 Growing Segment 更快 seal。同时 maxSize 保持 512MB 不变，避免产生过多小段。
>
> 第三层是热冷分区。最近一小时的数据写入"热分区"，用较小的 HNSW 索引（M=8, efConstruction=100），牺牲一些 Recall 换取更快的索引构建速度（通常 10 秒内完成）。超过一小时的数据由后台任务迁移到"冷分区"，用标准 HNSW 保证精度。
>
> 第四层是监控告警。我们在 Grafana 上监控 Growing Segment 的行数和数量、索引构建队列长度、搜索 P99 延迟等指标。一旦 Growing Segment 积压超过阈值就告警，说明写入速度超过了索引构建能力，需要扩容 Index Node。
>
> 效果上，我们把"写入到高效可搜"的延迟从默认的 2~3 分钟降到了 15~30 秒，P99 搜索延迟控制在 5ms 以内。

### Q："Growing Segment 太大会有什么问题？怎么解决？"

> Growing Segment 太大有三个问题。第一，搜索延迟增加——因为 Growing Segment 没有索引只能暴力扫描，20 万条数据的暴力扫描就要 20ms 左右，叠加到正常的 2ms 索引搜索上就是 10 倍增长。第二，延迟抖动——Growing Segment 大小在不断变化，导致每次查询的延迟不稳定，P99 比平均值恶化得更厉害。第三，内存压力——Growing Segment 驻留在 Data Node 内存中，太大会导致 OOM。
>
> 解决方案首先是预防：合理设置 seal 阈值让 Growing Segment 及时转为 Sealed；写入端做 batching 减少碎片。其次是应急：如果已经积压了大的 Growing Segment，可以手动 flush 强制触发 seal。长期来看，可以考虑升级到支持 streaming index 的方案，或者把写入和搜索的计算资源隔离开。

### Q："你的 Flink Sink 是怎么设计的？遇到过什么问题？"

> 我们的 Flink Sink 核心是微批聚合 + 指数退避重试 + 死信队列。batch_size=2000 和 flush_interval=5s 是经过压测确定的最优值——再大实时性不够，再小吞吐下降明显。
>
> 遇到的主要问题有三个。第一个是 Milvus 写入超时——当 Milvus 在做 compaction 或索引构建时，insert API 可能变慢甚至超时。我们用指数退避重试（最多 3 次）来解决，如果仍然失败就写入 Kafka 死信 topic，后续由补偿任务重新消费。第二个是 checkpoint 对齐问题——Flink checkpoint 时必须先 flush 缓冲区，否则未提交的数据在故障恢复时会丢失。我们在 sink 的 snapshotState 方法中强制 flush。第三个是背压——当下游 Milvus 写入变慢时，buffer 会快速增长。我们加了 buffer 上限（比如 10000 条），超过就触发背压让上游减速，而不是无限堆积导致 OOM。

## 七、调优 Checklist

```
□ 写入端是否做了 batching？(batch_size ≥ 1000)
□ Flush/Seal 阈值是否匹配业务实时性要求？
□ Growing Segment 大小是否有监控和告警？
□ 索引构建是否有足够的 Index Node 资源？
□ 搜索的 consistency level 是否选择了合适的级别？
□ 是否有热冷分区或其他实时性优化措施？
□ Compaction 策略是否合理（不会阻塞写入也不会积累太多小段）？
□ 端到端延迟（写入→可搜）是否有持续度量和 SLO？
```
