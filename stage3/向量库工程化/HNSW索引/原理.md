# HNSW 索引原理

## 一、HNSW 是什么

HNSW（Hierarchical Navigable Small World）是目前工业界最主流的向量索引算法，由 Yuri Malkov 和 Dmitry Yashunin 于 2016 年提出。它被 FAISS、Milvus、Qdrant、Weaviate、Elasticsearch、pgvector 等几乎所有向量数据库采用作为默认或首选索引。

**一句话概括**：HNSW 是一个多层跳表式近邻图，上层稀疏用于快速定位大致区域，下层稠密用于精确搜索最终结果。

---

## 二、从 Skip List 到 HNSW 的类比理解

### 2.1 Skip List 回顾

Skip List 是有序链表的多层版本：

```
Layer 3:  A ──────────────────────── Z        (最稀疏，大步跳跃)
Layer 2:  A ────── D ────────── M ── Z        (中等密度)
Layer 1:  A ── B ── D ── F ── H ── M ── Z    (较密)
Layer 0:  A-B-C-D-E-F-G-H-I-J-K-L-M-...-Z     (全量，逐个相连)
```

搜索时从最高层开始，每层尽量往右走，走不动就下一层继续。复杂度 O(log N)。

### 2.2 HNSW 的核心类比

| Skip List | HNSW |
|-----------|------|
| 一维有序数据 | 高维向量空间 |
| 节点按值排序连接 | 节点按向量距离连接 |
| "向右移动" = 值增大 | "向邻居移动" = 更接近目标 |
| 层级随机分配 | 层级随机分配（指数衰减概率） |
| 每层是有序链表 | 每层是 Navigable Small World 图 |

**关键区别**：Skip List 在一维有序空间中可以明确定义"向左/向右"，而高维向量空间没有全局序关系。HNSW 用**贪心搜索 + 图遍历**替代了 Skip List 中的方向性移动。

## 三、HNSW 的结构详解

### 3.1 多层图结构

```
                    Layer L (顶层，通常只有几个节点)
                   ╱    │    ╲
                  ╱     │     ╲
           Layer L-1 (更多节点)
          ╱   │   ╲   ╱   ╲
         ╱    │    ╲ ╱     ╲
        ...  中间层  ...    ...
       ╱   │   ╲   ╱   ╲   ╱
      ╱    │    ╲ ╱     ╲ ╱
   Layer 0 (底层，包含所有节点，连接最密集)
```

- **顶层**：极少数节点（约 log(N) 个），形成粗粒度的全局导航骨架
- **中间层**：节点数逐层指数增加，提供中粒度导航
- **底层（Layer 0）**：包含全部 N 个节点，每个节点连接到其局部近邻

### 3.2 节点的层级分配

每个新插入的节点被分配的层级 $l$ 服从指数衰减分布：

$$P(l) = \frac{1}{\ln(M)} \cdot e^{-l / \ln(M)}$$

其中 M 是最大邻居数参数。这意味着：
- 大部分节点只在 Layer 0
- 少数节点出现在 Layer 1
- 极少数节点出现在 Layer 2+
- 顶层通常只有 1~3 个节点

这种分布保证了上层的稀疏性和下层的稠密性。

### 3.3 每层的图结构

每一层都是一个 **Navigable Small World (NSW)** 图，具有以下性质：

1. **连通性**：任意两个节点之间可以通过图的边到达
2. **小世界特性**：任意两点之间的最短路径长度为 O(log N)
3. **贪心可搜索性**：从任意起点出发，贪心地走向离目标最近的邻居，能高效逼近目标

构建时使用启发式选边策略（不是简单地连最近的 K 个点），确保图的连通性和多样性：
- 优先连接不同方向的邻居（避免所有边都指向同一个密集区域）
- 保留长程边以维持小世界特性

## 四、搜索过程

### 4.1 算法伪代码

```
Search(query q, K, efSearch):
    # Phase 1: 从顶层向下贪心定位
    v = entry_point (顶层某个节点)
    for lc in [L, L-1, ..., 1]:
        v = GreedySearch(layer=lc, start=v, target=q)
        # GreedySearch: 反复走向离 q 最近的邻居，直到无法改善
    
    # Phase 2: 在 Layer 0 做 beam search
    candidates = priority_queue(min-heap by distance to q)
    results = priority_queue(max-heap by distance to q, size=efSearch)
    
    candidates.push(v)
    visited = {v}
    
    while candidates not empty:
        c = candidates.pop_nearest()
        if distance(c, q) > farthest_in_results:
            break  # 剪枝
        
        for neighbor in Layer0_neighbors(c):
            if neighbor not in visited:
                visited.add(neighbor)
                if distance(neighbor, q) < farthest_in_results 
                   OR results.size < efSearch:
                    candidates.push(neighbor)
                    results.push(neighbor)
    
    return top-K from results
```

### 4.2 搜索过程可视化

```
查询向量 q = ★

Layer 2:    A ●━━━━━━━━━━━━━● B          ← 从入口点 A 出发
                                 ↓ 下降到 Layer 1
Layer 1:    C ●━━━━● D ●━━━━━● E         ← 贪心走到 E（离★最近）
                      ↓ 下降到 Layer 0
Layer 0:    ...F●━━●G●━━●H●━━●I●━━●J...  ← 从 E 下方开始 beam search
                      ↑      ↑
                     找到 G 和 I 是最近的邻居
```

### 4.3 时间复杂度

- **Phase 1（上层导航）**：O(log N) 层 × O(log N) 步/层 = O(log²N)
- **Phase 2（底层搜索）**：O(efSearch × log(efSearch) × M)
- **总复杂度**：O(log²N + efSearch × M × log(efSearch))

注意 efSearch 是用户可调参数，与 N 无关，所以搜索复杂度对 N 是对数级别的。

## 五、核心参数详解

### 5.1 M（每层最大邻居数 / 图密度）

| M 值 | 图特征 | 内存开销 | 构建速度 | 搜索质量 |
|------|--------|---------|---------|---------|
| 8 | 稀疏图，连通性可能不足 | 低 | 快 | 较低，尤其在高维 |
| 16 | 默认推荐值，平衡选择 | 中等 | 中等 | 良好 |
| 32 | 稠密图，冗余连接多 | 较高 | 较慢 | 很好，适合高维 |
| 64 | 非常稠密 | 很高 | 很慢 | 边际收益递减 |

- **M 影响的是图的结构质量**：更大的 M 意味着每个节点有更多邻居，搜索时有更多路径可选，不容易陷入局部最优
- **M 同时影响内存**：每个节点在 Layer 0 存储 M×2 个邻居 ID（双向），上层存储 M 个。总内存 ≈ N × (M×2 + M×avg_layers) × sizeof(int)
- **经验法则**：768 维以下用 M=16；768~1536 维用 M=16~32；超过 1536 维考虑 M=32

### 5.2 efConstruction（构建时的搜索宽度）

- 构建索引时，每次插入新节点都要搜索其近邻来建边
- efConstruction 控制这个搜索的宽度
- **越大 → 构建出的图质量越好 → 搜索时 Recall 越高**
- **代价**：构建时间线性增长
- 推荐范围：100~500，默认 200 通常够用
- 这是一次性成本，建好后不影响搜索性能

### 5.3 efSearch（查询时的搜索宽度）

- **这是运行时最常调的参数**
- 控制 Layer 0 beam search 的候选集大小
- 必须 ≥ K（Top-K 的 K）
- **越大 → Recall 越高 → 延迟越大**
- 典型调参范围：K ~ 10×K

### 5.4 参数间的交互关系

```
efConstruction ↑ → 图质量 ↑ → 同等 efSearch 下 Recall ↑
M ↑ → 图连通性 ↑ → 同等 efSearch 下 Recall ↑，但单次搜索更慢
efSearch ↑ → Recall ↑ → Latency ↑（与 M 和 efConstruction 独立）
```

> 💡 **调参顺序建议**：先固定 M=16, efConstruction=200，只调 efSearch 找到满足 Recall 要求的值。如果 efSearch 需要设到很大才能达标，再考虑增大 M 或 efConstruction。

## 六、内存使用估算

### 6.1 原始向量存储

$$Memory_{vec} = N \times D \times sizeof(float)$$

### 6.2 图结构存储

Layer 0 每个节点存 2M 个邻居指针（双向边），上层平均每节点额外存约 M 个指针：

$$Memory_{graph} \approx N \times (2M + M \times \bar{l}) \times sizeof(int32)$$

其中 $\bar{l} \approx 1/\ln(M)$ 是平均层级数。

### 6.3 实际估算示例

| 参数 | 值 |
|------|-----|
| N | 10,000,000 (1千万) |
| D | 768 |
| M | 16 |

```
向量存储: 10M × 768 × 4B = 28.8 GB
图结构: 10M × (32 + 16×0.72) × 4B ≈ 1.74 GB
元数据: ~0.5 GB
─────────────────────────
总计: ≈ 31 GB
```

> ⚠️ HNSW 的主要缺点就是内存占用大。如果需要全量数据驻留内存，1 亿条 768 维向量大约需要 310 GB 内存，这对单机来说已经很吃力了。这也是为什么超大规模场景需要考虑 IVF-PQ 等压缩方案。

### 6.4 降低内存的方法

1. **量化**：float32 → float16（省一半），或用 SQ8（uint8量化）
2. **磁盘索引**：如 FAISS 的 OnDiskInvertedLists，向量放磁盘，图结构留内存
3. **分层部署**：热数据用 HNSW，冷数据用 IVF-PQ
4. **分布式**：Milvus/Qdrant 支持分片，将数据分散到多机

## 七、为什么 HNSW 是主流选择

### 7.1 与其他索引对比

| 维度 | HNSW | IVF-Flat | IVF-PQ | KD-Tree |
|------|------|----------|--------|---------|
| Recall@10 | ★★★★★ | ★★★★☆ | ★★★☆☆ | ★★☆☆☆ |
| 查询延迟 | ★★★★★ | ★★★★☆ | ★★★★★ | ★★☆☆☆ |
| 内存效率 | ★★☆☆☆ | ★★★☆☆ | ★★★★★ | ★★★☆☆ |
| 构建速度 | ★★☆☆☆ | ★★★★☆ | ★★★☆☆ | ★★★★☆ |
| 高维表现 | ★★★★★ | ★★★★☆ | ★★★★☆ | ★☆☆☆☆ |
| 动态更新 | ★★★★☆ | ★★☆☆☆ | ★★☆☆☆ | ★★☆☆☆ |
| 适用规模 | ≤1亿/机 | ≤10亿 | ≤100亿 | ≤100万 |

### 7.2 HNSW 胜出的原因

1. **Recall-Latency Pareto 最优**：在同等召回率下，HNSW 的延迟通常最低
2. **实现简单、调参直观**：主要就三个参数，且互相独立性好
3. **支持动态增删**：不需要像 IVF 那样重新训练聚类中心
4. **硬件友好**：图遍历模式对 CPU cache 相对友好（相比随机访问的哈希方法）
5. **生态完善**：所有主流向量数据库都以 HNSW 为默认索引

### 7.3 HNSW 不适用的场景

- **超大规模 + 内存受限**：10 亿级以上且单机内存不够 → IVF-PQ
- **极端低成本要求**：可以接受较低 Recall → PQ 量化
- **纯批量离线任务**：不需要低延迟 → Flat 暴力搜索保证精度

## 八、面试话术

### Q："HNSW 的原理是什么？为什么它是主流向量索引？"

> HNSW 本质上是一个多层的近邻图索引，灵感来自 Skip List。上层节点极少，形成全局导航骨架，搜索时从顶层入口点开始贪心向下逐层定位；到了最底层后做 beam search 精确定位 Top-K。这种结构把搜索复杂度从 O(N) 降到了 O(log²N + efSearch·M)，实现了毫秒级的高精度检索。
>
> 它成为主流的原因有三点：第一，在相同召回率下延迟最低，Recall-Latency 的 Pareto 前沿最优；第二，支持动态增删，不像 IVF 需要重训聚类中心；第三，调参简单直观——M 控图密度、efConstruction 控建图质量、efSearch 控查询精度，三者相对独立，工程师容易理解和调节。唯一的缺点是内存占用大，因为要存完整的图和原始向量，所以在超大规模且内存受限时会考虑 IVF-PQ 替代。

### Q："HNSW 的 M 和 efSearch 分别影响什么？怎么调？"

> M 是每个节点的最大邻居数，决定了图的密度和连通性。M 越大，图中路径越多，搜索越不容易陷入局部最优，但同时内存占用和单次搜索的计算量也增加。efSearch 是查询时 beam search 的候选集宽度，直接控制精度-延迟权衡：越大召回越高但延迟也越高。
>
> 调参策略：首先固定 M=16、efConstruction=200 作为起点。然后只调 efSearch，从 K 开始逐步翻倍（比如 10→20→40→80→160），画出 Recall-QPS 曲线。找到满足业务 Recall 要求的最小 efSearch。如果在 efSearch 调到很大时 Recall 仍然不够，说明图的质量本身有问题，这时再考虑增大 M 到 32 或提高 efConstruction 到 400 重新建索引。efConstruction 是一次性的构建成本，增大它只会让建索引变慢，不影响查询延迟，所以如果愿意等构建时间，提高 efConstruction 是最"免费"的提升方式。
