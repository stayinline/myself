# HNSW 索引实战

## 一、使用 FAISS 构建 HNSW 索引

### 1.1 基础构建

```python
import faiss
import numpy as np
import time

# ========== 准备数据集 ==========
D = 768          # 向量维度
N = 1_000_000    # 数据量
NQ = 200         # 查询数
K = 10           # Top-K

np.random.seed(42)
xb = np.random.randn(N, D).astype(np.float32)
xb /= np.linalg.norm(xb, axis=1, keepdims=True)  # 归一化

xq = np.random.randn(NQ, D).astype(np.float32)
xq /= np.linalg.norm(xq, axis=1, keepdims=True)

# ========== 构建 Ground Truth ==========
print("Building ground truth (Flat IP)...")
gt_index = faiss.IndexFlatIP(D)
gt_index.add(xb)
_, I_gt = gt_index.search(xq, K)
print(f"Ground truth ready: {I_gt.shape}")

# ========== 构建 HNSW 索引 ==========
M = 16
efConstruction = 200

print(f"\nBuilding HNSW (M={M}, efConstruction={efConstruction})...")
index = faiss.IndexHNSWFlat(D, M, faiss.METRIC_INNER_PRODUCT)
index.hnsw.efConstruction = efConstruction

t0 = time.perf_counter()
index.add(xb)
build_time = time.perf_counter() - t0
print(f"Build time: {build_time:.1f}s ({N/build_time:.0f} vectors/s)")
```

## 二、efSearch 扫描实验

### 2.1 完整扫描脚本

```python
def sweep_efsearch(index, xq, I_gt, K=10, ef_values=[16, 32, 64, 128, 256, 512]):
    """扫描不同 efSearch 值，返回 Recall/QPS/Latency"""
    NQ = xq.shape[0]
    results = []
    
    for ef in ef_values:
        index.hnsw.efSearch = ef
        
        # 预热
        index.search(xq[:10], K)
        
        # 正式测试（多次取平均）
        times = []
        recalls = []
        n_runs = 5
        
        for _ in range(n_runs):
            t0 = time.perf_counter()
            _, I_ann = index.search(xq, K)
            dt = time.perf_counter() - t0
            times.append(dt)
            
            recall = np.mean([
                len(set(I_gt[i]) & set(I_ann[i])) / K 
                for i in range(NQ)
            ])
            recalls.append(recall)
        
        avg_time = np.mean(times)
        avg_recall = np.mean(recalls)
        
        results.append({
            "efSearch": ef,
            "recall@K": round(avg_recall, 4),
            "avg_latency_ms": round(avg_time / NQ * 1000, 3),
            "qps": round(NQ / avg_time, 1),
            "p99_latency_ms": round(np.percentile(times, 99) / NQ * 1000, 3)
        })
        
        print(f"  efSearch={ef:>3d} | Recall@{K}={avg_recall:.4f} | "
              f"Avg={avg_time/NQ*1000:.2f}ms | QPS={NQ/avg_time:.0f}")
    
    return results

# 运行扫描
print("\n=== efSearch Sweep ===")
results = sweep_efsearch(index, xq, I_gt, K=10)
```

## 三、M 参数对比实验

### 3.1 固定 efConstruction，扫描不同 M

```python
M_values = [8, 16, 32, 48]
efC = 200
efS_target = 64  # 固定 efSearch 来比较图质量

print(f"{'M':>4s} | {'Build(s)':>8s} | {'Mem(MB)':>8s} | {'Recall@10':>10s} | {'Lat(ms)':>8s}")
print("-" * 60)

for m in M_values:
    idx = faiss.IndexHNSWFlat(D, m, faiss.METRIC_INNER_PRODUCT)
    idx.hnsw.efConstruction = efC
    
    t0 = time.perf_counter()
    idx.add(xb)
    build_t = time.perf_counter() - t0
    
    # 估算内存（FAISS 不提供直接接口，手动算）
    vec_mem = N * D * 4  # float32
    graph_mem = N * (2*m + m) * 4  # Layer0双向 + 上层单向, int32
    total_mb = (vec_mem + graph_mem) / 1024 / 1024
    
    idx.hnsw.efSearch = efS_target
    t0 = time.perf_counter()
    _, I_ann = idx.search(xq, K)
    search_t = time.perf_counter() - t0
    
    recall = np.mean([
        len(set(I_gt[i]) & set(I_ann[i])) / K 
        for i in range(NQ)
    ])
    
    print(f"{m:>4d} | {build_t:>8.1f} | {total_mb:>8.0f} | {recall:>10.4f} | {search_t/NQ*1000:>8.3f}")
```

### 3.2 预期结果分析

```
   M | Build(s) |  Mem(MB) | Recall@10 |  Lat(ms)
------------------------------------------------------------
   8 |     45.2 |     2920 |    0.9234 |    0.180
  16 |     78.6 |     3095 |    0.9703 |    0.260
  32 |    142.3 |     3445 |    0.9856 |    0.420
  48 |    215.8 |     3795 |    0.9891 |    0.580
```

**观察**：
- M=8 → 16：Recall 提升显著（+4.7%），内存增加不大
- M=16 → 32：Recall 继续提升（+1.5%），但构建时间和搜索延迟都翻倍
- M=32 → 48：边际收益很小（+0.35%），通常不值得

> 💡 **经验法则**：对于 768 维向量，M=16 是最佳性价比。只有在极高召回要求或超高维（>1536）时才考虑 M=32。

## 四、HNSW 适用场景判断

### 4.1 决策矩阵

```
你的数据规模是多少？
├── < 100万 → Flat Index 就够了，不需要 ANN
├── 100万 ~ 5000万
│   ├── 内存充足（≥ 数据量×30B）→ ✅ HNSW（首选）
│   └── 内存紧张 → IVF-PQ 或 SQ量化+HNSW
├── 5000万 ~ 5亿
│   ├── 单机大内存（≥256GB）→ HNSW + SQ8 量化
│   ├── 分布式 → Milvus/Qdrant 分片 + HNSW
│   └── 内存严格受限 → IVF-PQ
└── > 5亿
    ├── 可以接受稍低 Recall → IVF-PQ
    ├── 必须高 Recall → DiskANN / FreshDiskANN
    └── 混合策略 → 热数据 HNSW + 冷数据 IVF-PQ
```

### 4.2 HNSW 适用清单

✅ **适合 HNSW 的场景**：
- 千万级到亿级数据
- 需要高召回率（≥ 0.95）
- 低延迟要求（< 10ms）
- 内存预算充足
- 需要频繁增删更新
- RAG、语义搜索、实时推荐

❌ **不适合 HNSW 的场景**：
- 十亿级以上且单机内存不够
- 对内存成本极度敏感
- 纯离线批处理（不在乎延迟）
- 数据几乎不变且追求极致压缩

## 五、Milvus 中的 HNSW 实践

### 5.1 创建 HNSW 索引

```python
from pymilvus import connections, Collection, FieldSchema, CollectionSchema, DataType
import numpy as np

connections.connect(host="localhost", port="19530")

# 创建 Collection
fields = [
    FieldSchema(name="id", dtype=DataType.INT64, is_primary=True, auto_id=True),
    FieldSchema(name="embedding", dtype=DataType.FLOAT_VECTOR, dim=768),
    FieldSchema(name="text", dtype=DataType.VARCHAR, max_length=65535),
]
schema = CollectionSchema(fields, description="HNSW demo")
collection = Collection("hnsw_demo", schema)

# 插入数据
texts = ["这是第一条文档", "这是第二条文档"] * 500000
embeddings = np.random.randn(1000000, 768).astype(np.float32).tolist()
collection.insert([embeddings, texts])

# 创建 HNSW 索引
index_params = {
    "index_type": "HNSW",
    "metric_type": "IP",           # 归一化向量用 IP
    "params": {
        "M": 16,                   # 邻居数
        "efConstruction": 200      # 构建时搜索宽度
    }
}
collection.create_index("embedding", index_params)
print("Index created, building in background...")

# 加载到内存
collection.load()
```

### 5.2 查询时指定 efSearch

```python
# efSearch 在搜索时动态指定，不需要重建索引
search_params = {"metric_type": "IP", "params": {"ef": 128}}

query_vec = np.random.randn(1, 768).astype(np.float32).tolist()
results = collection.search(
    data=query_vec,
    anns_field="embedding",
    param=search_params,
    limit=10,
    output_fields=["text"]
)

for hit in results[0]:
    print(f"ID={hit.id}, Score={hit.score:.4f}, Text={hit.entity.get('text')}")
```

> ⚠️ Milvus 中 efSearch 的参数名是 `"ef"` 而不是 `"efSearch"`，这是常见的踩坑点。

## 六、面试话术

### Q："HNSW 的 M/efSearch 怎么调？"

> 我的调参方法论是分步控制变量法。第一步，固定 M=16、efConstruction=200 作为基线，这两个值是社区验证过的默认最优起点。第二步，只调 efSearch，从 K 开始逐步翻倍——比如 K=10 就从 10、20、40、80、160、320 依次测试。每个值跑一组标准查询集，计算 Recall@K 和 P99 延迟，画出 Recall-QPS 曲线。第三步，找到满足业务 Recall 阈值的最小 efSearch。比如 RAG 场景要求 Recall@10 ≥ 0.95，如果 efSearch=64 就能达到 0.97，那就没必要用 128。第四步，如果在 efSearch 调到很大（比如 512）时 Recall 仍不达标，说明图的底层质量不够，这时再考虑增大 M 到 32 或提高 efConstruction 到 400 重新建索引。efConstruction 是一次性成本，提高它只会让建索引变慢，不影响查询延迟，所以是最"便宜"的提升方式。整个过程一定要有标准化的评测数据集和自动化脚本，不能凭感觉调。

### Q："HNSW 在你的项目中实际表现怎么样？有没有遇到什么问题？"

> 我们的 RAG 系统用的 HNSW，大约 2000 万条 768 维文档向量，M=16，efConstruction=200，efSearch=64。实测 Recall@10=0.972，P99 延迟 3.2ms，QPS 约 3000。主要遇到的问题是内存占用——2000 万条全精度向量加图结构大约占了 65GB 内存。后来我们做了两件事优化：一是把向量从 float32 降到 float16（SQ8 也行），内存减半到 35GB，Recall 只掉了 0.3%；二是用了分区策略，按时间把数据分成热/温/冷三个区，只有最近一个月的热数据用 HNSW，历史数据用 IVF-PQ，整体内存降到了 20GB 以内，综合 Recall 仍然维持在 0.96 以上。

### Q："什么时候不用 HNSW？"

> 三种情况。第一，数据量特别小（百万以下），直接用 Flat Index，简单可靠无精度损失。第二，数据量超大且内存受限，比如十亿级向量但只有几百 GB 内存，这时候 HNSW 放不下全量数据，应该选 IVF-PQ，通过量化压缩到原来的 1/10~1/30。第三，纯离线批处理场景，不在乎延迟只在乎精度，那直接用暴力搜索保证 100% Recall。另外如果数据更新非常频繁且对实时性要求极高，HNSW 虽然支持增量插入，但大量删除后图结构会退化，可能需要定期重建，这种情况也可以考虑支持 streaming upsert 更好的索引方案。

## 七、常见问题排查

| 问题 | 可能原因 | 解决方案 |
|------|---------|---------|
| Recall 远低于预期 | 向量未归一化却用了 IP | 确认归一化或改用 L2 |
| Recall 不稳定 | efSearch 太小 | 增大 efSearch |
| 构建太慢 | efConstruction 太大或 M 太大 | 降低参数或用多线程 |
| 内存 OOM | 数据量 × 维度 × 4B 超出可用内存 | 量化 / 分片 / 磁盘索引 |
| 搜索延迟波动大 | GC / 内存碎片 / CPU 争抢 | 绑核 / hugepage / 隔离部署 |
| Milvus 搜索报错 | ef 参数名写错 | 用 `"ef"` 不是 `"efSearch"` |
