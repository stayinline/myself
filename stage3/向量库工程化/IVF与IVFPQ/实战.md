# IVF 与 IVF-PQ 实战

## 一、FAISS 构建 IVF_FLAT 索引

### 1.1 基础代码

```python
import faiss
import numpy as np
import time

# ========== 准备数据 ==========
D = 768
N = 5_000_000    # 500万向量
NQ = 200
K = 10

np.random.seed(42)
xb = np.random.randn(N, D).astype(np.float32)
xb /= np.linalg.norm(xb, axis=1, keepdims=True)

xq = np.random.randn(NQ, D).astype(np.float32)
xq /= np.linalg.norm(xq, axis=1, keepdims=True)

# Ground Truth
print("Building ground truth...")
gt = faiss.IndexFlatIP(D)
gt.add(xb)
_, I_gt = gt.search(xq, K)

# ========== IVF_FLAT ==========
nlist = 4096  # 聚类数，经验值 ≈ sqrt(N) ~ 2236，取 4096

quantizer = faiss.IndexFlatIP(D)  # 用于计算查询到centroid距离的索引
index_ivf_flat = faiss.IndexIVFFlat(quantizer, D, nlist, faiss.METRIC_INNER_PRODUCT)

# 训练（必须！）
print(f"Training IVF with nlist={nlist}...")
t0 = time.perf_counter()
index_ivf_flat.train(xb)  # 用全量或采样数据训练
train_time = time.perf_counter() - t0
print(f"Train time: {train_time:.1f}s")

# 添加数据
print("Adding vectors...")
t0 = time.perf_counter()
index_ivf_flat.add(xb)
add_time = time.perf_counter() - t0
print(f"Add time: {add_time:.1f}s")
```

### 1.2 nprobe 扫描

```python
def sweep_nprobe(index, xq, I_gt, K=10, nprobe_values=[1, 4, 8, 16, 32, 64, 128]):
    NQ = xq.shape[0]
    results = []
    
    for nprobe in nprobe_values:
        index.nprobe = nprobe
        
        # 预热
        index.search(xq[:10], K)
        
        times = []
        recalls = []
        for _ in range(5):
            t0 = time.perf_counter()
            _, I_ann = index.search(xq, K)
            dt = time.perf_counter() - t0
            times.append(dt)
            
            recall = np.mean([
                len(set(I_gt[i]) & set(I_ann[i])) / K
                for i in range(NQ)
            ])
            recalls.append(recall)
        
        avg_t = np.mean(times)
        avg_r = np.mean(recalls)
        results.append({
            "nprobe": nprobe,
            "recall": round(avg_r, 4),
            "latency_ms": round(avg_t / NQ * 1000, 3),
            "qps": round(NQ / avg_t, 1)
        })
        print(f"  nprobe={nprobe:>3d} | Recall@{K}={avg_r:.4f} | "
              f"Lat={avg_t/NQ*1000:.2f}ms | QPS={NQ/avg_t:.0f}")
    
    return results

print("\n=== IVF_FLAT nprobe Sweep ===")
ivf_flat_results = sweep_nprobe(index_ivf_flat, xq, I_gt)
```

## 二、FAISS 构建 IVF_PQ 索引

### 2.1 基础代码

```python
m = 96       # PQ 子空间数（必须整除 D: 768/96=8 ✓）
nbits = 8    # 每个子空间的编码位数 → 256 个 codeword

quantizer_pq = faiss.IndexFlatIP(D)
index_ivf_pq = faiss.IndexIVFPQ(
    quantizer_pq, D, nlist, m, nbits, faiss.METRIC_INNER_PRODUCT
)

# 训练
print(f"\nTraining IVF_PQ (nlist={nlist}, m={m}, nbits={nbits})...")
t0 = time.perf_counter()
index_ivf_pq.train(xb)
print(f"Train time: {time.perf_counter()-t0:.1f}s")

# 添加
t0 = time.perf_counter()
index_ivf_pq.add(xb)
print(f"Add time: {time.perf_counter()-t0:.1f}s")
```

### 2.2 OPQ 变体（推荐）

```python
# OPQ 通过旋转矩阵优化量化精度
opq_m = 96
index_opq = faiss.IndexIVFPQ(
    faiss.IndexFlatIP(D), D, nlist, opq_m, nbits, faiss.METRIC_INNER_PRODUCT
)

# 用 OPQMatrix 包装
opq_matrix = faiss.OPQMatrix(D, opq_m)
opq_matrix.niter = 50  # 迭代次数，越多越精确但越慢
index_opq_wrapper = faiss.IndexPreTransform(opq_matrix, index_opq)

print("Training IVF-OPQ...")
t0 = time.perf_counter()
index_opq_wrapper.train(xb)
print(f"Train time: {time.perf_counter()-t0:.1f}s")

index_opq_wrapper.add(xb)
```

### 2.3 内存对比

```python
def estimate_memory(index_type, N, D, nlist, m=None):
    """估算不同索引的内存占用"""
    vec_bytes = N * D * 4  # float32
    
    if index_type == "Flat":
        total = vec_bytes
    elif index_type == "IVF_Flat":
        centroid_bytes = nlist * D * 4
        ids_bytes = N * 8  # int64 IDs
        total = vec_bytes + centroid_bytes + ids_bytes
    elif index_type == "IVF_PQ":
        pq_code_bytes = N * m  # m bytes per vector (uint8 codes)
        codebook_bytes = m * 256 * (D // m) * 4
        centroid_bytes = nlist * D * 4
        ids_bytes = N * 8
        total = pq_code_bytes + codebook_bytes + centroid_bytes + ids_bytes
    elif index_type == "IVF_SQ8":
        sq_bytes = N * D  # uint8 per dimension
        centroid_bytes = nlist * D * 4
        ids_bytes = N * 8
        total = sq_bytes + centroid_bytes + ids_bytes
    
    return total / 1024 / 1024 / 1024  # GB

print("Memory Estimation (N=5M, D=768):")
for idx_type in ["Flat", "IVF_Flat", "IVF_SQ8", "IVF_PQ"]:
    mem = estimate_memory(idx_type, N, D, nlist, m=96)
    print(f"  {idx_type:>12s}: {mem:.2f} GB")
```

预期输出：
```
Memory Estimation (N=5M, D=768):
          Flat: 14.31 GB
     IVF_Flat: 14.37 GB
      IVF_SQ8:  3.66 GB
       IVF_PQ:  0.52 GB
```

> 💡 **关键发现**：IVF_PQ 的内存仅为 Flat 的 **1/28**！这就是它在超大规模场景中的核心价值。

## 五、nlist 选择经验

### 5.1 nlist 的经验公式

$$nlist \approx \sqrt{N} \quad \text{或} \quad nlist \approx 4\sqrt{N}$$

| N (数据量) | sqrt(N) | 推荐 nlist | 说明 |
|-----------|---------|-----------|------|
| 100万 | 1000 | 1024 | 小数据集 |
| 1000万 | 3162 | 4096 | 中等规模 |
| 1亿 | 10000 | 8192~16384 | 大规模 |
| 10亿 | 31623 | 32768~65536 | 超大规模 |

### 5.2 nlist 的影响

- **太小**：每个聚类内向量太多，nprobe 个聚类的搜索量仍然很大
- **太大**：聚类太细，nprobe 需要设得更大才能覆盖足够候选；训练也更慢
- **原则**：让每个聚类平均包含 N/nlist ≈ 几百到几千条向量

## 六、面试话术

### Q："HNSW vs IVF vs IVF-PQ 怎么选？"

> 选型主要看三个维度：数据规模、内存预算、Recall 要求。我的决策框架是这样的：
>
> 如果数据在千万级以内且内存充足，首选 HNSW——它的 Recall-Latency Pareto 最优，支持动态更新，调参也简单。我们的 RAG 系统 2000 万条文档就是用 HNSW，M=16, efSearch=64，Recall@10=0.97，P99 延迟 3ms。
>
> 如果数据到了亿级甚至十亿级，单机内存放不下 HNSW 的全量数据和图结构，就选 IVF-OPQ。OPQ 比标准 PQ 精度高 5~10%，内存压缩比可以到 20~30 倍。十亿条 768 维向量用 IVF-OPQ(m=96) 大约只需要 100GB 内存。代价是需要训练步骤，且不支持高效的增量更新。
>
> 如果是纯离线场景或者数据量很小，直接用 Flat Index，简单可靠无精度损失。
>
> 实际项目中我们还会做混合策略：热数据（最近一个月）用 HNSW 保证低延迟高召回，冷数据用 IVF-PQ 省内存，最后结果合并排序。这样兼顾了性能和成本。

### Q："IVF-PQ 的训练怎么做？训练数据有什么要求？"

> IVF-PQ 的训练分两步。第一步是 IVF 部分的 K-Means 聚类，得到 nlist 个 centroid。第二步是 PQ 部分，对每个子空间独立做 K-Means 得到码本。如果用 OPQ 还要额外训练旋转矩阵。
>
> 训练数据的要求：数量上至少 max(1000×256, 40×nlist)，实践中我一般用全量数据的 10%~20% 做随机采样。质量上必须有代表性——如果数据有明显的时间漂移或类别不均衡，采样时要分层抽样，不能纯随机。训练是一次性成本，通常几分钟到几十分钟，之后索引可以反复使用。需要注意的是，如果后续数据分布发生了显著变化，应该重新训练索引，否则聚类中心和码本都会过时，导致 Recall 下降。

### Q："nprobe 和 efSearch 有什么异同？"

> 它们都是控制"搜索范围"的参数，越大搜索越精确但越慢。区别在于作用的索引类型和机制不同。efSearch 是 HNSW 的参数，控制底层图搜索时 beam search 的候选集宽度，它影响的是图遍历的深度和广度。nprobe 是 IVF 的参数，控制查询时检查多少个聚类，它影响的是倒排列表的扫描范围。
>
> 从调参体验上看，efSearch 对 Recall 的提升更平滑连续，而 nprobe 可能有阶梯效应——因为聚类边界是离散的，增加一个 nprobe 可能突然多覆盖一大块区域。另外 efSearch 只影响搜索延迟不影响内存，nprobe 同理。两者都可以在线动态调整，不需要重建索引。

## 七、常见坑点

| 问题 | 原因 | 解决 |
|------|------|------|
| `train()` 报错数据不够 | 训练样本少于最低要求 | 增大采样量或减小 nlist/m |
| PQ Recall 很低 | m 选择不当（不整除 D） | 确保 D % m == 0 |
| IVF 搜索很慢 | nprobe 设太大 | 降低 nprobe 或增大 nlist |
| 训练后添加数据 Recall 差 | 新数据分布偏移 | 重新训练或用自适应方法 |
| OPQ 训练特别慢 | niter 设太大 | 降低 niter 或用 GPU 训练 |
| FAISS METRIC_INNER_PRODUCT 结果不对 | 向量未归一化 | 先归一化或改用 L2 |
