# IVF 与 IVF-PQ 原理

## 一、IVF（Inverted File Index）核心思想

### 1.1 从倒排索引到向量检索

在传统搜索引擎中，倒排索引将文档按词项分组：给定一个查询词，只需搜索包含该词的文档列表，而不需要遍历所有文档。

IVF 将这个思想迁移到向量空间：

> **用 K-Means 将整个向量空间划分为 nlist 个聚类（Voronoi cells），每个聚类的中心点称为 centroid。查询时只搜索距离最近的 nprobe 个聚类内的向量。**

```
原始空间（Flat）:           IVF 划分后:
┌─────────────────┐        ┌──────┬──────┬──────┐
│ ● ●   ●  ●     │        │ C1   │ C2   │ C3   │
│   ●  ★query     │        │ ●●   │★●●   │  ●   │
│ ●    ●   ●     │   →    ├──────┼──────┼──────┤
│  ●  ●     ●    │        │ C4   │ C5   │ C6   │
│    ●  ● ●      │        │ ●●   │ ●●   │ ●●   │
└─────────────────┘        └──────┴──────┴──────┘
                           查询 ★ 只需搜索 C2, C5 (nprobe=2)
```

### 1.2 IVF 的两个阶段

**训练阶段（Train）**：
1. 从数据集中采样（通常 10%~100%）
2. 运行 K-Means 聚类，得到 nlist 个 centroid
3. 将所有向量分配到最近的 centroid 对应的列表中

**搜索阶段（Search）**：
1. 计算查询向量与所有 nlist 个 centroid 的距离
2. 选取最近的 nprobe 个聚类
3. 在这 nprobe 个聚类的向量列表中做精确距离计算
4. 返回 Top-K

### 1.3 复杂度分析

| 操作 | Flat | IVF |
|------|------|-----|
| 训练 | 无 | O(nlist × iterations × sample_size × D) |
| 搜索 | O(N × D) | O(nlist × D + nprobe × N/nlist × D) |
| 当 nprobe << nlist | — | ≈ O(nprobe × N/nlist × D) |

当 nlist=1024, nprobe=8 时，搜索量约为全量的 8/1024 ≈ 0.78%，加速比约 128 倍。

## 二、Product Quantization（PQ）核心原理

### 2.1 为什么需要量化

即使 IVF 将搜索范围缩小到了 nprobe 个聚类，每个聚类内仍然可能有百万级向量。如果这些向量以 float32 存储，内存和带宽开销依然巨大。

PQ 的目标：**将高维向量压缩为短码（short code），在压缩空间中近似计算距离**。

### 2.2 PQ 的分段量化过程

```
原始向量 (D=768):
[████████████████████████████████████████████████]

Step 1: 分成 m 个子空间 (m=96, 每段 d*=8):
[████████][████████][████████]...[████████]
  sub-0     sub-1     sub-2       sub-95

Step 2: 每个子空间独立做 K-Means (k=256 → 8bit):
  sub-0 → codebook_0 (256个centroids, 每个8维)
  sub-1 → codebook_1 (256个centroids, 每个8维)
  ...
  sub-95 → codebook_95

Step 3: 原始向量被编码为 m 个 8-bit 整数:
[42, 187, 3, ..., 91]  ← 这就是 PQ code (96 bytes)

对比: float32 = 768×4 = 3072 bytes → PQ = 96 bytes
压缩比: 32x!
```

### 2.3 距离近似计算（ADC - Asymmetric Distance Computation）

PQ 的精妙之处在于不需要解码就能计算近似距离：

1. **离线**：对查询向量 q 的每个子空间段，预计算它到该子空间所有 256 个 centroid 的距离，得到一个 m × 256 的查找表（lookup table）
2. **在线**：对于数据库中的每个 PQ code `[c₀, c₁, ..., cₘ₋₁]`，近似距离 = `table[0][c₀] + table[1][c₁] + ... + table[m-1][cₘ₋₁]`

```
查询向量 q 拆分为 [q₀, q₁, ..., q₉₅]

Lookup Table (预计算一次):
         centroid_0  centroid_1  ... centroid_255
sub-0:   0.12        0.87            0.34
sub-1:   0.45        0.23            0.91
...
sub-95:  0.67        0.11            0.56

数据库向量的 PQ code = [42, 187, 3, ..., 91]
近似距离 = table[0][42] + table[1][187] + table[2][3] + ... + table[95][91]
         = O(m) 次查表加法，远快于 O(D) 次浮点乘法
```

### 2.4 PQ 的精度损失来源

1. **量化误差**：每个子空间只有 256 个离散代表点，连续向量被映射到最近的离散点
2. **子空间独立性假设**：PQ 假设各子空间相互独立，但实际向量的维度间存在相关性
3. **分段越细精度越高**：m 越大（每段越短），量化粒度越细，但码本也越大

## 三、Codebook（码本）详解

### 3.1 什么是码本

码本就是每个子空间的 K-Means 聚类中心集合：

$$Codebook_j = \{c_{j,0}, c_{j,1}, ..., c_{j,k-1}\}, \quad j = 0, 1, ..., m-1$$

- 共有 m 个码本
- 每个码本有 k=256 个 codeword（对应 8-bit 编码）
- 每个 codeword 是 d* = D/m 维的浮点向量

### 3.2 码本的存储开销

$$Memory_{codebooks} = m \times k \times d^* \times sizeof(float) = m \times 256 \times (D/m) \times 4 = D \times 1024 \text{ bytes}$$

对于 D=768：码本总大小 = 768 × 1024 = 768 KB，几乎可以忽略不计。

### 3.3 码本的训练

码本是通过对训练数据做 K-Means 得到的。训练质量直接影响 PQ 的精度：
- 训练数据要有代表性（覆盖数据的真实分布）
- 通常需要至少 max(1000×k, nlist×40) 条训练样本
- 训练是一次性的，之后索引可以反复使用

## 四、OPQ（Optimized Product Quantization）

### 4.1 PQ 的问题

标准 PQ 的子空间划分是固定的（前 d* 维、接下来 d* 维...），但实际数据的方差可能不均匀分布在各个维度上。某些子空间包含了大量信息，另一些几乎是噪声。

### 4.2 OPQ 的解决思路

在 PQ 之前加一个**旋转矩阵 R**，对向量做线性变换，使得变换后的各子空间的量化误差之和最小：

$$OPQ(x) = PQ(R \cdot x)$$

旋转矩阵 R 通过优化算法迭代求解，目标是使总的量化失真最小化。

### 4.3 OPQ vs PQ

| 指标 | PQ | OPQ |
|------|----|----|
| 精度 | 基准 | 提升 2%~10% Recall |
| 训练时间 | 基准 | 增加 2~5 倍 |
| 搜索速度 | 基准 | 相同（多一次矩阵乘法，可忽略） |
| 推荐度 | 基线 | ✅ 生产环境首选 |

> 💡 实践中几乎总是用 OPQ 替代 PQ，几乎没有 downside。FAISS 中对应 `OPQM` 参数。

## 五、SQ（Scalar Quantization）

### 5.1 原理

SQ 不对向量分段，而是对每个维度独立做标量量化：

- **SQ8**：float32 → uint8，每个维度映射到 [0, 255]，记录全局 min/max
- **SQ4**：float32 → uint4，更激进压缩
- **SQfp16**：float32 → float16，精度损失最小

### 5.2 SQ vs PQ

| 特性 | SQ8 | PQ |
|------|-----|-----|
| 压缩比 | 4x | 8x~64x |
| 精度损失 | 很小 | 中等 |
| 是否需要训练 | ❌ | ✅ |
| 实现复杂度 | 简单 | 较复杂 |
| 适用场景 | 内存优化但不想牺牲太多精度 | 极致压缩 |

### 5.3 SQ 的典型用法

SQ 经常作为 HNSW 或 IVF 的附加量化层：
- `HNSW_SQ8`：HNSW 图结构 + SQ8 量化向量，内存减半
- `IVF_SQ8`：IVF 倒排 + SQ8 量化，兼顾速度和内存

## 六、IVF-PQ 组合架构

### 6.1 完整流程

```
┌──────────────────────────────────────────────────────┐
│                  IVF-PQ Index                        │
│                                                      │
│  Train Phase:                                        │
│  1. K-Means → nlist centroids (IVF part)             │
│  2. PQ train → m codebooks (PQ part)                 │
│  3. (Optional) OPQ rotation matrix                   │
│                                                      │
│  Add Phase:                                          │
│  4. Assign each vector to nearest centroid           │
│  5. Encode residual (vec - centroid) with PQ         │
│  6. Store PQ code in inverted list                   │
│                                                      │
│  Search Phase:                                       │
│  7. Find nprobe nearest centroids                    │
│  8. Precompute lookup tables for query               │
│  9. Scan inverted lists with ADC                     │
│  10. Return Top-K                                    │
└──────────────────────────────────────────────────────┘
```

### 6.2 Residual Encoding（残差编码）

IVF-PQ 不是直接对原始向量做 PQ，而是对**残差**做 PQ：

$$residual = x - centroid_{assigned}$$

原因：同一聚类内的向量彼此接近，它们的残差分布在更小的范围内，PQ 量化这个更小范围的向量时精度更高。

### 6.3 训练步骤的重要性

IVF-PQ **必须先训练再使用**，这是与 HNSW 的关键区别：

```python
# FAISS 中的训练
index = faiss.IndexIVFPQ(faiss.IndexFlatL2(D), D, nlist, m, nbits)
index.train(training_data)  # ← 必须！否则 add() 会报错
index.add(database_vectors)
```

训练数据要求：
- 数量：至少 `max(1000 * 2^nbits, 40 * nlist)` 条
- 代表性：应该反映真实数据的分布
- 如果全量数据太大，随机采样即可

## 八、面试话术

### Q："IVF 和 PQ 分别是什么？为什么要组合使用？"

> IVF 是用 K-Means 把向量空间分成 nlist 个聚类，查询时只搜最近的 nprobe 个聚类，从而把搜索范围从全量缩小到 nprobe/nlist 的比例。它解决了"搜索范围大"的问题，但每个聚类内的向量仍然是原始的 float32，内存没有压缩。
>
> PQ 是把高维向量切成 m 段，每段独立做 K-Means 量化成 8-bit 编码。这样一条 768 维 float32 向量（3072 bytes）可以被压缩到 96 bytes（m=96 时），压缩比 32 倍。搜索时用查表法（ADC）在压缩空间直接算近似距离，不需要解码。
>
> 两者组合就是 IVF-PQ：先用 IVF 缩小搜索范围，再用 PQ 压缩每个聚类内的向量。这样既减少了搜索量又压缩了内存，特别适合十亿级数据且内存受限的场景。代价是精度有损——PQ 的量化误差会导致 Recall 下降，但通过 OPQ 旋转和优化 nprobe/m 参数，通常可以把 Recall@10 维持在 0.90 以上。

### Q："10 亿向量、内存有限，你选什么索引？"

> 十亿级 768 维向量如果用 HNSW 全精度存储，光向量就要 2.8 TB，加上图结构超过 3 TB，单机根本放不下。我会选 IVF-OPQ。具体来说：nlist 设为 sqrt(N) ≈ 32000 左右取整到 4096 或 8192；OPQ 的 m 设为 96 或 128（保证每段整除维度）；nbits=8。这样每条向量压缩到 96~128 bytes，十亿条约 96~128 GB，单机 256GB 内存可以容纳。搜索时 nprobe 从 32 开始调，根据 Recall 要求逐步增大。如果需要更高精度，可以用两级策略：IVF-OPQ 粗筛出 Top-100，再用原始向量精排 Top-10（前提是保留一份原始向量在磁盘上）。另外也可以考虑 DiskANN 方案，把图和向量放 NVMe SSD 上，内存只放 PQ 码本和导航结构，能在更小内存下支持十亿级高召回检索。

### Q："PQ 的精度损失怎么评估和控制？"

> PQ 的精度损失主要来自量化误差。控制手段有三个层面。第一，选合适的 m：m 越大（分段越细）精度越高，但码也越长。一般 m=D/8 是个好起点（768 维用 m=96）。第二，用 OPQ 代替 PQ：通过旋转矩阵让各子空间的方差更均匀，实测能提升 2~10% 的 Recall。第三，残差编码：IVF-PQ 中对 (vector - centroid) 而非原始向量做 PQ，因为残差的值域更小，量化精度更高。评估方面，一定要用 Flat Index 的结果作为 ground truth 来算 Recall，不能用另一个 ANN 索引当标准。同时关注不同 K 值下的 Recall 曲线——有时候 Recall@100 还行但 Recall@10 很差，说明排序质量有问题。
