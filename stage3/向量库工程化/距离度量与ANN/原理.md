# 距离度量与 ANN 原理

## 一、为什么需要向量相似度搜索

在传统数据库中，我们通过精确匹配（`WHERE id = ?`）或范围查询来检索数据。但在语义搜索、推荐系统、RAG 等场景中，我们需要回答的问题是：

> "哪些数据和我的查询**最相似**？"

这需要将非结构化数据（文本、图片、音频）通过 Embedding 模型映射到高维向量空间，然后在该空间中寻找最近邻。

---

## 二、三种核心距离度量的数学本质

### 2.1 欧氏距离（Euclidean / L2 Distance）

$$d_{L2}(a, b) = \sqrt{\sum_{i=1}^{D}(a_i - b_i)^2}$$

- **几何含义**：两点之间的直线距离
- **值域**：$[0, +\infty)$，越小越相似
- **特点**：对向量的绝对大小敏感；如果两个向量方向相同但模长不同，L2 距离不为零
- **适用场景**：图像特征（未归一化）、物理空间坐标、需要同时考虑方向和幅值的场景

### 2.2 内积（Inner Product / Dot Product）

$$IP(a, b) = \sum_{i=1}^{D} a_i \cdot b_i = \|a\| \cdot \|b\| \cdot \cos(\theta)$$

- **几何含义**：一个向量在另一个向量方向上的投影长度 × 被投影向量的模
- **值域**：$(-\infty, +\infty)$，越大越相似（注意方向与其他两种相反）
- **特点**：同时编码了方向相似性和模长信息
- **关键推论**：当 $\|a\| = \|b\| = 1$（单位向量）时，$IP(a,b) = \cos(\theta)$，此时内积等价于余弦相似度

### 2.3 余弦相似度 / 余弦距离

$$\cos(\theta) = \frac{a \cdot b}{\|a\| \cdot \|b\|}$$

$$d_{cosine}(a, b) = 1 - \cos(\theta)$$

- **几何含义**：仅衡量两个向量的夹角，忽略模长
- **余弦相似度值域**：$[-1, 1]$，越大越相似
- **余弦距离值域**：$[0, 2]$，越小越相似
- **特点**：纯方向度量，对向量缩放不变

### 2.4 三者关系总结表

| 度量 | 公式核心 | 归一化后等价关系 | 对模长敏感 | 典型取值方向 |
|------|---------|----------------|-----------|-------------|
| L2 | $\|a-b\|$ | $d_{L2}^2 = 2(1 - IP)$ 当 $\|a\|=\|b\|=1$ | ✅ | 越小越近 |
| IP | $a \cdot b$ | $IP = \cos(\theta)$ 当 $\|a\|=\|b\|=1$ | ✅ | 越大越近 |
| Cosine | $\frac{a \cdot b}{\|a\|\|b\|}$ | 本身就是归一化 IP | ❌ | 越大越近 |

> ⚠️ **核心结论**：对于已经归一化的向量（如 OpenAI text-embedding-3-small、BGE-M3 输出的向量），三种度量在排序上完全等价。选择哪种只影响索引结构和计算效率，不影响召回结果。

### 2.5 归一化后的数学证明

设 $\|a\| = \|b\| = 1$，则：

$$d_{L2}^2 = \|a - b\|^2 = \|a\|^2 + \|b\|^2 - 2a \cdot b = 2 - 2 \cdot IP(a,b)$$

即 $d_{L2}^2 = 2(1 - IP)$，是 IP 的单调递减函数。因此对归一化向量做 L2 搜索和 IP 搜索，Top-K 结果完全一致。

## 三、Embedding 模型与距离度量的配对

### 3.1 主流模型的输出特性

| 模型 | 维度 | 是否归一化 | 推荐度量 | 备注 |
|------|------|-----------|---------|------|
| OpenAI text-embedding-3-small/large | 1536/3072 | ✅ L2范数=1 | IP / Cosine | API 返回已归一化 |
| BGE-M3 / BGE-large-zh-v1.5 | 1024 | ✅ | IP / Cosine | 官方推荐 cosine |
| E5-mistral-7b-instruct | 4096 | ✅ | IP / Cosine | instruction-tuned |
| Cohere embed-v3 | 1024 | ✅ | IP / Cosine | search_document/search_query 区分 |
| Sentence-Transformers (all-MiniLM-L6-v2) | 384 | ✅ | Cosine | 默认 normalize=True |
| CLIP ViT-L/14 | 768 | ❌ 不一定 | L2 或先归一化再用 IP | 图文跨模态需注意 |
| Word2Vec / GloVe | 300 | ❌ | L2 | 传统词向量，通常不归一化 |

### 3.2 配对原则

1. **归一化模型 → IP 或 Cosine**：优先选 IP，因为 FAISS/Milvus 对 IP 有专门优化（如 SIMD），且避免重复归一化开销
2. **未归一化模型 → L2**：除非你手动归一化后再用 IP
3. **混合使用是大忌**：不同模型的向量空间不兼容，绝对不能混在同一索引中搜索
4. **查询和文档必须用同一模型**：即使是同一系列的不同版本（如 BGE-v1.5 vs BGE-M3），向量空间也不同

## 四、为什么暴力精确搜索不可行

### 4.1 计算复杂度分析

暴力搜索（Flat Index）对每个查询向量，需要与所有 N 个候选向量逐一计算距离：

$$T_{brute} = O(N \times D)$$

| 数据规模 N | 维度 D | 单次查询 FLOPs | GPU 10TFLOPS 耗时 | CPU 100GFLOPS 耗时 |
|-----------|--------|---------------|-------------------|-------------------|
| 1M | 768 | 7.68 × 10⁸ | 0.08ms | 7.7ms |
| 100M | 768 | 7.68 × 10¹⁰ | 7.7ms | 768ms |
| 1B | 768 | 7.68 × 10¹¹ | 77ms | 7.7s |
| 10B | 768 | 7.68 × 10¹² | 770ms | 77s |

> 实际系统中还需要考虑内存带宽瓶颈（memory-bound），真实延迟远高于理论 FLOPs 估算。10亿级向量暴力搜索在单机上完全不现实。

### 4.2 内存瓶颈

即使不考虑计算时间，内存也是问题：

- 1B × 768维 × float32(4B) = **3 TB**
- 1B × 768维 × float16(2B) = **1.5 TB**
- 单机内存上限通常 256GB~2TB，无法容纳全量数据做暴力搜索

## 五、ANN：用精度换速度的核心思想

### 5.1 什么是 ANN

ANN（Approximate Nearest Neighbor）是一类算法的统称，其核心思想是：

> **放弃保证找到真正的 Top-K 最近邻，转而以极高的概率找到"足够近"的邻居，将搜索复杂度从 O(N) 降到 O(log N) 甚至 O(1)。**

### 5.2 ANN 的核心权衡

```
                    精确搜索 (Flat)
                        │
            ┌───────────┴───────────┐
            │                       │
      图索引 (HNSW)           量化索引 (IVF-PQ)
     高精度 · 高内存          低内存 · 精度略损
     O(log N) 搜索           O(nprobe × m) 搜索
            │                       │
            └───────────┬───────────┘
                        │
              混合索引 (IVF-HNSW-PQ)
            兼顾规模 · 精度 · 速度
```

### 5.3 主要 ANN 算法族谱

| 算法族 | 代表 | 核心思路 | 优势 | 劣势 |
|--------|-----|---------|------|------|
| 树结构 | KD-Tree, Ball-Tree | 空间分割 | 低维有效 | 高维退化严重 |
| 哈希 | LSH | 相似向量落入同一桶 | 理论保证 | 需大量哈希表，内存高 |
| 图索引 | HNSW, NSG, Vamana | 多层导航图 | 高召回、低延迟 | 内存占用大 |
| 量化 | PQ, OPQ, SQ | 压缩向量 | 极致省内存 | 精度损失 |
| 混合 | IVF-PQ, HNSW-PQ | 组合以上策略 | 综合最优 | 调参复杂 |

## 六、维度灾难（Curse of Dimensionality）

### 6.1 直觉理解

在低维空间中，数据点之间有明确的"远近"之分。但随着维度增加：

- 所有点对之间的距离趋于集中（方差缩小）
- "最近邻"和"最远邻"的距离比值趋近于 1
- 基于距离的索引结构逐渐失效

### 6.2 数学表述

对于 D 维单位超球面上均匀分布的点，任意两点间距离的期望和方差满足：

$$E[d] \approx \sqrt{2}, \quad Var[d] \propto \frac{1}{D}$$

当 D → ∞ 时，所有距离趋向同一个常数，"近邻"概念失去意义。

### 6.3 实际影响

- **D < 20**：KD-Tree 等树结构高效
- **D = 100~300**：HNSW 表现优秀，IVF 可用
- **D = 768~4096**（现代 Embedding 常见维度）：树结构完全失效，只有图和量化方法可用
- **D > 10000**：即使 HNSW 也开始吃力，需要考虑降维（PCA/AutoEncoder）或稀疏化

### 6.4 为什么实际中 Embedding 仍然有效

虽然理论上存在维度灾难，但实际 Embedding 向量并非均匀分布在超球面上，而是集中在某个低维流形上（manifold hypothesis）。这种内在的低维结构使得 ANN 算法在实践中仍然高效。这也是为什么 768 维甚至 4096 维的向量检索仍然可以在毫秒级完成的原因。

## 七、Recall：ANN 的质量度量

### 7.1 Recall@K 定义

$$Recall@K = \frac{|ANN\_TopK \cap Exact\_TopK|}{K}$$

即用 ANN 搜到的 Top-K 结果中，有多少个确实在精确搜索的 Top-K 中。

### 7.2 典型 Recall 水平

| 场景 | 目标 Recall@10 | 说明 |
|------|---------------|------|
| RAG / 知识问答 | ≥ 0.95 | 召回不准直接影响生成质量 |
| 推荐系统 | ≥ 0.85 | 允许一定噪声，后续有重排 |
| 图片去重 | ≥ 0.99 | 漏掉就是事故 |
| 探索性搜索 | ≥ 0.70 | 用户容忍度高 |

### 7.3 Recall vs Latency 的 Pareto 前沿

每个 ANN 索引都有一组可调参数，改变参数会在 Recall-Latency 平面上画出一条曲线：

```
Recall ↑
1.0 |                              ● HNSW efSearch=512
    |                          ● efSearch=256
0.9 |                     ● efSearch=128
    |                 ● efSearch=64
0.8 |             ● efSearch=32
    |         ● efSearch=16
0.7 |     ● Flat(不可能到这)
    +────────────────────────────→ Latency(ms)
    0    1    2    5   10   20   50
```

> 没有免费的午餐：更高的 Recall 必然付出更多计算代价。工程化的核心就是在这条曲线上找到业务可接受的最优操作点。

## 八、面试话术要点

### Q1："Cosine 和 Inner Product 有什么区别？什么时候可以互换？"

> 余弦相似度只看向量夹角，内积同时看夹角和模长。当向量都已归一化为单位向量时，两者在排序上完全等价——因为 $IP = \cos(\theta)$，而 $L2^2 = 2(1-IP)$ 也是单调关系。OpenAI、BGE 等主流 Embedding 模型输出都是归一化的，所以实际工程中用 IP 代替 Cosine 没问题，还能省去运行时归一化的开销。但如果向量未归一化（比如原始 CLIP 输出），直接用 IP 会导致模长大的向量被过度偏好，这时应该用 L2 或者先做归一化。

### Q2："为什么向量检索要用近似搜索？精度怎么保证？"

> 十亿级向量做暴力搜索，单次查询要算几十亿次浮点运算，延迟在秒级甚至分钟级，完全不可接受。ANN 通过构建索引结构（如图索引 HNSW、倒排索引 IVF），把搜索范围从全量缩小到少量候选集，将复杂度从 O(N) 降到 O(logN) 级别。精度用 Recall@K 衡量——我们拿精确搜索的结果作为 ground truth，看 ANN 的 Top-K 里有多少是真的 Top-K。通过调节索引参数（如 HNSW 的 efSearch、IVF 的 nprobe），可以在 Recall 和延迟之间灵活取舍。一般 RAG 场景要求 Recall@10 ≥ 0.95，推荐场景 ≥ 0.85 就够了。

### Q3："维度灾难是什么？为什么 768 维的向量还能高效检索？"

> 维度灾难指的是在高维空间中，所有点对之间的距离趋于相同，导致基于距离的索引失效。理论上确实如此，但实际 Embedding 向量不是随机均匀分布的——它们集中在一个低维流形上。这是因为语义空间本身的内在维度远低于向量维度。所以虽然名义上是 768 维，但有效维度可能只有几十到一百多。这就是为什么 HNSW 等算法在实际 Embedding 数据上依然能保持很好的性能。不过如果维度真的非常高（比如超过 4096），还是建议先做 PCA 降维再建索引。
