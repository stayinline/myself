# 向量评测与更新删除原理

## 一、评测指标体系

### 1.1 Recall@K（召回率）

$$Recall@K = \frac{|ANN\_TopK \cap Exact\_TopK|}{K}$$

- **含义**：ANN 搜索返回的 Top-K 中，有多少个确实是精确搜索的 Top-K
- **值域**：[0, 1]，1 表示完美召回
- **特点**：只关心"有没有找对"，不关心排序是否完全一致
- **注意**：K 的选择很重要。Recall@10 和 Recall@100 可能差异很大

### 1.2 Precision@K（精确率）

$$Precision@K = \frac{|ANN\_TopK \cap Relevant|}{K}$$

- **含义**：返回的 Top-K 中有多少是真正相关的
- **与 Recall 的区别**：Recall 的分母是 K（或真实相关数），关注的是"该找到的找到了多少"；Precision 关注的是"找到的里面有多少是对的"
- **在 ANN 评测中**：如果用精确搜索 Top-K 作为"相关"标准，则 Precision@K = Recall@K

### 1.3 NDCG@K（归一化折损累积增益）

$$NDCG@K = \frac{DCG@K}{IDCG@K}$$

$$DCG@K = \sum_{i=1}^{K} \frac{rel_i}{\log_2(i+1)}$$

- **含义**：不仅关心结果是否正确，还关心排序质量——排在越前面的正确结果贡献越大
- **适用场景**：推荐系统、搜索结果排序等对顺序敏感的场景
- **对比 Recall**：Recall@10=0.9 只说明 10 个里对了 9 个，但不知道对的排在第几位；NDCG 能区分"对的排前面"和"对的排后面"

### 1.4 QPS / Latency

| 指标 | 定义 | 测量方式 |
|------|------|---------|
| QPS | 每秒查询数 | 总查询数 / 总耗时 |
| Avg Latency | 平均延迟 | 总耗时 / 查询数 |
| P50 Latency | 中位数延迟 | 50th percentile |
| P99 Latency | 尾部延迟 | 99th percentile |
| P999 Latency | 极端尾部 | 99.9th percentile |

> ⚠️ **P99 比 Avg 更重要**：用户感知的是最慢的那些请求。Avg 可能被大量正常请求拉低，掩盖了少数异常慢的请求。生产环境 SLA 通常以 P99 或 P999 为准。

### 1.5 指标之间的关系

```
              ┌──────────┐
              │  Recall   │ ← 质量指标（越高越好）
              └────┬─────┘
                   │ trade-off
              ┌────▼─────┐
              │ Latency  │ ← 性能指标（越低越好）
              └────┬─────┘
                   │ 受以下因素影响
        ┌──────────┼──────────┐
        ▼          ▼          ▼
    索引类型    参数配置     数据规模
   (HNSW/IVF) (efSearch/  (N, D)
               nprobe)
```

## 二、构建 Ground Truth

### 2.1 为什么需要 Ground Truth

ANN 是"近似"搜索，要评估"近似"的程度，必须有一个"精确"的标准答案。这个标准答案就是 Ground Truth。

### 2.2 Ground Truth 的构建方法

**方法一：Flat Index 暴力搜索（最常用）**

```python
import faiss
import numpy as np

# 用 Flat Index 做精确搜索作为 ground truth
gt_index = faiss.IndexFlatIP(D)
gt_index.add(database_vectors)
D_gt, I_gt = gt_index.search(query_vectors, K)

# I_gt[i] 就是第 i 个查询的精确 Top-K ID 列表
# 这是评估所有 ANN 索引的黄金标准
```

**方法二：人工标注（用于端到端评估）**

- 对于业务相关的评测，仅靠 Recall 不够——因为精确搜索的 Top-K 不一定都是"语义相关"的
- 需要人工判断每条结果的相关性等级（relevant / partially relevant / irrelevant）
- 然后用 NDCG 等考虑排序质量的指标来评估

**方法三：合成数据（用于算法研究）**

- 生成已知结构的向量数据（如聚类分布）
- 可以精确计算理论上的最近邻
- 适合算法论文中的对比实验

### 2.3 Ground Truth 的注意事项

| 注意点 | 说明 |
|--------|------|
| 数据集一致性 | GT 和 ANN 必须使用完全相同的数据集 |
| 度量一致性 | GT 和 ANN 必须使用相同的距离度量 |
| K 的一致性 | 评估 Recall@K 时 GT 也要搜 Top-K |
| 数据预处理 | 归一化等操作必须在 GT 和 ANN 之前统一完成 |
| 浮点精度 | float32 vs float64 可能导致边界 case 不同 |
| 重复向量 | 如果有完全相同的向量，Top-K 可能有多个正确答案 |

## 三、更新操作的原理

### 3.1 Upsert = Delete + Insert

在大多数向量数据库中，"更新"不是原地修改，而是**先标记删除旧版本，再插入新版本**：

```
Upsert(id=42, new_vector):
    Step 1: MarkDelete(id=42)     → 在 bitset 中标记
    Step 2: Insert(id=42, new_vector) → 写入新 Segment
```

### 3.2 Bitset（位图标记删除）

```
Segment 中的向量: [v₀, v₁, v₂, v₃, v₄, v₅, v₆, v₇]
Bitset:           [ 0,  1,  0,  0,  1,  0,  0,  0]
                    ↑      ↑
                  alive  deleted  alive  alive  deleted ...

搜索时:
  1. 正常执行 ANN 搜索
  2. 从结果中过滤掉 bitset=1 的向量
  3. 如果过滤后不足 K 个，可能需要扩大搜索范围
```

### 3.3 Bitset 的影响

- **空间开销**：每个向量 1 bit，1 亿条 ≈ 12 MB，几乎可忽略
- **搜索开销**：每次搜索结果需要做 bitwise AND 过滤，很快
- **Recall 影响**：如果大量删除导致有效数据稀疏，ANN 可能搜不到足够的有效邻居，Recall 下降
- **内存不释放**：被标记删除的向量仍然占据磁盘和内存空间

## 四、删除操作的原理

### 4.1 Soft Delete vs Hard Delete

| 操作 | 机制 | 立即生效 | 释放空间 | 性能影响 |
|------|------|---------|---------|---------|
| Soft Delete | Bitset 标记 | ✅ | ❌ | 小（过滤开销） |
| Hard Delete | Compaction 物理移除 | ❌ | ✅（Compaction 后） | 大（IO/CPU 密集） |

### 4.2 Compaction（合并压缩）

Compaction 是将多个 Segment 合并为一个新 Segment 的过程，同时：
- 物理移除被标记删除的向量
- 合并多个小 Segment 为大 Segment
- 重建索引（或在合并后的数据上应用已有索引）

```
Before Compaction:
Segment A: [v₀, ✗v₁, v₂, ✗v₃]    ← 50% 已删除
Segment B: [v₄, v₅, ✗v₆]           ← 33% 已删除
Segment C: [v₇]                      ← 小段

After Compaction:
Segment D: [v₀, v₂, v₄, v₅, v₇]   ← 干净、紧凑
(需要重建索引)
```

### 4.3 Compaction 触发条件

| 触发方式 | 条件 | 说明 |
|---------|------|------|
| 自动触发 | 删除比例 > 阈值（如 30%） | 避免过多无效数据 |
| 自动触发 | 小 Segment 数量 > 阈值 | 减少段碎片 |
| 手动触发 | API 调用 | 运维主动整理 |
| 定时任务 | Cron schedule | 低峰期执行 |

### 4.4 为什么删除不立即释放空间

1. **原子性和一致性**：立即物理删除需要修改索引结构（如 HNSW 图的边），这在并发环境下很难做到原子且安全
2. **性能考量**：物理删除涉及大量的磁盘 IO 和索引重建，如果每次删除都触发，写入性能会严重下降
3. **MVCC 思想**：类似数据库的 MVCC，标记删除让读操作不受影响（只需过滤 bitset），写操作也很快（只改一个 bit）
4. **批量效率**：Compaction 把多次删除的物理清理合并为一次批量操作，摊销了固定开销

## 五、评测驱动的索引选型方法论

### 5.1 标准化评测流程

```
Step 1: 准备评测数据集
  ├── 数据库向量（有代表性的样本）
  ├── 查询向量（覆盖真实查询模式）
  └── Ground Truth（Flat Index 精确搜索结果）

Step 2: 定义业务 SLA
  ├── Recall@K ≥ ?
  ├── P99 Latency ≤ ? ms
  ├── QPS ≥ ?
  └── Memory ≤ ? GB

Step 3: 候选索引 × 参数网格搜索
  ├── HNSW: M ∈ {8,16,32}, efSearch ∈ {16,32,...,512}
  ├── IVF_FLAT: nprobe ∈ {1,4,...,128}
  ├── IVF_PQ: m ∈ {48,96}, nprobe ∈ {8,...,128}
  └── ...

Step 4: 产出 Pareto 前沿
  ├── Recall vs QPS 曲线
  ├── Recall vs Memory 曲线
  └── 标注满足 SLA 的最优配置

Step 5: 线上验证
  ├── A/B Test 或 Shadow Traffic
  ├── 监控实际 Recall（采样验证）
  └── 持续观测延迟和资源使用
```

### 5.2 评测报告模板

| 索引配置 | Recall@10 | P99 Lat(ms) | QPS | Memory(GB) | 满足SLA |
|---------|-----------|-------------|-----|-----------|---------|
| HNSW_M16_efS64 | 0.972 | 3.2 | 3125 | 65 | ✅ |
| HNSW_M16_efS32 | 0.934 | 1.8 | 5556 | 65 | ❌ Recall |
| HNSW_M32_efS64 | 0.986 | 4.5 | 2222 | 68 | ✅ |

## 六、面试话术

### Q："你怎么证明你的向量检索效果好？"

> 我从三个层面来证明。
>
> 第一是离线评测。我们用 Flat Index 的精确搜索结果作为 Ground Truth，构建了包含 1000 条查询的标准评测集。对每种候选索引配置，计算 Recall@10、Recall@100、P99 延迟和 QPS，画出 Pareto 曲线。比如我们的 HNSW(M=16, efSearch=64) 配置，Recall@10=0.972，P99=3.2ms，QPS=3125，在满足 SLA（Recall≥0.95, P99≤5ms）的前提下 QPS 最高。
>
> 第二是端到端业务指标。Recall 高不等于业务效果好，因为 Embedding 模型本身的语义质量也会影响最终效果。所以我们还会看 RAG 系统的回答准确率、用户点击率等业务指标。我们做过 A/B Test，把向量检索的 Recall 从 0.93 提升到 0.97 后，RAG 回答的人工评分提升了 8%，这说明向量检索的质量确实传导到了业务价值。
>
> 第三是线上持续监控。我们在 Grafana 上搭建了实时监控面板，包括搜索 P99 延迟、QPS、Growing Segment 状态等基础设施指标。同时定期（每周）抽样做 Recall 回归测试，确保线上索引没有退化。如果数据分布发生了漂移导致 Recall 下降，我们会触发重新训练或重建索引。
>
> 总结就是：离线有标准评测集量化对比，在线有业务指标验证价值，持续有监控保障不退化。

### Q："Recall 和 Precision 有什么区别？你为什么主要看 Recall？"

> 在 ANN 评测的语境下，如果我们用精确搜索的 Top-K 作为"相关"的定义，那 Recall@K 和 Precision@K 在数值上是相等的——因为两者的分子都是"ANN Top-K ∩ Exact Top-K"，分母都是 K。但在更广泛的 IR 场景中它们是不同的：Recall 关注"该找到的找到了多少"，Precision 关注"找到的里面有多少是对的"。
>
> 我主要看 Recall 的原因是：对于 RAG 和语义搜索场景，漏掉相关文档的代价远大于混入一些不太相关的文档。因为下游的 LLM 有能力从一组候选中筛选出最有用的信息，但如果关键文档根本没被检索到，LLM 就无法弥补。所以 Recall 是更重要的指标。当然如果是去重或安全审核场景，Precision 就更重要了——误报的代价很高。

### Q："向量数据库的删除是怎么实现的？为什么删了空间没释放？"

> 向量数据库的删除通常是软删除——用一个 bitset 标记哪些向量被删除了，搜索时过滤掉这些被标记的向量。这类似于数据库的 MVCC 机制。真正的物理删除发生在 Compaction 过程中——后台任务把多个 Segment 合并成一个新 Segment，跳过被标记删除的向量，然后替换旧的 Segment。
>
> 不立即释放空间的原因有三点。第一，物理删除需要修改索引结构（比如 HNSW 图中移除节点和相关边），这在并发读写环境下很难做到安全且高效。第二，逐条物理删除的 IO 开销太大，不如攒一批在 Compaction 时批量处理。第三，软删除对搜索性能的影响很小——只是一个 bitwise AND 操作，而 Compaction 可以在低峰期异步执行，不影响在线服务。
>
> 在实际运维中需要注意：如果删除比例过高（比如超过 30%），应该主动触发 Compaction，否则大量无效向量会浪费内存和搜索时间。我们通过监控每个 Segment 的删除比例来自动触发 Compaction。
