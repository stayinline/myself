## 二、创建 Collection 与 Partition

### 2.1 定义 Schema

```python
from pymilvus import (
    connections, Collection, FieldSchema, 
    CollectionSchema, DataType, utility
)

connections.connect(host="localhost", port="19530")

# 定义字段
fields = [
    FieldSchema(name="id", dtype=DataType.INT64, is_primary=True, auto_id=True),
    FieldSchema(name="embedding", dtype=DataType.FLOAT_VECTOR, dim=768),
    FieldSchema(name="text", dtype=DataType.VARCHAR, max_length=65535),
    FieldSchema(name="tenant_id", dtype=DataType.VARCHAR, max_length=64),
    FieldSchema(name="created_at", dtype=DataType.INT64),  # Unix timestamp
]

schema = CollectionSchema(
    fields, 
    description="知识库文档向量集合",
    enable_dynamic_field=True  # 允许动态字段
)

# 创建 Collection
collection_name = "knowledge_base"
if utility.has_collection(collection_name):
    Collection(collection_name).drop()

collection = Collection(collection_name, schema)
print(f"Collection '{collection_name}' created")
print(f"Schema: {collection.schema}")
```

### 2.2 创建 Partition（按租户隔离）

```python
# 按租户创建分区
tenants = ["tenant_a", "tenant_b", "tenant_c"]
for tenant in tenants:
    partition_name = f"p_{tenant}"
    if not collection.has_partition(partition_name):
        collection.create_partition(partition_name)
        print(f"Partition '{partition_name}' created")

# 查看所有分区
partitions = collection.partitions
for p in partitions:
    print(f"  Partition: {p.name}, num_entities: {p.num_entities}")
```

### 2.3 Partition Key vs 手动 Partition

| 方式 | 适用场景 | 优点 | 缺点 |
|------|---------|------|------|
| 手动 create_partition | 分区数少且固定 | 完全控制、可单独搜索 | 需要自己管理路由 |
| Partition Key | 分区数多或动态 | 自动路由、透明 | 不能单独指定分区搜索 |

```python
# Partition Key 方式（自动路由）
fields_pk = [
    FieldSchema(name="id", dtype=DataType.INT64, is_primary=True, auto_id=True),
    FieldSchema(name="embedding", dtype=DataType.FLOAT_VECTOR, dim=768),
    FieldSchema(name="tenant_id", dtype=DataType.VARCHAR, max_length=64, 
                is_partition_key=True),  # ← 标记为分区键
]
# 写入时 Milvus 自动根据 tenant_id 路由到对应物理分区
# 搜索时如果 filter 包含 tenant_id，自动做分区裁剪
```

## 三、插入数据

### 3.1 批量插入

```python
import numpy as np
import time

N = 100_000  # 10万条

# 准备数据
texts = [f"这是第{i}条知识文档的内容描述" for i in range(N)]
embeddings = np.random.randn(N, 768).astype(np.float32).tolist()
tenant_ids = [f"tenant_{'abc'[i % 3]}" for i in range(N)]
timestamps = [int(time.time()) - i * 60 for i in range(N)]

# 插入到指定分区
insert_result = collection.insert(
    data=[embeddings, texts, tenant_ids, timestamps],
    partition_name="p_tenant_a"  # 指定分区
)
print(f"Inserted {insert_result.insert_count} rows")
```

### 3.2 Upsert（更新或插入）

```python
# 如果有主键，可以用 upsert 实现更新
# 注意：upsert = delete old + insert new，不是原地修改
update_embeddings = np.random.randn(100, 768).astype(np.float32).tolist()
update_texts = [f"更新后的文档内容 {i}" for i in range(100)]
update_ids = list(range(1, 101))  # 要更新的 ID

collection.upsert(
    data=[update_ids, update_embeddings, update_texts, 
          ["tenant_a"] * 100, [int(time.time())] * 100],
    partition_name="p_tenant_a"
)
print("Upsert completed")
```

## 四、观察 Segment 状态与索引构建

### 4.1 查看 Segment 信息

```python
from pymilvus import utility

# Flush 使 Growing Segment → Sealed
collection.flush()
print("Flush completed")

# 查看 Segment 状态
segments_info = utility.get_query_segment_info(collection_name)
for seg in segments_info:
    print(f"Segment ID: {seg.segmentID}")
    print(f"  State: {seg.state}")         # Sealed / Growing / Flushed
    print(f"  Rows: {seg.num_rows}")
    print(f"  Mem Size: {seg.mem_size}")
    print(f"  Index Name: {seg.index_name}")
    print(f"  Node IDs: {seg.nodeIds}")
    print()
```

### 4.2 创建索引并监控进度

```python
# 创建 HNSW 索引
index_params = {
    "index_type": "HNSW",
    "metric_type": "IP",
    "params": {"M": 16, "efConstruction": 200}
}

print("Creating index...")
t0 = time.time()
collection.create_index("embedding", index_params)
print(f"Index creation triggered ({time.time()-t0:.1f}s)")

# 监控索引构建进度
while True:
    progress = utility.index_building_progress(collection_name, "embedding")
    print(f"Index progress: {progress['indexed_rows']}/{progress['total_rows']} "
          f"({progress['indexed_rows']/max(progress['total_rows'],1)*100:.1f}%)")
    
    if progress['indexed_rows'] >= progress['total_rows'] and progress['total_rows'] > 0:
        break
    time.sleep(2)

print("Index building complete!")
```

## 五、搜索与分区裁剪

### 5.1 全 Collection 搜索

```python
search_params = {"metric_type": "IP", "params": {"ef": 64}}
query_vec = np.random.randn(1, 768).astype(np.float32).tolist()

results = collection.search(
    data=query_vec,
    anns_field="embedding",
    param=search_params,
    limit=10,
    output_fields=["text", "tenant_id", "created_at"],
    consistency_level="Eventually"  # 最终一致性，性能最好
)

for hit in results[0]:
    print(f"  ID={hit.id}, Score={hit.score:.4f}, "
          f"Text={hit.entity.get('text')[:30]}..., "
          f"Tenant={hit.entity.get('tenant_id')}")
```

### 5.2 分区搜索（Pruning）

```python
# 只在特定分区中搜索 → 减少搜索范围
results_partition = collection.search(
    data=query_vec,
    anns_field="embedding",
    param=search_params,
    limit=10,
    partition_names=["p_tenant_a"],  # ← 只搜这个分区
    output_fields=["text", "tenant_id"]
)
print(f"Searched in p_tenant_a only, got {len(results_partition[0])} results")
```

### 5.3 Filter 表达式搜索

```python
# 标量过滤 + 向量搜索混合
results_filtered = collection.search(
    data=query_vec,
    anns_field="embedding",
    param=search_params,
    limit=10,
    expr='tenant_id == "tenant_a" and created_at > 1700000000',
    output_fields=["text", "tenant_id", "created_at"]
)
print(f"Filtered search returned {len(results_filtered[0])} results")
```

### 5.4 Consistency Level 选择

| Level | 说明 | 延迟 | 数据新鲜度 |
|-------|------|------|-----------|
| Strong | 读到最新已确认的数据 | 最高 | 最强 |
| Bounded | 允许短暂滞后（默认几秒） | 中等 | 较好 |
| Eventually | 不保证实时一致 | 最低 | 最弱 |
| Session | 同一 session 内单调读 | 中等 | 会话内强 |

> 💡 **生产建议**：大多数 RAG 场景用 `Eventually` 或 `Bounded` 即可。只有金融/合规等强一致场景才需要 `Strong`。

## 六、Partition 策略实践

### 6.1 按时间分区

```python
# 按月分区，方便 TTL 管理和冷热处理
months = ["2024_01", "2024_02", "2024_03", "2024_04"]
for month in months:
    pname = f"time_{month}"
    if not collection.has_partition(pname):
        collection.create_partition(pname)

# 搜索最近两个月的数据
recent_partitions = ["time_2024_03", "time_2024_04"]
results = collection.search(
    data=query_vec,
    anns_field="embedding",
    param=search_params,
    limit=10,
    partition_names=recent_partitions  # 只搜近期数据
)

# 删除过期分区
collection.drop_partition("time_2024_01")
print("Dropped old partition")
```

### 6.2 按租户分区的优势

1. **数据隔离**：不同租户的数据物理分离，安全合规
2. **搜索裁剪**：查询时指定分区，避免扫描无关数据
3. **独立管理**：可以单独对某个租户的分区做 flush/load/drop
4. **资源隔离**：可以将不同分区分配到不同的 Query Node Replica

### 6.3 分区数量限制

> ⚠️ Milvus 默认最大分区数为 4096。超过此限制需要调整配置。如果分区数可能很多（如按天分区），建议使用 Partition Key 而非手动分区。

## 八、面试话术

### Q："你在项目中是怎么使用 Milvus 的？做了哪些优化？"

> 我们的 RAG 系统用 Milvus 存储约 3000 万条文档向量。主要做了几个层面的优化：
>
> 第一是分区策略。我们按租户+月份双维度分区，搜索时通过 partition_names 做裁剪，通常只搜当前租户的当月和上月数据，搜索范围缩小了 80% 以上。
>
> 第二是一致性级别。RAG 场景不需要强一致，我们用 Eventually 级别，避免了每次搜索都要同步最新 WAL 的开销，P99 延迟从 12ms 降到了 4ms。
>
> 第三是索引参数调优。我们在测试集上跑了完整的 efSearch sweep，最终选了 efSearch=64，Recall@10=0.971，满足业务要求的同时保持了 3ms 左右的平均延迟。
>
> 第四是 Segment 管理。我们发现默认 flush 间隔导致大量小 Segment，影响搜索性能。后来调整了 flush 阈值让每个 Segment 大约 200 万行，减少了 Segment 数量，搜索延迟降低了约 15%。

### Q："Milvus 的 Partition 和 Collection 怎么设计？"

> Collection 对应一个逻辑上的数据集，比如"知识库文档"或"用户行为向量"。Partition 是 Collection 内的物理分组，主要用于两个目的：数据隔离和搜索裁剪。
>
> 在我们的实践中，Partition 的设计取决于业务维度。对于多租户 SaaS，按租户 ID 分区是最常见的——每个租户一个 Partition，搜索时只查目标租户的分区，既实现了数据隔离又提升了性能。对于有时效性的数据，可以按时间（月/周）分区，方便 TTL 管理和冷热分层。
>
> 需要注意分区数不要太多——Milvus 默认上限 4096。如果分区键的基数很大（比如百万级用户 ID），应该用 Partition Key 功能让 Milvus 自动做哈希映射到有限的物理分区，而不是手动创建百万个分区。

### Q："Milvus 写入后多久能被搜到？"

> 这取决于几个阶段。写入后数据立即进入 Growing Segment，此时理论上已经可以被搜到（因为 Growing Segment 也参与搜索），但它是暴力扫描没有索引，延迟会比较高。当 Growing Segment 达到阈值被 Seal 并 Flush 到对象存储后，Index Node 开始构建索引。索引完成并被 Query Node 加载后，这段数据才能被高效检索。
>
> 在默认配置下，整个链路大约需要几十秒到几分钟。如果需要更快的可见性，有几个手段：一是调小 flush 阈值让 Segment 更快 seal；二是调用手动 flush API 强制触发；三是使用 Milvus 2.4+ 的 Streaming Node 特性实现近实时的索引更新。另外 consistency level 也会影响——如果用 Strong 级别，Proxy 会等待所有节点确认后才返回结果，延迟更高但数据更新鲜。

## 九、常见问题排查表

| 问题 | 排查方向 | 解决方案 |
|------|---------|---------|
| 搜索结果为空 | Collection 未 load | 调用 `collection.load()` |
| 搜索很慢 | Growing Segment 太多 | 手动 flush + 检查索引状态 |
| 写入失败 | MQ 满了或 Data Node 忙 | 检查 Pulsar/Kafka 积压 |
| 索引构建卡住 | Index Node 资源不足 | 增加 Index Node 或检查 GPU |
| 内存 OOM | Segment 太多/太大 | 增加 Query Node 或释放不用的 Collection |
| 分区超限 | 分区数 > 4096 | 改用 Partition Key 或合并分区 |
| 搜索结果不一致 | Consistency Level | 切换到 Bounded 或 Strong |
