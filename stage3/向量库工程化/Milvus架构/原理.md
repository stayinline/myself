# Milvus 架构原理

## 一、Milvus 概述

Milvus 是目前最成熟的开源云原生向量数据库，由 Zilliz 开发并捐赠给 LF AI & Data 基金会。它专为大规模向量检索设计，支持十亿级向量、多种索引类型、混合查询和云原生部署。

**核心设计理念**：计算与存储分离 + 微服务架构 + 日志即数据（Log as Data）。

### 2.2 各组件职责

| 组件 | 职责 | 可扩展性 | 备注 |
|------|------|---------|------|
| **Proxy** | 客户端接入、请求路由、结果合并、限流鉴权 | 水平扩展 | 无状态，可多副本 |
| **Coordinator Service** | 全局调度：Root Coord（DDL/DCL）、Query Coord（查询调度）、Data Coord（数据管理）、Index Coord（索引管理） | 每种 Coord 独立扩展 | 集群大脑 |
| **Query Node** | 加载 Segment 到内存、执行搜索/查询 | 水平扩展 | 有状态（内存中的 Segment） |
| **Data Node** | 接收写入数据、生成 Segment、Flush 到对象存储 | 水平扩展 | 从 MQ 消费写入日志 |
| **Index Node** | 为 Sealed Segment 构建索引 | 水平扩展 | CPU/GPU 密集型 |
| **Message Queue** | 持久化写入日志、变更通知、CDC | Pulsar/Kafka/RocksMQ | Log as Data 核心 |
| **Object Storage** | 持久化 Segment 文件、索引文件 | S3/MinIO/GCS | 存算分离的"存" |
| **Meta Store** | Collection/Partition/Segment/Index 元数据 | etcd/TiKV | 集群状态真相源 |

## 三、计算存储分离架构详解

### 3.1 为什么存算分离

传统数据库将数据和计算绑定在同一节点上，导致：
- 扩容时必须同时扩存储和计算，不灵活
- 某个节点故障可能丢失数据
- 读写互相干扰

Milvus 的存算分离：
- **存储层**：对象存储（S3/MinIO）持久化所有数据，天然高可用、低成本
- **计算层**：Query/Data/Index Node 是无状态或半状态的，可随时扩缩容
- **连接纽带**：Message Queue 保证写入的持久性和顺序性

### 3.2 数据流向

```
写入流程:
Client → Proxy → MQ (写入日志) → Data Node (消费日志, 缓冲)
                                      ↓ Flush
                                 Object Storage (Segment 文件)
                                      ↓ 通知
                                 Index Node (构建索引)
                                      ↓ 完成
                                 Object Storage (索引文件)
                                      ↓ 通知
                                 Query Node (加载 Segment+索引到内存)
                                      ↓ Ready
                                 可被搜索 ✓

读取流程:
Client → Proxy → Query Coord (路由) → Query Node(s) (并行搜索)
                                           ↓ 结果汇总
                                      Proxy → Client
```

### 3.3 云原生优势

- **弹性伸缩**：K8s HPA 根据 CPU/内存/QPS 自动扩缩 Query Node
- **故障恢复**：Node 挂了重新从对象存储加载 Segment，数据不丢
- **成本优化**：存储用廉价对象存储，计算按需分配
- **多租户隔离**：不同 Collection/Partition 可以分配到不同 Node 组

## 四、数据模型：Collection → Partition → Segment

### 4.1 层级关系

```
Collection (类似表)
├── Partition A (类似分区)
│   ├── Segment 1 (sealed, indexed)
│   ├── Segment 2 (sealed, indexing...)
│   └── Segment 3 (growing, unindexed)
├── Partition B
│   ├── Segment 4 (sealed, indexed)
│   └── Segment 5 (growing, unindexed)
└── _default Partition
    └── Segment 6 (sealed, indexed)
```

### 4.2 Segment 生命周期

```
Growing Segment                 Sealed Segment
(活跃写入中)                     (只读，等待/已有索引)
     │                                │
     │  达到阈值触发 Seal               │  Index Node 构建索引
     ├────────────────────────→       │
     │                                │  索引完成
     │                                ├────────────────→ Indexed Segment
     │                                │                  (可高效搜索)
     │                                │
     │                                │  Compaction
     │                                ├────────────────→ Compacted Segment
     │                                │                  (合并小段、清理删除)
```

### 4.3 Growing vs Sealed 的关键区别

| 特性 | Growing Segment | Sealed Segment |
|------|----------------|---------------|
| 状态 | 正在接收写入 | 不再接收写入 |
| 索引 | ❌ 无索引 | ✅ 有（或正在建） |
| 搜索方式 | 暴力扫描 | 索引加速 |
| 存储位置 | Data Node 内存 | 对象存储（Query Node 加载到内存） |
| 可见性 | 写入即可见（但慢） | 索引完成后高效可见 |
| 大小限制 | 默认 2M 行或 ~512MB | 固定大小 |

### 4.4 Segment 触发 Seal 的条件

满足以下任一条件即触发：
1. 行数达到 `dataCoord.segment.maxSize`（默认约 200万行 / 512MB）
2. 超过最大存活时间 `dataCoord.segment.sealProportion` × timeout
3. 手动调用 `flush()` API

## 五、索引构建时机

### 5.1 自动索引流程

```
Segment sealed
      ↓
Data Coord 通知 Index Coord
      ↓
Index Coord 分配 Index Node
      ↓
Index Node 从对象存储读取 Segment 数据
      ↓
在内存中构建索引（HNSW/IVF等）
      ↓
将索引文件写回对象存储
      ↓
通知 Query Coord
      ↓
Query Node 加载 Segment + 索引到内存
      ↓
该 Segment 可被高效搜索 ✓
```

### 5.2 关键时间窗口

从数据写入到可被高效检索之间存在延迟：

```
写入时刻 t₀
    ↓ Growing Segment 内暴力搜索可达（但慢）
Seal 时刻 t₁ (t₀ + flush间隔)
    ↓ 等待 Index Node 调度
索引开始 t₂
    ↓ 构建耗时取决于数据量和索引类型
索引完成 t₃
    ↓ Query Node 加载
可高效搜索 t₄
```

典型延迟：
- Growing → Sealed：几秒到几分钟（取决于 flush 配置）
- Seal → 索引完成：几十秒到几分钟（取决于数据量和索引类型）
- 总计：**写入到高效可搜 ≈ 几十秒到几分钟**

> ⚠️ 这就是"实时写入与检索冲突"的根源，后续章节会详细讨论。

### 6.2 各组件扩展策略

| 组件 | 扩展触发条件 | 扩展方式 | 注意事项 |
|------|------------|---------|---------|
| Proxy | QPS 瓶颈 | 增加副本 | 无状态，简单水平扩展 |
| Query Node | 内存不足/搜索延迟高 | 增加节点 | Segment 会自动 rebalance |
| Data Node | 写入吞吐不足 | 增加节点 | Channel 会自动分配 |
| Index Node | 索引构建排队 | 增加节点 | GPU 节点加速 HNSW/IVF |
| MQ | 消息积压 | 增加 broker/partition | 影响写入和消费延迟 |

## 七、Log as Data 设计哲学

### 7.1 核心思想

Milvus 不像传统数据库那样维护一份独立的 WAL（Write-Ahead Log），而是**把消息队列本身作为数据的唯一真相源**：

- 所有写入操作先写入 MQ
- Data Node 从 MQ 消费并物化为 Segment
- Query Node 通过 MQ 获知数据变更
- 即使所有 Node 都重启，只要 MQ 和对象存储在，就能完整恢复

### 7.2 好处

1. **解耦**：写入和读取完全异步，互不影响
2. **可靠性**：MQ 提供持久化和重试保证
3. **CDC**：其他系统可以订阅 MQ 获取实时变更
4. **时间旅行**：保留 MQ 历史消息可以实现时间点恢复

### 7.3 代价

- MQ 成为关键依赖，MQ 故障 = 全集群不可用
- MQ 的消息保留策略需要精心配置
- 引入了额外的运维复杂度

## 八、面试话术

### Q："Milvus 的架构是怎样的？为什么要这样设计？"

> Milvus 采用计算存储分离的微服务架构。核心组件包括 Proxy（接入层）、四种 Coordinator（调度层）、三种 Worker Node（计算层），加上消息队列、对象存储和元数据存储三个基础设施。
>
> 这种设计的核心动机有三个。第一，弹性伸缩——Query Node 可以根据搜索负载水平扩展，Data Node 根据写入量扩展，Index Node 根据索引构建需求扩展，各自独立不耦合。第二，高可用——数据持久化在对象存储上，任何 Node 挂了都能从存储恢复，不存在单点数据丢失。第三，读写解耦——写入走 MQ → Data Node → 对象存储的路径，搜索走 Query Node 直接从内存读的路径，两者互不阻塞。
>
> "Log as Data" 是另一个关键设计：消息队列不仅是传输通道，更是数据的真相源。所有变更先入 MQ，各 Node 从 MQ 消费来保持状态一致。这让 CDC、时间旅行、故障恢复都有了天然的基础。

### Q："Milvus 的数据从插入到可被索引检索经历了什么？"

> 完整链路分五步。第一步，客户端调用 insert API，Proxy 将数据写入消息队列。第二步，Data Node 从 MQ 消费数据，放入内存中的 Growing Segment。此时数据已经可以被搜索，但因为 Growing Segment 没有索引，只能做暴力扫描，效率很低。第三步，当 Growing Segment 达到大小阈值（默认约 200万行或 512MB）或超时后，被 Seal 变成 Sealed Segment，同时 Flush 到对象存储。第四步，Index Coord 感知到新的 Sealed Segment，调度 Index Node 从对象存储读取数据，在内存中构建索引（如 HNSW），然后将索引文件写回对象存储。第五步，Query Coord 通知 Query Node 加载该 Segment 及其索引到内存，此后这个 Segment 的数据就可以被高效检索了。
>
> 整个过程的延迟取决于几个因素：flush 间隔决定了 Growing → Sealed 的时间；Index Node 的资源和索引类型决定了构建时间；Query Node 的加载速度决定了最后一步。通常在默认配置下，从写入到高效可搜大约需要几十秒到几分钟。如果需要更快的可见性，可以调小 flush 阈值或使用 streaming upsert 特性。

### Q："Milvus 和 Elasticsearch 的向量检索有什么区别？"

> 定位不同。ES 本质是全文搜索引擎，向量检索是后来加的功能；Milvus 是专门为向量检索从头设计的。具体差异：第一，索引类型——Milvus 支持 HNSW、IVF-PQ、DiskANN 等多种专业向量索引，ES 主要支持 HNSW。第二，规模——Milvus 的存算分离架构天然支持十亿级，ES 在大规模向量场景下资源消耗更高。第三，功能——Milvus 有原生的 Partition、Segment 管理、量化压缩等向量专用特性；ES 的优势在于全文检索+向量的混合查询和成熟的生态。选型建议：如果以向量检索为主且规模大，选 Milvus；如果需要全文+向量的混合搜索且已有 ES 基础设施，可以用 ES 的 knn 功能。
