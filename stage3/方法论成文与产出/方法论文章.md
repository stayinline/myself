## 一、引言：为什么 RAG 需要"数据基础设施"

### 1.1 RAG 的现状与痛点

当前大多数 RAG 实现停留在"Demo 级别"：

```
典型 Demo 级 RAG:
  文档 → LangChain Loader → 固定切片 → OpenAI Embedding → ChromaDB → 检索 → GPT生成

这个流程的问题:
  ❌ 数据更新靠手动重跑，知识库永远是"昨天的"
  ❌ 没有数据血缘，回答出错了不知道是哪份文档的问题
  ❌ 没有质量监控，准确率下降无人知晓
  ❌ 无法支撑多源异构数据的统一接入
  ❌ 扩展性差，百万级文档就扛不住
```

### 1.2 数据工程师的独特价值

AI 应用的质量上限取决于数据质量，而非模型能力。这正是数据工程师的主场：

| AI 应用需求 | 对应数据工程能力 |
|------------|-----------------|
| 知识库实时更新 | CDC + 流处理 |
| 多源数据融合 | ETL + Schema 治理 |
| 数据可追溯 | 数据血缘 + Lineage |
| 大规模向量检索 | 分布式存储 + 索引优化 |
| 质量持续保障 | 数据质量框架 + 监控 |
| 成本可控 | 分层存储 + 计算优化 |

> **核心观点**：RAG 不是一个 AI 问题，而是一个数据基础设施问题。模型只是链条中的一环，而数据工程师掌控着其余 80% 的链路。

### 2.1 架构全景

### 2.2 设计原则

我们在设计之初确立了三个核心原则：

**原则一：可信 (Trustworthy)**
- 每条 RAG 回答都能追溯到原始数据源和时间点
- 数据变更有完整的审计日志
- 模型输出有置信度评分和人工兜底机制

**原则二：实时 (Real-time)**
- 从数据源变更到知识库可查询，端到端 <10s
- 告别 T+1 的知识库更新模式
- 支持增量更新而非全量重建

**原则三：可观测 (Observable)**
- 全链路 Metrics/Traces/Logs/Alerts
- RAG 质量指标自动化评测
- 异常可在 15 分钟内定位根因

### 2.3 技术选型决策

（此处插入选型对比表，参见《旗舰架构整合/原理.md》中的选型矩阵）

重点解释几个关键决策：

**为什么选 Paimon 而不是 Iceberg？**
> 我们的核心计算引擎是 Flink，Paimon 作为 Flink 社区原生项目，在流式写入性能、主键表 Upsert 支持、以及与 Flink Checkpoint 的协同上都优于 Iceberg。特别是 CDC 场景下频繁的 Update/Delete 操作，Paimon 的 LSM-Tree 存储结构天然适配。当然，如果团队以 Spark 为主，Iceberg 会是更好的选择。

**为什么选 StarRocks 而不是 ClickHouse？**
> RAG 场景下的分析查询往往涉及多表 Join（事实表关联维度表），StarRocks 的 CBO 优化器在 Join 性能上显著优于 ClickHouse。同时 StarRocks 原生支持 Paimon External Catalog，无需额外的数据同步链路。ClickHouse 在单表宽表查询上仍有优势，但在我们的场景中 Join 需求更普遍。

## 三、关键技术深入

### 3.1 Flink CDC 实时入湖

#### 挑战
- MySQL Binlog 解析的性能瓶颈
- Schema Evolution 的自动适配
- Exactly-Once 语义的保证

#### 解决方案

```java
// Flink CDC Source 配置示例
MySqlSource<String> source = MySqlSource.<String>builder()
    .hostname("mysql-host")
    .port(3306)
    .databaseList("production")
    .tableList("production.*")
    .username("cdc_user")
    .password("***")
    .deserializer(new JsonDebeziumDeserializationSchema())
    .includeSchemaChanges(true)  // 关键：开启 Schema 变更捕获
    .build();
```

关键配置解读：
- `includeSchemaChanges(true)`: 允许上游 DDL 变更自动传播到下游
- Parallelism 设置为与源表分片数匹配，避免热点
- Snapshot 阶段使用增量快照，减少大表初始同步时间

### 3.2 Flink 中的 Embedding 生成

#### 挑战
Embedding API 调用是外部 IO 密集型操作，直接同步调用会严重阻塞 Flink 流水线。

#### 解决方案：Async I/O + 批量聚合

```java
// 异步批量 Embedding UDF
public class BatchEmbeddingFunction extends AsyncTableFunction<Row> {

    private final int batchSize = 32;
    private final long flushIntervalMs = 100;
    private List<Row> buffer = new ArrayList<>();
    private ScheduledExecutorService scheduler;

    @Override
    public void open(FunctionContext context) {
        scheduler = Executors.newSingleThreadScheduledExecutor();
        scheduler.scheduleAtFixedRate(this::flush, 
            flushIntervalMs, flushIntervalMs, TimeUnit.MILLISECONDS);
    }

    public void eval(Row input) {
        buffer.add(input);
        if (buffer.size() >= batchSize) {
            flush();
        }
    }

    private synchronized void flush() {
        if (buffer.isEmpty()) return;
        List<Row> batch = new ArrayList<>(buffer);
        buffer.clear();

        CompletableFuture.supplyAsync(() -> embeddingApi.batchEncode(batch))
            .thenAccept(results -> {
                for (Row result : results) {
                    collect(result);
                }
            });
    }
}
```

效果对比：
| 方案 | 吞吐量 | P99延迟 | API调用次数 |
|------|--------|---------|-----------|
| 同步单条 | 50/s | 200ms | 50/s |
| 异步单条 | 500/s | 250ms | 500/s |
| 异步批量(32) | 2000/s | 300ms | 63/s ← 最优 |

### 3.3 Paimon 流批一体存储

#### 数据分层设计

```
ODS (Operational Data Store):
  - CDC 原始事件，append-only
  - 保留所有变更历史
  - TTL: 30天

DWD (Data Warehouse Detail):
  - 清洗后的明细数据，primary key upsert
  - 去重、字段标准化、异常标记
  - 永久保留

DWS (Data Warehouse Summary):
  - 窗口聚合结果
  - 小时/天粒度预计算
  - TTL: 1年

DIM (Dimension):
  - SCD Type 2 维度表
  - 用户画像、产品目录等
  - 永久保留

Vector Index:
  - 文档 Embedding + Metadata
  - 定期同步到 Milvus
  - 保留原始文本用于溯源
```

### 3.4 RAG 检索增强策略

#### 混合检索 + Rerank

```python
def hybrid_search(query, top_k=5):
    # Step 1: 向量检索 (语义相似)
    vector_results = milvus.search(
        embedding=embed(query),
        top_k=20,
        filter_expr="category == 'product_docs'"
    )

    # Step 2: BM25 检索 (关键词匹配)
    keyword_results = elasticsearch.search(
        query=query,
        top_k=20,
        fields=["title^2", "content"]
    )

    # Step 3: RRF 融合
    fused = rrf_fusion(vector_results, keyword_results, k=60)

    # Step 4: Cross-Encoder Rerank
    reranked = cross_encoder.rerank(
        query=query,
        documents=fused[:20],
        top_k=top_k
    )

    return reranked
```

效果：相比纯向量检索，混合检索+Rerank 将 Top-5 命中率从 72% 提升到 91%。

## 四、经验教训

### 4.1 踩过的坑

**坑 1: Flink 状态膨胀导致 Checkpoint 超时**
- 现象：运行 3 天后 CP 耗时从 5s 增长到 60s，频繁超时
- 根因：CDC 数据的 Retraction 消息积累了大量过期状态
- 解决：设置 State TTL=7d + RocksDB Incremental CP + 调整 Compaction 策略
- 教训：**永远要给状态设 TTL**，除非你确定状态不会无限增长

**坑 2: Embedding API 限流导致反压**
- 现象：高峰期 Flink 出现严重反压，吞吐骤降
- 根因：Embedding API Rate Limit 100 QPS，突发流量超限后被 throttle
- 解决：Async I/O + Token Bucket 限流器 + 本地缓存热门查询
- 教训：**外部 API 调用必须有完善的限流和降级机制**

**坑 3: RAG 检索"看起来相关但实际无关"**
- 现象：向量检索返回高分结果，但内容与问题不匹配
- 根因：短查询 vs 长文档的语义不对称；同义词/缩写未覆盖
- 解决：Query Expansion + HyDE (Hypothetical Document Embedding) + Metadata Filter
- 教训：**向量相似度 ≠ 语义相关性**，必须加后处理层

**坑 4: Paimon Compaction 影响写入延迟**
- 现象：Compaction 触发时写入延迟飙升 5x
- 根因：默认 Compaction 策略过于激进
- 解决：调整 compaction.min-file-num=10 + 分离 Compaction 线程
- 教训：**流式写入场景要特别关注 Compaction 对延迟的影响**

### 4.2 给后来者的建议

1. **先跑通最小闭环再优化**：不要一开始就追求完美架构，先用最简单的方案验证端到端可行性
2. **评估先行**：在建设 RAG 系统之前，先建立评估体系（Golden Test Set + 自动化评测），否则优化就是盲猜
3. **数据质量 > 模型调优**：80% 的 RAG 质量问题出在数据和检索环节，而非模型
4. **监控从第一天就开始**：不要等到上线后才补监控，Day 1 就要有基本的 Metrics 和 Alerting
5. **文档化每个 Trade-off**：技术选型的理由比选型本身更重要，三个月后你会感谢自己

## 六、结语：数据工程师在 AI 时代的定位

AI 应用的竞争正在从"谁的模型更强"转向"谁的数据基础设施更好"。模型可以被 API 调用，但高质量、实时、可信的数据管道是无法被替代的核心壁垒。

数据工程师不需要变成算法专家，但需要理解 AI 应用对数据的需求，并将传统数据工程的严谨方法论（血缘、质量、可观测性）带入 AI Infra 领域。

**这才是数据工程师在 AI 时代的独特价值。**
