## 一、简历改写原则

### 1.1 去水三问

对简历中的每一句话，自问：

```
Q1: 我能用 2 分钟解释清楚这个技术点吗？
    → 不能 → 降级或删除

Q2: 面试官追问"具体怎么做的"时，我能给出 3 个以上细节吗？
    → 不能 → 补充实操经验或弱化表述

Q3: 如果被问到"遇到过什么问题"，我有真实的踩坑经历吗？

```

## 三、STAR 故事库（按技术领域分类）

### 3.1 Flink Window / Join / 容错

**S**: 实时订单统计存在乱序数据导致窗口结果偏差 8%，且大状态下 Checkpoint 频繁超时。

**T**: 设计正确处理乱序的窗口聚合方案，同时解决状态膨胀问题。

**A**:
1. Event Time + Watermark(delay=5s) + AllowedLateness(1min) + SideOutput 三层迟到数据处理
2. HashMap → RocksDB + Incremental Checkpoint，CP 模式改为 EXACTLY_ONCE
3. Interval Join 替代 Regular Join 控制状态生命周期
4. 自动化对账机制：每小时与 Hive 离线结果比对

**R**: 窗口准确率 92%→99.7%，CP 耗时 45s→8s，对账差异 <0.1%

**可追问准备**:
- Watermark 设 5s 的依据？→ 分析了源数据的乱序分布 P99=4.2s
- RocksDB 为什么比 HashMap 好？→ 内存溢出风险、增量 CP、支持超大状态
- Side Output 的数据怎么处理？→ 写入单独的迟到数据表，定期合并回主表

### 3.2 数据建模 / SCD

**S**: 用户画像表每日全量快照，存储成本高且无法精确追溯变更时间点。

**T**: 基于 CDC 实现 SCD Type 2，支持秒级历史版本查询。

**A**:
1. Flink CDC 捕获 Binlog UPDATE/DELETE 事件
2. Flink 中实现 SCD2 逻辑：旧记录标记 end_time，新记录 start_time=now
3. Paimon 主键表 Upsert 高效维护有效/失效记录
4. Schema Evolution 自动适配：上游加字段无需停服

**R**: 存储成本 -70%，历史查询精度从天级→秒级，Schema 变更零停机

**可追问准备**:
- SCD Type 2 vs Type 3 怎么选？→ Type 2 保留完整历史适合趋势分析，Type 3 只保留前一版本
- Paimon 如何处理 DELETE？→ Lookup Compaction 标记删除，下次 Compaction 物理清除
- 维度表和历史事实表怎么关联？→ 用事实表的 event_time 匹配维度的 start/end_time 区间

### 3.3 CDC 深入

**S**: 上游 MySQL 频繁 DDL 变更，CDC 链路每次都要手动调整，平均每月中断 2-3 次。

**T**: 实现 Schema Evolution 自适应，DDL 变更零停机。

**A**:
1. Flink CDC 开启 includeSchemaChanges=true
2. Paimon Sink 启用 schema.evolution.enabled=true
3. 新增字段自动追加，删除字段标记废弃而非报错
4. 建立 Schema Registry 做变更审计和通知

**R**: DDL 变更导致的链路中断从月均 3 次降至 0 次，运维工时减少 20h/月

**可追问准备**:
- 类型不兼容的变更怎么处理？→ 告警 + 暂停写入 + 人工确认
- 字段重命名怎么办？→ CDC 无法识别重命名，视为删旧增新，需要在应用层映射
- 多表 CDC 并行度怎么设？→ 按表大小和数据量动态分配，大表独立并行度

### 3.4 OLAP 引擎选型

**S**: 需要一个能支撑多表 Join 分析的 OLAP 引擎，同时对接 Paimon 数据湖。

**T**: 评估 ClickHouse/Doris/StarRocks 并做出选型。

**A**:
1. 构建标准化 Benchmark：单表扫描、多表 Join、实时更新、并发查询
2. StarRocks Join 性能优于 CH 5x（CBO 优化器），与 Doris 持平但 Paimon 集成更好
3. 利用物化视图预计算高频查询模式，P99 从 8s 降至 1.2s
4. Paimon External Catalog 免数据同步，查询直达湖仓

**R**: 复杂 Join 查询 P99 <3s，免去 ETL 同步链路，运维组件减少 1 个

**可追问准备**:
- CH 什么时候比 SR 好？→ 单表宽表高并发扫描、日志分析场景
- 物化视图怎么维护？→ 异步刷新 + TTL，权衡新鲜度和资源消耗
- Paimon Catalog 查询性能如何？→ 比原生表慢 20-30%，但省去同步成本总体更优

### 3.5 数据湖 (Paimon)

**S**: 需要流批一体的存储层，同时支持 CDC 实时写入和分析查询。

**T**: 选型并落地 Paimon 作为统一湖仓存储。

**A**:
1. 对比 Iceberg/Hudi/Paimon 在 Flink 流式写入、Upsert、读取性能的差异
2. Paimon 主键表 + Lookup Compaction 支撑 CDC Upsert
3. Time Travel 支持任意时间点快照查询
4. 配置合理的 Compaction 策略避免写入延迟飙升

**R**: 消除 Lambda 双链路，存储成本 -40%，数据一致性从"最终一致"提升为"强一致"

**可追问准备**:
- Compaction 策略怎么调？→ min-file-num=10, max-size=256MB, 分离 Compaction 线程
- Time Travel 的性能开销？→ 快照元数据查找 O(1)，读取性能取决于数据版本跨度
- Paimon vs Delta Lake？→ Delta 偏 Spark 生态，Paimon 偏 Flink，我们选 Flink 所以选 Paimon

### 3.6 向量数据库 / RAG

**S**: RAG 系统纯向量检索准确率低（65%），大量相关问题未被召回。

**T**: 构建混合检索 + Rerank 管道，将准确率提升至 85%+。

**A**:
1. 文档切片优化：语义感知切分 + Metadata 标注
2. 向量检索(Milvus HNSW) + BM25(ES) + RRF 融合排序
3. Cross-Encoder Rerank Top-20→Top-5
4. 构建 200 条 Golden Test Set + 自动化评测框架

**R**: 准确率 65%→89%，幻觉率 18%→4%，Top-5 命中率 72%→91%

**可追问准备**:
- HNSW vs IVF 怎么选？→ HNSW 查询更快但内存占用高，IVF 适合超大规模低内存场景
- RRF 的 k 参数怎么调？→ 默认 60，通过 Grid Search 在评测集上找最优
- Rerank 增加多少延迟？→ ~50ms（cross-encoder batch inference），可接受

### 3.7 Agent 稳定性

**S**: Agent 在生产环境任务完成率低（48%），经常陷入循环或输出格式错误。

**T**: 设计多层防护体系，将任务完成率提升至 85%+。

**A**:
1. Max Steps(5) + Per-step Timeout(30s) + Global Timeout(120s)
2. Loop Detector 检测重复动作提前终止
3. Pydantic 输出校验 + 最多 3 次重试修正
4. 置信度评分 + 低置信度转人工兜底
5. 全链路 Audit Trail 支持事后排查

**R**: 任务完成率 48%→87%，P99 延迟 <15s，异常率 <3%，MTTR <5min

**可追问准备**:
- 误差累积怎么量化？→ 单步 90% × N 步 = p^N，5 步只有 59%
- Loop Detector 的误判率？→ ~2%，宁可误杀不可漏放
- 为什么不用更强的模型？→ 试过 GPT-4o，提升 5% 但成本 3x，ROI 不如工程约束
